## 実装（handlers/ / data/）
- 修正した内容
  - `src/jpoke/handlers/item.py`: `ヒメリのみ_restore_pp_on_item_enabled` という実装済みだが
    どこからも呼ばれていない（未登録の）関数が存在していたのを発見。`ヒメリのみ_restore_pp_if_any_move_empty`
    に改名し、`Event.ON_SWITCH_IN` / `Event.ON_ITEM_ENABLED` / `Event.ON_FORCE_BERRY_TRIGGER` の
    3イベント共通ハンドラとして実際に使われるようにした
  - `src/jpoke/data/item.py`: `"ヒメリのみ"` の `handlers` に上記3イベントの登録を追加
    （`ON_SWITCH_IN` は既存の「きのみの発動」枠に合わせデフォルトpriority=100、
    `ON_ITEM_ENABLED`/`ON_FORCE_BERRY_TRIGGER` は既存の状態異常回復きのみに合わせ priority=50）
  - `src/jpoke/handlers/item.py`: PPが0の技が複数ある場合の回復対象選定ロジック
    （最後に使われた技＝`pp_consumed_move` がPP0ならそれを優先し、なければ技リスト先頭の
    PP0技を対象にする、一次情報の「第五世代以降」仕様）を `himeri_select_move_to_restore()` として
    共通化。従来は単に技リストの先頭からPP0の技を探すだけで「最後に使われた技」を優先しない
    実装ミスがあったため修正
  - `src/jpoke/handlers/move_attack.py`: なげつける・むしくいの `ヒメリのみ` 分岐で上記の
    `himeri_select_move_to_restore()` を使うよう変更（同じ選定ロジックの重複実装を解消）
  - `src/jpoke/handlers/move_status.py`（`うらみ_deplete_pp`）・
    `src/jpoke/handlers/move_attack.py`（`ぶきみなじゅもん_reduce_defender_pp`）:
    `Move.modify_pp()` は通常のPP消費経路（`move_executor._consume_pp`）を経由しないため
    `Event.ON_PP_CONSUMED` が発火せず、うらみ・ぶきみなじゅもんでPPが0になってもヒメリのみが
    反応しない実装漏れがあった。PP減少後に `ctx.derive(attacker=..., defender=..., move=...)` で
    「PPが減った側」を attacker ロールに読み替えたコンテキストを作り、`Event.ON_PP_CONSUMED` を
    明示的に再発火するよう修正
