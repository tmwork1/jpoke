## 実装計画書

## 実装（handlers/ / data/）
以下の2件のバグを修正した。

1. **PP誤り**: `data/moves/move_ha.py` の `pp=5` は Gen9 本家基準の値であり、
   champions ルール（`docs/champions/move_list.txt` 692行目 `PP=8`）への移行漏れだった。
   `pp=8` に修正。
2. **`target` 未設定によるまもる/マジックコートの誤ブロック**: `MoveData.target` を指定していなかった
   ため、デフォルト値 `"foe"` が適用され、`Move.is_blocked_by_protect` / `Move.is_reflectable` が
   誤って `True` を返していた。実際に相手が `まもる` を使った状態で `はねやすめ` を実行すると
   「攻撃を防いだ」ログが出て回復が発生しないことを確認し、`target="self"` を追加して修正した。

また、失敗チェックと回復処理が `ON_BEFORE_APPLY_MOVE`（`はねやすめ_check`）と `ON_STATUS_HIT`
（`はねやすめ_heal_and_remove_flying`）の2ハンドラに分割されていたが、`move_executor.run_move` は
`ON_BEFORE_APPLY_MOVE` が `False` を返しても `MOVE_FAILED` ログを自動生成しない
（`ON_STATUS_HIT` の失敗時のみ `_execute_status_hit` が自動でログを出す）ため、HP満タン失敗時に
`MOVE_FAILED` ログが欠落する不整合があった。`じこさいせい_heal_self` / `あさのひざし_heal_self` と
同じパターン（単一の `ON_STATUS_HIT` ハンドラ内で失敗チェックと回復・タイプ除去をまとめて処理）に
統合し、`はねやすめ_check` は削除した。

`removed_types` を用いたタイプ除去の実装（`model/pokemon.py` の `types` プロパティ、
`handlers/volatile.py` の `はねやすめ_remove_flying` / `はねやすめ_restore_flying`、
`data/volatile.py` の `"はねやすめ"` volatile エントリ、`ON_TURN_END priority=120`）は
仕様・`docs/spec/turn.md` と整合しており、変更なし。
