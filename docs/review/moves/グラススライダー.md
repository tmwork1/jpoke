# グラススライダー レビュー結果

## 仕様書
- 新規作成（`docs/spec/moves/グラススライダー.md`）。
- 一次情報（docs/wiki/moves/グラススライダー.html）から以下を記載:
  - 基本情報（タイプ・分類・威力55/命中100/PP20・優先度: グラスフィールド下で+1、それ以外0）
  - 優先度上昇の判定条件（グラスフィールド展開中 かつ 使用者が接地）
  - 判定フラグ（急所・命中判定・追加効果なし・まもる・おうじゃのしるし・マジックコート/よこどり対象外）
  - テラスタルとの関係（くさタイプ一致時の威力60底上げが、優先度+1時でも例外的に適用される仕様と、
    本プロジェクトでの実現方法）
  - 優先度上昇をグラススライダー自身ではなく地形「グラスフィールド」側に実装する理由
  - 既知の制限（サイコフィールド/ファストガード等が動的な優先度上昇を認識しない）

## 実装（handlers/ / data/）
- バグ修正: `src/jpoke/data/moves/move_ka.py` の `"グラススライダー"` エントリから
  `priority=1` を削除（デフォルト値0に変更）。従来は常に優先度+1として扱われており、
  グラスフィールド非展開時にも先制技として行動順が誤って計算されていた。
- 新規実装: `src/jpoke/handlers/field.py` に `グラスフィールド_boost_move_priority` を追加。
  グラススライダー使用時、使用者が接地していれば優先度を+1する。
- 新規登録: `src/jpoke/data/field/terrain.py` の `"グラスフィールド"` エントリに
  `DomainEvent.ON_MODIFY_MOVE_PRIORITY` ハンドラとして上記関数を登録（`DomainEvent` の
  import を追加）。
  - 技自身にハンドラを持たせる方式は不可（`SpeedCalculator.calc_move_priority` は
    技実行前のターン開始時に発火するが、技のハンドラは `move_executor` が技実行時にのみ
    登録するため間に合わない）。地形側のハンドラは地形展開中ずっと登録されているため、
    トリックルームの素早さ反転と同じパターンで正しく機能する。
- PP・威力・命中率・分類・タイプは元々 docs/champions/move_list.txt（PP20・威力55・命中100）
  と一致しており変更なし。
