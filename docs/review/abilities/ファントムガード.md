## 実装計画書
- 新規作成（`docs/plan/abilities/ファントムガード.md`）。既存実装済みの内容を追認する形で作成した。
  - `flags={"full_hp_damage_modifier"}` のみを設定し、`mold_breaker_ignorable` は付与しない

  - `docs/progress/ability.md` で「計画書」列が `x` になっていたにもかかわらず実ファイルが
    存在しなかった不整合を解消した。

## 実装（handlers/ / data/）
- 変更なし。`src/jpoke/data/ability.py` の `ファントムガード` エントリ、
  `src/jpoke/handlers/ability.py` の `マルチスケイル_reduce_damage`（共有ハンドラ）を確認した結果、
  仕様どおりに実装されていた。
  - `ファントムガード` は `full_hp_damage_modifier` フラグのみを持ち `mold_breaker_ignorable` を
    持たない。`かたやぶり_disable_foe_ability`（`src/jpoke/handlers/ability.py`）は
    `mold_breaker_ignorable` を持つ特性のみを無効化するため、ファントムガードは
    かたやぶり下でも正しくダメージ半減が適用され続ける。
  - マルチスケイルは `mold_breaker_ignorable` を持つため、かたやぶり下では無効化される。
    両特性の差分が正しく表現されていることを確認した。

## リーサル計算
- 変更なし。`docs/progress/ability.md` の「リーサル実装」列は `x`。
  `src/jpoke/core/lethal.py` の `_calc_damage_dist` が `full_hp_damage_modifier` フラグを汎用的に
  見て満タン枝/非満タン枝のダメージ分布を分けて計算する仕組みになっており、
  マルチスケイル・ファントムガード共通で個別のハンドラ登録（`lethal_handlers`）は不要。
  この汎用ロジックは内部で `battle.calc_damages`（通常のダメージ計算経路）を呼ぶため、
  かたやぶりによる無効化判定（`mold_breaker_ignorable` フラグ）も自動的に反映される。
  仕様・実装と照合し、誤り・欠落は見つからなかったため新規実装は行わなかった。
