# ぎたい レビュー結果

## 実装計画書
- 新規作成（`docs/plan/abilities/ぎたい.md` が存在しなかったため新規作成した）
  - 実装方針: `Pokemon.ability_override_type` を用いてフィールドに応じた単タイプへ変化させる
    （へんげんじざい・リベロ・へんしょくと同じ既存機構を再利用）
  - `Event.ON_SWITCH_IN`（priority=120, `docs/spec/turn.md` 準拠）、`Event.ON_FIELD_CHANGE`、
    `Event.ON_ABILITY_ENABLED` の3イベントで発動する設計とした
  - 既知の実装上の制約（スコープ外）を明記した:
    - へんしん（変身）が本エンジン未実装のため、仕様書記載の「へんしん時の分岐」は対応しない
    - `Pokemon.types` の解決順序（`move_override_types > ability_override_type >
      volatile_override_type > 本来のタイプ`）は既存の設計であり、フィールド発生中に
      みずびたし/まほうのこなを後から受けた際の「一時的にそちらのタイプへ変わり、次に
      ぎたいが発動したときに戻る」という時系列上の見た目は再現されない
      （ぎたいのタイプが常に優先される）。これは `ability_override_type` を共有する
      他の特性にも共通する既存の簡略化であり、ぎたい固有の追加実装は行わない

## 実装（handlers/ / data/）
- 「実装=保留」だった理由がドキュメント上に一切記録されていなかったため調査したところ、
  既存の型変更機構（`ability_override_type`、`へんげんじざい`/`リベロ`/`へんしょく`と同じ仕組み）
  で実装可能と判断し、保留を解消して実装した。
- `src/jpoke/handlers/ability.py` に `ぎたい_フィールドタイプ対応表` と `ぎたい_change_type` を新規実装。
  - フィールド種別を対応する単タイプ（でんき/くさ/フェアリー/エスパー）に変換し、
    フィールドが無いときは `mon.base_types`（本来のタイプ）に戻す。
  - 現在の実効タイプ（`mon.types`）と変化先が同じ場合は発動しない。
- `src/jpoke/data/ability.py` の `ぎたい` に `ON_SWITCH_IN`（priority=120）・`ON_FIELD_CHANGE`・
  `ON_ABILITY_ENABLED` の3イベントへ同一ハンドラを登録した。
- かがくへんかガス・かたやぶりの無効化チェックはハンドラ内に書かず、`AbilityHandler(source="ability")`
  の既定の仕組み（`enabled_ignoring`）に委ねている。

## テスト
- `tests/abilities/test_ability_ka.py` にテストを新規追加（`ぎゃくじょう` テストの直前に挿入）。

    （パラメトライズ、4ケース）
  - 自身が場にいる状態でフィールドが変化すると即座にタイプが変わる

  - フィールドが無い状態で登場すると本来のタイプのまま発動しない（`ability.revealed is False`）
  - 交代すると本来のタイプに戻り、再度フィールドがある場に出ると再びタイプが変わる
  - `change_ability` で特性がぎたいに変わったとき、即座にフィールドに応じたタイプになる
  - かがくへんかガスの効果中は発動せず、解除後に即座に発動する
