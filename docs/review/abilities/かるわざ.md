## 実装（handlers/ / data/）
- 重大なバグを発見し修正した。旧実装は `Event.ON_ABILITY_ENABLED` で常に「現在のアイテム
  所持状況からstateを再初期化」していたため、かがくへんかガス解除時（`remove_disabled_reason`
  経由。同一の `Ability` インスタンスが有効化に戻るだけで新規生成されない）にも初期化処理が
  走ってしまい、以下の仕様に違反していた。
  - 「かるわざが発動中にかがくへんかガスが発動した場合、その間かるわざの効果は無くなる。
    かがくへんかガスが解除されればかるわざは再度発動する」→ 旧実装ではガス解除後、
    アイテムを既に失っているにもかかわらず `state` が `"inactive"` に上書きされ、
    すばやさ2倍が復活しないバグがあった。
  - 修正: `かるわざ_init_state` は `state == ""`（未初期化。交代での再入場、または
    スキルスワップ等で新しい `Ability` インスタンスに差し替わった直後）のときのみ
    アイテム所持状況から初期化するよう変更。ガス解除等で同一インスタンスが復帰した
    場合は `state` を保持し、発動中だった場合はそのまま発動状態を維持するようにした。
  - あわせて、「未発動時にかがくへんかガスが発動し、その間にアイテムを消費してから
    かがくへんかガスが解除された場合、かるわざは発動しない」という仕様も検証したところ、
    旧実装のロジック（`ON_CALC_SPEED` 内で `idle` かつアイテムなしなら即 `active` へ遷移）
    では、無効化中に失ったアイテムもガス解除後の速度計算時に検知され誤って発動して
    しまう構造的リスクがあった。これに対応するため、`idle → active` の遷移ロジックを
    `DomainEvent.ON_CALC_SPEED` 内の遅延判定から、道具喪失時に発火する既存イベント
    `Event.ON_ITEM_LOST`（`item_manager._change_item` が発火。なげつける・はたきおとす・
    どろぼう・トリック・ふしょくガス等、道具を失うあらゆる経路が最終的にこの1箇所を
    経由する）をトリガーとする方式に変更した（`かるわざ_activate_on_item_lost` を新設）。
    特性が無効化されている間はこのハンドラ自体が `event_manager` のスキップ判定により
    発火しないため、無効化中の道具喪失は `idle → active` 遷移を起こさなくなり、
    仕様どおり「ガス解除後も発動しない」を再現できるようになった。
  - `Event.ON_ABILITY_DISABLED` に登録していた `かるわざ_reset_state`
    （`state` を `""` にリセットする処理）は、上記のとおり `state` を無効化中も
    保持する必要があるため削除した（同一インスタンス上で `state` を保持し続けることが
    仕様上正しいため、無効化イベントで何もしなくてよい）。
  - `かるわざ_modify_speed` は `state == "active"` のみを見るシンプルな条件に整理し、
    アイテム再取得時に `idle` へ戻す処理はそのまま維持した。
- 挙動を実機で手動検証（`battle.add_ability_disabled_reason` /
  `battle.remove_ability_disabled_reason` を使ったスクリプト）し、修正前後で上記2つの
  仕様ケースの結果が反転する（修正前:バグを再現、修正後:仕様どおり）ことを確認した。
