## 実装（handlers/ / data/）
- `src/jpoke/handlers/ability.py` の `アイスフェイス_restore_on_snow` から、`ON_FIELD_CHANGE` ハンドラは特性登録が場に出ているポケモンにのみ行われる（switch_manager が登場時に登録・退場時に解除する）ため常に真になる不要な重複ガード（`mon is None or not battle.is_active(mon)`）を削除した（CLAUDE.md「イベント発火側で前提が保証されている場合、ハンドラ側の重複ガードは不要」に準拠、姉妹関数 `アイスフェイス_restore_on_switch_in` と同様のスタイルに統一）。
- 以下の点は仕様との整合を確認し、修正不要と判断した。
  - `ON_MODIFY_MOVE_DAMAGE` priority=40（`turn.md` L460「アイスフェイスに被弾」）、`ON_SWITCH_IN` priority=140（`turn.md` L105、他の場に出たときに発動する特性より優先度が低い仕様と一致）。
  - `mold_breaker_ignorable` フラグにより、かたやぶり系の物理技のダメージ軽減・あられ/ゆき系技によるフォルムチェンジ復帰のいずれも、`ON_BEGIN_MOVE`〜`ON_END_MOVE` の間 `ability.enabled` が false になることで自動的に防止される（ハンドラ側で個別のかたやぶりチェックを書く必要はない）。
  - `battle.weather.name`（`WeatherManager.active`）は `ON_CHECK_WEATHER_ENABLED` を介してノーてんき/エアロックによる無効化を反映するため、ハンドラ側で個別のノーてんき/エアロックチェックは不要。
  - 物理ダメージを0にする際に `hits_taken` を明示的にインクリメントしているのは、`move_executor._execute_hit` が `actual_damage <= 0` のとき通常のインクリメントをスキップするための補完であり、`ばけのかわ_block_damage` と同じパターンで妥当。

## リーサル計算
