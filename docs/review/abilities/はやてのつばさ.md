## 実装計画書
- 変更なし（`docs/plan/abilities/はやてのつばさ.md` は `DomainEvent.ON_MODIFY_MOVE_PRIORITY` で HP満タン かつ ひこうタイプ技のときに優先度+1する方針で、仕様書の主要効果と一致している）。

## 実装（handlers/ / data/）
- 変更なし。`src/jpoke/handlers/ability.py` の `はやてのつばさ_modify_priority`（`ctx.attacker.hp == ctx.attacker.max_hp` かつ `ctx.move.type == "ひこう"` で value+1）、`src/jpoke/data/ability.py` の `ON_MODIFY_MOVE_PRIORITY` ハンドラ登録（`subject_spec="attacker:self"`）は既に実装済みで、進捗表の「実装」列が未更新（`-`）だっただけと判明した。
- 仕様書にあるテラバースト等「タイプが変化する技」の除外について確認した。`Move.type` は `resolve_move_type()`（`move_executor.py`）で技実行時に上書きされる（`ctx.move.type = self.resolve_move_type(...)`）が、`calc_move_priority()`（`speed_calculator.py`、行動順決定時に呼ばれる）はその上書き前の `move.type`（= `move.data.type` の静的値）を参照するため、テラバースト等の「実行時に解決されるタイプ」はGale Wingsの優先度補正対象にならない。仕様書の記載どおりの挙動になっていることをコードから確認した（現状 めざめるパワー・しぜんのめぐみ・マルチアタックは未実装のためこの経路は通らないが、テラバースト・めざめるダンス・さばきのつぶては実装済みで仕様どおり除外される）。
- 仕様書にある「HP満タン判定はGen8以降、行動直前の実効値で動的に再判定される」点について、`ファストガード_protect` / `じょおうのいげん_block_priority` が `battle.speed_calculator.calc_move_priority()` を都度呼び直して実効優先度を動的に再計算する既存パターンで担保されており、はやてのつばさ側に追加実装は不要と判断した。
