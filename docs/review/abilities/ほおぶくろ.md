## 実装（handlers/ / data/）
- `src/jpoke/handlers/ability.py` に `ほおぶくろ_heal_on_berry_consumed` を新規実装
  （`Event.ON_BERRY_CONSUMED` / `subject_spec="source:self"`）。実装は空だった
  `"ほおぶくろ": AbilityData()` に対する新規実装であり、`data/ability.py` にもハンドラ登録を追加した。
- `battle.modify_hp(mon, r=1/3)` を使用。`battle.modify_hp` が内部で発火する
  `Event.ON_MODIFY_HEAL` により、かいふくふうじ状態時の回復ブロックとHP満タン時のクランプ
  （変化量0）が自動的に処理されるため、ほおぶくろ側で個別の条件分岐は不要とした。
- `src/jpoke/core/context.py` の `EventContext` に `is_self_fling: bool = False` を追加。
  なげつけるで自分の持ち物のきのみを投げて手放したとき（`なげつける_consume_item`）だけ
  `True` を渡すようにした。ほおぶくろはこの経路では「食べる」に該当しないため発動しない
  仕様だが、はんすうは同じ経路でも対象になる仕様（`docs/spec/abilities/はんすう.md`）のため、
  両特性で挙動を分岐させる必要があった。
- `src/jpoke/handlers/move_attack.py` の `なげつける_apply_item_effect` を修正:
  既存実装は「なげつけるで効果のあるきのみの集合（`_NAGETSUKERU_BERRY_EFFECT_ITEMS`）に
  含まれる」ことだけで無条件に `Event.ON_BERRY_CONSUMED` を発火しており、状態異常でない
  ときにラムのみを受けた場合など「追加効果があっても実際には発動しなかった場合」にも
  誤って発火していた（一次情報のラムのみの例と矛盾）。各分岐で実際に効果が適用されたかを
  `effect_applied` に記録し、`effect_applied` が真の場合のみ発火するよう是正した
  （はんすうにも影響する共通コードのため、なげつける全体の正確性が向上した）。
- 実装完了後、`handlers/ability.py` 内での関数の並び順が誤っていた（5十音順で
  「ほうし」より前に置いていた）ため、「ほうし」の後・「ほのおのからだ」の前に移動して
  五十音順を修正した（`data/ability.py` 側の `"ほおぶくろ"` エントリはもともと正しい位置に
  あったため変更なし）。

## テスト
- `tests/abilities/test_ability_ha.py` に以下を追加:
  - HPが満タンのときは発動しない
  - かいふくふうじ状態では回復しない
  - かがくへんかガス中は発動しない
  - きのみを食べると最大HPの1/3を回復する（端数切り捨て）
  - なげつけるで相手から効果のあるきのみを受けると発動する（ラムのみ・状態異常あり）
  - なげつけるで相手から効果の無いきのみを受けても発動しない（ラムのみ・状態異常なし、
    `なげつける_apply_item_effect` の是正を検証）
  - なげつけるで自分のきのみを投げて手放しても発動しない（`is_self_fling` を検証）
  - むしくいで奪ったきのみでも発動する
  - HPしきい値で自動発動する持ち物（オボンのみ等）を使うと、テストの準備操作（HPを
    減らすための `modify_hp`）自体がきのみの自動消費とほおぶくろの発動を誘発してしまい
    テストの意図が崩れるため、直接の消費テストでは自動発動条件を持たないオッカのみを使用した。
  - なげつけるはそれ自体が物理ダメージを与える技のため、回復量を正確に検証するテストでは
    `t.fix_damage(battle, 0)` でダメージを0に固定し、ほおぶくろの回復量だけを分離して検証した。
- `python -m pytest tests/ -v` で5478件成功・1件skip（既存の未実装技マグニチュードのテストで
  今回の変更と無関係）を確認した。
