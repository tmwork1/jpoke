## 実装（handlers/ / data/）
- 修正した内容:
  - **バグ修正**: `handlers/ability.py` の共通ヘルパー `_activate_weather` は、
    `WeatherManager.apply` が `False` を返した場合（＝天候が変わらなかった場合）を一律に
    「特性は発動しなかった」として扱い、`_announce_ability_triggered`（特性バー表示・
    `ability.revealed = True`）を呼んでいなかった。
    しかし仕様書には「場がおおひでり・おおあめ・らんきりゅう状態のときは、ひでりの特性バーは
    現れるが晴れに変えることができない」と明記されており（あめふらし・すなおこし・ゆきふらし
    の一次情報にも同様の記載あり）、既に同じ天候が有効で不発になる場合（特性バーも表示されない
    べきケース。地形始動特性の同一挙動が `tests/abilities/test_ability__group.py::
    test_地形始動特性_同一地形が有効時は継続ターンを更新しない` で `not ...revealed` として
    既に codify されている）と、より強い天候に阻まれて不発になる場合（特性バーは表示されるべき
    ケース）を区別できていなかった。
    `battle.weather_manager.current_name != weather`（＝阻まれたのが同一天候ではなく、より強い
    天候によるものだったか）で両者を判定するように修正し、強天候による不発時にも
    `_announce_ability_triggered` を呼ぶようにした。
    この修正は `_activate_weather` を共用する あめふらし・すなおこし・ゆきふらし・ひひいろのこどう・
    おわりのだいち・はじまりのうみ・デルタストリーム にも波及するが、いずれも仕様書に同じ
    「特性バーは表示される」旨の記載があり、既存テストとも矛盾しないことを確認済み（後述）。
  - それ以外は変更なし。`data/ability.py` の `ひでり`（`Event.ON_SWITCH_IN` / `Event.ON_ABILITY_ENABLED`、
    `subject_spec="source:self"`）はあめふらし・すなおこし・ゆきふらしと同一パターンで登録されている。
    天候の上書き判定（`WeatherManager.apply` の優先度比較、`WEATHER_PRIORITY`）、あついいわによる
    継続ターン延長（`item.py` の `あついいわ_resolve_field_count`）は天候共通ロジックに委譲されており
    個別の欠落なし。複数の天候始動特性が同時入場したときの速度順上書きも `EventManager` 側の
    Priority+素早さソートで実現されており、特性側の実装は不要であることを確認した。
