  `docs/spec/abilities/めんえき.md` の内容（効果・特性の仕様・備考）はすべて一致しており
  誤り・欠落はなかった。

## 実装計画書
- 新規作成（`docs/plan/abilities/めんえき.md` が存在しなかったため）。既存実装
  （`_prevent_ailment` / `_cure_ailment_on_enable` 共通ヘルパーの再利用、
  `mold_breaker_ignorable` フラグ）と整合する内容で作成し、じゅうなん等の既存プランと
  同じ構成に揃えた。どくびし・どくどくだまとの相互作用が既存の共通実装（`ailment_manager.apply`
  経由での `ON_BEFORE_APPLY_AILMENT` 発火）で自然に解決される旨も明記した。

## 実装（handlers/ / data/）
- 変更なし。`data/ability.py` の `めんえき` エントリは `ON_BEFORE_APPLY_AILMENT`
  （`prevent_poison_ailment` 共通ハンドラ）と `ON_ABILITY_ENABLED`
  （`めんえき_cure_poison_on_enable`）の2ハンドラ・`mold_breaker_ignorable` フラグで構成され、
  同系統の `じゅうなん`（まひ版）と完全に同一パターンであることを確認した。かたやぶり系の
  `add_ability_disabled_reason`/`remove_ability_disabled_reason` が `ON_ABILITY_DISABLED`/
  `ON_ABILITY_ENABLED` を発火する仕組みにより、「かたやぶりの技でどく状態にされても技終了
  直後に回復する」という仕様も正しく実現されていることをコードで確認した。どくびし
  （`handlers/field.py` の `どくびし_apply_poison`）・どくどくだま（`handlers/item.py` の
  `どくどくだま_apply_poison`）もいずれも `ailment_manager.apply` 経由のため、めんえき固有の
  追加実装なしに自然にブロックされることを確認した。

## テスト
- `tests/abilities/test_ability__group.py` の `test_状態異常無効` / `test_状態異常無効_かたやぶりで無効`
  に既にめんえきのどく・もうどく両方のケースが含まれていることを確認した（変更なし）。
- 仕様書の「すでにどく・もうどく状態になっているポケモンの特性がめんえきに書き換わった場合、
  即座に状態異常を治す」を検証するテストが存在しなかったため、
  `tests/moves_status/test_move_sa.py` に
  `test_スキルスワップ_めんえき取得ですでにどく状態なら即座に回復する` を新規追加した
  （どく状態のピカチュウがスキルスワップでめんえきを取得すると即座に状態異常が治ることを確認）。
- （めんえきとは無関係な既存テストの flaky 修正・§共通13）`python -m pytest tests/ -v` の全件実行中に
  `tests/test_examples_smoke.py::test_examples_エラーなく実行できる[05_benchmark/01_step_time_benchmark.py]`
  が `returncode == -1`（stdout/stderrともに空）で失敗することを発見した。単体で
  `examples/05_benchmark/01_step_time_benchmark.py` を縮小パラメータ（`--n-battles 10`）で
  実行すると正常終了することを確認し、既定パラメータ（300バトル×最大100ターン）のままだと
  実行時間・CPU負荷が非常に大きく、このリポジトリのような多数のエージェントが並列に
  フルテストスイートを実行する環境下ではOSによる強制終了が間欠的に発生することを特定した
  （同時刻に他のworktree・ループが同一スクリプトを並列実行していたことをプロセス一覧で確認）。
  スモークテストの目的は「エラーなく終了すること」の確認のみで実測値の精度は不要なため、
  `tests/test_examples_smoke.py` にスクリプトごとの追加引数マップ `EXTRA_ARGS` を導入し、
  このベンチマークスクリプトのみ `--n-battles 5 --max-turns 20` に縮小して実行するよう修正した。
  修正後、単体で4回連続成功（0.5秒程度で完了）することを確認し、全件再実行でも
  5626 passed, 1 skipped（0:01:02）で収束することを確認した。
