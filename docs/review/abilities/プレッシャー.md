## 実装計画書
- 更新日・内容を全面的に見直した（既存計画書は残っていたが記述に誤りがあったため修正）。
  - `subject_spec="target:self"` という誤った記述を `subject_spec="defender:self"` に修正した。
    `ON_MODIFY_PP_CONSUMED` は `AttackContext`（`attacker`/`defender` ロールのみを持つ）で発火するため、
    `EventContext` 用のロール名 `"target"` を指定すると `AttackContext.resolve_role` が
    `ValueError` を送出する不正な指定だった（実際のコード `data/ability.py` は
    `"defender:self"` で正しく登録されていたため実害はなかったが、計画書の記述が実装と食い違っていた）。
  - 「自分を対象にする技の除外と例外」節を新設し、下記「実装」節で発見した
    self-target 技の除外ロジックとふういん・のろいの例外を明記した。
  - 「現時点で個別対応が不要な例外」節を新設し、カウンター/ミラーコート/メタルバースト/ほうふく・
    テラバースト・ドラゴンアロー・きりふだ・よこどり・マジックコート・ものまね・さきどり・へんしんの
    それぞれについて、シングル対戦専用の本リポジトリでは追加対応が不要（または該当技が未実装）と
    判断した根拠を記載した。

## 実装（handlers/ / data/）
一次情報の「特性の仕様」節の各項目を実装と突き合わせた結果、以下の欠落を発見し修正した。

- **自分を対象にする技（つるぎのまい等）が誤ってPP消費を+1していた**: `プレッシャー_extra_pp`
  （`handlers/ability.py`）が `ctx.move.target` を一切確認せず、`AttackContext.defender`
  （常に「敵」に設定される）がプレッシャー所持者であれば無条件に+1していた。このため
  自分対象の技（`target="self"`）でも敵にプレッシャー所持者がいるだけでPPが余分に減る
  バグがあった（一次情報「効果範囲内にプレッシャーのポケモンが含まれていない行動に
  プレッシャーの効果は発動しない。基本的に自分が対象の技はプレッシャーの影響を受けない。」に反する）。
  `ctx.move.target` が `"foe"` / `"foe_side"` / `"field"` のときのみ+1するよう修正した
  （シングル対戦専用のため `"foe_side"`/`"field"` は実質 `"foe"` と同じ唯一の相手を指す）。
- **ふういんの例外が (偶然) 動作していたのを明示的なロジックに変更**: ふういんは
  `target="self"` だが一次情報の例外規定により相手のプレッシャーの影響を受けるべき技。
  修正前は target チェックが無かったため偶然正しく動いていたが、上記の target チェックを
  追加すると除外されてしまうため、`move_name == "ふういん"` の例外分岐を明示的に追加した。
- **のろいのタイプ別分岐が未実装だった**: 一次情報「相手の場にプレッシャーがいても、
  ゴーストタイプ以外が使う"鈍い"にプレッシャーの効果は無い」に対し、`のろい` の `target` は
  既定値 `"foe"` のため修正前・修正後ともに無条件で+1されてしまう状態だった。
  `ctx.attacker.has_type("ゴースト")` が偽のとき（鈍い）は+1しないよう分岐を追加した。
- かたやぶりとの相互作用（一次情報「かたやぶりの影響を受けず、この効果が発動した技に
  対してもPPを多く減らす」）は、PP消費（`_consume_pp`）が `ON_BEGIN_MOVE`（かたやぶりが
  相手の特性を無効化するタイミング）より前に発生するアーキテクチャのため、追加の実装なしで
  仕様どおり動作することを確認した（テストで明示的に検証）。
- 型注釈を `ctx: EventContext` から実際の型 `ctx: AttackContext` に修正した。

## リーサル計算
- 対象外（`docs/progress/ability.md` のリーサル実装列が `n/a`）。プレッシャーはPP消費量にのみ
  影響し、ダメージ量やリーサル判定に直接影響しないため対象外。
