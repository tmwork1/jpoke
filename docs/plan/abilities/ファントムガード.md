# ファントムガード 実装計画

更新日: 2026-07-12

## スコープ

- 仕様書: `docs/spec/abilities/ファントムガード.md`
- 実装状態: 実装済み（本ドキュメントは既存実装の追認・整理）
- 実装方針: マルチスケイルと全く同じ被ダメージ半減ロジックを流用し、`かたやぶり`
  （及び特性を無視する効果全般）に無効化されない点だけをフラグ構成で切り分ける。

## 仕様要点（実装に反映）

- 自分のHPが満タンのとき、攻撃技で受けるダメージを0.5倍にする。
- かたやぶり効果を持つ技・特性を無視する技のダメージも半減できる（＝この特性自体は
  `mold_breaker_ignorable` フラグを持たない）。
- それ以外の細かい仕様（ダメージ固定技は対象外・こんらんの自傷は対象外・HP回復後の
  再発動・みがわり状態での判定基準・連続攻撃技のヒットごとの判定）はマルチスケイルと
  共通のため、マルチスケイル#特性の仕様を参照する。

## 実装対象イベント

- `Event.ON_CALC_DAMAGE_MODIFIER` — ダメージ計算内部イベント。`docs/spec/turn.md` に
  掲載がないため、既存の同種ハンドラ（マルチスケイル）の登録済み priority
  （未指定・デフォルト）に合わせる。

## 変更対象（ファイル/関数単位・実績）

- `src/jpoke/data/ability.py`
  - `ファントムガード` エントリに `flags={"full_hp_damage_modifier"}` を設定する
    （`mold_breaker_ignorable` は付与しない — マルチスケイルとの唯一の差分）。
  - `Event.ON_CALC_DAMAGE_MODIFIER` に `h.マルチスケイル_reduce_damage`
    （`subject_spec="defender:self"`）を登録する。関数は新規実装せず、マルチスケイル用の
    既存ハンドラをそのまま再利用する。
- `src/jpoke/handlers/ability.py`
  - 新規実装なし。既存の `マルチスケイル_reduce_damage`（HP満タン判定 + `apply_fixed_modifier`
    による0.5倍化）をそのまま利用する。
- `src/jpoke/core/lethal.py`
  - 新規実装なし。`full_hp_damage_modifier` フラグを汎用的に見て満タン枝/非満タン枝を
    分けて計算する既存の仕組み（`_calc_damage_dist`）がそのまま適用される。
- `tests/abilities/test_ability_ha.py`
  - `test_ファントムガード_HP満タンのとき半減`（基本の発動確認）
  - `test_ファントムガード_かたやぶりで無効化されない`（マルチスケイルとの差分確認）

## 実装フロー（実績）

1. 仕様書でマルチスケイルとの差分（かたやぶりに無効化されない）を確認する。
2. `data/ability.py` の `ファントムガード` に `full_hp_damage_modifier` フラグのみを付与し、
   `mold_breaker_ignorable` は付与しない。
3. `Event.ON_CALC_DAMAGE_MODIFIER` へ既存の `マルチスケイル_reduce_damage` を
   `subject_spec="defender:self"` で登録する。
4. `tests/abilities/test_ability_ha.py` に基本の半減テストとかたやぶり無効化耐性テストを追加する。

## テスト観点

1. 発動: HPが満タンのとき、攻撃技のダメージが0.5倍になる（`damage_modifier == 2048`）。
2. 非発動: HPが満タンでないとき、ダメージは半減しない（`damage_modifier == 4096`）。
3. 無効化耐性: 攻撃側が `かたやぶり` を持っていても半減が適用される
   （マルチスケイルとの違いの回帰確認）。
4. その他のマルチスケイル共通仕様（固定ダメージ技・こんらん自傷・回復後の再発動・
   連続攻撃技のヒットごとの判定）は、共通ハンドラの回帰テスト
   （`tests/abilities/test_ability_ma.py` のマルチスケイル系テスト）で担保されており、
   本特性専用の重複テストは追加しない。

## 検証コマンド

```powershell
python -m pytest tests/abilities/test_ability_ha.py -k "ファントムガード" -v
python -m pytest tests/ -v
```

## リスクと対策

- リスク: マルチスケイルの実装を変更すると、フラグ・ハンドラ共有によりファントムガードの
  挙動も連動して変わる。
  - 対策: マルチスケイル側を修正する際は、`mold_breaker_ignorable` フラグの有無だけが
    両特性の差分であることを踏まえ、ファントムガード側のテスト（かたやぶり回帰）も
    合わせて確認する。
