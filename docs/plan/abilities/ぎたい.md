# 特性実装計画: ぎたい

更新日: 2026-07-11

## スコープ

- 仕様書: `docs/spec/abilities/ぎたい.md`
- 実装状態: 実装済み
- 実装方針: フィールド変化時・登場時・特性有効化時に、フィールドに対応する単タイプへ
  `Pokemon.ability_override_type` を設定する（へんげんじざい・リベロ・へんしょくと同じ仕組み）。
  フィールドが無いときは `ability_override_type = None` にして本来のタイプへ戻す。

## 実装ゴール（完了定義）

- 自身が場にいる状態でフィールドが変化したとき、即座に対応する単タイプへ変化する。
- フィールドが発生している場に繰り出されたとき、`ON_SWITCH_IN` で登場時に効果が発動する
  （`docs/spec/turn.md` の `ON_SWITCH_IN` priority=120 に準拠）。
- 特性がぎたいに変わったとき（スキルスワップ・トレース等で `ON_ABILITY_ENABLED` が発火したとき）
  即座に発動する。かがくへんかガスの効果が切れたときも同様（`ON_ABILITY_ENABLED` 経由）。
- フィールドが無いときは本来のタイプ（`Pokemon.base_types`）に戻る。
- 現在のタイプと変化先のタイプが同じ場合は発動しない（ログを出さない）。
- 交代・ひんし・戦闘終了でタイプが元に戻る（`Pokemon.reset_on_switch_out()` が
  `ability_override_type` を毎回リセットするため、追加実装は不要）。
- かがくへんかガス・かたやぶりによる無効化は `AbilityHandler(source="ability")` の
  既定の仕組みで自動的に処理される（本特性のハンドラ内に無効化チェックを書かない）。

## 実装対象イベント

- `Event.ON_SWITCH_IN`（priority=120, `docs/spec/turn.md` 準拠）— フィールドが発生している場に
  繰り出されたときの発動
- `Event.ON_FIELD_CHANGE`（priority=100, 既定値）— 自身が場にいる間にフィールドが変化したときの発動
- `Event.ON_ABILITY_ENABLED`（priority=100, 既定値）— 特性がぎたいに変わった／
  かがくへんかガスが切れたときの発動

## 変更対象（ファイル/関数単位）

- `src/jpoke/data/ability.py` — `ぎたい` に上記3イベントの handlers を登録する
- `src/jpoke/handlers/ability.py` — `ぎたい_change_type` を実装する
- `tests/abilities/test_ability_ka.py` — テストを追加する

## 実装ステップ

1. `src/jpoke/handlers/ability.py` にフィールド名→タイプの対応表
   （`ぎたい_フィールドタイプ対応表`）と `ぎたい_change_type` を実装する。
   - `mon.types == target_types` なら発動しない。
   - 発動する場合は `mon.ability_override_type` を更新し、`_announce_ability_triggered` でログを出す。
2. `src/jpoke/data/ability.py` の `ぎたい` に `ON_SWITCH_IN`（priority=120）・
   `ON_FIELD_CHANGE`・`ON_ABILITY_ENABLED` の3イベントへ同一ハンドラを登録する。
3. テストを追加し、全テストが通ることを確認する。

## テスト観点

1. フィールドが発生している場に繰り出すと、対応するタイプに変化する（4フィールド分）。
2. 自身が場にいる状態でフィールドが変化すると、即座にタイプが変わる。
3. フィールドが解除されると本来のタイプ（`base_types`）に戻る。
4. フィールドが無い状態で登場した場合は本来のタイプのまま（発動ログが出ない）。
5. 交代すると本来のタイプに戻り、フィールドがある場に再度出るとまた変化する。
6. 特性がぎたいに変わったとき（`change_ability`）、フィールドの有無に応じて即座にタイプが変わる。
7. かがくへんかガスの効果中は発動しない（`AbilityHandler` の既定処理で自動的にスキップされる）。

## 既知の実装上の制約（スコープ外）

- **へんしん（変身）状態時の挙動**: 仕様書に記載されている「へんしん状態でフィールドが無い場合に
  変身前後どちらのタイプへ戻るか」という分岐は、`へんしん`（技）自体が本エンジンで未実装
  （`docs/plan/moves/へんしん.md` 参照）のため、対応するテスト・実装は行わない。
  `へんしん` 実装時に本ハンドラとの相互作用を再検証すること。
- **みずびたし/まほうのこな/ミラータイプとの時系列的な優先順位**: `Pokemon.types` の解決順序は
  `move_override_types > ability_override_type > volatile_override_type > 本来のタイプ` に
  固定されている（`へんげんじざい・リベロ` 実装時に決定された既存の優先順位で、本特性のために
  変更しない）。そのため、フィールド発生中に `ぎたい` でタイプ変化済みのポケモンへ後から
  みずびたし/まほうのこなを付与しても、ウィキが記す「みずタイプ/エスパータイプへ一時的に
  変わり、次に `ぎたい` が発動したときにフィールドタイプへ戻る」という時系列上の見た目は
  再現されない（常に `ぎたい` のフィールドタイプが優先される）。一方、ミラータイプ・テクスチャー等
  の `move_override_types` は本特性より優先順位が高いため、そちらの「即座に戻らない」仕様は
  正しく再現される。この制約は `ability_override_type` を共有する他の特性
  （へんげんじざい・リベロ・へんしょく）にも共通する既存の設計上の簡略化であり、
  ぎたい固有の追加実装は行わない。
