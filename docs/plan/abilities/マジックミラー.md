# マジックミラー 実装計画

## スコープ
- 優先度: H
- 仕様書: `docs/spec/abilities/マジックミラー.md`（詳細な反射仕様は `docs/spec/volatiles/マジックコート.md` を参照）
- 実装方針: 既存の `Event.ON_CHECK_REFLECT` 経路へ特性ハンドラを追加し、反射対象の変化技をマジックコートと同一フェーズで跳ね返す。

## 実装対象イベント
- `Event.ON_CHECK_REFLECT`（`docs/spec/turn.md` に個別記載はないが、`docs/spec/volatiles/マジックコート.md` の
  「ターン処理の優先度」に準拠し、`ON_TRY_MOVE_1` 通過後・`ON_PAY_HP` 直後に発火する `ON_CHECK_REFLECT` を使う）

## 仕様要点（実装に反映）
- 反射対象は `Move.is_reflectable`（`category=="status"` かつ `target` が `foe`/`foe_side` かつ `unreflectable`
  フラグなし）で判定する。個別技名の判定ではなく、`unreflectable` フラグを持つ技だけを対象外とする
  デノイリスト方式（例: `トリック` / `すりかえ` に `unreflectable` を付与）。
- 攻撃技（`category` が `physical`/`special`）は `is_reflectable` が常に False になるため自然に反射しない。
- `かたやぶり` / `ターボブレイズ` / `テラボルテージ` により防御側特性が無効化される場合は反射しない。
  → 個別実装は不要。`マジックミラー` の `AbilityData.flags` に `mold_breaker_ignorable` を設定しておけば、
  `かたやぶり` 側ハンドラ（`ON_BEGIN_MOVE` で `battle.add_ability_disabled_reason(defender, "かたやぶり")`）が
  防御側特性を無効化し、`EventManager` の `enabled_ignoring` チェック（`core/event_manager.py`）により
  `ON_CHECK_REFLECT` ハンドラ自体が自動的にスキップされる。
- `かがくへんかガス` 等で `マジックミラー` が無効化されている間も同じ `enabled_ignoring` チェックにより
  自動的に反射しない（ハンドラ内で個別に確認不要）。
- 反射判定は `_execute_move` 内で技実行につき1回だけ `ON_CHECK_REFLECT` を発火する設計のため、
  跳ね返した効果が再び跳ね返されることは構造上起こらない（無限反射防止のための追加ガードは不要）。
- ミラーアーマーとの相互作用（`ON_BEFORE_MODIFY_STAT` という別イベント経路で処理される）も、
  上記の「1回だけ発火」という設計により自然に整合する。追加のハンドラ間連携コードは不要。

## 変更対象（想定）
- `src/jpoke/data/ability.py`
  - `マジックミラー` に `Event.ON_CHECK_REFLECT` ハンドラを登録する。
  - `flags={"mold_breaker_ignorable"}`、`subject_spec="defender:self"`、`priority=200`
    （`マジックコート` 状態のハンドラと同一優先度。シングルバトルでは競合しない）。
- `src/jpoke/handlers/ability.py`
  - `マジックミラー_reflect(battle, ctx: AttackContext, value: bool) -> HandlerReturn` を追加する。
  - 判定は `ctx.move.is_reflectable` のみで完結する（防御側特性の有効性チェックは
    `EventManager` 側で自動的に行われるため、ハンドラ内では不要）。
  - 反射成功時は `_announce_ability_triggered(battle, ctx.defender)` でログ出力する。

## 実装フロー
1. `handlers/ability.py` に反射判定ハンドラを追加する。
2. `data/ability.py` の `マジックミラー` 定義へ `ON_CHECK_REFLECT` 登録を追加する（`mold_breaker_ignorable` フラグも付与）。
3. `Event.ON_CHECK_REFLECT` に対する既存の `マジックコート` と競合しないことを確認する（`_execute_move` 内で
   1回のみ発火・スワップされるため、両者が同時に反射することはない）。
4. `tests/abilities/test_ability_ma.py` に発動・非発動・無効化・相互作用系の回帰テストを追加する。

## テスト観点
- 発動: 相手の反射対象変化技（例: なきごえ）を跳ね返す。
- 発動: 反射後、元使用者側が効果を受ける。
- 発動: 設置技（ステルスロック等）を跳ね返し、使用者側に設置する。
- 非発動: `unreflectable` フラグを持つ変化技（うつしえ等）は通常どおり通る。
- 非発動: 攻撃技は反射しない。
- 無効化: 攻撃側が `かたやぶり` 系なら反射しない。
- 無効化: `かがくへんかガス` 下では反射しない。
- 境界: 自分の `マジックミラー` で跳ね返ってきた効果を再び跳ね返さない。
- 相互作用: ミラーアーマー持ちから能力低下技を受けた場合、跳ね返した効果がミラーアーマーで再度跳ね返される。
- 相互作用: マジックミラー持ちがミラーアーマー持ちのランクを下げようとした場合、
  ミラーアーマーの反射はマジックミラーで再び跳ね返されない。

## 実装済み事項の確認（レビュー時点）
- 反射成功時のログ出力は `_announce_ability_triggered` による `ABILITY_TRIGGERED` ログで運用している。
- 反射対象技セットは `Move.is_reflectable` プロパティと `unreflectable` フラグに一本化されており、
  個別技名判定は増えていない。
