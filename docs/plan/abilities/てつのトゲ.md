# 特性実装計画: てつのトゲ

更新日: 2026-07-12

## スコープ

- 仕様書: `docs/spec/abilities/てつのトゲ.md`
- 実装状態: 実装済み
- 実装方針: `ON_DAMAGE_HIT` で `is_contact_reaction()` を満たしたときに攻撃者へ固定割合ダメージを与える。
  効果・発動条件が完全に同一の「さめはだ」と共通のヘルパー関数 `_apply_contact_counter_chip()` を再利用する。

## 実装ゴール（完了定義）

- 直接攻撃（接触技）でダメージを受けたとき、攻撃してきた相手に最大HPの1/8（切り捨て、最低1）のダメージを与える。
- 非接触技・技が無効化された場合（まもる等）・みがわりに防がれた場合は発動しない。
- まもる/みがわりを無視する直接攻撃で本体がダメージを受けた場合は発動する。
- 攻撃者がマジックガードを持つ場合はダメージを受けない（`ON_MODIFY_NON_MOVE_DAMAGE` 側で汎用的に処理される）。
- ぼうごパット所持者に対しては発動しない（`ON_CHECK_CONTACT_REACTION` 側で汎用的に処理される）。
- 連続攻撃技は1発ごとに発動し、途中で攻撃者がひんしになると以降のヒットは発生しない。
- てつのトゲ側がひんしになったときも発動する。
- 反撃ダメージはダメおしの「同ターン中にダメージを受けた」判定に加算される。
- 既存テストを壊さない。

## 実装対象イベント

- `Event.ON_DAMAGE_HIT` — 技のダメージが実際にヒットし、防御側の実HPが減少した直後（`turn.md` に個別記載なし。同系統の他特性〔さめはだ等〕と同様デフォルト優先度100を使用）。

## 変更対象（ファイル/関数単位）

- `src/jpoke/data/ability.py` — `てつのトゲ` に `Event.ON_DAMAGE_HIT` ハンドラを登録する（実装済み）
- `src/jpoke/handlers/ability.py` — 既存の `さめはだ_chip_contact_attacker()`（内部で共通ヘルパー `_apply_contact_counter_chip()` を呼び出す）をそのまま再利用する（実装済み。てつのトゲ専用の関数は新設しない）
- `tests/abilities/test_ability_sa.py` — さめはだと共通のパラメータ化テストで両特性を検証する（実装済み）

## 実装ステップ

1. 仕様書 `docs/spec/abilities/てつのトゲ.md`（および参照元の `docs/spec/abilities/さめはだ.md#特性の仕様`）で発動条件・例外条件を確認する。
2. 効果・発動条件が `さめはだ` と完全に同一であるため、新規ハンドラは実装せず `さめはだ_chip_contact_attacker()` を共有する。
   接触判定は `battle.query.is_contact_reaction(ctx)` に委譲し、ぼうごパット等の反応効果ブロックを共通処理に任せる。
   ダメージ適用は `battle.modify_hp(ctx.attacker, r=-1/8, reason="")` を用い、マジックガードの無効化は
   `reason=""`（`ON_MODIFY_NON_MOVE_DAMAGE`）側の共通処理に任せる。
3. `src/jpoke/data/ability.py` の `てつのトゲ` に `Event.ON_DAMAGE_HIT` ハンドラを `subject_spec="defender:self"` で登録する。
4. `tests/abilities/test_ability_sa.py` の共通パラメータ化テスト（`test_さめはだ_接触ダメージ`）に `てつのトゲ` のケースを含める。

## テスト観点

1. 接触技を受けたとき、相手の最大HPの1/8（切り捨て、最低1）のダメージを与える。
2. 非接触技（例: みずでっぽう）では発動しない。
3. さめはだと実装を共有しているため、マジックガード・ぼうごパット・みがわり・ひんし時発動・ダメおし連携などの
   個別分岐テストは `さめはだ` 側の既存テストで検証済み（共通ヘルパーのため重複実装は不要）。

## 検証コマンド

```powershell
python -m pytest tests/abilities/test_ability_sa.py -k "てつのトゲ or さめはだ" -v
python -m pytest tests/ -v
```

## リスクと対策

- リスク: さめはだと実装を共有しているため、片方だけを修正すると差異が生じる。
  - 対策: 両特性とも共通ヘルパー `_apply_contact_counter_chip()` を経由させ、ロジックを一本化する。
    仕様変更が必要な場合は `docs/spec/abilities/さめはだ.md` と `docs/spec/abilities/てつのトゲ.md` の
    両方を確認し、差異が生じていないか照合する。
