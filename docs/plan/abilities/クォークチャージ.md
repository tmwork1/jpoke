# 特性実装計画: クォークチャージ

更新日: 2026-07-11

## スコープ

- 仕様書: `docs/spec/abilities/クォークチャージ.md`
- 実装状態: 実装済み
- 実装方針: こだいかっせいと共通のパラドックス特性基盤（`src/jpoke/handlers/ability_paradox.py`）を
  参照し、参照する場の条件（エレキフィールド）とログ表現のみをクォークチャージ側で差し替える。
  共通基盤の設計意図は `docs/plan/abilities/こだいかっせい.md` を参照。

## 実装ゴール（完了定義）

- 場がエレキフィールド状態のとき、またはブーストエナジーを持っているとき、
  自分のランク補正込みで最も高い能力（HP除く）が上がる。
  - 攻撃・防御・特攻・特防は1.3倍（5325/4096）、素早さは1.5倍（6144/4096）。
  - 同値時は 攻撃→防御→特攻→特防→素早さ の順で先に判定した能力が選ばれる。
- 場条件・アイテム条件のどちらか一方のみで発動し、二重発動しない
  （場条件を優先し、場条件由来の効果が切れたときのみ未消費のブーストエナジーへフォールバックする）。
- かがくへんかガスの発動中は補正が無効化され、解除されると同じ能力に即座に復帰する
  （能力の選び直しは行わない）。
- 交代すると発動状態・上昇対象・発動源がリセットされる。
- スキルスワップは使用者・対象のどちらかがこの特性であれば失敗する（コピー不可）。
- 既存テストを壊さない。

## 実装対象イベント

- `Event.ON_SWITCH_IN`（priority=140, `docs/spec/turn.md` ON_SWITCH_IN 表の
  「140 クォークチャージ（特性）」に対応） — 場に出た直後、いかく等の入場効果
  （priority 100/120）より後、しろいハーブ等（priority 160）より前に発動判定を行う。
- `Event.ON_ABILITY_ENABLED`（priority=200, turn.md 未掲載のためこだいかっせいの
  既存登録に合わせる） — かがくへんかガス解除やとくせいなし解除で特性が再有効化された
  ときに補正状態を再評価する。
- `Event.ON_FIELD_CHANGE`（priority=200, turn.md 未掲載のためこだいかっせいの
  既存登録に合わせる） — 場に出た後にフィールドがエレキフィールドへ/から変化したときに
  即座に再評価する。
- `Event.ON_REFRESH_PARADOX_BOOST`（priority=200） — ブーストエナジーの入手など、
  アイテム側から補正状態の再評価を要求するための内部イベント。
- `DomainEvent.ON_CALC_SPEED` — 素早さ実数値計算時に1.5倍補正を適用する。
- `Event.ON_CALC_ATK_MODIFIER` / `Event.ON_CALC_DEF_MODIFIER` — ダメージ計算の
  攻撃側/防御側実数値補正時に1.3倍補正を適用する。

## 変更対象（ファイル/関数単位）

- `src/jpoke/data/ability.py` — `クォークチャージ` に `uncopyable` フラグと上記 handlers を登録する。
- `src/jpoke/handlers/ability_paradox.py` — こだいかっせいと共通の
  `refresh_paradox_charge_state` / `modify_speed` / `apply_atk_modifier` / `apply_def_modifier`
  を実装する（特性名で場条件を分岐）。
- `src/jpoke/handlers/move_status.py` — `スキルスワップ_can_apply` にクォークチャージ/
  こだいかっせい/かがくへんかガスの base_name 個別除外を実装する
  （`protected` フラグは付与しない。Ver.3.0.0以降は他の上書き手段〈いえき・なりきり等〉に
  対しては上書き可能であり、スキルスワップだけが例外的に失敗するため）。
- `tests/abilities/test_ability__group.py` — こだいかっせいと共通のパラメトライズテストを追加する。

## 実装ステップ

1. 仕様書 `docs/spec/abilities/クォークチャージ.md` で発動条件・例外条件を確認する。
2. `ability_paradox.py` の共通 helper（こだいかっせい実装時に作成済み）を利用し、
   `クォークチャージ` の AbilityData に handler を登録する。
3. `docs/spec/turn.md` の ON_SWITCH_IN 表でクォークチャージの priority（140）を確認し、
   いかく等の入場効果より後・しろいハーブ等より前になるよう登録する。
4. かがくへんかガス解除時に補正状態を再評価できるよう `Event.ON_ABILITY_ENABLED` を登録する
   （こだいかっせいの既存登録と揃える）。
5. `スキルスワップ_can_apply` にクォークチャージ/こだいかっせいの base_name 除外を追加し、
   スキルスワップが正しく失敗することを確認する。
6. `tests/` にテストを追加し、全テストが通ることを確認する。

## テスト観点

1. エレキフィールド下で発動し、最も高い能力（ランク補正込み）が上がる。
2. ブーストエナジー所持時に発動し、アイテムを消費する（場条件が無いとき）。
3. 場条件が優先され、場条件由来の効果が切れたときのみ未消費のブーストエナジーへ切り替わる。
4. 交代で発動状態・アイテム消費状態がリセットされる。
5. かがくへんかガスの発動中は補正が無効化され、解除後に同じ能力へ復帰する
   （能力の選び直しは行わない）。
6. しろいハーブ等の入場効果より先にクォークチャージの発動判定が行われる
   （バトンタッチで引き継いだ下降ランクを参照して能力を選ぶ）。
7. スキルスワップは使用者・対象のどちらかがこの特性であれば失敗する。
8. イカサマ・ボディプレスなど参照する実数値が特殊な技でも、補正は使用者自身の
   攻撃側スロット（物理なら攻撃、特殊なら特攻）にのみ適用される。

## リスクと対策

- リスク: `ability_paradox.py` はこだいかっせいと共通実装のため、クォークチャージ側の
  修正がこだいかっせいの挙動にも影響する。
  - 対策: 場条件の分岐はすべて `mon.ability.name` で判定しているため、
    フィールド固有の条件（エレキフィールド/はれ）以外は同一ロジックを再利用してよい。
    priority など `data/ability.py` 側の登録値は特性ごとに個別管理する。
