# ふしょく 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/ふしょく.md`
- 実装方針: どく・もうどく付与の共通経路を単一の制御点に寄せ、`ふしょく` は「どく/はがねタイプへの通常無効だけを外す」例外として扱う。状態異常そのものを防ぐ他要因は既存の `Event.ON_BEFORE_APPLY_AILMENT` 判定をそのまま通す。

## 現状整理
- `src/jpoke/data/ability.py` の `ふしょく` は `AbilityData()` のみで、ハンドラ未登録。
- 状態異常付与は `src/jpoke/handlers/common.py` の `apply_ailment()` から `battle.ailment_manager.apply()` に集約されている。
- `src/jpoke/core/pokemon_state.py` の `AilmentManager.apply()` は `Event.ON_BEFORE_APPLY_AILMENT` を発火し、`めんえき`・`ふみん`・`やるき`・`ミストフィールド`・`しんぴのまもり` などの無効化要因を共通処理している。
- `どくどく` のような状態異常技は `common.apply_ailment()` を経由している一方、`どくびし` は直接 `ailment_manager.apply()` を呼んでおり、こちらも共通制御点側を直せば追従できる。
- `src/jpoke/core/move_executor.py` の変化技処理は `Event.ON_CHECK_IMMUNE` を発火するが、`src/jpoke/handlers/move.py` のタイプ無効判定 `_is_type_immune()` は一撃必殺技ハンドラからしか参照されていない。
- 実挙動確認では、`ふしょく` なしの `どくどく` でも `コイル`・`フシギダネ` に通常どおり `もうどく` が入る。つまり、毒タイプ/鋼タイプへの毒無効は現状未実装で、`ふしょく` 以前の共通仕様欠落になっている。

## 仕様要点（実装に反映）
- `ふしょく` 持ちが与える `どく` / `もうどく` は、通常は無効の どくタイプ・はがねタイプ にも通る。
- `めんえき`、`ミストフィールド`、`しんぴのまもり`、`みがわり`、既存の別状態異常など、「タイプ相性以外」の無効要因は貫通しない。
- 特性無効化中や特性変更後は通常どおりの毒無効判定に戻る。
- 対象外は「毒付与そのもの」であり、ダメージ・命中・追加効果抽選など他の処理順は変えない。

## 未対応・要確認（レビューで解消）
- 毒タイプ/鋼タイプへの毒無効は「分散している」のではなく、少なくとも通常の毒付与経路では未実装だった。`AilmentManager.apply()` の `_can_apply_by_type()` に一元化して解消済み。
- `ON_BEFORE_APPLY_AILMENT` は現在「状態異常そのものを防ぐ」用途で使われているため、`ふしょく` はここへ直接載せず、タイプ無効の判定ヘルパー `_can_apply_by_type()` 側に寄せた。
- `どくびし` や `トーチカ` など、技以外の毒付与経路にも `source` が渡る。特に `どくびし` は `handlers/field.py` の `どくびし_apply_poison()` で `source=ctx.source`（=繰り出したポケモン自身）を渡しているため、繰り出したポケモン自身が `ふしょく` 持ちのはがね/どくタイプ（テラスタル含む）だと従来 `_can_apply_by_type()` が `source.ability.name == "ふしょく"` を True と誤判定し、本来無効なはずのどく付与を通してしまうバグがあった（一次情報「どくびし状態の場に、はがねタイプにテラスタルしたふしょくのポケモンを繰り出してもどく状態にはならない」に反する）。
  - 対処: `AilmentManager.apply()` に `allow_type_immunity_bypass: bool = True` を追加し、`どくびし_apply_poison()` からの呼び出しのみ `allow_type_immunity_bypass=False` を渡してタイプ無効貫通を無効化した。`source=ctx.source` 自体は据え置き（`シンクロ` 等の `is_foe_target()` 判定に必要なため変更しない）。
  - `どくどくだま`（`handlers/item.py` の `_apply_ailment_from_item()`）も `source=mon`（自分自身）だが、こちらは一次情報どおり「ふしょく持ちが自身のどくどくだまで、はがね/どくタイプでももうどくになる」が正しい仕様のため `allow_type_immunity_bypass` は既定の `True` のまま変更していない。

## 実装フロー（実装済み）
1. `AilmentManager.apply()` に、どく/もうどく付与時のタイプ無効判定ヘルパー `_can_apply_by_type()` を追加した。
2. `_can_apply_by_type()` では「対象が どく/はがね か」「付与元 `source` が存在し、`ふしょく` が有効か」を判定し、`ふしょく` 有効時だけタイプ無効を解除する。
3. `めんえき` などの既存 `ON_BEFORE_APPLY_AILMENT` はそのまま残し、`ふしょく` はそれらを素通ししない。
4. `src/jpoke/core/move_executor.py` / `src/jpoke/handlers/move.py` の `ON_CHECK_IMMUNE` 側には手を入れず、毒付与そのものの可否だけを状態異常レイヤで扱う。
5. `src/jpoke/data/ability.py` の `ふしょく` は `AbilityData()` のまま（専用ハンドラなし）。
6. `AilmentManager.apply()` に `allow_type_immunity_bypass` パラメータを追加し、`どくびし` のようにタイプ無効貫通の対象外とすべき経路で `False` を渡せるようにした。
7. 追加効果毒技、変化技、場由来の毒付与（どくびし・どくどくだま）で挙動差が出ないことをテストで固定した。

## テスト観点
- 発動ケース: `ふしょく` 持ちの `どくどく` が はがねタイプ・どくタイプ の相手に成功する。
- 発動ケース: `ふしょく` 持ちの追加効果毒技で、ダメージ後の毒付与判定が成功する。
- 非発動ケース: `ふしょく` なしでは従来どおり、はがねタイプ・どくタイプ への毒付与は失敗する。
- 無効化ケース: 相手が `めんえき` のときは `ふしょく` 持ちでも失敗する。
- 無効化ケース: `ミストフィールド`・`しんぴのまもり`・`みがわり`・既存状態異常中には失敗する。
- 境界ケース: `かがくへんかガス` や `かたやぶり` 系の既存特性有効/無効判定と矛盾しない。
- 境界ケース: `どくびし` や `トーチカ` など、`ふしょく` を持たない付与元の経路は従来どおりである。
- 境界ケース: `どくびし` は繰り出したポケモン自身が `ふしょく` 持ちのはがねタイプでもタイプ無効貫通の対象外で、どく状態にならない（`allow_type_immunity_bypass=False`）。
- 境界ケース: `どくどくだま` は持ち主が `ふしょく` 持ちなら、はがね・どくタイプでも自身のもうどくが発動する（`allow_type_immunity_bypass` は既定の `True`）。
- 境界ケース: どく技自体がはがねタイプへのタイプ相性無効でダメージが通らない場合、`ふしょく` 持ちでも追加効果のどく付与は発動しない。
- 境界ケース: 相手の特性 `どくのトゲ` による付与は `ふしょく` の効果対象外であり、`ふしょく` 持ちがはがね・どくタイプなら通常どおり毒状態にならない。
