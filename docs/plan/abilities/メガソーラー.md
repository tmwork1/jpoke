## 仕様まとめ（`docs/spec/abilities/メガソーラー.md` より）

1. 自分が行動するとき、天気がにほんばれ状態であるものとして技を使用する（実際の天候は変化しない）
2. ノーてんきのポケモンがいてもメガソーラーの効果は有効
3. ソーラービーム/ソーラーブレード: 溜めターンをスキップ（既存の溜めハンドラが `battle.weather.sunny` を参照するため自動対応）
4. ウェザーボール: 威力100のほのおタイプ（同上）
5. ダメージ補正: ほのおタイプ1.5倍、みずタイプ0.5倍
6. 命中補正: かみなり・ぼうふうが50%になる
7. こうごうせい/つきのひかり/あさのひざし: 最大HPの2/3回復
8. 岩タイプの特防補正・こおりタイプの防御補正を無視（各ハンドラが `battle.weather.name` を参照するため自動対応）
9. 相手のリーフガードを発動させない（リーフガード実装時に確認）

## 実装ゴール（完了定義）

- メガソーラー持ちが攻撃するとき、天候に関係なくほのお技が1.5倍・みず技が0.5倍になる
- ノーてんきが相手にいてもメガソーラーの効果が有効である
- 攻撃後（次のポケモンの行動前）に天候が元に戻る
- ソーラービームがはれ扱いで即撃てる（既存のソーラービームハンドラで対応）
- テストがすべて通る

## 実装方針

### なぜ直接書き換えか

`weather_manager.apply()` は以下の副作用を持つため使用しない：
- `ON_FIELD_ACTIVATE` / `ON_FIELD_DEACTIVATE` の発火（すなおこし等のカウントダウン等が動く）
- `ON_FIELD_CHANGE` の発火（こだいかっせい等の再判定が走る）
- ハンドラの登録・解除

直接 `weather_manager.current_name` と `fields["はれ"].count` を書き換えることで、
`battle.weather` / `battle.raw_weather` を参照する既存ハンドラが自動的に「はれ」として動作する。

### ノーてんき対策

`battle.weather` は `weather_manager.active` を呼び出し、`ON_CHECK_WEATHER_ENABLED` を発火する。
ノーてんきが有効なら `False` が返って inactive Field が返されるため、
`ON_CHECK_WEATHER_ENABLED` に `stop_event=True, value=True` を返すハンドラを登録する。
subject_spec は `"source:self"`。

### 発動・復元のタイミング

`move_executor.py` の `run_move` は `try/finally` で攻撃前後をブラケット。
現状このブラケットを利用できるイベントは `ON_BEGIN_MOVE` / `ON_END_MOVE`（priority: 既存の かたやぶりハンドラを参照）。

- **発動**: `ON_BEGIN_MOVE`（`"attacker:self"`）
  - 元の `weather_manager.current_name` と元 Field の `count` を `ability.state` に保存
  - `weather_manager.current_name = "はれ"` にセット
  - `weather_manager.fields["はれ"].count` を `1` にセット（`Field.name` が空文字を返すのを防ぐ）
- **復元**: `ON_END_MOVE`（`"attacker:self"`）
  - `ability.state` から元の `current_name` と `count` を復元

### 状態の保存形式

`AbilityState`（`ability.state`）は文字列 Literal 型のためタプルを格納できない。
そのため `Ability` クラスに専用属性を追加し、状態を保存する（`ability.state` 自体は
「上書き中かどうか」を示す `"active"` / `""` のフラグとしてのみ使用する）：

- `ability.saved_weather_name: WeatherName` — 上書き前の本来の天候名
- `ability.saved_weather_count: int` — 上書き前の「はれ」フィールドのカウント
- `ability.weather_override_depth: int` — 下記「ねごと・まねっこ等のネスト対応」参照

### ねごと・まねっこ等のネスト対応

ねごと・まねっこは、選択/コピーした技を `battle.run_move(attacker, ...)` で
**同じ攻撃者に対して再帰的に** 実行する。この場合、外側の技（ねごと自身等）の
`ON_BEGIN_MOVE` で天候が「はれ」に上書きされた状態のまま、内側の技の
`ON_BEGIN_MOVE` / `ON_END_MOVE` が同じ攻撃者に対してもう一度発火する。

単純に `ability.state` の `"active"` / `""` だけで管理すると、内側の `ON_END_MOVE` が
先に `state = ""` にリセットしてしまい、外側の `ON_END_MOVE` が
「既に非アクティブだから何もしない」と誤判定し、天候が「はれ」のまま
永久に復元されなくなる（実際の天候が壊れたまま残るバグ）。

これを避けるため `ability.weather_override_depth` で発動階層を数える。
- `メガソーラー_activate`: `depth == 0` のときのみ本来の天候を保存し、常に `depth += 1`
- `メガソーラー_deactivate`: 常に `depth -= 1`。`depth > 0` の間は復元処理をスキップし、
  `depth` が0に戻った最も外側の呼び出し時のみ本来の天候へ復元する

### 天候を変更する技との相互作用

にほんばれ・あまごい・すなあらし・ゆきげしき・さむいギャグは、いずれも
`weather_manager.apply(name, count)` を呼んで実際に天候を変更する技である。
`apply()` は `if self.current_name == name: return False` で
「既に同じ天候なら失敗」と判定するため、メガソーラーが天候を「はれ」に仮想上書きした
状態のまま にほんばれ を使うと、常に「既にはれ」と誤判定されて技が失敗し、
かつ `ON_END_MOVE` で本来の天候（例: あめ）へ巻き戻されてしまう
（一次情報: 「この特性であっても技のにほんばれは使用でき、場をにほんばれ状態に
変えることができる」という記述と矛盾する）。

対策として、`ctx.move.name` が上記5技のいずれかに一致する場合は
`メガソーラー_activate` / `メガソーラー_deactivate` の両方で天候の仮想上書き自体を
スキップする（`_メガソーラー_WEATHER_SETTING_MOVES` 定数を参照）。これにより、
これらの技は実際の天候に対して通常どおりの成功/失敗判定・変更を行える。

### 既知の非対応

- **せいちょうとの相互作用**: 一次情報によれば、メガソーラー所持者がせいちょうを
  使用した場合、判定基準が「仮想的なはれ」ではなく「実際の天候」になる
  （実際に晴れていなければ1段階も上昇せず、メッセージなしで行動終了し、
  次のじだんだ/やけっぱちの威力が2倍になる）。これは damage 計算・充填ターン省略等
  他の全ての効果が「仮想はれ」を参照するのとは異なる、せいちょう固有の例外的挙動である。
  メガニウム自身はせいちょうを覚えないため通常のパーティ構築では到達不能で、
  とくせいコピー・トレース等でメガソーラーを獲得した別のポケモンがせいちょうを
  使う場合のみ発現する極めて限定的なケースのため、実装コストとのバランスから
  今回は非対応とする（`docs/spec/abilities/メガソーラー.md` 特性の仕様に明記済み）。
- **ねごと等で選ばれた技が天候変更技だった場合の二重ネスト**: 「天候を変更する技との
  相互作用」の対策は `ctx.move.name` を直接比較するため、ねごとの外側の
  `ON_BEGIN_MOVE`（ねごと自身、非天候技なので通常どおり仮想上書きが有効）が先に
  発動している状態で、ねごとが選んだ内側の技がたまたま天候変更技だった場合、
  内側の `apply()` は依然として外側が上書きした「はれ」を参照してしまい、
  正しく動作しない。ねごとが天候変更技を選択できる確率で発生する非常に稀な
  組み合わせのため、今回は対応を見送る。
- **攻撃対象（相手）の天候依存効果への波及**: 現在の実装は
  `weather_manager.current_name` を物理的に書き換えるため、メガソーラー所持者の
  攻撃中（`ON_BEGIN_MOVE`〜`ON_END_MOVE`の間）は、相手側のすながくれ・ゆきがくれ等
  天候依存の回避率補正も理論上「はれ」を参照してしまう可能性がある
  （一次情報に明記された挙動ではないため確証はない）。正しく分離するには
  「読み取り時にのみ攻撃者視点でオーバーレイする」設計への刷新が必要で、
  影響範囲が広いため今回は対応を見送る。

## 実装ステップ

1. `handlers/ability.py` に3つのハンドラ関数を実装する
   - `メガソーラー_activate(battle, ctx, value)` — 天候を直接「はれ」に変更
   - `メガソーラー_deactivate(battle, ctx, value)` — 天候を元に戻す
   - `メガソーラー_override_weather_enabled(battle, ctx, value)` — `ON_CHECK_WEATHER_ENABLED` で True を返す
2. `data/ability.py` に登録する
3. テストを追加し、全テストが通ることを確認する

## テスト観点

- メガソーラー持ちが攻撃するとき: ほのお技が1.5倍になる（天候なしでも）
- メガソーラー持ちが攻撃するとき: みず技が0.5倍になる（天候なしでも）
- 攻撃後に天候が元の状態に戻る（あめ中なら攻撃後もあめのまま）
- ノーてんきが相手にいても: ほのお技が1.5倍になる
- 相手が攻撃するとき: 天候補正は元の天候のまま（メガソーラーの効果は相手に適用されない）
- ソーラービームを使用したとき: はれ扱いで即撃てる（溜めなし）
- にほんばれを使用したとき: 実際の天候（あめ等）をはれに変更でき、変更後も維持される
- まねっこ等で技実行がネストしたときも、行動全体の終了後に天候が正しく元に戻る

## 既知の考慮事項

- `ON_BEGIN_MOVE` / `ON_END_MOVE` を借用しているため、
  かたやぶり以外の用途で使われていることを明確にするコメントを付ける
- `ability.state` は「上書き中かどうか」を示すフラグとしてのみ使用し、
  元の天候情報は `ability.saved_weather_name` / `ability.saved_weather_count` に保存する
  （メガソーラーはこれら専用属性以外の `ability.state` を使用しない）
- リーフガードが実装済みの場合、リーフガードハンドラ側での対応が別途必要
- 「既知の非対応」節（せいちょう相互作用・ねごとの二重ネスト・相手への波及）は
  将来的な精度向上の余地として残す
