# へんげんじざい・リベロ 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/へんげんじざい.md`, `spec/ability/リベロ.md`
- 実装方針: 同一効果のため、共通ハンドラ1本を `へんげんじざい` と `リベロ` の両方に登録する。

## 実装ゴール（完了定義）
- 技実行直前に、使用者のタイプがその技タイプへ変化する。
- 1回の場滞在につき1回だけ発動する（SV仕様）。
- 行動不能や技不発時は発動しない。
- 交代でタイプ変化状態と発動回数がリセットされる。
- `へんげんじざい` と `リベロ` で挙動差が出ない。

## 現状整理
- `src/jpoke/data/ability.py` の `へんげんじざい` / `リベロ` は `AbilityData()` のみで、専用ハンドラ未登録。
- 行動フローは `src/jpoke/core/move_executor.py` の `_execute_move()` で進み、`ON_CHECK_ACTION` / `ON_MOVE_CHARGE` / `ON_CHECK_MOVE` を通過後に実ダメージ処理へ入る。
- `src/jpoke/model/pokemon.py` の `types` は現在 `data.types`（またはテラスタル時の型）を返すのみで、特性による一時タイプ変更状態を保持していない。
- `src/jpoke/core/switch.py` の `run_switch()` は退場時に `old.bench_reset()` を呼ぶため、場依存の状態はここでリセットできる。

## 仕様反映ポイント
- 発動タイミングは「行動成立後、威力/一致補正計算前」に固定する。
- 発火イベントは `spec/turn.md` に合わせて `Event.ON_MOVE_CHARGE` を採用する。
- テラスタル中は非発動（仕様書どおり）にする。
- 変化技も発動対象に含める。
- `かがくへんかガス` による特性無効化中は発動しない。
- すでにタイプ変化が成立した後に `かがくへんかガス` で特性が無効化されても、その場では元タイプに戻さない（交代時リセットのみ）。

## 変更対象（ファイル単位）
- `src/jpoke/model/pokemon.py`
  - 特性由来の一時タイプを保持する属性（例: `ability_override_type`）を追加。
  - `types` プロパティでテラスタル未使用時に一時タイプを優先参照する。
  - `bench_reset()` で一時タイプを初期化する。
  - 特性の `enabled` 切り替えでは一時タイプを変更しない（解除は交代時のみ）。
- `src/jpoke/handlers/ability.py`
  - 共通ハンドラ（例: `タイプ追従_change_type_on_move`）を追加。
  - 発動条件（行動成立、未発動、非テラスタル、特性有効、技タイプあり）をここで判定する。
  - 発動時にタイプ変更と `ability.count += 1` を行い、特性発動ログを記録する。
- `src/jpoke/data/ability.py`
  - `へんげんじざい` / `リベロ` の両方に同一ハンドラを登録する。
  - `Event.ON_MOVE_CHARGE` へ登録し、優先度は `spec/turn.md` 記載の 100 に合わせる。
- `tests/test_ability.py`
  - へんげんじざい・リベロ共通テスト（発動/非発動/リセット）を追加する。

## 実装ステップ
1. `Pokemon` に場依存タイプ上書き状態を追加し、`bench_reset()` で確実に初期化する。
2. 共通ハンドラを実装し、以下を順に処理する。
   - 特性有効/行動成立/未発動/非テラスタルを確認
   - 技の有効タイプを取得して、現在タイプとの差分があるときのみ更新
   - 発動回数カウントとログ出力
  - 特性が後で無効化されても、タイプ上書き状態は保持する
3. `data/ability.py` で `へんげんじざい` と `リベロ` を `Event.ON_MOVE_CHARGE` (priority=100) の同一ハンドラへ接続する。
4. テストを追加し、既存のタイプ補正・テラスタル回帰を確認する。

## テスト計画
- 発動ケース: 異なるタイプの攻撃技で先にタイプが変化し、STAB が乗る。
- 発動ケース: 変化技でもタイプが変化する。
- 非発動ケース: 行動不能（`ON_CHECK_ACTION=False`）なら変化しない。
- 非発動ケース: 同一場滞在2回目は発動しない。
- 非発動ケース: テラスタル中は発動しない。
- 回復ケース: 交代後に再度1回だけ発動できる。
- 同一性ケース: 同条件で `へんげんじざい` と `リベロ` の結果が一致する。
- 相互作用ケース: `かがくへんかガス` 展開中は両特性とも発動しない。
- 相互作用ケース: タイプ変化成立後に `かがくへんかガス` が展開されても、交代するまで変化後タイプを維持する。

## リスクと対策
- リスク: 発火イベントが早すぎると、行動失敗時にもタイプが変わる。
  - 対策: 行動成立後にしか通らないイベントへ接続する。必要なら専用イベントを追加する。
- リスク: 一時タイプ状態のリセット漏れで交代後にタイプが残留する。
  - 対策: `bench_reset()` に初期化を集約し、交代テストで検出する。
- リスク: 特性有効状態の再計算時に、タイプ変化まで巻き戻してしまう。
  - 対策: タイプ変化状態は `ability.enabled` と独立して保持し、解除トリガーを交代時に限定する。
- リスク: テラスタル中のタイプ計算と干渉する。
  - 対策: `types` の優先順を `テラスタル > 特性上書き > 元タイプ` に固定し、回帰テストで担保する。
