# 実装計画: いたずらごころ

作成日: 2026-07-10

## 効果
全ての変化技の優先度を1加算して使用する。
第七世代以降では、この特性の効果で優先度を上げた技は敵のあくタイプに無効化される（PPは消費される）。

## スコープ（シングルバトル前提の簡略化）
- 本プロジェクトはポケモンチャンピオンズシングルバトルのみが対象。仕様書にあるダブルバトル固有の記述
  （味方対象の技・おさきにどうぞ／さいはいで発動する技・自分含む全員が対象の技など）は、シングルバトルでは
  「味方」が存在しないため実質的に該当しない。
- `model/move.py` の `MoveTarget` は `"foe" | "foe_side" | "own_side" | "self" | "field"` の5値のみを持つ
  （シングル専用のため「相手全員」「自分含む全員」等の区別は不要）。あくタイプ無効化は `target == "foe"`
  （相手単体を対象とする変化技）のときのみ判定すればよい。

## イベント／ハンドラ

| イベント | priority | subject_spec | 関数名 |
|---------|---------|-------------|--------|
| `DomainEvent.ON_MODIFY_MOVE_PRIORITY` | 既定値（100） | `attacker:self` | `いたずらごころ_modify_move_priority` |
| `Event.ON_BEFORE_APPLY_MOVE` | 既定値（100） | `attacker:self` | `いたずらごころ_blocked_by_dark` |

### priority の根拠
- `DomainEvent.ON_MODIFY_MOVE_PRIORITY` は `docs/spec/turn.md` に専用セクションが無いが、
  `enums/event.py` のコメントで「いたずらごころ等」の優先度加算専用イベントと明記されており、
  他に同イベントを使う特性が無いため既定 priority のままでよい。
- `Event.ON_BEFORE_APPLY_MOVE` は `turn.md` 上では見出しが `Event.ON_APPLY_MOVE`（技が無効化されないかの
  最終判定）という現行 `enums/event.py` には存在しない旧称のセクションに対応する
  （`ON_TRY_MOVE_2` → 命中判定 Interrupt → 当該セクション、という並びが実際のコード順序
  `ON_TRY_MOVE_2` → 命中判定 → `ON_BEFORE_APPLY_MOVE` と一致するため）。
  従来 `turn.md` は本特性のあくタイプ無効化を誤って `Event.ON_TRY_MOVE_2`（priority=120）の表に
  記載していたが、`ON_TRY_MOVE_2` を実際に使用するハンドラは `src/jpoke/data/*.py` に1件も無く
  （コード全体で未使用）、実装は `Event.ON_BEFORE_APPLY_MOVE` に存在する。本レビューで `turn.md` の
  当該行を `Event.ON_APPLY_MOVE`（実体は `Event.ON_BEFORE_APPLY_MOVE`）のテーブルへ priority=100 として
  移設した。同イベントを使う既存の免疫判定系ハンドラ（`そうしょく_absorb_grass` 等）はいずれも priority
  未指定（既定値100）で、判定同士は「他の判定と両立しても最終的にどちらか片方が `stop_event=True` を
  返せば無効化が成立する」ため相互の順序に依存しない。
  `じょおうのいげん`/`ビビッドボディ`/`テイルアーマー`（`ON_TRY_MOVE_1` priority=40）、
  `サイコフィールド`/`ファストガード`（`ON_TRY_MOVE_1` priority=100）は
  いずれも `ON_TRY_MOVE_1` で `ON_BEFORE_APPLY_MOVE` より前に発火するイベントであるため、
  仕様書が定める無効化の優先順位
  「じょおうのいげん/ビビッドボディ/テイルアーマー → サイコフィールド → ファストガード → あくタイプ」
  は、イベントの発火順（`ON_TRY_MOVE_1` → `ON_BEFORE_APPLY_MOVE`）で自然に満たされる。
  よって `いたずらごころ_blocked_by_dark` は既定 priority のままでよい。

## 実装

```python
def いたずらごころ_modify_move_priority(battle: Battle, ctx: AttackContext, value: int) -> HandlerReturn:
    """いたずらごころ特性: 変化技の優先度を+1する。"""
    if ctx.move.category == "status":
        value += 1
    return HandlerReturn(value=value)


def いたずらごころ_blocked_by_dark(battle: Battle, ctx: AttackContext, value: bool) -> HandlerReturn:
    """いたずらごころ特性: 優先度が上がった変化技はあくタイプ相手に無効化される。"""
    if (
        ctx.move.category == "status"
        and ctx.move.target == "foe"
        and ctx.defender.has_type("あく")
    ):
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_IMMUNED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="いたずらごころ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

- あくタイプ無効化は「+1した結果の優先度が0以下でも無効になる」（仕様書 `ほえる` の例）ため、
  実際の優先度値を再計算せず、変化技かつ相手単体対象であれば常に判定する。
- `ctx.move.target == "foe"` により、自己対象技（例: ふるいたてる／かえんのまもり）や
  場対象技（`own_side` / `foe_side` / `field`）はあくタイプ無効化の対象外になる（仕様通り）。
- 無効化しても PP は消費される（`ON_BEFORE_APPLY_MOVE` は PP 消費〈`ON_CONSUME_PP`〉より後段のイベントのため、
  既存の PP 消費フローに手を加える必要はない）。
- かがくへんかガス・かたやぶり等による特性無効化は共通の特性無効化チェックで処理されるため、
  本ハンドラ側に重複ガードは書かない。

## data/ability.py 登録

```python
"いたずらごころ": AbilityData(
    handlers={
        DomainEvent.ON_MODIFY_MOVE_PRIORITY: h.AbilityHandler(
            h.いたずらごころ_modify_move_priority,
            subject_spec="attacker:self",
        ),
        Event.ON_BEFORE_APPLY_MOVE: h.AbilityHandler(
            h.いたずらごころ_blocked_by_dark,
            subject_spec="attacker:self",
        ),
    }
),
```

## テストケース
- 変化技（でんじは等）を使うと `speed_calculator.calc_move_priority` が+1される。
- あくタイプの相手に変化技（でんじは）を使うと不発になる（状態異常が付与されない・`move_applied is False`）。
- 自己対象の変化技（かえんのまもり／ふるいたてる）はあくタイプの相手でも成功する。
- ファストガードは、技データ上は優先度0でも、いたずらごころによって動的に+1された変化技を防ぐ
  （`tests/moves_status/test_move_fa.py`）。

## リーサル計算
- 対象外（`n/a`）。優先度変更・技無効化はダメージ算出に関与しないため `handlers/lethal.py` に対応不要。
