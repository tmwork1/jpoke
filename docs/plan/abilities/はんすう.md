# はんすう 実装計画

## スコープ
- 優先度: -
- 仕様書: `docs/spec/abilities/はんすう.md`
- 実装方針: 「きのみ消費の確定」を単一イベントに集約し、2ターン目の終了時に同じきのみを
  もう一度発動させる。HP割合等の発動条件は無視して強制発動させる既存の
  `ItemManager.force_trigger_berry`（ほおばる・おちゃかい・むしくい・ついばむが使用）の
  仕組みをそのまま再利用する。
- 非スコープ（既知の制約として実装に明記する）:
  - はんすう発動タイミングで既に別のアイテムを持っている場合（例: きのみ消費後に
    トリック等で新しい持ち物を得た場合）は、持ち物を書き換えずに効果の再現を諦める
    （`特性でアイテムが復活するわけではない`という仕様上、既存の持ち物を上書きしてはならないが、
    持ち物を保持したまま効果だけを個別に再現する汎用の仕組みが無いため）。
  - スカーレット・バイオレット Ver.1.2.0 の「2ターンごとに無限発動する」不具合は対象外
    （本リポジトリは最新パッチの仕様＝Ver.1.3.0以降の修正済み挙動を実装する）。

## 実装対象イベント（第一候補）
- 新規イベント `Event.ON_BERRY_CONSUMED`（`ItemManager.consume_item` 内で
  `mon.item.is_berry()` の場合に emit する。source:self で使う `EventContext` に
  `item_name: ItemName` フィールドを追加して消費したきのみ名を伝える）。
  - `consume_item` は通常のきのみ発動に加えて `force_trigger_berry`
    （ほおばる・おちゃかい・むしくい・ついばむ）からも呼ばれるため、これらの経路も
    自動的にはんすうの起点になる。
  - なげつける（技）は `ItemManager.consume_item` を経由せず `remove_item` を直接呼ぶため、
    `handlers/move_attack.py` の `なげつける_consume_item`（使用者自身の消費）と
    `なげつける_apply_item_effect`（対象がきのみ効果を受けたとき）から個別に
    `Event.ON_BERRY_CONSUMED` を emit する。
- `Event.ON_TURN_END`（priority=150。`docs/spec/turn.md` の `Event.ON_TURN_END` 節
  「150 はんすう」を参照。同じ priority のかそく/ムラっけ/スロースタート/ナイトメア/
  くっつきバリ/どくどくだま/かえんだま/ものひろい/しゅうかく等とはすばやさ順、
  同速なら登録順でタイブレークされる。くっつきバリ等の持ち物発動より前、
  だっしゅつパック/ものまねハーブ（priority=160、ランク変化後）より前に判定させる必要がある）

## イベント選定理由
- 「きのみを消費した」ことが確定するのは `ItemManager.consume_item` の呼び出し時点であり、
  ここに新規イベントを集約すれば、通常の発動条件付き消費・強制消費（ほおばる/おちゃかい/
  むしくい/ついばむ）を1箇所でカバーできる。
- なげつけるはアイテムシステム（`mon.item`）を経由せず個別に効果を再現しているため、
  該当ハンドラから明示的に発火する必要がある。
- ターン終了時のカウント処理は、しゅうかく等と同じ「素早さ順で発動判定を行う」性質を持つため
  `Event.ON_TURN_END` の priority=150 に合わせる。

## 仕様要点（実装に反映）
- きのみ消費時点でカウントを2にセットする。カウントを持つポケモンの `Event.ON_TURN_END`
  （priority=150）判定のたびに1減算し、0になった時点で同じきのみをもう一度食べる
  （消費した ターン自身の end-of-turn 判定も1回分としてカウントされるため、
  「次のターンの終了時」に発動する。優先度150の判定タイミングを過ぎてから
  消費した場合は、その回の減算に間に合わないため自然に「2ターン後」になる）。
- 再発動時は、HP割合・状態異常・ランクなどの発動条件を無視して強制的に効果を得る
  （`force_trigger_berry` が内部で `Event.ON_HP_CHANGED(value=max_hp)` →
  `Event.ON_FORCE_BERRY_TRIGGER` → 未消費なら明示消費、の順で処理するため、
  そのまま流用すれば発動条件は自然に無視される。半減系きのみ等、いずれの
  イベントにも登録されていないきのみは効果を発動させずに消費だけが行われ、
  「アニメーションは流れるが効果が無い」という仕様を追加コードなしで再現できる）。
- 再発動によって消費された同じきのみが再びはんすうの対象になってはならない
  （Ver.1.2.0の不具合は再現しない）。`ItemManager` に一時停止フラグ
  （例: `suppress_berry_consumed_event: bool`）を持たせ、はんすうの再発動処理中は
  内部の `consume_item` 呼び出しが `Event.ON_BERRY_CONSUMED` を発火しないようにする。
- 交代すると発動しなくなる。`Pokemon.reset_on_switch_out` から呼ばれる
  `Ability.reset_on_switch_out` でカウント・対象きのみ名をクリアすれば、
  次に場に戻ってきても引き継がれない（`Pokemon.memory` のスコープ付き辞書ではなく
  `Ability` インスタンス側に状態を持たせることで、特性そのものが変わった場合にも
  新しい `Ability` インスタンスに切り替わり自然にリセットされる。スキルスワップ等で
  はんすう同士を交換してもカウントが引き継がれない、という仕様とも整合する）。
- かがくへんかガス等による特性無効化中はイベント発火自体が抑止される
  （`_check_handler_validity` が `ability.enabled_ignoring(...)` を見て無効時は
  ハンドラをスキップする既存の仕組みをそのまま使うため、追加の無効化チェックは書かない）。
  無効化が解除されればハンドラが再び呼ばれ、カウントの減算を再開できる
  （＝一時停止・再開の仕様を追加コードなしで満たす）。
- 無効化中にきのみを消費した場合は `Event.ON_BERRY_CONSUMED` のはんすう側ハンドラ自体が
  呼ばれないため、カウントが開始されず「ガスが解除されてもはんすうは発動しない」仕様も
  自然に満たす。

## 実装フロー
1. `src/jpoke/enums/event.py` に `Event.ON_BERRY_CONSUMED` を追加する
   （`ON_FORCE_BERRY_TRIGGER` の直後、emit/handle コメント付き）。
2. `src/jpoke/core/context.py` の `EventContext` に `item_name: ItemName = ""` を追加する。
3. `src/jpoke/core/item_manager.py`
   - `ItemManager.__init__` に `self.suppress_berry_consumed_event: bool = False` を追加する。
   - `consume_item` で `mon.item.is_berry()` のとき、`suppress_berry_consumed_event` が
     False なら `Event.ON_BERRY_CONSUMED` を `EventContext(source=mon, item_name=mon.item.base_name)`
     で emit してから `remove_item` を呼ぶ。
4. `src/jpoke/model/ability.py`
   - `Ability.__init__` に `self.cud_chew_item: ItemName = ""` と
     `self.cud_chew_turns: int = 0` を追加する（`is_hangry` 等と同様、特定特性専用の
     属性として直接持たせる）。
   - `Ability.reset_on_switch_out` で上記2フィールドをリセットする。
5. `src/jpoke/handlers/ability.py`
   - `はんすう_start_counter(battle, ctx: EventContext, value) -> HandlerReturn`:
     `ctx.source.ability.cud_chew_item = ctx.item_name`、`cud_chew_turns = 2` をセットする。
   - `はんすう_on_turn_end(battle, ctx: EventContext, value) -> HandlerReturn`:
     `cud_chew_turns <= 0` なら何もしない。1減算し、0より大きければ終了。
     0になったら `cud_chew_item` を取り出してクリアし、`mon.has_item()` なら
     （既知の制約により）何もしない。そうでなければ特性発動ログを出し、
     `battle.item_manager.suppress_berry_consumed_event = True` の間に
     `gain_item` → `force_trigger_berry(mon, track_loss=False)` を呼んで
     `finally` で必ずフラグを戻す。
6. `src/jpoke/data/ability.py` の `はんすう` に
   `Event.ON_BERRY_CONSUMED`（`subject_spec="source:self"`）と
   `Event.ON_TURN_END`（`subject_spec="source:self"`, `priority=150`）を登録する。
7. `src/jpoke/handlers/move_attack.py`
   - なげつけるの効果が定義されている全きのみ名の集合
     （ラムのみ/クラボのみ/カゴのみ/モモンのみ/チーゴのみ/ナナシのみ/キーのみ/ヒメリのみ/
     オレンのみ/オボンのみ/フィラのみ/ウイのみ/マゴのみ/バンジのみ/イアのみ/チイラのみ/
     リュガのみ/アッキのみ/カムラのみ/ヤタピのみ/ズアのみ/タラプのみ/サンのみ/スターのみ/
     ミクルのみ）を定数として定義し、`なげつける_apply_item_effect` の末尾で
     対象のきのみがこの集合に含まれる場合、`ctx.defender` を source とした
     `Event.ON_BERRY_CONSUMED` を emit する（半減系きのみ等、効果の無いきのみは対象外）。
   - `なげつける_consume_item` で、消費前の使用者のアイテムがきのみ
     （`attacker.item.is_berry()`）なら `remove_item` の前に使用者を source とした
     `Event.ON_BERRY_CONSUMED` を emit する（本リポジトリには努力値を下げるきのみが
     実装されていないため、除外リストは不要）。

## 既知の制約
- はんすう発動タイミングで既に別のアイテムを持っている場合、効果を再現せずスキップする
  （上記スコープ参照）。
- `force_trigger_berry` を経由するため、しゅうかくと同様「復活直後の即時使用」に関する
  制約は引き継がない（はんすうは強制発動イベントを使うため、この点はしゅうかくより忠実）。

## テスト観点
- きのみ消費（自然発動・むしくい/ついばむ・ほおばる/おちゃかい）の次のターン終了時ではなく、
  さらにその次のターン終了時に同じきのみをもう一度消費する。
- HPがしきい値を満たさない状態でオボンのみ等を消費しても、はんすう再発動時は無条件に
  効果を得る。
- 複数のきのみを消費してから発動するまでの間は、最後に消費したきのみのみが対象になる。
- 発動前に交代すると発動しない（場に戻ってきても発動しない）。
- 発動前にかがくへんかガスで無効化されている間はカウントが進まず、解除後に再開する。
- 発動前にきのみを消費した状態でかがくへんかガスが発動すると、そのきのみでは発動しない。
- 特性が書き換わる（スキルスワップ等）とカウントが失われ、はんすうに戻っても発動しない。
- なげつけるで自分のきのみを投げた場合、次のターンに使用者が効果を得る。
- なげつけるを受けてきのみの効果が発動した場合、次のターンに対象が効果を得る
  （半減系きのみ等、効果の無いきのみを受けた場合は対象にならない）。
