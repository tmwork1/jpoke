# 特性実装計画: クイックドロウ

更新日: 2026-06-03

## スコープ

- 優先度: H
- 仕様書: `docs/spec/abilities/クイックドロウ.md`
- 実装状態: 未実装
- 実装方針: あとだしと同種の「行動順ティア補正」方式。優先度は変更せず、`DomainEvent.ON_CALC_SPEED` で攻撃技選択時に 30% の確率で先攻ティア補正を付与する。

## 実装ゴール（完了定義）

- 攻撃技選択時に 30% の確率で「クイックドロウで行動が はやくなった!」ログが出て先制する。
- 変化技では発動しない。
- 優先度そのものは変わらない（優先度+1技に後攻になることはない）。
- トリックルーム状態でも発動する（行動順が早くなる）。
- 既存テストを壊さない。

## 実装対象イベント

- `DomainEvent.ON_CALC_SPEED` — 攻撃技選択時に 30% で先攻ティア補正を付与する（あとだしの後攻補正と対称）

## 変更対象（ファイル/関数単位）

- `src/jpoke/data/ability.py` — `クイックドロウ` に handler を登録する
- `src/jpoke/handlers/ability.py` — `クイックドロウ_calc_speed` を実装する
- `src/jpoke/core/speed.py` — 先攻ティア補正の拡張が必要な場合
- `tests/test_ability.py` — テストを追加する

## 実装ステップ

1. 仕様書 `docs/spec/abilities/クイックドロウ.md` で攻撃技判定・トリックルームとの相互作用を確認する。
2. `src/jpoke/core/speed.py` の行動順ティア実装（あとだしの後攻ティア）を確認し、先攻ティアが追加済みか確認する。
3. `src/jpoke/handlers/ability.py` に `クイックドロウ_calc_speed` を実装する（`battle.rng.check(0.3)` で 30% 判定、攻撃技のみ対象）。
4. `src/jpoke/data/ability.py` の `クイックドロウ` に handler を登録する。
5. テストを追加し、全テストが通ることを確認する。

## テスト観点

1. 攻撃技使用時に 30% の確率で先制する（確率テストは乱数固定で確認）。
2. 変化技選択時は発動しない。
3. 相手が優先度+1技を使用した場合はクイックドロウ発動でも後攻になる（優先度帯が異なるため）。
4. トリックルーム状態でも先制する。
