# 特性実装計画: スロースタート

更新日: 2026-07-11

## スコープ

- 仕様書: `docs/spec/abilities/スロースタート.md`
- 実装状態: 実装済み（レビューで すばやさ半減ハンドラの登録漏れ・
  ON_ABILITY_ENABLED 未登録を修正）
- 実装方針: `mon.ability.count` を「場に出てからの経過ターン数」として使い、
  count < 5 の間はこうげき・すばやさの両方を0.5倍にする

## 実装ゴール（完了定義）

- 場に出てから5ターンの間（場に出たターンを1ターン目として含む）、
  自身のこうげきとすばやさが1/2になる
- 5ターン経過後は補正が外れる（`調子を 取り戻した!` に相当する処理として、
  6ターン目の ON_TURN_END でカウント==5に到達したことを検知する）
- 交代して場を離れると、次に場に出たときカウントがリセットされる
  （`Ability.reset_on_switch_out` により count=0 に戻り、加えて次回
  `ON_SWITCH_IN` で明示的に0を再設定する）
- 特性を変えるわざでスロースタートを新規に得た場合、そのターンから
  カウントを開始する（`ON_ABILITY_ENABLED` で count=0 にリセット）
- かがくへんかガスなどで無効化されている間はカウントが進まず
  （`ON_TURN_END` ハンドラ自体が発火しないため）、無効化が解除されたとき
  （`ON_ABILITY_ENABLED`）はカウントが0にリセットされる
  （消費していたカウントは引き継がれない）
- 死に出しで場に出た場合、そのターンの ON_TURN_END は既に発火済みのため
  カウントされない（`TurnController._run_end_phase` で `ON_TURN_END` の後に
  `run_faint_switch()` を呼ぶ既存のフェーズ順序により自動的に満たされる。
  個別実装は不要）
- 特性がスロースタート以外に変わった場合、下がっていたステータスは
  即座に元に戻る（`ability_manager.change_ability` がハンドラを
  unregister するため、個別実装は不要。「調子を 取り戻した!」の
  メッセージも出ない）

## 実装対象イベント

- `Event.ON_SWITCH_IN` — 場に出たときにカウントを0にリセットし、
  「調子が あがらない！」を通知する（`docs/spec/turn.md` に個別の
  記載はないが、既存実装の `スロースタート_start` を踏襲）
- `Event.ON_ABILITY_ENABLED` — 特性を変えるわざやかがくへんかガス解除で
  スロースタートが（再度）有効化された瞬間にカウントを0にリセットする。
  `turn.md` に未掲載のイベントだが、`サイコメイカー` 等と同様に
  `ON_SWITCH_IN` と同じ関数を再利用する（`docs/plan/abilities/サイコメイカー.md`
  参照）
- `DomainEvent.ON_CALC_SPEED` — 実効すばやさ計算時に count < 5 なら
  0.5倍にする。ダメージ計算外のドメインイベントであり `turn.md` には
  掲載されないため、同種の `くろいてっきゅう`（`ItemData`）・`まひ`
  （`AilmentData`）の `ON_CALC_SPEED` ハンドラを参考にする。優先度は
  指定せず、他の `ON_CALC_SPEED` ハンドラと同様にデフォルトを使う
  （ドメインイベントは登録順に実行され priority ソートされないため
  問題ない、`EventManager.emit` 参照）
- `Event.ON_CALC_ATK_MODIFIER` — ダメージ計算内の物理攻撃補正。count < 5
  かつ技分類が物理のとき 0.5倍にする（`core/damage.py` 参照、`turn.md`
  未掲載のためダメージ計算内部イベントの慣例に従う）
- `Event.ON_TURN_END` — 経過ターンをカウントする。priority=150
  （`docs/spec/turn.md` の「ターンの終了時に発動するとくせい」の
  優先度帯を踏襲、既存実装のまま変更なし）

## テスト観点

1. 場に出てから5ターンの間、実効すばやさが半分になる
   （`battle.speed_calculator.calc_effective_speed(mon) == mon.stats["spe"] // 2`）。
2. 6ターン目には実効すばやさが元に戻る。
3. こうげきについては既存テストの通り、5ターン未満で0.5倍・以降は等倍
   （`test_スロースタート_登場5ターン未満は攻撃補正0_5倍`）。
4. とくこうには補正がかからない（既存テストの通り）。

## リスクと対策

- リスク: `DomainEvent.ON_CALC_SPEED` はソートされず登録順に実行されるため、
  まひ（`AilmentData`、状態異常適用時に動的登録）との適用順が
  仕様書の記載（第五世代以降はまひをスロースタート等の後に計算する）と
  一致するか要確認。
  - 対策: 両者とも「値を半分にする」操作であり、`floor(floor(x/2)/2)` は
    適用順によらず同じ結果になるため、実害はない。
- リスク: 能力ランクによるすばやさ変動が `DomainEvent.ON_CALC_SPEED` の
  基準値（`mon.stats["spe"]`、ランク未反映）に反映されない可能性がある
  （`SpeedCalculator.calc_effective_speed` はランク補正前の実数値を
  ハンドラチェーンに渡している）。
  - これはスロースタート固有の問題ではなく、すばやさ計算システム全体に
    影響する既存の欠陥であり、本レビューのスコープ外とする
    （別途フローで対応が必要）。
