# 検討記録: examples/07_janken_nash_fictitious_play.py のNash均衡近似における代替手法

更新日: 2026-07-12

## 対象

`examples/07_janken_nash_fictitious_play.py`（旧 `scripts/janken_nash.py`）が採用している、
エビワラーの3すくみ技（マッハパンチ／はやてがえし／きあいパンチ）の混合戦略Nash均衡を
モンテカルロ推定 + 反復最適反応（fictitious play）で近似する手法について、他に
Nash均衡を求める方法がないか検討した記録。

## 結論

**`examples/07_janken_nash_fictitious_play.py` のシミュレーションベースの手法
（`simplex_grid()` による格子探索 + `approximate_nash()` の fictitious play）は
そのまま維持する。** 実行コストが大きい点は冒頭の実行設定（`GRID_STEP` /
`EVAL_TRIALS` / `ITERATIONS`）で調整可能にする形で対処済み（既定値は実行時間を
抑えた粗い近似、精度優先の推奨値をdocstringに明記）。

一方、案4（CFR）で指摘した「毎ターン同じ確率で技を選び続ける固定戦略しか
求められない」という07の限界には教材的価値があると判断し、これは代替ではなく
**追加のサンプル `examples/08_janken_nash_cfr.py` として実装した**（HP状況に
応じて確率を変える適応的な戦略を、`Battle.copy()` によるロールアウトベースの
regret matchingで学習する）。詳細は案4を参照。

## 検討した代替案

### 案1: 3×3利得行列 + 線形計画法（LP）で厳密に解く

このゲームの純戦略はマッハパンチ／はやてがえし／きあいパンチの3つだけなので、
以下の手順で高速に解ける可能性がある。

1. 3×3の純戦略同士（計9通り）の対戦をモンテカルロ推定し、利得行列を1つ作る
2. その利得行列をゼロサム行列ゲームとみなし、`scipy.optimize.linprog` や
   `nashpy` のようなライブラリで混合戦略Nash均衡を厳密かつ瞬時に解く

**採用しなかった理由:** この方法が正しく機能するのは「毎ターンの技選択が独立で、
1ターン分の利得行列を確率で線形結合すれば複数ターンの対戦の期待値になる」
（= 価値関数が (p, q) について双線形である）場合に限られる。しかし
`examples/07_janken_nash_fictitious_play.py` のゲームは、選んだ技によって受けたダメージが
次ターン以降のHP状況（ひいては行動選択の有利不利）に影響する多ターンの
逐次ゲームであり、価値関数が (p, q) について非線形になりうる。そのため、
純戦略同士の利得行列だけから求めた均衡は、実際の多ターン戦での均衡の
近似に過ぎず、両者が一致する保証がない。現状のようにあらゆる混合戦略の
組を実際に多ターンシミュレートする方式の方が、この非線形性を正しく
扱える。

ただし、時間計算量の都合で「まず高速なLP解を初期値・当たりをつける
目的で使い、そこから重い方のシミュレーションベース探索を絞り込む」
という組み合わせ方（ウォームスタート）は今後の高速化案として有効。

### 案2: regret matching / Hedge（乗法的重み更新）

fictitious play は「毎反復で候補全件に対する最適反応を格子探索で求める」ため、
1反復あたりのコストが `候補数 × trials` に比例して大きい。regret matching /
Hedge アルゴリズムは、各手の「後悔（実際に選んでいたらどれだけ得だったか）」を
累積し、その累積後悔に応じた確率で次の手を選ぶという更新を1ゲームごとに行う
方式で、格子探索が不要なため反復あたりのコストが小さい。ゼロサムゲームでは
両者の平均戦略がNash均衡に収束することが知られており、fictitious playと
同等の理論的裏付けを持つ。

**採用しなかった理由:** 現状の実装（grid + fictitious play）で十分な教材的
価値があり、regret matchingへの置き換えは既存コードの構造を大きく変える
（`simplex_grid()` による候補列挙という中核の設計を捨てることになる）ため、
「examplesへの昇格」というタスクの範囲を超える。将来、実行速度や近似精度が
問題になった場合の改善案として記録しておく。

### 案3: Double Oracle

候補戦略を最初から格子で全列挙するのではなく、少数の候補（例: 3つの純戦略）
から始め、「現在の候補プールに対するLPでの均衡」→「その均衡に対する全体
（連続空間）での最適反応を求めて候補プールに追加」を繰り返す方法。
`GRID_STEP` を細かくするほど候補数が組合せ的に増える格子探索と比べ、
候補プールが必要な分だけ緩やかに増えるため理論上は効率が良い。

**採用しなかった理由:** 案2と同様、実装の中核を置き換える規模の変更になる
ため見送り。3手番のみの小さいゲームでは格子探索でも実用的な時間で近似
できており、優先度は低い。

### 案4: CFR（Counterfactual Regret Minimization）— 採用（examples/08として追加）

案1〜3はいずれも「毎ターン同じ確率で技を選び続ける」という07の定式化
（stationary mixed strategy）を前提にした改善案だった。CFRは本来、
逐次的な意思決定点（information set）ごとに後悔を蓄積するポーカー等の
imperfect-information extensive-form game向けの手法で、**状態（HP状況など）に
応じて確率を変える適応的な戦略**まで含めてNash均衡（に収束する平均戦略）を
求められる。「マッハパンチ/はやてがえし/きあいパンチをどんな比率で混ぜるか」
という固定戦略の探索よりも一段階野心的で、実戦の読み合いに近い。

**採用した理由と実装方針:** 07が「07だけでは学べない発展的な観点」を
提供できると判断し、置き換えではなく `examples/08_janken_nash_cfr.py` として
追加した。ただし厳密な教科書通りのCFR（ゲーム木を全展開し、到達確率で
重み付けした反実仮想価値を使うoutcome-sampling MCCFR等）は、本物のダメージ
乱数・急所判定を含む多ターンのバトルシミュレーションに対してそのまま
適用するには実装難度・計算量ともに大きすぎる。そこで以下の簡略化を行った:

* 状態（information set）を `(自分のHP割合バケット, 相手のHP割合バケット)`
  （既定3段階ずつ）に離散化する。ターン数は状態に含めていない
* 反実仮想価値は、`TreeSearchPlayer` と同じ `Battle.copy()` の仕組みで
  盤面をフォークし、「もし別の手を選んでいたら」を実際にロールアウト
  （現在の戦略テーブルで決着まで進める）して推定する近似手法とした
  （詳細は `examples/08_janken_nash_cfr.py` のモジュールdocstring参照）

この簡略化により、07と同様に「実装が理解しやすい」水準を保ちつつ、
状態依存の戦略という07にはない教材的価値を追加できた。実行時間は
`Battle.copy()` を伴うぶん07より重く（既定設定で数十秒程度）、
`N_EPISODES` / `HP_BUCKETS` で調整可能にしている。

## 今後について

`examples/07_janken_nash_fictitious_play.py`（固定混合戦略）と `examples/08_janken_nash_cfr.py`
（HP状況に応じた適応的戦略）は、どちらも教材用途のサンプルとして
「実装が理解しやすく、理論的裏付けもある」手法をそのまま維持する。
より本格的な研究用途（技構成が増える、状態にターン数も加える、精度を
大きく上げたい等）で計算コストが問題になった場合は、案1（LPによる
ウォームスタート）・案2/3（07の反復あたりのコスト削減）・より厳密な
outcome-sampling MCCFRへの発展（08の反実仮想価値推定の精緻化）を
個別の改善タスクとして検討すること。
