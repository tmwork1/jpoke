## 2026-07-15 再検討の経緯

初回調査（本ドキュメント旧版）の後、ps-champ-ja 側で以下の整備が行われた（`main` の直近コミット、
`008c4c7b`〜`a27a283e`）。再ダウンロードして差分を確認し、結論を更新する。

- **`jpoke/` への依存を撤去**（`efea3a80`）: 従来は本プロジェクト自身の `pokedex.json` を
  ps-champ-ja のパイプラインに取り込んで和名を補完していたが、この循環参照をやめ、
  `langmap/`（poke-langmap）のみを和名の出典とするよう変更された。ライセンス上の整理としても
  望ましい変更。
- **スキーマを対戦必要データのみに刷新**（`008c4c7b`）: `pokedex.json` の収録数が
  1339件→**1288件**に減少（コスメティックのみのフォルム等を整理）。jpoke 実装数1286件との
  差が縮小した。
- **基本フォルムの和名サフィックス統一**（`c1feb4b5`, `a27a283e`）: `FORM_RULES` による
  フォルム和名解決を `langmap/_form_langmap.csv` に一本化し、対応済みの種についてはjpokeと
  同じ `種族名(フォルム名)` 形式に統一された（詳細は [missing_forms.md](missing_forms.md)）。
- **`showdown_id` / `showdown_name` / `num` / `forme` / `baseForme` を全件に付与**: 旧版では
  `pokedex.json` は和名キーのみのフラットな辞書だったが、現在は各エントリが Pokemon Showdown
  側の安定した識別子一式を保持している。これは**名前の表記ゆれに依存しない突合キーとして
  使える**、今回の再検討で最も重要な変化。
- ライセンスを「コード(MIT) / データ(CC BY-NC-SA 4.0)」に分離（`b43219a6`, `25b780fb`）。
  jpoke の `LICENSE-DATA` と同じ枠組みで、`jpoke/` 依存撤去によりデータの出所も
  ポケモンWiki(poke-langmap経由)＋Pokemon Showdown champions mod のみとシンプルになった。

## ps-champ-ja の構成（更新後）

1. `data_raw/` `data_en/`: [Pokemon Showdown](https://github.com/smogon/pokemon-showdown) の
   `champions` mod から抽出（英語・showdown ID ベース）
2. `langmap/*.csv`: [poke-langmap](https://github.com/tmwork1/poke-langmap)（ポケモンWiki由来の
   和名対応表）をビルド時にダウンロード（リポジトリにはコミットされない）
3. 上記を `scripts/translate-data.py` で合成し `data_jp/` を生成（**jpoke自身のデータへの
   依存は撤去済み**）

| ファイル | 内容 |
|---|---|
| `pokedex.json` | 種族データ。和名キー＋`showdown_id`/`showdown_name`/`num`/`forme`/`baseForme` |
| `pokedex_excluded.json` | 和名重複・コスメティックフォルム等で除外されたエントリ一覧 |
| `moves.json` | 技の静的パラメータ。和名キー＋`showdown_id`/`showdown_name`/`num` |
| `learnsets.json` | 種族ごとの習得技リスト（技名のみ） |

**abilities.json・items.json は依然として存在しない。** 特性・アイテムは対象外のまま。

## ライセンス: 問題なし（むしろ改善）

`data_raw/` `data_en/` `data_jp/` は CC BY-NC-SA 4.0、`scripts/` は MIT。jpoke 自身のデータへの
依存が無くなったため、取り込み時に注意すべき循環参照の懸念も解消された。実際に取り込む場合は
`LICENSE-DATA` に ps-champ-ja / Pokemon Showdown champions mod への言及を追記する。

## 再検証結果

### 1. 技データ: カバー率は変わらず約70%（500/716）

champions mod のスコープ自体は変わっていないため、moves.json は前回調査と同じ500件のまま。
jpoke実装済み716技のうち500件がカバーされ、jpokeにあってps-champ-jaに無い技が216件。
**結論は変わらず**：技データの単独基盤にはできず、既存実装済み技の**値の照合・差分検出用途**
として使う。

パワーシフト（ps-champ-jaにあってjpoke未実装の1件）は引き続き `docs/champions/moves.md` に
未掲載で公式仕様として実在するか未確認（`docs/progress/move.md` 参照）。

[ユーザー方針・維持]
- moves.jsonからmove.pyに値を転写するスクリプトを作成する
- MoveDataに exist=(ps-champ-jaに存在するかどうか) という変数を追加する
- flagsは現在の実装を維持する

### 2. ポケモンデータ: 名前ベースの不一致が56件→21件に半減以上

直接の文字列一致で再検証すると、jpoke実装済み1286件中 **21件**が ps-champ-ja の和名キーと
一致しない（初回56件→31件→26件→23件→本更新で21件）。内訳:

| 分類 | 件数 |
|---|---|
| 真の欠落（jpoke独自メガシンカ9体） | 9 |
| 構造突合（num+baseStats+types）で一意に解決 | 6 |
| 構造突合だけでは決まらない（abilities/forme併用 or 本質的に1対多） | 6 |
| 合計 | 21 |

9体のメガシンカ（メガグソクムシャ・メガシャリタツ・メガジガルデ・メガセグレイブ・
メガゼラオラ・メガダークライ・メガニャオニクス・メガヒードラン・メガマギアナ）は
PS champions mod 自体に存在しないため、ps-champ-jaが今後どれだけ整備されても解消しない
**恒久的な欠落**。取り込みは追加専用マージ（jpoke側にしか無いエントリは保持）で設計する。

残り12件（コレクレー各フォルム・タイプ：ヌル・テラパゴス(ノーマル)・ヒヒダルマ(ノーマル)・
ポリゴン２・ウッウ各フォルム・ウーラオス(キョダイ)・ケルディオ(いつも)・
ストリンダー(キョダイ)・モルペコ）は、`FORM_RULES` がまだ対応していない種の基本フォルム
無印名・全角/半角ゆれ・状態違いで、**ps-champ-ja側に対応するエントリ自体は存在する**
（詳細は [missing_forms.md](missing_forms.md)）。

### 3. 重要な方針転換: 文字列一致ではなく構造キーで突合する

初回調査は「jpokeとps-champ-jaの和名表記を完全一致させる」ことを前提に、ps-champ-ja側の
`FORM_RULES` 拡充を待つ方針だった。しかし `num`（図鑑番号）・`forme`・`baseForme`・
`showdown_id` が全件に付与されたことで、**文字列一致に依存しない突合が可能**と判明した。
実際、2〜4回目の再確認では上流の `FORM_RULES` 拡充（`5e0d3db2`, `5d4529e3`, `b3f45df5`）
により計10件が文字列完全一致に昇格したが、それを待たずとも構造キーで先に解決できていたことが
確認できている。

残り12件について `num` + `baseStats`（H/A/B/C/D/S）+ `types` で自動突合を試したところ
（`pokedex.json` 本体だけでなく `pokedex_excluded.json` も候補プールに含める必要がある。
コレクレー(とほ)＝`gimmighoulroaming` のように、ps-champ-ja内部の和名重複回避のために
除外ファイル側へ追いやられているだけで実データは存在するケースがあるため）：

- **6件は一意に対応先が確定**（コレクレー各フォルム・タイプ：ヌル・テラパゴス(ノーマル)・
  ヒヒダルマ(ノーマル)・ポリゴン２）
- **残り6件は種族値・タイプだけでは一意に決まらない**（ウッウのうのみ/まるのみ、
  ウーラオス(キョダイ)・ストリンダー(キョダイ)のダイマックスフォルム、ケルディオ(いつも)、
  モルペコ）。これらは種族値・タイプが同一で `abilities` 配列（例: ストリンダーはハイ/ローで
  発動特性が異なる）や `forme` 文字列（`Gmax`等）まで見ないと確定できない。中にはウッウ・
  モルペコのように、jpoke側では区別しない見た目違い/状態違いを ps-champ-ja 側が複数の
  showdown forme として持つため、本質的に1対多になるケースも含まれる（マージ上は許容できる）。

→ **推奨方針を変更する**: ps-champ-ja側の `FORM_RULES` 拡充を待ち続ける（初回調査時の方針）
のではなく、jpoke側で「jpoke内部キー（`0741-000`等）↔ ps-champ-ja `showdown_id`」の**対応表を
一度だけ構築し、リポジトリにチェックインする**。構築は `num` + `baseStats` + `types` の
構造的突合で大半（残り12件中6件）を自動化し、自動で決着しない残り6件のみ `abilities`/`forme`
も併用した個別確認・手動補完で埋める。この対応表さえあれば、以降の再取り込み・照合は
ps-champ-ja側の和名表記の変化（表記ゆれ修正・リネーム等）に影響されず安定する。

### 4. スキーマの差異（変更なし）

- pokedex: `types` 配列 vs `type-1`/`type-2`、`baseStats` ネスト辞書 vs `H/A/B/C/D/S` フラット、
  `prevo`/`evos` vs `pre_evolution` のみ（`evos`・`genderRatio`・`requiredItem` は jpoke の
  `PokemonData` に対応するフィールドが無い）
- moves: 静的パラメータのみで、`src/jpoke/data/moves/*.py` が持つハンドラ登録は当然含まれない

→ 変換スクリプトは引き続き必須。ただし突合キーが安定したことで変換スクリプトの設計難易度は
下がった（名寄せロジックが不要になり、対応表を引くだけで済む）。

### 5. データ鮮度・継続同期

`pushed_at` は調査当日（2026-07-15）と非常に新しい。再ダウンロード時の差分検知・
バージョン管理の仕組みは依然ない。対応表方式にすることで、ps-champ-ja側の和名リネームが
今後起きても対応表の再構築（またはピンポイント修正）だけで追従でき、影響範囲が局所化される。

## 推奨スコープ（更新）

1. **pokedex（種族データ）**: jpoke内部キー ↔ ps-champ-ja `showdown_id` の対応表を
   `num`+`baseStats`構造突合で構築し、jpoke独自エントリ（カスタムメガ9体等）は対応表に
   含めず素通し（追加専用マージ）として実装する。対応表は再利用可能なデータファイルとして
   チェックインする
2. **moves**: 差し替えではなく、既存716技の power/PP/accuracy 等の**照合・差分検出ツール**
   として利用（変更なし）
3. **abilities/items**: 対象外。既存のポケモンWikiベースのパイプライン（`scripts/wiki/`）を
   継続使用（変更なし）
