# トレース 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/トレース.md`
- 実装方針: 入場時に相手の有効特性からコピー先を決定し、特性差し替えを共通処理化する。交代時に元の `トレース` へ確実に戻す。

## 実装対象イベント（第一候補）
- `Event.ON_SWITCH_IN`
- `Event.ON_SWITCH_OUT`

## 仕様要点（実装に反映）
- 入場時に、相手アクティブのコピー可能な特性を候補化し、候補が複数ならランダムに1つ選ぶ。
- 候補が存在しない場合は不発（特性変更なし）。
- `かがくへんかガス` などで `トレース` 自体が無効な場合は発動しない。
- 交代でベンチに戻る際、現在の特性が何であっても元の `トレース` に戻す。
- コピー不可判定は特性フラグ `uncopyable` を基準に統一する。

## 変更対象（想定）
- `src/jpoke/data/ability.py`
  - `トレース` に `Event.ON_SWITCH_IN` / `Event.ON_SWITCH_OUT` の `handlers` を登録。
- `src/jpoke/handlers/ability.py`
  - `トレース_on_switch_in`: 候補抽出、ランダム選択、特性差し替え、発動ログ出力。
  - `トレース_on_switch_out`: 元特性へ復元。
- `src/jpoke/core/ability_manager.py`
  - 特性差し替え時にハンドラ登録状態を壊さないための共通更新API（例: `set_ability`）を追加。
- `src/jpoke/core/battle.py`
  - `AbilityManager` の特性差し替えAPIを委譲公開。
- `src/jpoke/model/pokemon.py`
  - 交代時復元のため、選出時の元特性名を保持する属性（例: `base_ability_name`）を追加。

## 実装フロー
1. `Pokemon` に「元特性名」を保持する属性を追加し、初期化時に設定する。
2. `AbilityManager` に特性差し替えAPIを追加し、以下を一括で処理する。
   - 旧特性ハンドラ解除
   - 特性オブジェクト更新
   - 必要時の新特性ハンドラ再登録
   - 有効/無効状態再計算
3. `ability.py` データ定義で `トレース` に入場/退場イベントを登録する。
4. `handlers/ability.py` に `トレース` の本体処理を実装する。
5. 発動ログは `ABILITY_TRIGGERED` を使い、成功時のみ記録する。

## テスト観点
- 発動: 相手1体のコピー可能特性を正しくコピーする。
- 発動: いかくをコピーしたときに直ちにいかくが発動する。
- 非発動: `uncopyable` 特性のみが相手にいる場合は不発。
- 非発動: `かがくへんかガス` 下で `トレース` が無効化されている場合は不発。
- 境界: コピー後に交代すると `トレース` に戻る。
- 境界: 再入場時に再度コピー判定が走る。

## 実装前確認事項
- コピー後に得た特性の「入場時効果」を即時発火させるかどうかを統一方針として明記する。
- 交代以外の特性変更（将来のスキルスワップ実装等）と競合しないよう、特性差し替えAPIを単一経路に固定する。
