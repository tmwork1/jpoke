# マイティチェンジ 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/マイティチェンジ.md`
- 関連仕様: `spec/turn.md`
- 実装方針: 交代成立時の `ON_SWITCH_OUT` でフォルム更新を確定する。

## 実装ゴール（完了定義）
- イルカマン(ナイーブ) が生存したまま引っ込むと、待機中に イルカマン(マイティ) へ変化する。
- ひんし退場時は変化しない。
- 既に イルカマン(マイティ) の場合は追加変化しない。
- 通常交代・交代技・だっしゅつ系割り込み交代のいずれでも、引っ込み成立時に同じ判定で変化する。
- 次回登場時にマイティフォルムが反映される。

## 現状整理
- `src/jpoke/data/ability.py` の `マイティチェンジ` は `flags` のみで、イベントハンドラ未登録。
- `src/jpoke/model/pokemon.py` には `set_form(alias, keep_damage=True)` が実装済みで、フォルム更新 API は流用可能。
- `src/jpoke/core/switch.py` の `run_switch()` は退場時に必ず `Event.ON_SWITCH_OUT` を emit し、その後で `bench_reset()` を呼ぶ。
- 同実装では瀕死交代でも `ON_SWITCH_OUT` が発火するため、マイティチェンジ側で `alive` 判定が必要。
- `tests/test_ability.py` にマイティチェンジ専用テストは未作成。

## 仕様反映ポイント
- 発動イベントは `Event.ON_SWITCH_OUT` を採用する。
- 判定条件は以下に固定する。
  - 使用者が `マイティチェンジ` を所持し、特性が有効。
  - 使用者がイルカマン。
  - 現在フォルムが `イルカマン(ナイーブ)`。
  - 使用者が生存している（`alive` が `True`）。
- 非発動条件は仕様書どおりに固定する。
  - ひんし退場。
  - 既に `イルカマン(マイティ)`。
- 交代契機は `SwitchManager.run_switch` 経由なら共通化されるため、追加の分岐実装は作らない。

## 変更対象（ファイル単位）
- `src/jpoke/handlers/ability.py`
  - マイティチェンジ本体ハンドラ（`ON_SWITCH_OUT`）を追加する。
  - 発動ログは `LogCode.ABILITY_TRIGGERED` を利用する。
- `src/jpoke/data/ability.py`
  - `マイティチェンジ` に `handlers` を登録する。
- `tests/test_ability.py`
  - マイティチェンジ専用テストを追加する。

## 実装ステップ
1. マイティチェンジハンドラ実装
- `ON_SWITCH_OUT` で対象・フォルム・生存を判定し、該当時のみ `set_form("イルカマン(マイティ)")` を実行。
- 発動時に能力発動ログを記録する。
2. Ability データ登録
- `マイティチェンジ` にイベントハンドラを登録する。
3. テスト追加と回帰確認
- 発動・非発動・交代契機差分の回帰を固定する。

## テスト計画
- 発動ケース: イルカマン(ナイーブ) が通常交代で引っ込むと、控えで イルカマン(マイティ) へ変化する。
- 発動ケース: だっしゅつボタン等の割り込み交代でも同様に変化する。
- 非発動ケース: ひんし退場時は変化しない。
- 非発動ケース: 既に イルカマン(マイティ) の個体は再変化しない。
- 維持ケース: マイティ化後の再登場でフォルムが維持される。
- ログケース: 発動時のみ `ABILITY_TRIGGERED` が記録される。

## リスクと対策
- リスク: ひんし交代でも `ON_SWITCH_OUT` が発火するため、誤って変化する。
  - 対策: `alive` 判定を必須条件にし、専用テストで固定する。
- リスク: フォルム alias の直書きが散在し、表記ゆれで切替に失敗する。
  - 対策: イルカマン用 alias をハンドラ内定数化し、テストでも同名を利用する。
