# マジックミラー 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/マジックミラー.md`
- 実装方針: 既存の `Event.ON_CHECK_REFLECT` 経路へ特性ハンドラを追加し、反射対象の変化技をマジックコートと同一フェーズで跳ね返す。

## 実装対象イベント（第一候補）
- `Event.ON_CHECK_REFLECT`

## 仕様要点（実装に反映）
- 相手の反射対象変化技（`reflectable` ラベル付き）を無効化し、使用者側へ跳ね返す。
- 攻撃技本体や反射対象外の変化技は反射しない。
- `かたやぶり` / `ターボブレイズ` / `テラボルテージ` により防御側特性が無効化される場合は反射しない。
- `かがくへんかガス` 等で `マジックミラー` が無効化されている間は反射しない。
- 反射判定は1回で完結させ、無限反射を起こさない（既存 `ON_CHECK_REFLECT` の bool 返却モデルを維持）。

## 変更対象（想定）
- `src/jpoke/data/ability.py`
  - `マジックミラー` に `Event.ON_CHECK_REFLECT` ハンドラを登録する。
  - `subject_spec="defender:self"`、`priority=200` を基本案とする（マジックコートと同等優先度）。
- `src/jpoke/handlers/ability.py`
  - `マジックミラー_reflect`（仮名）を追加する。
  - 判定順は以下を基本とする。
    1. `ctx.check_def_ability_enabled(battle)` で防御側特性の有効性を確認。
    2. `ctx.move.has_label("reflectable")` を確認。
    3. 条件を満たす場合 `HandlerReturn(value=True)` を返す。
- `src/jpoke/enums/event.py`（必要時のみ）
  - `ON_CHECK_REFLECT` コメントの handle 先に ability 側の利用を追記する。

## 実装フロー
1. `handlers/ability.py` に反射判定ハンドラを追加する（防御側特性有効性チェック込み）。
2. `data/ability.py` の `マジックミラー` 定義へ `ON_CHECK_REFLECT` 登録を追加する。
3. `Event.ON_CHECK_REFLECT` に対する既存の `マジックコート` と競合しないことを確認する。
4. `tests/test_ability.py` に発動・非発動・無効化系の回帰テストを追加する。
5. 必要に応じて `event.py` コメントを更新し、イベント利用先のドキュメントを同期する。

## テスト観点
- 発動: 相手の `reflectable` 変化技（例: なきごえ）を跳ね返す。
- 発動: 反射後、元使用者側が効果を受ける。
- 非発動: 反射対象外変化技は通常どおり通る。
- 非発動: 攻撃技は反射しない。
- 無効化: 攻撃側が `かたやぶり` 系なら反射しない。
- 無効化: `かがくへんかガス` 下では反射しない。
- 境界: `マジックコート` 併用時でも処理が1回で完了し、反射ループしない。

## 実装前確認事項
- 反射成功時のログ出力方針（`ABILITY_TRIGGERED` を追加するか、既存ログだけで運用するか）を統一する。
- 反射対象技セットを `reflectable` ラベルに一本化し、個別技名判定を増やさない。
