# はらぺこスイッチ 実装計画

## スコープ
- 優先度: M
- 仕様書: `spec/ability/はらぺこスイッチ.md`
- 実装方針: 「フォルム状態管理」と「技タイプ変更」を分離して段階実装し、最後にターン終了順序と例外条件を統合検証する。

## 現状ギャップ
- `src/jpoke/data/ability.py` の `はらぺこスイッチ` は `flags` のみで、`handlers` 未登録。
- `src/jpoke/data/zukan.json` はモルペコを `0877-000` (まんぷくもよう) だけで保持しており、はらぺこもようの状態を保持するランタイム表現が未定義。
- `オーラぐるま` は `src/jpoke/data/move.py` で固定でんきタイプ定義。

## 実装対象イベント
- `Event.ON_SWITCH_OUT`
- `Event.ON_TURN_END_5`
- `Event.ON_MODIFY_MOVE_TYPE`

## 事前整備タスク
1. モルペコのフォルム状態は `Ability` インスタンス内の軽量プロパティ（例: `is_hangry`）で管理する。
2. `src/jpoke/handlers/ability.py` に状態更新ヘルパーを用意し、以下を一か所で扱う。
	- 現在フォルムの切替
	- 交代時の通常リセット（まんぷくもようへ戻す）
3. テラスタル中はフォルム固定にするガード条件をヘルパーに集約する。

## 実装フロー
1. `src/jpoke/data/ability.py` の `はらぺこスイッチ` に以下の `handlers` を登録する。
	 - `ON_SWITCH_OUT`: 交代時フォルム変更処理（通常はまんぷく化、テラスタル時は維持）
	 - `ON_TURN_END_5`: ターン終了時の交互切替
	 - `ON_MODIFY_MOVE_TYPE`: オーラぐるまをフォルムに応じて でんき/あく へ変換
2. `src/jpoke/handlers/ability.py` に `はらぺこスイッチ` 用ハンドラ群を追加する。
3. `src/jpoke/handlers/ability.py` にフォルム状態更新ヘルパーを置き、`はらぺこスイッチ` 用ハンドラから直接呼び出す。
4. `src/jpoke/core/move_executor.py` の `ON_MODIFY_MOVE_TYPE` 経路で、オーラぐるまの変換が反映されることを確認する。

## 順序・優先度設計
- ターン終了処理は `Event.ON_TURN_END_5` を利用し、`ON_TURN_END_1..4`（スリップダメージ・状態処理・場の解除）より後に実行する。
- モルペコ同士の順序は、現行のイベントソート（実効素早さ基準）で満たせる範囲と、トリックルーム考慮の不足があれば別タスク化して解消する。

## 仕様反映チェックポイント
- 特性無効化（`かがくへんかガス` / `とくせいなし`）中は切替停止。
- テラスタル中は切替停止。
- へんしん個体ははらぺこスイッチ対象外。
- ターン終了時の切替と交代時のフォルム変更のみを実装し、交代直後の即時はらぺこ化は行わない。

## テスト観点
- `tests/test_ability.py`
	- ターン終了時にのみ はらぺこもよう/まんぷくもよう が交互に切り替わる。
	- 交代時にフォルム変更（通常はまんぷく化、テラスタル時は維持）が正しく行われる。
	- 特性無効化中はターン終了でも遷移しない。
	- テラスタル中はターン終了でも遷移しない。
- `tests/test_move.py` または `tests/test_ability.py`
	- オーラぐるまがフォルムに応じて でんき/あく へ変化する。
	- へんしん個体のオーラぐるまタイプは仕様どおりに扱う。

- 新規テストが追加され、関連テストが全て通過する。
- 仕様書 (`spec/ability/はらぺこスイッチ.md`) の必須項目に対応する実装・テストの突合が取れている。
