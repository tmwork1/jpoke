# どろぼう 実装計画書

## 仕様要約（docs/spec/moves/どろぼう.md 参照）

* タイプ: あく / 分類: 物理 / 威力: 60 / 命中: 100 / PP: 25
* 対象: 相手1体 / 優先度: 0 / 接触あり
* 効果: 自分が道具を持っていない場合、命中時に相手の持ち物を奪って自分の持ち物にする。

## 現状の実装状況

`src/jpoke/data/move.py` に既に登録済み。

```python
"どろぼう": MoveData(
    type="あく",
    category="physical",
    pp=25,
    power=60,
    accuracy=100,
    flags={"contact"},
    handlers={
        # docs\spec\turn.md ON_DAMAGE: 「100 はたきおとす等のアイテム効果」
        # くっつきバリの転移判定（priority=30）より後に発動する必要があるため ON_DAMAGE_HIT を使用する。
        Event.ON_DAMAGE_HIT: h.MoveHandler(
            ha.どろぼう_steal_item,
        )
    }
),
```

技「ほしがる」も同一のハンドラ（`ha.どろぼう_steal_item`）を共有しており、これは仕様上正しい
（威力・命中・PP以外は完全に同じ追加効果を持つため）。

## ハンドラ構成

| Event | subject_spec | 関数 | 役割 |
|---|---|---|---|
| `ON_DAMAGE_HIT` | `attacker:self` （デフォルト） | `どろぼう_steal_item` | ダメージ命中後、`battle.item_manager.take_item(ctx.defender)` を呼び出し、条件を満たせば道具を奪う |

`take_item` 内部で以下を判定する（`core/item_manager.py`）:

1. 相手が道具を持っていない、または自分がすでに道具を持っている場合は失敗（`take_item` 内の早期リターン）。
2. 上記を満たせば `swap_items()` を呼び、`ON_CHECK_ITEM_CHANGE` イベントで両者のアイテム変更可否を判定する。
   * ねんちゃく・マルチタイプ（プレート以外）・こだいかっせい/クォークチャージ（ブーストエナジー）・
     だいこんごうだま等の専用道具ガードは、すべてこの `ON_CHECK_ITEM_CHANGE` イベント経由で
     既存の特性・アイテムハンドラ側が実装済みであり、本技側の実装は不要。

## Priority 根拠

`docs/spec/turn.md` の `ON_DAMAGE` セクション（コード上は `Event.ON_DAMAGE_HIT` に対応）に
「100 はたきおとす等のアイテム効果」と明記されている。
くっつきバリの転移判定（`ON_DAMAGE_HIT` priority=30）や、さめはだ・てつのトゲ等の
接触ダメージ特性（`ON_DAMAGE_HIT` デフォルト優先度=100、同時刻ハンドラ）より後に
道具を奪う必要があるため、`MoveHandler` のデフォルト優先度（100）をそのまま使用する
（`はたきおとす`・`ほしがる` と同一パターン）。

第五世代以降の仕様では、どろぼうの効果は相手のダメージ被弾特性・ひんし判定より後に発動し、
相手をひんしにさせた場合でも道具を奪える。`ON_DAMAGE_HIT` はダメージ適用後
（`core/move_executor.py` の `_execute_hit` 内、`modify_hp` 実行後）に発火するため、
この仕様を自然に満たす。またダメージが0（例: みがわりに完全に防がれた場合）は
`actual_damage <= 0` の分岐で `ON_DAMAGE_HIT` 自体が発火しないため、
「みがわりに防がれた場合は奪えない」という仕様も自動的に満たされる。

## 実装箇所

### handlers/move_attack.py

既存の `どろぼう_steal_item` で完結しており、追加実装は不要。

```python
def どろぼう_steal_item(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """どろぼう・ほしがるのアイテム奪取効果。"""
    battle.item_manager.take_item(ctx.defender)
    return HandlerReturn(value=value)
```

### data/move.py

登録済み・修正不要（`どろぼう`・`ほしがる` ともに同一ハンドラを参照）。

### core/item_manager.py（本レビューで修正）

`swap_items()` で相手に道具が渡る場合、渡した側の `last_lost_item_name` が誤って更新され、
リサイクル・しゅうかく・ものひろいが「奪われて相手が保持している道具」を復元・拾得できてしまう
不具合を修正した。`_change_item` に `track_loss` 引数を追加し、`swap_items()` からは
`track_loss=False` を指定することで、道具が場に存在したまま相手へ渡る場合
（どろぼう・ほしがる・すりかえ・特性マジシャン等、いずれも `take_item`/`swap_items` 経由）は
`last_lost_item_name` を更新しないようにした。
`remove_item`（はたきおとす等、道具が場から完全に消滅するケース）は従来通り
`track_loss=True`（デフォルト）のままで、リサイクル等の復元対象になる。

## 注意点・エッジケース（review-test 向け）

1. **自分がすでに道具を持っている場合** → 追加効果不発（ダメージのみ）。`take_item` の
   `source.has_item()` チェックで対応済み。
2. **相手が道具を持っていない場合** → 追加効果不発。`take_item` の `target.has_item()` チェックで対応済み。
3. **相手の特性ねんちゃく** → `ON_CHECK_ITEM_CHANGE` で `ねんちゃく_prevent_item_change` が
   `source != target`（＝相手起因の変更）をブロックするため、既存の特性側実装で対応済み。
4. **相手がみがわり状態でみがわりに防がれた場合** → `ON_DAMAGE_HIT` 自体が発火しないため対応済み。
5. **はたきおとすで自分の道具が無くなっている場合** → `source.has_item()` が `False` になるため、
   通常通り相手から奪える（第五世代以降の仕様と一致）。
6. **リサイクル・しゅうかく・ものひろいとの相互作用** → 上記 `item_manager.py` の修正により、
   奪われた側がこれらで道具を復元・拾得できないことを保証する。
