# 計画書: おちゃかい

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 10 / 命中: なし（必中） / 対象: 自分含む全員
- 専用技（ポットデス）
- 使用すると「おちゃかいをして みんなで きのみを 食べた!」というメッセージが出る
- 場にいるポケモン全員（自分・相手）が持っているきのみを消費し、その効果を受ける
- きのみを消費できるポケモンがいない場合もメッセージは出て「しかし 何も 起こらなかった!」が続く
- 発動条件があるきのみ（1/2以下でHP回復等）でも条件を無視して消費させ効果を受ける
- 姿を隠しているポケモンはきのみを消費しない
- みがわり状態のポケモンはきのみを消費する
- まもる・トリックガードを無視する（ダイウォールには防がれる）
- ねんちゃく・マジックルーム・ぶきよう状態でもきのみを消費して効果を受ける
- きんちょうかんの影響下でもきのみを消費する
- 消費順: 使用者 → 使用者の味方 → 相手左 → 相手右（シングルでは使用者 → 相手の2体）
- まもる: × / マジックコート: × / 命中判定: ×
- おちゃかいで消費したきのみはものひろい・しゅうかく・リサイクルで復活する

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT` の内部 priority 表がない。既存の変化技ハンドラ（`おきみやげ_faint_and_modify_defender_stats`、`おかたづけ_cleanup` など）がすべて priority 100（デフォルト）を使用している。おちゃかいも同一パターンとして priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「お」行 — `おきみやげ_faint_and_modify_defender_stats` の後、`おたけび_modify_defender_stats` の前に挿入する。

```python
def おちゃかい_force_consume_berries(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """おちゃかい: 場の全ポケモンのきのみを強制消費し、その効果を発動する。

    消費順はシングルバトルでは使用者→相手の順。
    きのみを持つポケモンがいない場合でもメッセージは表示され、
    「しかし 何も 起こらなかった!」に相当する処理（何もしない）で終わる。
    姿を隠しているポケモンはスキップする。
    """
    # シングルバトルでは使用者→相手の順に処理
    # battle.actives は [player0_active, player1_active] の順のため、
    # attacker が先頭になるよう並べ替える
    attacker = ctx.attacker
    others = [m for m in battle.actives if m is not attacker]
    targets = [attacker] + others

    any_consumed = False

    for mon in targets:
        # 姿を隠しているポケモン（そらをとぶ等）はスキップ
        if mon.has_volatile("そらをとぶ") or mon.has_volatile("あなをほる") or mon.has_volatile("みずのなか"):
            continue

        if not mon.item.is_berry():
            continue

        # きのみを強制消費して効果を発動する
        # ON_ITEM_LOST イベントを通じてきのみのハンドラが解除される前に
        # 専用のイベント発火でその場での効果を発動させる必要がある
        # 現状フレームワークではきのみ効果は発動条件イベント（ON_HP_CHANGED 等）に
        # 紐付いているため、おちゃかい専用の強制発動ロジックが必要
        # → 実装方針については「注意点・エッジケース」を参照
        battle.add_event_log(mon, LogCode.ITEM_TRIGGERED, payload={"item": mon.item.name})
        battle.consume_item(mon)
        any_consumed = True

    return HandlerReturn(value=value)
```

> **注意**: 上記の実装は「消費のみ」の骨格コードである。実際のきのみ効果発動については下記の「注意点・エッジケース」に記載した実装方針を採用すること。

## 注意点・エッジケース

### きのみ効果の強制発動方法（最重要）

現在のフレームワークでは、きのみの効果（例: オボンのみのHP回復）は発動条件に対応するイベント（`ON_HP_CHANGED` 等）が発火した際に自動でトリガーされる仕組みになっている。`battle.consume_item(mon)` はアイテムを消費（`remove_item`）するだけで、きのみの効果は **発動しない**。

おちゃかいで「発動条件を無視してその場できのみ効果を受ける」を実現する方法は以下の2案が考えられる：

**方案A: `ON_STATUS_HIT` 内で `ON_HP_CHANGED` 等の条件チェックイベントを手動発火する**
- 消費前に `ON_HP_CHANGED` を擬似的に発火して HP 閾値チェックを走らせる
- 実装が複雑で副作用リスクが高い

**方案B: `move_status` ハンドラ内できのみ効果を直接判定・実行する（推奨）**
- `mon.item.is_berry()` で確認後、`battle.consume_item(mon)` で消費
- きのみ個別の効果（HP回復量など）は `handlers/item.py` の既存ロジックを再利用または参照して `おちゃかい` ハンドラ内で直接 `battle.modify_hp` 等を呼ぶ
- ただし全きのみの効果を網羅する実装コストが高い

**方案C: `Item.force_trigger()` 相当のAPIを `battle` または `item_manager` に追加する**
- `battle.force_trigger_item(mon)` のようなAPIを新設し、ハンドラを一時的に別コンテキストで発火させる
- フレームワーク変更を伴うが最も拡張性が高い
- impl エージェントへの指示: `battle.py` への API 追加要否をまず確認し、既存の `_announce_and_consume_item` パターンを参考に設計すること

**実装推奨**: まずはシンプルに `battle.consume_item(mon)` でアイテムを消費し、きのみの効果発動については方案C（専用API新設）の可否を impl が判断する。判断が困難な場合は効果発動なしの「消費のみ」として計画書を限定し、別チケットで追跡する。

### 「しかし 何も 起こらなかった!」メッセージ

wiki仕様によれば、きのみを消費できるポケモンがいない場合でも技自体は発動し「おちゃかいをして みんなで きのみを 食べた!」のメッセージが出た後に「しかし 何も 起こらなかった!」が続く。

実装では：
- `ON_APPLY_MOVE` での失敗 (`stop_event=True`) は行わない
- `ON_STATUS_HIT` ハンドラ内で `any_consumed` フラグの有無に応じてメッセージを切り替える
- ただし「しかし 何も 起こらなかった!」相当の `LogCode` が `TEXT_LOG` しかない場合、その使用で問題ないか確認すること

### 姿を隠しているポケモンのスキップ

wiki仕様「姿を隠しているポケモンはきのみを消費しない」に対応するため、volatile 状態を確認する。現在 `has_volatile` に対応する揮発状態名を確認すること（`そらをとぶ`・`あなをほる`・`みずのなか`・`かげぶんしん` 等）。

### ねんちゃく・マジックルーム・ぶきよう

wiki仕様では「ねんちゃく・マジックルーム・ぶきよう状態のポケモンもきのみを消費する」とある。通常の `battle.consume_item()` は `can_change_item()` を内部で呼び、そこで `ON_CHECK_ITEM_CHANGE` イベントが発火するが、ねんちゃくは `remove_item` ではなく `take_item` に対してのみ無効化するため問題ない可能性が高い。実装時に `can_change_item` の動作を確認すること。

### きんちょうかん

wiki仕様「きんちょうかんの影響を受けているポケモンもきのみを消費する」は、`きんちょうかん_check_nervous` が `ON_CHECK_NERVOUS` で `True` を返すが、きのみの消費は `ON_CHECK_NERVOUS` を経由しないため問題なし（自動的に仕様通りになる）。

### ほおぶくろ・はんすう

wiki仕様では「消費したきのみが効果を発動した場合にほおぶくろ・はんすうも発動する」とある。ただし「効果が発動しなかった場合は発動しない（例: 状態異常でないときのラムのみ）」。これは実装方針（方案A/B/C）が確定してから追加で検討する。

### 失敗条件なし・`ON_APPLY_MOVE` ハンドラ不要

おちゃかいはきのみを持つポケモンが0体でも技が成功扱いになるため、`ON_BEFORE_APPLY_MOVE` での失敗チェックハンドラは不要。

### 対象指定（target）

`MoveData` の `target` フィールドはデフォルト `"foe"` だが、おちゃかいは「自分含む全員」が対象。ただし `ON_STATUS_HIT` のハンドラ内で `battle.actives` を直接走査して全員を処理するため、`target` フィールドの変更が必要かどうかを impl が確認すること。変更が必要な場合、`"all"` または `"self_and_foes"` に相当する `MoveTarget` 型の値が存在するか `utils/type_defs.py` を確認する。

### ランクがすでに最大・最小の場合

きのみの効果によるランク変化（チイラのみ等）がすでに最大の場合でも、きのみの消費は行われる（おかたづけの仕様と同様）。`ON_TRY_MOVE_2` の priority 130 で「ランク補正上げ: すでに最大」による `stop_event` が発生しないよう注意。技自体が `stop_event` されると全ポケモンのきのみ処理がスキップされる。
