# とんぼがえり 実装計画書

## 仕様要約（docs/spec/moves/とんぼがえり.md 参照）

* タイプ: むし / 分類: 物理 / 威力: 70 / 命中: 100 / PP: 20
* 対象: 相手1体 / 優先度: 0 / 接触あり
* 効果: 攻撃後、使用者が手持ちのポケモンと交代する（追加効果とはみなされない）。

## 現状の実装状況

`src/jpoke/data/move.py` に既に登録済み。

```python
"とんぼがえり": MoveData(
    type="むし",
    category="physical",
    pp=20,
    power=70,
    accuracy=100,
    flags={"contact"},
    handlers={
        Event.ON_HIT: h.MoveHandler(
            ha.pivot,
        )
    }
),
```

内容は仕様書と一致しており、修正不要。

## ハンドラ構成

| Event | 関数 | subject_spec | priority | 役割 |
|---|---|---|---|---|
| ON_HIT | `pivot` | attacker:self | 100（デフォルト） | 交代可能な控えがいれば `player_states[player].interrupt = Interrupt.PIVOT` を設定 |

`pivot`（`handlers/move_attack.py`）は `ボルトチェンジ` と共通の実装であり、専用の新規関数は不要。
実際の交代処理（プレイヤーが控えを選択して入れ替える）は `SwitchManager.run_interrupt_switch(Interrupt.PIVOT)`
（`turn_controller.py` の `_run_move_phase` 内、だっしゅつボタン・ききかいひの後、だっしゅつパックの前に実行）
が担う共通インフラであり、本技側の実装には含まれない。

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_HIT` 節（ダメージ発生後の処理、みがわりに被弾しても発動）を確認した。
一覧に交代技専用の行は明記されていないが、既存の `ボルトチェンジ` も同じ `pivot` 関数をデフォルト
priority（100）で `Event.ON_HIT` に登録しており、他の priority=100 の処理（コアパニッシャー・なげつける等）
とは独立した状態変更（`interrupt` フラグ設定のみ）であるため相互作用がなく、priority の新規検討は不要。
既存パターンを踏襲しデフォルト値のままとする。

## 注意点・エッジケース（review-test 向け）

1. **控えがいない場合**
   - `pivot` は `battle.command_manager.get_available_switch_commands(player)` が空の場合、
     interrupt を設定しない。ダメージは通常通り与えられるが交代は発動しない。実装済み・動作確認済み。

2. **まもる・命中しなかった場合**
   - `Event.ON_BEFORE_APPLY_MOVE` で無効化される、または命中判定に失敗した場合は
     `Event.ON_HIT` 自体が発火しないため、交代も発動しない。既存の move_executor.py の
     フロー（本技固有の実装ではない）に依存しており、対応済み。

3. **対象を倒した場合の交代順序**
   - `Event.ON_HIT` は攻撃側の interrupt を設定し、`_run_move_phase` 内で
     攻撃側の PIVOT 交代→（ターン終了時の）防御側の瀕死交代、の順に処理されるため、
     「とんぼがえりの交代先を見せた後に相手が次のポケモンを選択する」という仕様と一致する。
     動作確認済み（手動シミュレーションで攻撃側→防御側の順に交代されることを確認）。

4. **レッドカード・だっしゅつボタン・にげごし/ききかいひとの相互作用**
   - これらの防御側アイテム・特性は `Event.ON_DAMAGE_HIT`（レッドカードは実ダメージ0の場合のみ
     `Event.ON_HIT`）で `battle.run_switch(...)` を直接呼び出し、防御側プレイヤーの交代を
     即座に実行する。攻撃側の `interrupt = Interrupt.PIVOT` フラグとは別プレイヤーの状態を
     操作するため、通常のケース（実ダメージ>0）では相互に干渉しない。
   - 唯一の理論的なエッジケース: 防御側がばけのかわ/アイスフェイスで実ダメージ0のまま
     かつレッドカードを持つ場合、`レッドカード_force_switch_on_zero_damage` と `pivot` が
     同一イベント（`Event.ON_HIT`）に登録されるが、操作対象のプレイヤーが異なる
     （防御側 vs 攻撃側）ため、ハンドラの実行順序に関わらず両者の処理は独立して成立する。
     既存実装のまま対応不要。

5. **攻撃側がゴツゴツメット等の反動で瀕死になった場合**
   - Wiki記載: 「対象の特性さめはだ/てつのトゲ/ゆうばく/とびだすなかみ/うのミサイル、
     持ち物ゴツゴツメット/ジャポのみ、みちづれ状態の効果を受けて、攻撃側がひんしになった場合、
     技の効果は発動せず、次のポケモンはターンの終了時に繰りだす」とある。
   - 本プロジェクトの `pivot` は `Event.ON_HIT`（ダメージ判定直後）で interrupt を設定するため、
     後続の `Event.ON_DAMAGE_HIT` で発動するゴツゴツメット等により攻撃側が瀕死になった場合でも
     interrupt フラグは `PIVOT` のまま残る。
   - 実機シミュレーションで確認した結果、`_run_move_phase` はプレイヤーごとに逐次処理されるため、
     瀕死になった攻撃側は PIVOT 交代によって（本来の「ターン終了時の瀕死交代」ではなく）
     ただちに次のポケモンに入れ替わる。シングルバトルにおいては、後続の行動順のポケモンが
     交代先に対して行動する点で最終的な結果は「瀕死による強制交代」と変わらず、
     現状 `Interrupt.PIVOT` と `Interrupt.FAINTED` を区別して参照する効果
     （おいうち等）は本プロジェクトに未実装のため、実害のある不整合は確認されなかった。
   - おいうちを実装する際は、`pivot` が interrupt を設定した後に攻撃側が瀕死になった場合、
     `Interrupt.PIVOT` を `Interrupt.FAINTED` 相当として扱う必要がないか改めて確認すること
     （既知の制約として記録。今回は対応しない）。
