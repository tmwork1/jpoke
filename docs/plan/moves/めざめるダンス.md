# 実装計画書: めざめるダンス

## 仕様要約

- タイプ: ノーマル（技のタイプは使用者依存で変化） / カテゴリ: 特殊 / 威力: 90 / 命中率: 100 / PP: 15 / 対象: 1体選択
- 使用するポケモンのタイプ1と同じタイプの技になる。
- テラスタルしている場合はその時のタイプ1（＝基本的にテラスタイプ）を参照する。ステラタイプにテラスタルしている場合は元のタイプ1を参照する。
- 使用者の特性 ノーマルスキン / フェアリースキン / フリーズスキン / スカイスキン / エレキスキン の効果は発動しない。
- そうでん状態の効果を受けると、使用者のタイプに関係なくでんきタイプになる。

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| `data/move.py` エントリ | 実装済み |
| `handlers/move_attack.py` 関数 | 実装済み（めざめるダンス_modify_type） |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_MODIFY_MOVE_TYPE` | 100（デフォルト） | `attacker:self`（デフォルト） | `ctx.attacker.types[0]` を技タイプとして返す |

## Priority 根拠・既存ハンドラとの整合性

`ON_MODIFY_MOVE_TYPE` は turn.md に priority 表が存在しないイベントであり、以下の既存ハンドラとの相互作用を検証した上でデフォルト優先度(100)を採用した。

- `Pokemon.types` プロパティは既にテラスタル・ステラタイプ・タイプ除去（もえつきる/はねやすめ等）を考慮した「現在のタイプ1」を返すため、`ctx.attacker.types[0]` を参照するだけでテラスタル対応の仕様を満たす。追加の分岐処理は不要。
- スキン系特性（ノーマルスキン等、`_skin_modify_move_type` ベース）はポケモン入場時から登録済みの特性ハンドラであり、EventManager 内部では移動技ハンドラ（`ctx.move.register_handlers` で技実行直前に登録）より先に評価される（同一priorityの場合、登録順が早いものから安定ソートされるため）。そのため `めざめるダンス_modify_type` がスキン系特性の後に実行され、`value` を無条件で `types[0]` に上書きする。これにより「スキン系特性の効果が発動しない」という仕様を自動的に満たすことを実機テストで確認した。
- そうでん状態（`そうでん_move_type`）も本来はスキン系特性と同様に事前登録されるハンドラであり、同一priorityのままでは めざめるダンス に上書きされてしまう（実機テストで確認したバグ）。そうでん が最終的に勝つ必要があるため、`data/volatile.py` の `そうでん_move_type` に `priority=110` を明示し、めざめるダンス（および他の同優先度の技タイプ変更ハンドラ）より後に評価されるよう修正した。

## 既知の限界（対応不要と判断）

- 単タイプの使用者が もえつきる／はねやすめ でタイプ無しになった状態でこの技を使用した場合、`Pokemon.types` が空リストになり `めざめるダンス_modify_type` は元の `value`（デフォルトのノーマルタイプ）を返す。仕様上は「タイプ無し」の技になるべきだが、本シミュレータには「無タイプ技」の概念自体が実装されていないため対応範囲外とする。
