# 実装計画書: いやしのねがい

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 12（`docs/champions/move_list.txt` 準拠） / 命中: なし（必中） / 対象: 自分
- 使用者が **ひんしになる** 代わりに、次に場に出た自分のポケモンの **HP を全回復** し、**状態異常を回復** する
- PP は回復しない（みかづきのまい との違い）
- `heal` ラベルを持つ
- `target="self"` を持つ（自分専用技。`まもる` に阻まれない）
- 控えのポケモンがいない（生存個体がいない）場合は失敗し、使用者はひんしにならない
- 技使用直後に使用者がひんしになり、死に出し交代が発生する
- 次に場に出たポケモンが「いやしのねがい」効果を受け取った時点でフィールドは消える

## ハンドラ構成

| イベント | priority | subject_spec | 役割 | 実装場所 |
|----------|----------|--------------|------|----------|
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self` | 控えの生存確認 → 失敗ならログを出して終了。成功時は自陣営に「いやしのねがい」フィールドを設置し、使用者をひんしにする | `handlers/move_status.py::いやしのねがい_apply` → `data/moves/move_a.py` |

## Priority 根拠

### ON_STATUS_HIT (priority=100)

`docs/spec/turn.md` に `ON_STATUS_HIT` に関する直接的な記載はないが、同種の類似ハンドラ（`いのちのしずく_heal_self`、`はらだいこ_apply`、`みがわり_apply` 等）はすべて `ON_STATUS_HIT` priority=100（デフォルト）を使用している。`いやしのねがい` も同パターンとする。

### ON_SWITCH_IN (priority=100)

`docs/spec/turn.md` の `Event.ON_SWITCH_IN` セクションに以下の記載がある：

> | 100 | いやしのねがいによる回復 |

priority=100 を明示する。

## 実装箇所（現状のコードに準拠）

### 1. `src/jpoke/data/moves/move_a.py`

```python
"いやしのねがい": MoveData(
    type="エスパー",
    category="status",
    pp=12,
    target="self",
    flags={"heal"},
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.いやしのねがい_apply,
        ),
    }
),
```

`target="self"` を明示しないと `Move.target` が既定値 `"foe"` になり、
`Move.is_blocked_by_protect` が `True` を返して `まもる` に阻まれる誤動作を起こす
（自分専用技のため、まもるは効かないのが正しい）。

### 2. `src/jpoke/handlers/move_status.py`

```python
def いやしのねがい_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """いやしのねがい: 使用者をひんしにし、自陣営に「いやしのねがい」フィールドを設置する。

    次に場に出たポケモンの HP が全回復し、状態異常が回復する。
    PP は回復しない（みかづきのまいとの違い）。
    控えのポケモンがいない場合は失敗し、使用者はひんしにならない。
    """
    mon = ctx.attacker
    player = battle.get_player(mon)
    if not battle.query.can_switch(player):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="いやしのねがい_交代不可")
        )
        return HandlerReturn(value=False, stop_event=True)
    side = battle.get_side(mon)
    side.activate("いやしのねがい", 1)
    battle.faint(mon)
    return HandlerReturn(value=value)
```

### 3. `src/jpoke/handlers/field.py`

```python
def いやしのねがい_heal(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """いやしのねがい: 場に出たポケモンの HP を全回復し、状態異常を回復する。

    HP 満タンかつ状態異常なしのポケモンには発動せず、フィールドは保留される（第八世代以降仕様）。
    HP またはAilment に変化があった場合のみフィールドを解除する。
    """
    mon = ctx.source
    side = battle.get_side(mon)
    if mon.hp == mon.max_hp and not mon.ailment.is_active:
        return HandlerReturn(value=value)
    battle.modify_hp(mon, v=mon.max_hp - mon.hp)
    battle.ailment_manager.remove(mon)
    side.deactivate("いやしのねがい")
    return HandlerReturn(value=value)
```

### 4. `src/jpoke/data/field/side_field.py`

```python
"いやしのねがい": FieldData(
    handlers={
        Event.ON_SWITCH_IN: h.FieldHandler(
            h.いやしのねがい_heal,
            subject_spec="source:self",
            priority=100,
        ),
    },
),
```

## 注意点・エッジケース

- **控えのポケモン不在時の失敗判定**: `battle.query.can_switch(player)` が `False` の場合は失敗し、フィールドも設置されず使用者もひんしにならない（`しっぽきり_check` 等と同じ判定 API を使用）
- **`target="self"` の必要性**: `Move.is_blocked_by_protect` は `target == "foe"` のときのみ `True` を返す。`target` 未指定（既定値 `"foe"`）のままだと `まもる` が誤って `いやしのねがい` を防いでしまうため、`target="self"` を必ず指定する
- **使用者がひんしになる前提**: `ON_STATUS_HIT` で `side.activate("いやしのねがい", 1)` を先に呼んでからひんし処理（`battle.faint`）を行う。ひんし後に `run_faint_switch` が呼ばれ、死に出しポケモンの `ON_SWITCH_IN` で回復が発動する
- **SideField のカウント管理**: `activate("いやしのねがい", 1)` で 1 ターン分のフィールドを設置するが、`ON_SWITCH_IN` 内で `side.deactivate` を直接呼ぶため、カウントのデクリメントより先にフィールドが消える（`ON_TURN_END` tick は不要）
- **みかづきのまいとの違い**: みかづきのまいは PP も回復するが、いやしのねがいは HP と状態異常のみの回復。両者の実装を混同しないよう注意
- **ailment_manager.remove の uncurable ガード**: `ailment_manager.remove` 内部で `uncurable` フラグをチェックするため、ハンドラ側での追加ガードは不要
- **HP 満タン時の回復**: `modify_hp(mon, v=mon.max_hp - mon.hp)` は HP が満タンの場合 v=0 となり変化しない。`modify_hp` は v=0 のとき早期リターンするため問題ない
- **SideField が設置される陣営**: `battle.get_side(ctx.attacker)` で使用者陣営のフィールドに設置される。ON_SWITCH_IN のハンドラも `ctx.source` から同陣営を取得するため、陣営の一致が保証される
- **かいふくふうじとの相互作用**: `battle.modify_hp` は内部で `ON_MODIFY_HEAL` を発火し、`かいふくふうじ_block_heal` が回復量を 0 にする。ただし本プロジェクトはシングルバトルのみが対象であり、ベンチのポケモンが `かいふくふうじ` の巻き添えを受けたまま場に出るケース（ダブル/トリプル固有の "MOVE" 移動シナリオ）は発生しないため、実質的にこの分岐が単体で問題になることはない
- **ダブル/トリプルバトル固有仕様は対象外**: サイドチェンジ・トリプルバトルの MOVE・げんきのかけらとの組み合わせなど、複数体バトル特有の仕様は本プロジェクト（シングルバトル専用）では実装しない
- **「いやしのねがい」フィールドの既存判定**: `ON_APPLY_MOVE`（`ON_BEFORE_APPLY_MOVE`）での失敗判定（`ねがいごと: 前ターン効果残り` に相当）は現段階では実装しない。「いやしのねがい」は使用直後に使用者がひんしになるため、同一ターン内での二重設置は不可能
