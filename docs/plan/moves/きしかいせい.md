# 計画書: きしかいせい（再レビュー）

## 仕様要約

* 使用者の残りHPが少ないほど威力が大きくなる（`docs/spec/moves/きしかいせい.md` 参照）。
* X = floor(残りHP × 48 / 最大HP) を算出し、X の範囲に応じて威力 20〜200 の6段階から決定する。
* じたばたと全く同じ威力計算式を使用する。

## 既存実装の確認結果

`src/jpoke/handlers/move_attack.py` の `きしかいせい_calc_power` と共通関数 `_hp_low_to_power`
（`じたばた_calc_power` と共用）を確認した結果、仕様書の式・威力段階と完全に一致しており、
`src/jpoke/data/move.py` の `MoveData`（type/category/pp/power/accuracy/flags/handlers）も
一次情報と一致している。実装の修正は不要。

## ハンドラ構成（既存・変更なし）

| 技名 | イベント | priority | subject_spec | 役割 |
|------|----------|----------|--------------|------|
| きしかいせい | `Event.ON_CALC_POWER_MODIFIER` | デフォルト（100） | `attacker:self`（`MoveHandler` デフォルト） | 使用者の残りHP割合から技の実威力を算出して返す |

* `Event.ON_CALC_POWER_MODIFIER` は `docs/spec/turn.md` に priority の記載がないため、
  既存の同種ハンドラ（`じたばた_calc_power` ほか、`ON_CALC_POWER_MODIFIER` を使う全ての
  `MoveHandler`）が明示的な priority 指定なし＝デフォルト値（100）で登録されている慣例に
  合わせ、今回もデフォルトのままとする。

```python
def きしかいせい_calc_power(battle: Battle, ctx: AttackContext, value: int) -> HandlerReturn:
    """きしかいせい: 自分の残りHPが少ないほど威力が高くなる。

    計算式: X = floor(残りHP × 48 / 最大HP)
    X ≥ 33 → 20 / 17-32 → 40 / 10-16 → 80 / 5-9 → 100 / 2-4 → 150 / 0-1 → 200
    """
    return HandlerReturn(value=_hp_low_to_power(ctx.attacker.hp, ctx.attacker.max_hp) * 4096)
```

```python
"きしかいせい": MoveData(
    type="かくとう",
    category="physical",
    pp=15,
    power=1,
    accuracy=100,
    flags={"contact"},
    handlers={
        Event.ON_CALC_POWER_MODIFIER: h.MoveHandler(
            ha.きしかいせい_calc_power,
        ),
    }
),
```

## リーサル計算

`docs/progress/move.md` の該当行は「リーサル実装: n/a」「リーサルテスト: n/a」のため対象外。
理由: 威力は攻撃側自身の現在HPにのみ依存し、相手（`defender`）のHP分布 `StateDist` を
直接左右する要素ではないため、`lethal.py` 側の専用ハンドラは不要。

## テスト

既存の `tests/moves_attack/test_move_ka.py` に威力段階の境界を確認するテストが
一通り揃っている（HP満タン→20、X=32→40、X=16→80、X=5→100、X=2→150、X=0→200）。
今回のレビューで確認した範囲では計算式・実装ともに正しいため、テストの追加は不要。
ただし `test_きしかいせい_HP約10パーセントのとき威力100` の docstring に、実際のテストコード
（`hp=26`）と矛盾する記述（`hp=24 → x=4 → 威力150`）が残っていたため修正する。

## 注意点・エッジケース
