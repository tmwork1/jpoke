# 実装計画書: うらみ

## 仕様要約（`docs/spec/moves/うらみ.md` 参照）

- タイプ: ゴースト / カテゴリ: 変化 / PP: 12（`src/jpoke/data/ps-champ-ja/moves.json` 準拠） / 命中: 100 / 対象: 1 体選択
- 相手が直前に使った技の PP を 4 減らす
- 失敗条件:
  1. 相手が技を使っていない場合（`executed_move` が None）
  2. 相手の直前の技が「わるあがき」の場合
  3. 相手の直前の技のPPがすでに0の場合
- みがわり状態の相手にも効果を貫通させる（`bypass_substitute` フラグが必要）

## 現状の実装状況（レビュー時点）

| 項目 | 状態 |
|------|------|
| `data/moves/move_a.py` エントリ | 登録済みだが `pp=10`（誤り、`12` が正）、`flags` 未設定（`bypass_substitute` 欠落） |
| `handlers/move_status.py` 関数 | 実装済みだが失敗判定が `executed_move is None` のみで、わるあがき・PP0 のケースが未対応 |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 失敗条件（未行動・わるあがき・PP0）の判定 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手の `executed_move` の PP を 4 減らす |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「うらみ: 失敗条件」が priority 100 として明記されている
- `ON_STATUS_HIT` priority 100（デフォルト）: 同種ハンドラと同じパターン

## PP 情報の保持場所

- `Pokemon.executed_move` は `Move | None` 型で、直前に使用した技が格納される（`model/pokemon.py` で確認済み）
- `Move.modify_pp(v: int)` メソッドが存在し、PP の増減に使用できる（`model/move.py` で確認済み）
- `Move` オブジェクトは `Pokemon.moves` リスト内の要素と同一オブジェクトであるため、`executed_move.modify_pp(-4)` で次ターン以降の PP に反映される

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「う」行 — `うたう_apply_sleep` の後、`エレキフィールド_activate_terrain` の前に挿入する。

```python
def うらみ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """うらみの失敗チェック。

    以下のいずれかに該当する場合は失敗する。
    - 相手が技を使っていない（executed_move が None）
    - 相手の直前の技が「わるあがき」（PPが概念上無限で対象にならない）
    - 相手の直前の技のPPがすでに0（他の効果で0まで減っていた場合）
    """
    move = ctx.defender.executed_move
    if move is None or move.name == "わるあがき" or move.pp <= 0:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="うらみ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def うらみ_deplete_pp(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """うらみの効果: 相手が直前に使った技のPPを4減らす。"""
    move = ctx.defender.executed_move
    move.modify_pp(-4)
    return HandlerReturn(value=value)
```

`うらみ_can_apply` で失敗条件を確定させているため、`うらみ_deplete_pp` 側の
`move is None` ガードは不要（`ON_BEFORE_APPLY_MOVE` が `False` を返せば
`ON_STATUS_HIT` には到達しない）。

### `src/jpoke/data/moves/move_a.py`

既存の `"うらみ"` エントリの `pp` を `12` に修正し、`flags={"bypass_substitute"}` を追加する。

```python
"うらみ": MoveData(
    type="ゴースト",
    category="status",
    pp=12,
    accuracy=100,
    flags={"bypass_substitute"},
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.うらみ_can_apply,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.うらみ_deplete_pp,
        ),
    }
),
```

## 注意点・エッジケース

- `Move.modify_pp(-4)` は `max(0, min(max_pp, pp + v))` でアンダーフロー・オーバーフローを防ぐ実装になっているため、PP が 0 のときでも安全に呼び出せる（ただし `うらみ_can_apply` で事前に弾くため実際に0のまま呼ばれることはない）
- `executed_move` が格納されるタイミングを確認すること: 相手が行動する際に設定され、交代時にリセットされる（`reset_on_switch_out` で `executed_move = None`）
- 相手が「わるあがき」を使った場合、`executed_move` が「わるあがき」の `Move` オブジェクト（`pp=PP_INFINITE`）になる。PPが0にならないため「PPが0の技」チェックでは検出できず、`move.name == "わるあがき"` の明示チェックが必須
- 相手がターン内でまだ行動していない場合（自分が先攻）、`executed_move` は前ターンの技を保持している。失敗条件として turn.md で「うらみ: 失敗条件」が priority 100 で記載されている
- 相手の直前の行動が状態異常・状態変化・特性なまけ等でPPを消費せずに失敗していた場合、`executed_move` はその行動では更新されない（`core/move_executor.py` の `run_move` で `ON_TRY_ACTION` が偽の場合、PP消費および `executed_move` 更新処理に到達しないため）。そのため「最後にPPを消費した技を対象にする」仕様はエンジン側の設計により自然に満たされ、うらみ側の追加対応は不要
- みがわり貫通のため `flags={"bypass_substitute"}` が必須（`core/move_executor.py` の `check_hit_substitute` で `bypass_substitute` フラグを持たない対象 `foe` の技はみがわりでブロックされるため）
