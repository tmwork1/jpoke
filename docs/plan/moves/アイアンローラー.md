# 実装計画書: アイアンローラー

参照仕様書: docs/spec/moves/アイアンローラー.md

## 実装方針

アイアンローラーは「場にフィールドが発生しているときのみ成功する」通常の物理
ダメージ技で、命中してヒットした後にフィールドを解除する（一次情報:
docs/wiki/moves/アイアンローラー.html 技の仕様節）。フィールド解除は
「追加効果」ではなく技の主効果の一部であるため、ちからずく・りんぷん・
おんみつマントの影響を受けない（`apply_volatile_to_defender` 等の
確率つき追加効果ヘルパーは使わず、無条件に `battle.terrain_manager.remove()`
を呼ぶ）。

### データ定義（`src/jpoke/data/moves/move_a.py`）

* `type="はがね"`
* `category="physical"`
* `power=130`
* `accuracy=100`
* `pp=8`（`src/jpoke/data/ps-champ-ja/moves.json` に基づく。Gen9本家は5だが、チャンピオンズでは
  全技PPが8/12/16/20に統一されているため8とする）
* `flags={"contact"}`
* `target`: 指定なし（デフォルトの `"foe"` を使用）
* `handlers`:
  * `Event.ON_TRY_MOVE_1`: `アイアンローラー_check_terrain`
    （`subject_spec="attacker:self"`, `priority=30`）
    * `docs/spec/turn.md` `Event.ON_TRY_MOVE_1` の
      「30 | アイアンローラー: フィールドが無い」に基づく。
  * `Event.ON_DAMAGE_HIT`: `アイアンローラー_clear_terrain`
    （`subject_spec="attacker:self"`, `priority=180`）
    * `docs/spec/turn.md` `Event.ON_DAMAGE`（コード上は `Event.ON_DAMAGE_HIT`
      に対応）の「180 | アイアンローラー等でフィールド消失」に基づく。
    * `Event.ON_DAMAGE_HIT` は対象の実HPダメージが0（`actual_damage<=0`）の
      場合は発火しない仕様（`core/move_executor.py` `_execute_hit`）のため、
      通常時（実HPダメージ>0）のみをこのハンドラで処理する。
  * `Event.ON_HIT`: `アイアンローラー_clear_terrain_on_zero_damage`
    （`subject_spec="attacker:self"`, `priority=180`）
    * みがわりに被弾した場合など、実HPダメージが0（`value<=0`）だった場合の
      フィールド解除を処理する。`Event.ON_HIT` は「ダメージ発生後の処理
      （みがわりに被弾しても発動）」（`docs/spec/turn.md` `Event.ON_HIT`）と
      明記されており、みがわり被弾時にも必ず発火するため採用する。
      `docs/spec/turn.md` の `Event.ON_HIT` セクションには本効果に対応する
      掲載がないため、`Event.ON_DAMAGE_HIT` 側と同じ「180」を踏襲し、
      既存の同種の二段構え実装（アイテム`レッドカード`の
      `レッドカード_force_switch` / `レッドカード_force_switch_on_zero_damage`
      パターン、`src/jpoke/data/item.py`）を参考にする。
      実HPダメージが正の場合は `Event.ON_DAMAGE_HIT` 側で既に処理済みのため、
      ここでは `value<=0` のときのみ `battle.terrain_manager.remove()` を呼ぶ。

### ハンドラ実装（`src/jpoke/handlers/move_attack.py`）

* `アイアンローラー_check_terrain(battle, ctx, value)`
  * `battle.terrain.is_active` が `False` の場合、失敗ログを出して
    `HandlerReturn(value=False, stop_event=True)` を返す（既存実装のまま）。
* `アイアンローラー_clear_terrain(battle, ctx, value)`
  * `battle.terrain_manager.remove()` を呼び `HandlerReturn(value=value)` を
    返す（既存実装のまま）。
* `アイアンローラー_clear_terrain_on_zero_damage(battle, ctx, value)`（新規）
  * `value <= 0` の場合のみ `battle.terrain_manager.remove()` を呼ぶ。
  * `value > 0` の場合は何もせず `HandlerReturn(value=value)` を返す
    （`Event.ON_DAMAGE_HIT` 側で既に解除済みのため）。

## 相互作用の確認

* **まもる**: `flags` に `"unprotectable"` を付与しないため通常通り防がれる
  （一次情報の「まもる: ○」に合致）。防がれた場合は `ON_BEFORE_APPLY_MOVE` で
  `move_applied=False` となり `_execute_hit` 自体が呼ばれないため、
  フィールド解除は発生しない。
* **マジックコート**: 物理技のため対象外（そもそも `ON_CHECK_REFLECT` の
  対象になるのは変化技のみ）。
* **みがわり**: `Event.ON_HIT` 側の新規ハンドラが `value<=0` を検知して
  フィールドを解除する。みがわりの実装（`みがわり_block_damage`,
  `Event.ON_MODIFY_MOVE_DAMAGE`）はダメージを0にするのみで技自体を無効化
  しない（`みがわり_immune` は変化技のみを無効化）ため、`_execute_hit` は
  最後まで実行され `Event.ON_HIT` は発火する。
* **ちからずく・りんぷん・おんみつマント**: いずれも
  `Event.ON_MODIFY_SECONDARY_CHANCE` を通じて「追加効果」の確率にのみ
  作用する（`りんぷん_block_secondary_chance`,
  `おんみつマント_negate_secondary` 等）。アイアンローラーのフィールド解除は
  これらのイベントを経由しない無条件処理のため、影響を受けない
  （一次情報の記載通り）。
* **レッドカード・にげごし・ききかいひ・だっしゅつパック等の強制交代**:
  これらは `docs/spec/turn.md` `Event.ON_DAMAGE`（レッドカード優先度150）や
  `Event.ON_HP_CHANGED`（ききかいひ等）等、既存の共有アーキテクチャで
  タイミングが規定されており、アイアンローラー側の優先度（180）は
  それらとの相対順序も含めて `docs/spec/turn.md` の既存記載
  （「180 | アイアンローラー等でフィールド消失」）にそのまま従う。
  この相対順序自体の妥当性検証（レッドカード等、複数の仕組みにまたがる
  大域的な優先度設計）は本計画のスコープ外とし、本タスクでは変更しない。

## リーサル計算

ダメージ技だが、対象を確実に倒せるかどうかの判定はフィールドが存在するか
どうかにのみ依存する成立条件があるものの、既存の一般的なダメージ技用リーサル
計算ロジック（威力・命中率に基づく通常のダメージ分布計算）で十分カバーできる
特別な追加効果（回復・追加ダメージ・ランク変化等）を持たないため、専用の
リーサルハンドラの追加は不要（`docs/progress/move.md` 上も `n/a` 扱い）。

## テスト方針

`tests/moves_attack/test_move_a.py` に以下を追加・維持する。

* フィールドが存在するとき命中してダメージを与え、フィールドを解除すること
  （既存）
* フィールドが存在しない場合は失敗し、ダメージを与えないこと（既存）
* みがわり状態の相手に対して、実HPダメージが0でもフィールドが解除されること
  （新規追加。修正前の実装では `Event.ON_DAMAGE_HIT` のみに依存しており
  発火しなかったための回帰確認）
* まもるで防がれた場合はフィールドが解除されないこと（新規追加）
