# 計画書: スピードスワップ

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 10 / 命中: - / 対象: 相手（1体選択）
- 使用者と相手の「すばやさ（S）」の**実数値**を入れ替える
- ランク変化（`rank["S"]`）は変化しない。実数値のみを入れ替える
- 命中判定: なし（必中）/ まもる: ○ / マジックコート: × / よこどり: × / オウムがえし: ○
- `data/move.py` の `accuracy=100` は他の変化技に倣った設定値であり、命中判定は発生しない（命中率フィールドは `-`）

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT` の内部 priority 表は掲載されていない。既存の変化技ハンドラ（`ガードシェア_equalize_stats`・`ガードスワップ_swap_ranks`・`すりかえ_swap_items` 等）はすべて priority 100（`MoveHandler` デフォルト）を使用している。スピードスワップも同一パターンで priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「す」行 — `すりかえ_swap_items` の直前に挿入する（す→すり の順）。

実数値の入れ替えは `_stats_manager.stats[5]`（インデックス5 = すばやさ）を直接操作する。ガードシェアが `stats[2]`, `stats[4]`（ぼうぎょ・とくぼう）を操作する実績パターンを踏襲する。`battle.modify_stat()` は実数値の書き換えAPIではなくランク変化量の加算APIであるため使用しない。

```python
def スピードスワップ_swap_speed(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """スピードスワップ: 使用者と相手のすばやさの実数値を入れ替える。

    ランク変化は行わず、実数値のみを入れ替える。
    インデックス 5 = すばやさ（[HP, 攻撃, 防御, 特攻, 特防, 素早さ]）。
    """
    attacker = ctx.attacker
    defender = ctx.defender
    a_spd = attacker._stats_manager.stats[5]
    d_spd = defender._stats_manager.stats[5]
    attacker._stats_manager.stats[5] = d_spd
    defender._stats_manager.stats[5] = a_spd
    return HandlerReturn(value=value)
```

### `src/jpoke/data/move.py`

既存の `"スピードスワップ"` エントリに `handlers` を追加する。

```python
"スピードスワップ": MoveData(
    type="エスパー",
    category="変化",
    pp=10,
    accuracy=100,
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.スピードスワップ_swap_speed,
        ),
    }
),
```

## 注意点・エッジケース

- **実数値直接操作の根拠**: ガードシェア（`_stats_manager.stats[2]`, `[4]`）と同じパターンで実数値を直接書き換える。`battle.modify_stat()` はランク変化量加算APIのため「入れ替え」に使用不可
- **ランク変化は変わらない**: `mon.rank["S"]` は変化しない。よっておいかぜやすりかえ等によるランク補正は入れ替え後も各ポケモンの rank["S"] に従う
- **ON_MODIFY_STAT 非発火**: 直接代入のため、ランク変化に反応する特性（かちき・まけんき等）は発動しない。これはゲーム仕様上も正しい（スピードスワップは「ランク変化」ではなく「実数値交換」の技）
- **トリックルーム中の行動順への影響**: 入れ替え後のすばやさ実数値はその後のターンから行動順計算に反映される。入れ替えたターン自体の行動順は変化しない（行動順はターン開始時に決定済み）
- **おいかぜ・ルームサービス等との相互作用**: これらはすばやさのランク変化（rank["S"]）やフィールド状態で速度を修正するため、実数値スワップ後もそれぞれの補正が独立して適用される。すばやさ実測値（計算後最終値）はランク補正や場の状態も含むが、本関数が書き換えるのはあくまで種族値・努力値から計算された実数値の箇所
- **ワンダールーム中**: ワンダールームはぼうぎょとくぼうの参照を入れ替えるが、すばやさには影響しない。スピードスワップと干渉なし
- **0値・極端な実数値**: 双方のすばやさが同値の場合は入れ替えても変化なし。`ON_TRY_MOVE_2` の失敗条件（ランク上限・下限）はランク変化技に対する判定であり、実数値スワップ技のスピードスワップには適用されない
- **ガードシェア・パワーシェアとの類似**: シェア系（平均化）ではなく「入れ替え」であることに注意。コードパターンはガードシェア（stats[2], stats[4]）に近いが、平均化ではなく swap である
