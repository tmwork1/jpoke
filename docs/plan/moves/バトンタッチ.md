## 仕様要約

- タイプ: ノーマル / 分類: 変化 / PP: 40 / 命中: なし / 優先度: 0
- 効果: 控えのポケモンと交代し、**能力ランク変化と一部の揮発性状態を交代先に引き継ぐ**。
- 失敗条件:
  - 交代できる控えポケモンがいない（`turn.md` ON_APPLY_MOVE priority=100「バトンタッチ等: 交代不可」）
- 引き継ぐもの（第9世代シングルバトル）:
  - 能力ランク変化（A/B/C/D/S/ACC/EVA すべて、プラス・マイナス両方）
  - みがわり（残りHPごと）
  - こんらん（残ターン込み）
  - きゅうしょアップ
  - ちいさくなる
  - まるくなる
  - アクアリング
  - ねをはる
  - やどりぎのタネ
  - じゅうでん
  - でんじふゆう（残ターン込み）
  - ちょうはつ（残ターン込み）
  - しおづけ
  - たくわえる（count ごと）
- 引き継がない主なもの:
  - 状態異常（ねむり・まひ・やけど等）
  - 強制行動系volatile（あばれる・そらをとぶ・あなをほる等）
  - アンコール・いちゃもん・かなしばり等の拘束系
  - こだわり
  - バインド・にげられない等
- バトンタッチによる交代はとらわれ状態（かげふみ/じりょく/ありじごく/ねをはる/バインド）を無視して実行できる。
- バトン先も設置技の効果を受ける。

if baton_data is not None:
    # ランク引き継ぎ（クリアボディ等を経由しない直接代入）

    # volatile 引き継ぎ
    for volatile_name, v_data in baton_data["volatiles"].items():
        count = v_data.get("count")

        self.battle.volatile_manager.apply(new, volatile_name, count=count, **kwargs)

### 3. `src/jpoke/handlers/move_status.py`

五十音順で「バ（ば）」は「は」行に属するため、`はきだす_check_can_use` より前に挿入する。
モジュール先頭付近のモジュールレベル定数として `_BATON_PASS_VOLATILES` を追加する。

```python
# バトンタッチで交代先に引き継ぐ揮発性状態の名前セット
_BATON_PASS_VOLATILES: frozenset[str] = frozenset({
    "みがわり",
    "こんらん",
    "きゅうしょアップ",
    "ちいさくなる",
    "まるくなる",
    "アクアリング",
    "ねをはる",
    "やどりぎのタネ",
    "じゅうでん",
    "でんじふゆう",
    "ちょうはつ",
    "しおづけ",
    "たくわえる",
})


def バトンタッチ_check(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """バトンタッチの失敗条件チェック: 交代できる控えのポケモンがいない場合は失敗する。"""
    mon = ctx.attacker
    player = battle.get_player(mon)
    if not battle.get_available_switch_commands(player):
        battle.add_event_log(
            mon,
            LogCode.MOVE_FAILED,
            payload={"reason": "バトンタッチ_交代不可"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def バトンタッチ_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """バトンタッチの効果: ランク・揮発性状態を保存し、ピボット交代する。

    能力ランク変化（プラス・マイナス両方）と一部の揮発性状態を交代先に引き継ぐ。
    クリアボディ等のハンドラを経由させないため、ランクは直接代入で引き継ぐ。
    """
    mon = ctx.attacker
    player = battle.get_player(mon)

    # 退場時にランクがリセットされるため、事前にコピーを作成
    rank_copy = dict(mon.rank)

    # 引き継ぎ対象の volatile をスナップショット
    volatile_copy: dict[str, dict] = {}
    for name, v in mon.volatiles.items():
        if name not in _BATON_PASS_VOLATILES:
            continue
        v_data: dict = {}
        if v.count is not None:
            v_data["count"] = v.count
        if name == "みがわり" and v.hp is not None:
            v_data["hp"] = v.hp
        volatile_copy[name] = v_data

    # PlayerState に引き継ぎデータを格納
    battle.player_states[player].baton_pass_data = {
        "rank": rank_copy,
        "volatiles": volatile_copy,
    }

    # ピボット交代（交代先の選択をプレイヤーに委ねる）
    battle.player_states[player].interrupt = Interrupt.PIVOT
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

1. **交代不可チェックの判定関数**
   - バトンタッチはとらわれ状態を無視して交代できるため、`battle.can_switch(player)` ではなく
     `battle.get_available_switch_commands(player)` を使う必要がある。
   - `can_switch` は `is_trapped` チェックを含むが、`get_available_switch_commands` は
     生存している控えポケモンの有無のみ確認する（実装を確認の上、挙動に差異があれば調整する）。

2. **ランクの直接代入**
   - `switch_manager` でのランク適用は `new.rank[stat] = v` で直接代入する。
   - `battle.modify_stats` を経由するとクリアボディ・あまのじゃく等が発動してしまうため不可。

3. **volatile 適用タイミングと設置技**
   - `baton_pass_data` の適用は `_process_events_after_switch`（設置技発火を含む ON_SWITCH_IN）の前に行う。
   - そのため、引き継いだ `でんじふゆう` や `ねをはる` は ON_SWITCH_IN 発火時にはまだ存在する。
   - ねをはる状態を引き継いだ場合、バトン先は地面にいるものとして設置技を受ける（仕様通り）。

4. **みがわりの残HP**
   - みがわりを引き継ぐ際は `Volatile.hp` もコピーして渡す。
   - `volatile_manager.apply(new, "みがわり", hp=残HP)` で復元する。
   - バトン先の最大HPにかかわらず同じ残HPで引き継がれる。

5. **`inherit_migawari` との分離**
   - バトンタッチはみがわりを `baton_pass_data["volatiles"]` 経由で引き継ぐ。
   - `inherit_migawari = True` は設定しない（両方設定するとみがわりが二重付与されるバグになる）。

6. **ネガティブランクの引き継ぎ**
   - マイナスのランクもそのまま引き継がれる。
   - バトン先がしろいハーブを持つ場合は `ON_SWITCH_IN` で自然に発動する。
   - バトン先がだっしゅつパックを持つ場合は発動しない（仕様）。だっしゅつパック側のハンドラが判断する。

7. **バトンタッチ連鎖**
   - バトン先もバトンタッチを使える場合、引き継がれたランク・volatile はさらに次の交代先に伝播する。
   - `baton_pass_data` を再度格納して交代を繰り返すだけで自然に動作する。

8. **`volatile_manager.apply` の重複チェック**
   - `volatile_manager.apply` は既にその volatile が存在する場合に失敗する（False を返す）。
   - 引き継ぎ時は退場（全 volatile クリア）→ 入場後に適用するため、重複付与は起こらない。

9. **とくせいなし引き継ぎは対象外**
   - `とくせいなし` volatile は `_BATON_PASS_VOLATILES` に含めない。
   - 仕様上引き継いだ場合、バトン先が場に出たときに発動する特性も発動しないが、
     これは他の volatile との複雑な相互作用を生む可能性があり、初期実装では対象外とする。

10. **VolatileData の `forced` フラグ**
    - `あばれる`・`そらをとぶ`・`あなをほる`・`シャドーダイブ`・`ダイビング` 等は
      `_BATON_PASS_VOLATILES` に含まれないため、自動的に引き継ぎ対象外となる。
