# じんらい 実装計画書

## 仕様要約

- タイプ: でんき、分類: 特殊、威力: 70、命中: 100、PP: 5、優先度: +1、直接攻撃: ×
- **発動条件**: 対象がそのターンに「ダメージを与える攻撃技（物理または特殊）」を選択していること
- **失敗条件（以下のいずれか）**:
  - 対象がそのターンに変化技を選択している
  - 対象が技以外の行動（交代・テラスタル・どうぐ使用など）を選択している
  - 対象がすでに行動済み（先行）でコマンドが消費されている
- タイプ・分類・接触判定を除けば **ふいうち** と全く同じ発動条件を持つ（一次情報より）。

## 現状の実装状況

`data/move.py` に `MoveData` が骨格のみ定義済みで `handlers={}`（追加効果なしのコメント）となっており、
**発動条件チェックが一切実装されていない**（対象が変化技を選択していてもダメージを与えてしまうバグ）。

```python
"じんらい": MoveData(
    type="でんき",
    category="special",
    pp=5,
    power=70,
    accuracy=100,
    priority=1,
    handlers={},  # 追加効果なし
),
```

## ハンドラ構成

| イベント | Priority | subject_spec | 役割 |
|----------|----------|-------------|------|
| `Event.ON_TRY_MOVE_1` | 30 | `"attacker:self"`（デフォルト） | 失敗チェック: 対象行動済みまたは攻撃技未選択なら失敗 |

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` セクションに以下の記載がある：

> | 30 | ふいうち: 対象行動済み・変化技選択 |

じんらいはふいうちと同一の発動条件を持つため、同じ `priority=30` を使用する。

## 参照した類似実装

### ふいうち（`handlers/move_attack.py`）

ふいうちは既に `_ふいうち_can_apply` / `ふいうち_try_move` として同種の判定ロジックを実装済みである。
一次情報（Wiki）に「タイプ・分類・接触判定を除けばふいうちと同様の効果を持つ」と明記されているため、
判定ロジックはふいうちと共有する（新規に複製しない）。

- 共有する判定関数を `_ふいうち_can_apply` → `_ふいうち系_can_apply` にリネームし、
  ふいうち・じんらい両方の `_try_move` ハンドラから呼び出す。

## 実装箇所

### `src/jpoke/handlers/move_attack.py`

1. `_ふいうち_can_apply` を `_ふいうち系_can_apply` にリネームする（ふいうち・じんらいで共有するため）。
2. 「じ」の位置（`じんつうりき_apply_flinch` の直後、`すいとる_drain` の前）に
   `じんらい_try_move` を追加する。

```python
def じんらい_try_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """じんらいの発動条件を判定する。

    タイプ・分類・接触判定を除けばふいうちと同様の効果を持つため、
    判定ロジックを共有する（_ふいうち系_can_apply）。
    対象が未行動かつ攻撃技を選択している時のみ成功する。
    """
    if not _ふいうち系_can_apply(battle, ctx):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="じんらい")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### でんきタイプ関連特性との相互作用

- `_ふいうち系_can_apply` は `ON_TRY_MOVE_1`（priority=30）で発火し、
  ちくでん・でんきエンジン・ひらいしんの無効化ハンドラは `Event.ON_BEFORE_APPLY_MOVE` で発火する。
  イベント発火順は `ON_TRY_MOVE_1 → ON_TRY_MOVE_2 → ON_BEFORE_APPLY_MOVE` のため、
  じんらいの失敗判定は特性による無効化より必ず先に評価される。
  → 技の仕様による失敗が優先され、失敗時は相手の特性が発動しない、という一次情報の記載と一致する。
  特別な追加実装は不要（既存のイベント順序で自然に満たされる）。
- ひらいしんによる対象変更（リダイレクト）は目標選択フェーズで解決され、`ctx.defender` は
  リダイレクト後のポケモンを指す。したがって `_ふいうち系_can_apply` はリダイレクト後の
  ポケモンの行動をそのまま参照すればよく、追加のリダイレクト処理は不要。
