# 計画書: ねむる

## 仕様要約

- HPと状態異常をすべて回復した後、2ターンの間ねむり状態になる（3ターン目に必ず目を覚ます）
- タイプ: エスパー、変化、PP: 8（championsルールにより8/12/16/20の4段階へ統一。
  `docs/champions/move_list.txt` 準拠。Gen9本家は5）、対象: 自分、命中: -（命中判定なし）
- 回復技（`heal` ラベル）
- `MoveData.target="self"` を明示する（未設定だと既定値 `"foe"` になり、相手の
  まもる／マジックコートで誤って防がれる／跳ね返される）

出典: `docs/spec/moves/ねむる.md`

---

## 失敗条件

`docs/spec/turn.md` を踏まえ、以下の失敗条件が存在する。

| 優先度 | 失敗条件 | 処理場所 |
|---|---|---|
| ON_TRY_MOVE_1 Priority 30 | HP が満タン（最大HP = 現在HP） | `ねむる_check` ハンドラ内 |
| ON_TRY_MOVE_1 Priority 30 | すでにねむり状態（`mon.ailment.name == "ねむり"`） | `ねむる_check` ハンドラ内 |
| ON_TRY_MOVE_1 Priority 30 | エレキフィールド下で接地している | `ねむる_check` ハンドラ内 |
| ON_TRY_MOVE_1 Priority 30 | ミストフィールド下で接地している | `ねむる_check` ハンドラ内 |
| ON_TRY_MOVE_1 Priority 30 | 場に（自分・相手いずれかを問わず）さわぐ状態のポケモンがいる | `ねむる_check` ハンドラ内（`さわぐ`/`さわがしい` volatile の有無を判定） |
| ON_BEFORE_APPLY_MOVE Priority 100 | HP 満タン・すでにねむり状態の再チェック | `ねむる_check_apply` ハンドラ内（`ねむる_check` を再利用） |
| ON_STATUS_HIT | 特性（やるき・ふみん・スイートベール・きよめのしお・リミットシールド・
  リーフガード等）によるねむり付与不可、ゆめうつつ（uncurable）による上書き不可 | `ねむる_apply` 内の `ailment_manager.apply()` 呼び出しが `False` を返した場合 |

以下は ailment_manager 内の `ON_BEFORE_APPLY_AILMENT` で自動処理されるため、ハンドラ側での個別実装は不要:

- やるき・ふみん・スイートベール・きよめのしお特性による無効化
  （`prevent_sleep_ailment` / `きよめのしお_prevent_ailment` が処理済み）
- リミットシールドのりゅうせいのすがたによる無効化（`リミットシールド_prevent_ailment` が処理済み）
- リーフガード特性＋にほんばれ／おおひでりによる無効化（`リーフガード_prevent_ailment` が処理済み）
- しんぴのまもり・フラワーベールによる無効化は自分自身への付与を防がない
  （`しんぴのまもり_prevent_ailment` は `source != target` のときのみ発動するため、自己付与技である
  ねむるでは無効化されない。仕様通り）
- ゆめうつつ（uncurable フラグ）による上書き不可は `ailment_manager.apply()` 自体が判定する

**設計上の重要な注意（過去の不具合と修正方針）**:
特性による無効化はすべて `Event.ON_BEFORE_APPLY_AILMENT` の中で実行時に判定される
（かたやぶり・かがくへんかガスの無効化判定を含む正規の経路）。もし `ねむる_apply` で
「HP回復 → 状態異常解除 → ねむり付与」という順序で実装すると、ねむり付与だけが特性で
ブロックされてもHP回復・状態異常解除はすでに完了してしまっており、**対象ポケモンが
「ねむらずに全回復だけする」という誤った結果になる**（実際にやるき・ふみん・きよめのしお・
リミットシールド・リーフガード・ゆめうつつ（ぜったいねむり）でこの不具合を確認した）。
これを避けるため、`ねむる_apply` では **ねむり付与を先に試行し、成功した場合のみHPを
回復する** 順序に統一する。

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_TRY_MOVE_1` | 30 | `attacker:self` (省略) | HP満タン・すでにねむり状態・エレキ/ミストフィールド・さわぐ状態の失敗チェック |
| `ON_BEFORE_APPLY_MOVE` | 100 | `attacker:self` (省略) | turn.md「ステータス無効化」対応（HP満タン・すでにねむり状態の重複チェック） |
| `ON_STATUS_HIT` | 100 (省略) | `attacker:self` (省略) | ねむり付与を先に試行（count=3, overwrite=True）→成功時のみHP全回復 |

## Priority 根拠

- `ON_TRY_MOVE_1 Priority 30`: `docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` 表に「ねむる: 失敗条件多数」として Priority 30 に明記されている
- `ON_BEFORE_APPLY_MOVE Priority 100`: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 表に「ねむる: ステータス無効化」として Priority 100 に明記されている（実装コード上は `ON_BEFORE_APPLY_MOVE`）。
  同じ優先度に「自己回復技: HP満タン」という同型の再チェックパターンが並んでいるため、
  ここでは HP満タン・すでにねむり状態の再チェックのみを行う（特性による無効化判定は
  `ON_STATUS_HIT` 内の実際の `ailment_manager.apply()` 呼び出しに委ねる）
- `ON_STATUS_HIT Priority 100`: 既存の `じこさいせい_heal_self`・`なまける_heal_self` が priority=100（デフォルト）で登録されているため同一とする

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: `ねむりごな_apply_ailment_to_defender` の後、`ねばねばネット_set_side_field` の前（「ね」の中で「ねむりごな」の後）

```python
def ねむる_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねむるの効果: 3ターンのねむり付与に成功した場合のみHPと状態異常を全回復する。

    さわぐ・特性（やるき・ふみん・スイートベール・きよめのしお・リミットシールド等）・
    にほんばれ下のリーフガードなど、ON_BEFORE_APPLY_AILMENT で寝られないと判定される
    ケースでは、状態異常の上書き付与自体が失敗する。その場合は技全体を失敗させ、
    HPも回復しない（「ステータス無効化」時にHPだけ回復してしまう不具合を避けるため、
    ねむり付与を先に試行してから回復するという順序にしている）。

    Notes:
        Champions仕様: ねむるのカウントは3固定（Wiki「ねむるによってねむり状態になったときは
        カウントは3で固定」）。説明文「技ねむるでは3度目まで回復しない」と一致。
    """
    mon = ctx.attacker
    # ねむり付与を先に試行（count=3固定、既存の状態異常は上書き）
    if not battle.ailment_manager.apply(mon, "ねむり", count=3, source=mon, overwrite=True):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ねむる"),
        )
        return HandlerReturn(value=False, stop_event=True)
    # ねむり付与に成功した場合のみHP全回復
    battle.modify_hp(mon, v=mon.max_hp - mon.hp)
    return HandlerReturn(value=value)


def ねむる_check(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねむるの失敗条件チェック。

    以下のいずれかに該当する場合は失敗する:
    - HP が最大HP（すでに満タン）
    - すでにねむり状態
    - エレキフィールド下で接地している
    - ミストフィールド下で接地している
    - さわぐ・さわがしい状態（場にさわぐ状態のポケモンがいる）

    特性（やるき・ふみん・スイートベール・きよめのしお・リミットシールド等）や
    にほんばれ下のリーフガードによる無効化は、ここでは判定せず ねむる_apply 内の
    ailment_manager.apply（ON_BEFORE_APPLY_AILMENT）に委ねる
    （かたやぶり等の無効化判定を含む正規の特性判定経路を再利用するため）。
    """
    mon = ctx.attacker
    if mon.hp == mon.max_hp or mon.has_ailment("ねむり"):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ねむる"),
        )
        return HandlerReturn(value=False, stop_event=True)
    # エレキフィールド下で接地しているポケモンのねむるは失敗する
    if (
        battle.terrain.name == "エレキフィールド"
        and not battle.query.is_floating(mon)
    ):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="エレキフィールド"),
        )
        return HandlerReturn(value=False, stop_event=True)
    # ミストフィールド下で接地しているポケモンのねむるは失敗する
    if (
        battle.terrain.name == "ミストフィールド"
        and not battle.query.is_floating(mon)
    ):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ミストフィールド"),
        )
        return HandlerReturn(value=False, stop_event=True)
    # 場にさわぐ状態のポケモンがいるときのねむるは失敗する
    if mon.has_volatile("さわぐ") or mon.has_volatile("さわがしい"):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="さわぐ"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ねむる_check_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねむるのステータス無効化チェック（ON_BEFORE_APPLY_MOVE）。

    HP 満タン・ねむり状態を再チェックする。ねむる_check と同条件。
    """
    return ねむる_check(battle, ctx, value)
```

## 注意点・エッジケース

- **`target="self"` の明示が必須**: `MoveData.target` の既定値は `"foe"` であり、これを明示しないと
  `Move.is_blocked_by_protect` / `Move.is_reflectable` が誤って `True` を返し、相手のまもるで防がれたり
  マジックコートで跳ね返されたりする（実際にレビュー時にこの不具合を検出・修正した）。
- **ねむり付与を先に試行する順序**: 「HP回復 → 状態異常解除 → ねむり付与」の順で実装すると、
  特性等によるねむり付与失敗時にHPだけ回復してしまう不具合が生じる。`ねむる_apply` では
  `ailment_manager.apply(mon, "ねむり", count=3, source=mon, overwrite=True)` を先に呼び、
  戻り値が `False` の場合は技全体を失敗させてHPを回復しない。
- **count=3 の意味**: `ailment.py` の `ねむり_check_action` では `battle.ailment_manager.tick(mon)` を
  行動前に呼ぶため、count=3 で開始すると 1ターン目・2ターン目は行動不能（tick後 count=2→1）、
  3ターン目の tick で count=0 となり目を覚ます。`docs/champions/move_list.txt` の
  「2ターンの間『ねむり』状態になり、3ターン目で目を覚ます」と一致する。
- **さわぐ・さわがしい状態でのねむり無効**: `さわぐ` volatile はさわぐ使用者自身のねむりを、
  自動的に相手へ付与される `さわがしい` volatile が相手側のねむりを防ぐ（シングルバトル限定の
  ペア実装）。`mon.has_volatile("さわぐ") or mon.has_volatile("さわがしい")` で両方を判定すれば、
  自分がさわぐ使用者か、相手がさわぐ使用者かのいずれのケースもカバーできる。
- **しんぴのまもり・フラワーベールによる無効化**: `しんぴのまもり_prevent_ailment` は
  `ctx.source != ctx.target` のときのみ発動するため、自己付与技であるねむるでは無効化されない
  （一次情報通り、しんぴのまもり状態でもねむるは成功する）。
- **ゆめうつつ（ぜったいねむり特性）との相互作用**: ゆめうつつは `uncurable=True` のため、
  `ailment_manager.apply(..., overwrite=True)` を呼んでも上書きされず `False` が返る。
  結果としてねむるは技全体が失敗し、HPも回復しない（一次情報の「じょうかを受けたときは
  失敗する」と整合）。
- **かいふくふうじ状態での使用**: `labels=["heal"]`（実装上は `flags={"heal"}`）が付いているため、
  かいふくふうじ状態では技自体が使用不可（`かいふくふうじ_try_action` が汎用的に処理する）。
- **PPはchampions仕様で8**: `docs/champions/move_list.txt` 準拠。Gen9本家は5だが、
  championsでは全技PPが8/12/16/20の4段階に統一されているため8。
