# もえつきる 実装計画書

## 仕様要約（docs/spec/moves/もえつきる.md 参照）

* タイプ: ほのお / 分類: 特殊 / 威力: 130 / 命中: 100 / PP: 8
* 対象: 相手1体 / 優先度: 0 / 接触なし
* 効果:
  1. 自分が「ほのお」タイプでないと失敗する。
  2. 自分が「こおり」状態の場合、こおり状態を治してから攻撃する。
     ただし、自分がほのおタイプを持っていない（＝技が失敗する）場合はこおりを解凍しない。
  3. 命中してダメージを与えると、自分の「ほのお」タイプを無くす
     （テラスタル中は消失処理を無視するため、`Pokemon.types` の実装済み仕様に委ねられる）。

## 現状の実装状況（修正前）

`src/jpoke/data/moves/move_ma.py` に登録済みで、`もえつきる_fail_if_no_fire_type` /
`もえつきる_remove_fire_type` / `もえつきる_thaw_attacker` の3ハンドラが実装されている。
前2つは仕様と一致しているが、`もえつきる_thaw_attacker`（`src/jpoke/handlers/move_attack.py`）に
以下の誤りがある。

```python
def もえつきる_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """もえつきる: こおり状態でも使用可能にし、こおりを解凍する。
    ...
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

このハンドラは、自分が「ほのお」タイプかどうかを確認せずに無条件でこおりを解凍している。
しかし一次情報（ポケモンWiki）には次の記述がある：

> こおり状態のときでも使用可能で、使用すると自分のこおり状態を回復してから攻撃する。
> 自分がほのおタイプを持っていないときに使用した場合はこおり状態は回復しない。

`かえんぐるま` 等の通常のほのお技（`かえんぐるま_thaw_attacker` 等）はタイプ依存の失敗条件を
持たないため無条件解凍で正しいが、`もえつきる` は「ほのおタイプでないと技自体が失敗する」
専用技であるため、ハンドラ内でも `has_type("ほのお")` を条件に含める必要がある。

## ハンドラ構成（修正後）

| Event | Priority | subject_spec | 役割 |
|---|---|---|---|
| `Event.ON_TRY_ACTION` | 5 | （省略、デフォルト） | 自分がほのおタイプかつこおり状態のとき、こおりを解凍する |
| `Event.ON_TRY_MOVE_1` | 10 | `attacker:self` | 自分が「ほのお」タイプでない場合に技を失敗させる |
| `Event.ON_DAMAGE_HIT` | 180 | `attacker:self` | 命中後、自分の「ほのお」タイプを `removed_types` に追加して消失させる |

（`ON_TRY_MOVE_1` / `ON_DAMAGE_HIT` の priority は既存実装のまま変更なし。
`ON_TRY_ACTION` の priority=5 も既存実装のまま変更なし。修正対象は
`もえつきる_thaw_attacker` の条件分岐のみ。）

## Priority 根拠

* `docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` 表に
  `10 | ほのおタイプでないもえつきるの失敗` と明記されており、既存実装はこれと一致する。
* `docs/spec/turn.md` の `Event.ON_DAMAGE` 表に
  `180 | もえつきるによるタイプ消失` と明記されており、既存実装はこれと一致する。
* `ON_TRY_ACTION`（こおり解凍）の priority=5 は、`こおり_action`（priority=10、
  `src/jpoke/data/ailment.py`）より先に発火させることで、ailment 除去後の状態で
  `こおり_action` の判定がスキップされる既存パターン（`かえんぐるま_thaw_attacker` 等）を
  踏襲する。この部分は既存実装のまま変更しない。

## 実装箇所

### handlers/move_attack.py（修正）

```python
def もえつきる_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """もえつきる: ほのおタイプを持つ場合のみ、こおり状態でも使用可能にし、こおりを解凍する。

    自分がほのおタイプでない場合は技自体が失敗するため、こおり状態は解凍されない
    （ポケモンWiki: 「自分がほのおタイプを持っていないときに使用した場合はこおり状態は
    回復しない」）。
    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり" and mon.has_type("ほのお"):
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

`もえつきる_fail_if_no_fire_type` / `もえつきる_remove_fire_type` は変更なし。

### data/moves/move_ma.py

変更なし（既存の登録のまま）。

## 注意点・エッジケース（review-test 向け）

1. `has_type("ほのお")` はテラスタル込みの現在タイプで判定されるため、
   ほのおタイプにテラスタルしたポケモンがこおり状態で使用した場合は正しく解凍される。
2. `Pokemon.types` はテラスタル中 `removed_types` を無視する実装（`model/pokemon.py`）に
   なっているため、ほのおタイプにテラスタルした状態で成功しても実際のタイプは消えない
   （仕様書「テラスタル中はほのおタイプが消えない」を自動的に満たす）。
3. まもる・タイプ相性による無効化・命中判定失敗時は `ON_DAMAGE_HIT` に到達しないため、
   ほのおタイプは消えない（仕様と一致）。
4. Champions 固有の「コマンド選択時点でほのおタイプでなければ選択自体ができない」制限は、
   本プロジェクトが「コマンド選択の合法手フィルタリング」を実装対象外としているため見送る
   （`でんこうそうげき` の計画書と同様の判断）。
5. テストに「こおり状態 + ほのおタイプでないポケモン」の組み合わせを追加し、
   技が失敗し、かつこおり状態が解凍されないことを確認する（既存テストにはこのケースが無く、
   バグを検出できていなかった）。
