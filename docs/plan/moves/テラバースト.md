# テラバースト 実装計画書

## 仕様要約（docs/spec/moves/テラバースト.md 参照）

* タイプ: ノーマル（テラスタル時はテラスタイプに変化） / 分類: 特殊（テラスタル時は可変） /
  威力: 80（ステラテラスタル時 100） / 命中: 100 / PP: 10
* 対象: 相手1体 / 優先度: 0 / 直接攻撃なし
* 効果:
  1. テラスタルしている場合、技タイプが使用者のテラスタイプに変化する
     （テラスタイプがノーマルでも「ノーマル技のタイプを変える特性」の対象にはならない）。
  2. テラスタルしている場合、自分の実数値（ランク補正込み）がこうげき＞とくこうなら物理技になる
     （同値・特殊が高い場合は特殊技のまま）。てんねん所持の相手に対してもランクを考慮して分類を
     判定し、分類確定後のダメージ計算でランク無視が適用される（テラバーストにはかたやぶり効果が
     無いため、フォトンゲイザーとは異なる）。
  3. ステラタイプにテラスタルしたときのみ威力が100になり、命中すると自分のこうげき・とくこうが
     1段階ずつ下がる（あまのじゃく・たんじゅん等の反転・増幅は既存の汎用ランク変化ハンドラに委ねる）。
  4. ひこうタイプに変化してもはやてのつばさは発動しない
     （行動順決定＝`_resolve_action_order` はテラスタル実行フェーズ＝`_run_terastal_phase` より前に
     行われるため、優先度判定時点では `move.type` がまだ変化前のノーマルのままとなり、構造上自動的に
     この仕様を満たす。専用コードは不要）。
  5. 相手の場にプレッシャー所持ポケモンがいる場合はPP消費が増える（汎用のプレッシャー実装で
     自動的に満たされるため専用コードは不要）。

## ハンドラ構成

| Event | Priority | subject_spec | 役割 |
|---|---|---|---|
| `Event.ON_MODIFY_MOVE_TYPE` | 100（デフォルト） | `attacker:self` | テラスタル時、技タイプを使用者のテラスタイプに変更する |
| `Event.ON_MODIFY_MOVE_CATEGORY` | 100（デフォルト） | `attacker:self` | テラスタル時、こうげき＞とくこう（ランク補正込み）なら物理技にする |
| `Event.ON_CALC_POWER_MODIFIER` | 100（デフォルト） | `attacker:self` | ステラタイプテラスタル時、威力を100にする |
| `Event.ON_HIT` | 100（デフォルト） | `attacker:self` | ステラタイプテラスタル時、命中後に自分のこうげき・とくこうを1段階ずつ下げる |

## Priority 根拠

* `ON_MODIFY_MOVE_TYPE` / `ON_MODIFY_MOVE_CATEGORY` / `ON_CALC_POWER_MODIFIER` は
  `docs/spec/turn.md` に掲載が無いイベントのため、同種の既存実装を参照して決定した。
  `フォトンゲイザー_modify_move_category`（同じ「ランク込みの実数値比較で分類を切り替える」
  効果）が `priority` を指定せずデフォルト（100）で登録されているため、同様にデフォルトを採用する。
  同一ポケモンの技固有ハンドラは他ハンドラと競合しないため、優先度指定の必要が無い。
* `ON_HIT` は `docs/spec/turn.md` に「10: 自分のランクが下がる反動」と明記されているが、
  同じ「命中後に自分のランクを下げる」効果を持つ既存実装（`オーバーヒート_lower_attacker_spa`、
  `リーフストーム_sharply_lower_spa_C`）がいずれも `priority` 指定なし（デフォルト100）で
  登録されているため、既存パターンとの一貫性を優先しデフォルトを採用する。

## 実装箇所（既存実装済み）

### handlers/move_attack.py

```python
def テラバースト_modify_move_category(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テラバーストの分類（物理/特殊）を判定する。"""
    mon = ctx.attacker
    if mon.terastallized:
        atk = mon.ranked_stats["atk"]
        spa = mon.ranked_stats["spa"]
        value = "physical" if atk > spa else "special"
    return HandlerReturn(value=value)


def テラバースト_modify_move_type(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テラバーストのタイプを判定する。"""
    mon = ctx.attacker
    if mon.terastallized:
        value = mon.active_tera_type
    return HandlerReturn(value=value)


def テラバースト_stellar_power(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ステラテラスタル状態ではテラバーストの威力が100になる補正。"""
    if ctx.attacker.active_tera_type == 'ステラ':
        value = 5120  # = 4096 * 100 / 80
    return HandlerReturn(value=value)


def テラバースト_stellar_stat_drop(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ステラテラスタル時のテラバースト発動後の攻撃・特攻-1段階効果。"""
    mon = ctx.attacker
    if mon and mon.active_tera_type == 'ステラ':
        battle.modify_stats(mon, {"atk": -1, "spa": -1}, source=mon)
    return HandlerReturn(value=value)
```

## リーサル計算（新規実装）

ダメージ計算自体（タイプ・分類・ステラ威力補正）は `battle.calc_damages()` が通常の
`ON_MODIFY_MOVE_TYPE` / `ON_MODIFY_MOVE_CATEGORY` / `ON_CALC_POWER_MODIFIER` ハンドラを
そのまま経由するため、専用のリーサル実装は不要。ステラ時の自己ランクダウンのみ
`LethalEvent.ON_HIT` で追加実装する。

```python
def テラバースト_lower_attacker_atk_spa(battle: Battle, ctx: LethalContext, hp_dist: StateDist) -> StateDist:
    """テラバースト: ステラタイプにテラスタルして命中した場合、こうげき・とくこうを1段階ずつ下げる。"""
    if ctx.attacker.active_tera_type != "ステラ":
        return hp_dist
    ctx.attacker.rank["atk"] = clamp_stats(ctx.attacker.rank["atk"] - 1)
    ctx.attacker.rank["spa"] = clamp_stats(ctx.attacker.rank["spa"] - 1)
    return hp_dist
```

既存の `オーバーヒート_lower_attacker_spa` 等と同様、あまのじゃく・たんじゅんによる
反転・増幅はリーサル計算では考慮しない（他の自己ランクダウン技のリーサル実装と同じ簡略化方針）。

## 注意点・エッジケース（review-test 向け）

1. **PP**: `src/jpoke/data/ps-champ-ja/moves.json` にテラバースト・テラクラスターの行が存在しない
   （威力・分類・タイプが可変の技はテーブル形式の一次情報から除外されていると考えられる）。
   同種のテラクラスターも移行前の PP をそのまま維持しているため、テラバーストも PP=10（Gen9基準）
   を維持する。
2. **分類判定とてんねん**: `テラバースト_modify_move_category` は `ctx.attacker.ranked_stats` を
   参照するため、防御側のてんねん（`てんねん_ignore_rank`、ON_CALC_ATTACK系のランク無視ハンドラ）
   の影響を受けない。分類判定後のダメージ計算では、てんねん側のハンドラが通常通りランクを無視する
   ため、既存実装のままで仕様と一致する。
3. **ノーマルスキン等の「ノーマル技のタイプを変える特性」との競合**: 特性ハンドラ
   （`_skin_modify_move_type` 系）と `テラバースト_modify_move_type` はどちらも
   `ON_MODIFY_MOVE_TYPE` に `attacker:self` / デフォルト優先度で登録される。特性ハンドラは
   ポケモン登場時から登録済み、技ハンドラは技使用時に動的登録されるため、イベント発火時の
   ソート（`(priority, -speed)`、同キーは登録順で安定ソート）で特性ハンドラが先に実行され、
   `テラバースト_modify_move_type` が後から上書きする。そのためテラスタイプがノーマルの場合も
   仕様通り特性は無効化される（動作確認済み、専用コード不要）。
4. **はやてのつばさ**: 行動順決定がテラスタル実行より前に行われる仕様上の順序により、
   `はやてのつばさ_modify_priority` が参照する `ctx.move.type` はテラスタイプ変化前の値になるため、
   ひこうタイプに変化しても優先度は上昇しない（構造上自動的に満たされる）。
5. **プレッシャーによるPP消費増加**: `プレッシャー_extra_pp` は技非依存の汎用実装のため
   専用コード不要。テラバースト固有のテストは追加しない。
6. **まもる・マジックコート**: 通常の攻撃技と同じ扱いで、専用コード不要。
