# 実装計画書: さいはい

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 16（`docs/champions/move_list.txt` L308 準拠） / 命中: - / 対象: 1 体選択（シングルバトルでは相手固定）
- 対象（`ctx.defender`）が最後に PP を消費して使用した技（`executed_move`）を、対象自身の PP を消費して再度実行させる
- 使用者自身は行動を消費せずに対象へ技を使わせる（さいはい自体の行動が技の実行トリガーとなる）
- 命中判定なし、急所なし、追加効果なし、まもる: ○（まもる状態の対象には防がれる。`MoveData` に `unprotectable` フラグを付けないことで既存の汎用まもる判定に委譲する）
- マジックコート: × / よこどり: × / オウムがえし: ×
- 本プロジェクトはシングルバトル専用（`battle.actives` は常に2体以下）だが、さいはいは
  「対象のポケモンが直前に使用した技をもう一度使わせる」という効果自体が単体対象でも意味を持つ
  （相手に使えば相手の直前の技を、使用者自身に向けて再度撃たせることができる）。
  `docs/champions/move_list.txt` L309 の効果文も「相手が最後に使用した技をその場でもう一度使わせる」と
  明記しており、`ダブルバトル用` の注記が無い（さきおくり・てだすけ等の真のダブル専用技には明記されている）。
  そのため、進捗表の「ダブル専用」扱いは誤りであり、本レビューで実装対象に修正する。

### 失敗条件（`Event.ON_BEFORE_APPLY_MOVE` priority 100）

- 対象が場に出てから PP を消費する行動を一度もしていない（`executed_move is None`）
- 対象の `executed_move` が指示できない技（後述の `_INSTRUCT_BLOCKED_MOVES`）である場合
- 対象の `executed_move` の PP がすでに 0 の場合
  （0 のまま `battle.run_move` に渡すと、`move_executor.run_move` 内の
  `if move.pp == 0: move = Move("わるあがき")` によりわるあがきへ自動置換されてしまうため、
  さいはい側で事前に弾く必要がある）

### 指示後の挙動（`Event.ON_STATUS_HIT` priority 100）

- `battle.run_move(ctx.defender, move)` を呼び、対象自身を攻撃者としてその技を実行させる
- `move_executor.run_move` 内で `defender = self.battle.foe(attacker)` により、
  再実行時の対象（＝技の宛先）は自動的に「さいはいの使用者」になる
  （シングルバトルでは対象の相手は常にさいはいの使用者しかいないため、追加のターゲット指定処理は不要）
- 対象の PP 消費・状態異常/状態変化による行動失敗判定（ねむり・こんらん・まひ等）・
  ちょうはつ/かなしばり等による技封じは、`run_move` 内の通常フロー（`ON_TRY_ACTION` 等）にすべて委譲される
- 対象の PP が 0 の技を指示した場合は上記の失敗条件チェックで弾かれるため、
  `run_move` 側のわるあがき自動置換が発生することはない

#### さいはいのハンドラ関数

```python
def さいはい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """さいはいの失敗条件を判定する。

    - 相手が場に出てからPPを消費する行動を一度もしていない（executed_move が None）場合は失敗する
    - 相手の直前の技が指示できない技（_INSTRUCT_BLOCKED_MOVES）の場合は失敗する
    - 相手の直前の技のPPがすでに0の場合は失敗する
      （0のまま battle.run_move に渡すと わるあがき に自動置換されてしまうため、ここで弾く）
    """
    move = ctx.defender.executed_move
    if move is None or move.name in _INSTRUCT_BLOCKED_MOVES or move.pp <= 0:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="さいはい")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def さいはい_instruct(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """さいはいの効果: 相手が直前に使用した技を、相手のPPを消費してもう一度使わせる。

    battle.run_move(ctx.defender, move) により通常の技実行フローが走るため、
    状態異常・状態変化による行動失敗判定やPP消費、ちょうはつ等による技封じは
    通常の技実行と同様に処理される。

    さいはい自身の ON_BEFORE_APPLY_MOVE / ON_STATUS_HIT ハンドラは、この呼び出しが完了するまで
    イベントマネージャーに登録されたままになる。登録されたままだと、指示された技の実行中にも
    さいはいの使用者（ctx.attacker）に対してこれらのイベントが発火し、さいはい_can_apply が
    無関係な技（指示された技自身）に対して誤って再評価されてしまう
    （この時点で ctx.attacker.executed_move はすでにさいはい自身に更新済みのため、
    「さいはいはさいはいを指示できない」という自己参照チェックに誤って引っかかり、
    指示した技のPPだけ消費されて不発になってしまう）。これを避けるため、
    指示された技の実行中はさいはい自身の ON_BEFORE_APPLY_MOVE / ON_STATUS_HIT ハンドラを
    一時的に解除する（ねごと_select_and_execute と同じパターン）。
    """
    move = ctx.defender.executed_move
    suppressed_events = (Event.ON_BEFORE_APPLY_MOVE, Event.ON_STATUS_HIT)
    handlers_data = ctx.move.data.handlers
    for event in suppressed_events:
        handler = handlers_data.get(event)
        if handler is None:
            continue
        for h in (handler if isinstance(handler, list) else [handler]):
            battle.events.off(event, h, ctx.attacker)
    try:
        battle.run_move(ctx.defender, move)
    finally:
        for event in suppressed_events:
            handler = handlers_data.get(event)
            if handler is None:
                continue
            for h in (handler if isinstance(handler, list) else [handler]):
                battle.events.on(event, h, ctx.attacker)
    return HandlerReturn(value=value)
```

#### 実装時に発見した再帰呼び出しの不具合（発見・修正済み）

実装当初は `battle.run_move(ctx.defender, move)` を単純に呼ぶだけの実装だったが、これは
`MoveExecutor` のイベント発火が再帰時にも共有される問題を引き起こしていた。
`Event.ON_BEFORE_APPLY_MOVE` は物理/変化を問わずすべての技実行で発火するため、指示された技
（例: たいあたり）の実行中にも `さいはい_can_apply` が再度呼ばれてしまう。このとき
`ctx.defender`（＝再帰呼び出し内での「たいあたりの対象」＝さいはいの使用者自身）の
`executed_move` は、さいはい自身の実行開始時点（`move_executor.py` の
`ctx.attacker.executed_move = ctx.move` 代入）ですでに「さいはい」に更新済みであるため、
`_INSTRUCT_BLOCKED_MOVES` の自己参照チェック（`"さいはい"` を含む）に誤って合致し、
指示した技（たいあたり）のPPだけ消費されて不発になる不具合が生じていた
（`ねごと_select_and_execute` の docstring に記載されている既知のパターンと同種の問題）。
`ねごと_select_and_execute` と同じ「実行中は自身のハンドラを一時的に解除する」パターンを
適用して修正した。

## 注意点・エッジケース

### `executed_move` の保持と判定

- `Pokemon.executed_move` は `Move` オブジェクトへの参照。ターン内で技が成功した時点
  （`move_executor.py` の `ctx.attacker.executed_move = ctx.move` 代入箇所）に設定される
- 交代時（`reset_on_switch_out` 相当の処理）で `None` にリセットされるため、
  場に出たばかりのポケモンには必ず失敗する
- `executed_move` は「PP を消費した技」である。`ON_TRY_ACTION` で行動失敗した場合
  （まひ、こんらん等）は PP を消費しないため `executed_move` は更新されない。
  仕様書の「PP 消費を伴う行動を一度もしていないとき」に合致する

### PP チェックと消費

- 仕様書: 「指示された技の PP が無くなった技を指示したときは失敗する」
  → `さいはい_can_apply` で `move.pp <= 0` をチェックする
- `battle.run_move(ctx.defender, move)` 内で `_consume_pp` が呼ばれ、対象の PP が 1 消費される。
  さいはい自身の PP も別途消費されるため、1 ターンに合計 2 つの技の PP が消費されることになる

### `run_move` による再実行の挙動

- `battle.run_move(ctx.defender, move)` は `move_executor.run_move` に委譲され、通常の技実行フローが走る。
  これにより、ねむり・こんらん・まひ等の状態異常判定（`ON_TRY_ACTION`）、ちょうはつ・かなしばり等の
  技封じ判定、場に出て最初の行動でしか成功しない技（ねこだまし等）の失敗判定も、
  すべて対象ポケモンに対して自動的に再実行される（さいはい側で個別に実装する必要はない）
- `defender = self.battle.foe(attacker)` により、再実行時の宛先は自動的に「さいはいの使用者」になる。
  シングルバトルでは対象は常に使用者 1 体のみのため、ダブルバトルのような複雑なターゲット位置管理は
  不要（`docs/wiki/moves/さいはい.html` の「さいはいで出された技」節にある注目の的/よびみず/
  ひらいしんによる引き寄せ位置の再現、アンコールでランダム選択された位置の再現等は、
  ダブル/トリプルバトル特有の複数対象選択に関する仕様であり、本プロジェクトの対象外）

### 指示できない技リストの管理

- `_INSTRUCT_BLOCKED_MOVES` に収録されていない技は指示できる
  （まもる自体は「連続使用で成功率が下がる」だけで指示不可ではないため、リストに含めない）
- 技名の直接比較で判定する（`labels`/`flags` に汎用の「指示不可」フラグが存在しないため。
  `non_encore` フラグはアンコールの対象技セットが異なり流用できない）
- ソーラービーム/ソーラーブレードは「にほんばれ状態のときも失敗する」仕様があるが、
  溜め技として名前で一律ブロックしているため、天候に関わらず正しく失敗する

### まもる・よこどり・マジックコート

- まもる状態の対象には防がれる: `MoveData` に `unprotectable` フラグを付与しないことで、
  既存の汎用まもる判定（`ON_TRY_MOVE_1` 相当）に委譲する。さいはいハンドラ側での対処は不要
- よこどり・マジックコート: さいはいは対象外（既存の汎用判定が変化技全般に対して行われる中で、
  さいはい固有のフラグ付与は不要。既存の他の変化技と同様の扱いで問題ない）

### `ON_STATUS_HIT` の `value` 戻り値

- `battle.run_move` は戻り値を返さない（`None`）。`HandlerReturn(value=value)` で
  元の `value`（`True`）を維持して返す

## リーサル計算

- 対象外（`n/a`）。さいはいは相手にダメージを与えず、また相手の技を再実行させても
  その宛先は「さいはいの使用者自身」になるため、相手（`defender`）の HP 分布には影響しない。
  `docs/progress/move.md` の他の非ダメージ変化技（さいみんじゅつ等）と同様に `n/a` とする

## 対象外とした仕様（実装スコープ外）

以下は一次情報に記載があるが、シングルバトル専用の本プロジェクトでは再現不能・再現不要と判断し、
実装しない：

- マックスレイドバトル/テラレイドバトルでの失敗（Dynamax/レイドバトル自体が本プロジェクトの対象外）
- Zワザ/ダイマックスわざの失敗（両システムとも本プロジェクトの対象外）
- 「さいはいの指示で技を出したが自身の行動を終えていないポケモン」への
  しっぺがえし威力2倍・エラがみ/でんげきくちばし威力不変・コアパニッシャーでの特性消失・
  ふいうち/さきおくり/さきどり/そうでんの失敗などの特殊相互作用
  （対象追跡用の専用フラグが必要になる非常にニッチな相互作用のため見送り）
- 特性アナライズの発動タイミング調整（「さいはいを受けて行動したときのみ発動」等）
- りんしょうの行動順繰り上げ（りんしょう自体が本プロジェクトで未実装）
- ソード・シールド初期バージョンの不具合、通信対戦開始時のバグ（過去バージョンの既知バグは対象外）
