# 計画書: ハバネロエキス

## 仕様要約

- くさタイプ / 変化 / 必中 / PP16（`docs/champions/move_list.txt` 準拠）/ 対象: 1体選択
- 相手の『こうげき』ランクを2段階上げる
- 相手の『ぼうぎょ』ランクを2段階下げる
- 両方とも境界（こうげき+6・ぼうぎょ-6）で変化不可の場合のみ技全体が失敗する

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_STATUS_HIT` | 100（デフォルト） | attacker:self（省略） | こうげき+2・ぼうぎょ-2 を1回の `modify_stats` にまとめて適用 |

## Priority 根拠

`docs/spec/turn.md` の `ON_STATUS_HIT` セクションにハバネロエキスの明示的な記載はない。ただし同種の変化技（いばる・おだてる・くすぐる・フェザーダンス等）はすべて `ON_STATUS_HIT` priority 100 のデフォルト値を使用しており、本技も同パターンを採用する。

## 実装箇所

### handlers/move_status.py

`hs.modify_defender_stats` ユーティリティ（`handlers/move.py`）を利用し、2つのランク変化を
1回の `battle.modify_stats` 呼び出しにまとめる。これにより：

- しろいハーブ・クリアボディ等のランク変化検知系アイテム・特性が正しく1回のイベントで反応する
- 両方とも境界に達していて変化できない場合、`modify_stats` の戻り値が空dict（偽値）になり、
  `modify_defender_stats` がそれをそのまま `HandlerReturn.value` として返すことで、
  `move_executor._execute_status_hit` が技全体を失敗と判定する
- 片方のみ境界に達している場合は、変化可能な方のみ適用され `actual_changes` が非空になるため
  技は成功する

```python
def ハバネロエキス_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ハバネロエキスの効果: 相手のこうげきを2段階上げ、ぼうぎょを2段階下げる。

    2つのランク変化を1回のmodify_stats呼び出しにまとめることで、こうげき・ぼうぎょの
    どちらも変化できない場合（こうげき+6かつぼうぎょ-6）に戻り値が空dict（偽値）になり、
    技全体が失敗として扱われる。片方のみ変化可能な場合はそちらのみ適用され技は成功する。
    """
    return modify_defender_stats(battle, ctx, value, stats={"atk": 2, "def": -2})
```

五十音順の挿入位置: `はらだいこ` 関連の前、`はねる` の後（は行内、五十音順で「はねる」<「ハバネロ」<「はらだいこ」）。

### data/moves/move_ha.py

```python
    "ハバネロエキス": MoveData(
        type="くさ",
        category="status",
        pp=16,
        handlers={
            Event.ON_STATUS_HIT: h.MoveHandler(
                hs.ハバネロエキス_apply,
            ),
        }
    ),
```

`accuracy` は必中のため指定しない（省略時 `None` で命中判定をバイパスする。`あくび`と
同じパターン）。

## 注意点・エッジケース

- こうげきとぼうぎょのランク変化は `modify_stats` に1回でまとめて渡すことで、境界判定と
  技全体の成否判定が自動的に整合する（`おたけび`・`おきみやげ`等の複数ランク同時変更技と
  同じ実装パターン）。2回に分けて `modify_stats` を呼ぶ実装は、しろいハーブ等の複数ランク
  変化検知系の効果が2回に分かれて発火してしまう・技全体の失敗判定が行われない、という
  問題があるため避けること。
- みがわり状態の相手に対するランク変化はフレームワーク側で無効化されるため、ハンドラ内の
  追加ガードは不要
- クリアボディ・しろいけむり等によるランク低下無効化も `modify_stats` 内で処理されるため、
  ハンドラ内ガード不要
