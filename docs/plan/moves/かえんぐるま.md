# かえんぐるま 実装計画書

参照: docs/spec/moves/かえんぐるま.md

## 現状

* `src/jpoke/data/move.py` の `"かえんぐるま"` エントリは以下の通り実装済み。
  - type="ほのお", category="physical", pp=25, power=60, accuracy=100
  - flags={"contact", "secondary_effect", "thaw"}
  - `Event.ON_DAMAGE_HIT`: `かえんぐるま_apply_burn_to_defender`（10%でやけど付与）
* 上記は仕様と一致しており修正不要。
* 一方、仕様書に記載した「使用者がこおり状態の場合、こおり状態を治してから攻撃する」効果は
  未実装。`flags` の `"thaw"` は現状 `こおり_cure_by_thaw_move`（`Event.ON_DAMAGE_HIT`,
  被弾側＝defender)にのみ使われており、使用者自身の無条件解凍処理は個別実装が必要。

## 追加実装

### ハンドラ

`src/jpoke/handlers/move_attack.py` に `かえんぐるま_thaw_attacker` を追加する。
`もえつきる_thaw_attacker`（同ファイル内、Event.ON_TRY_ACTION, priority=5）と全く同じパターンで実装する。

```python
def かえんぐるま_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """かえんぐるま: こおり状態でも使用可能にし、こおりを解凍する。

    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

### 登録

`src/jpoke/data/move.py` の `"かえんぐるま"` エントリに以下を追加する。

```python
Event.ON_TRY_ACTION: h.MoveHandler(
    ha.かえんぐるま_thaw_attacker,
    priority=5,
),
```

### priority の根拠

* `docs/spec/turn.md` の `Event.ON_CHECK_ACTION`（コード上は `Event.ON_TRY_ACTION`）節には
  `10 | こおりの回復判定` と `170 | こおりを回復するわざでこおり回復` の2行がある。
* しかし `170` は `10`（`こおり_action`）より後に実行される（`_sort_handlers` は priority 昇順）。
  `こおり_action` は priority=10 で通常の25%解凍抽選を行い、解凍しなければ
  `stop_event=True` で即座にイベントを打ち切るため、priority=170 のハンドラは
  「25%抽選で解凍しなかった場合」には実行されない。これでは「無条件解凍」という
  仕様（フレアドライブ・もえつきる等と同様）を満たせない。
* 既存の同種実装 `もえつきる_thaw_attacker` は priority=5（`こおり_action` の10より前）で
  登録されており、`こおり_action` が発火する前に ailment を除去することで
  `_check_handler_validity` の `ailment` ソースチェック（`subject.ailment.is_active`）により
  `こおり_action` 自体がスキップされる、という設計になっている。
  実際に `tests/moves_attack/test_move_ma.py::test_もえつきる_こおり状態で使うと解凍されて攻撃できる`
  でこの動作が検証済みであり、正しく機能する実装パターンであることを確認した。
* 以上より、`docs/spec/turn.md` に記載の priority=170 は本ケースでは機能しないため、
  既存の同種ハンドラ（もえつきる）に倣い priority=5 を採用する。
