## 実装箇所

### `src/jpoke/handlers/move_attack.py` に追加する関数

五十音順の位置（ア行・「アイアンローラー」は現在存在するアイアンテール等より後、アイスハンマーの前に配置）。

```python
def アイアンローラー_check_terrain(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """アイアンローラーの使用可否: フィールドが存在しない場合に失敗させる。"""
    if not battle.terrain.is_active:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload={"reason": "アイアンローラー_フィールドなし"}
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def アイアンローラー_clear_terrain(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """アイアンローラー: ダメージ後にフィールドを解除する。"""
    battle.terrain_manager.remove()
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### フィールド存在チェック
- `battle.terrain.is_active` が `False`（フィールドなし）のとき、`ON_TRY_MOVE_1` で `HandlerReturn(value=False, stop_event=True)` を返して失敗させる
- フィールドが存在すれば命中・ダメージ計算は通常通り進む

### フィールド解除のタイミング
- turn.md に「ON_DAMAGE priority 180（= 実装上は `ON_DAMAGE_HIT` イベント）」と明記されているため、`ON_HIT` ではなく `ON_DAMAGE_HIT` を使う
- `terrain_manager.remove()` はフィールドが存在しない場合 `False` を返すが副作用はないため、ON_TRY_MOVE_1 で通過を保証済みのコンテキストでは問題ない

### みがわりとの相互作用
- `ON_DAMAGE_HIT` はみがわりに被弾しても発動する（turn.md「ON_HIT: ダメージ発生後の処理（みがわりに被弾しても発動）」の記述が `ON_HIT` 側にあるが、`ON_DAMAGE_HIT` はダメージ付与後のイベントであり、フィールド解除は対象がみがわりであっても実行される）
- ただし、実際のみがわり処理との関係は既存ハンドラの動作に準じる

### アイスピナーとの差異
- アイスピナーはフィールドが存在しなくても使用できる（フィールド解除はヒット時のみ）
- アイアンローラーはフィールドが存在しない場合に「失敗」する点が決定的な差異

### アイスピナーの実装不足
- `アイスピナー` の MoveData にもフィールド解除の `ON_DAMAGE_HIT` ハンドラが未実装である
- アイアンローラーとは異なり `ON_TRY_MOVE_1` の失敗チェックは不要

### フィールドが複数種ある場合
- ポケモンチャンピオンズはシングルバトルのみを対象とし、フィールドは `TerrainManager`（排他的管理）なので同時に複数フィールドが存在することはない
- `terrain_manager.remove()` で現在アクティブな1つのフィールドを解除するだけで十分
