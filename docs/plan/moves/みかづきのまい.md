# 実装計画書: みかづきのまい

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 10 / 命中: なし（必中） / 対象: 自分
- 使用者が **ひんしになる** 代わりに、次に場に出た自分のポケモンの **HP・全ての技のPP・状態異常**
  を全回復する
- `dance`（おどりこの対象）・`heal` ラベルを持つ
- 控えのポケモンがいない場合は失敗し、使用者はひんしにならない
- 詳細は `docs/spec/moves/みかづきのまい.md` `docs/spec/fields/みかづきのまい.md` を参照

## Priority 根拠

### ON_BEFORE_APPLY_MOVE (priority=100)

`docs/spec/turn.md` に `ON_BEFORE_APPLY_MOVE` に関する直接的な記載はないが、同種の失敗判定
ハンドラ（`ねがいごと_can_apply` 等）はすべてデフォルト priority=100 を使用している。
本技も同パターンとする。

### ON_STATUS_HIT (priority=100)

同上。`いのちのしずく_heal_self` 等、既存の自己完結型変化技ハンドラのデフォルトに合わせる。

### ON_SWITCH_IN (priority=100)

`docs/spec/turn.md` の `Event.ON_SWITCH_IN` セクションに以下の記載がある：

> | 100 | みかづきのまいによる回復 |

priority=100 を明示する。

## 実装箇所（レビュー時点で追加した内容）

### `src/jpoke/handlers/move_status.py`

`みかづきのまい_apply` の直後に挿入。

```python
def みかづきのまい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """みかづきのまいの失敗チェック: 控えのポケモンがいない場合は失敗し、使用者はひんしにならない。"""
    player = battle.get_player(ctx.attacker)
    if not battle.query.has_available_bench(player):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="みかづきのまい")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

`battle.query.has_available_bench` はトラップ状態（にげられない・バインド等）を無視して
控えの有無のみを判定する API（`だっしゅつパック` 等と同じ用途）。みかづきのまいは
使用者自身がひんしになる技であり、とらわれ状態の影響を受けずに判定すべきため
`can_switch` ではなく `has_available_bench` を使う。

### `src/jpoke/data/moves/move_ma.py`

既存の `"みかづきのまい"` エントリに `ON_BEFORE_APPLY_MOVE` ハンドラを追加。

```python
"みかづきのまい": MoveData(
    type="エスパー",
    category="status",
    pp=10,
    target="self",
    flags={"dance", "heal"},
    handlers={

        Event.ON_STATUS_HIT: h.MoveHandler(hs.みかづきのまい_apply),

```

## 注意点・エッジケース

- **控え無し失敗判定が既存実装に欠落していた**: 修正前は控えのポケモンがいない状態で
  みかづきのまいを使うと、フィールド設置後に使用者がひんしになり、そのまま試合終了扱いに
  なってしまうバグがあった（原作では技が失敗し、使用者はひんしにならない）。本レビューで
  `ON_BEFORE_APPLY_MOVE` チェックを追加して修正した
- **PP について**: `docs/champions/move_list.txt` に本技のエントリが存在しないため、
  Gen9本家準拠の PP=10 を維持する（`docs/spec/moves/みかづきのまい.md` 参照）
- **かいふくふうじの uncurable ガード**: `ailment_manager.remove` 内部で `uncurable` フラグを
  チェックするため、ハンドラ側での追加ガードは不要
- **HP 満タン時の回復**: `modify_hp(mon, v=mon.max_hp - mon.hp)` は HP が満タンの場合 v=0 と
  なり変化しない。`modify_hp` は v=0 のとき早期リターンするため問題ない
- **PP 満タン時のリセット**: `move.reset(reset_pp=True)` は type/category/power も
  データ値へ再代入するが、`ON_SWITCH_IN` の時点でこれらは常にデータ値と一致しているため
  副作用はない（技実行後に `MoveExecutor._execute_move` の `finally` 節で毎回
  `ctx.move.reset()` が呼ばれ、一時的な変更は都度元に戻されるため）
- **1箇所への重複設置**: `side.activate("みかづきのまい", 1)` は `FieldManager.activate` が
  既にアクティブな場合は何もしないため、同じ位置に重ねて設置されることはない
  （使用者はそれでもひんしになる＝技としては成功扱いだが、フィールド効果は延長されない）
- **いやしのねがいとの重複**: 「みかづきのまい」フィールドと「いやしのねがい」フィールドは
  別名で管理されるため同時に存在しうる。`ON_SWITCH_IN` の同一 priority(100) 内では
  `EventManager._sort_handlers` が安定ソートであり、`FieldManager._activate_field` が
  フィールド有効化時にハンドラを動的登録するため、先に設置された側のハンドラが先に
  実行される。追加の実装対応は不要
