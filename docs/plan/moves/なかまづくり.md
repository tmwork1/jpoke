# 計画書: なかまづくり

## 仕様要約

- タイプ: ノーマル / 変化技 / PP 16（champions/move_list.txt準拠） / 命中 100 / 優先度 0
- 効果: 相手の特性を**使用者と同じ特性**に書き換える
- まもる状態で防がれる（wiki: まもる ○）
- マジックコート状態で反射される（wiki: マジックコート ○）
- よこどりでは奪われない（wiki: よこどり ×）
- みがわり状態の相手には防がれる（wiki「技の仕様」節に明記: 「みがわり状態に防がれる」）
- 変更された特性は交代で元に戻る
- 失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100、turn.md 明記）:
  - 対象の特性が `なまけ` である（wiki の個別記述）
  - 対象の特性が `protected` フラグを持つ（「上書きできない特性」）
  - 使用者の特性が `uncopyable` フラグを持つ（「コピーできない特性」）
  - 使用者と対象が同じ特性である
  - 対象の特性変更がとくせいガード等により防がれる
    （`battle.ability_manager.is_change_blocked(ctx.defender)`）

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 節に以下の記載がある。

```
| 100 | なかまづくり: 失敗条件 |
```

実装上は `ON_BEFORE_APPLY_MOVE` priority 100 に対応する（いえき・シンプルビーム・スキルスワップが同じパターンで登録されていることを確認済み）。

`ON_STATUS_HIT` は priority 100（MoveHandler デフォルト）とする。これはシンプルビームを含む全変化技効果のパターンと一致する。

## 実装箇所

### `handlers/move_status.py` に追加する関数（五十音順: な の位置）

```python
def なかまづくり_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なかまづくりの失敗条件チェック。

    以下の場合は失敗する:
    - 対象の特性が「なまけ」である
    - 対象の特性が protected フラグを持つ（上書きできない特性）
    - 使用者の特性が uncopyable フラグを持つ（コピーできない特性）
    - 使用者と対象が同じ特性である
    - 対象の特性変更がとくせいガード等で防がれる
    """
    attacker_ability = ctx.attacker.ability.base_name
    defender_ability = ctx.defender.ability.base_name
    if (
        defender_ability == "なまけ"
        or ctx.defender.ability.has_flag("protected")
        or ctx.attacker.ability.has_flag("uncopyable")
        or attacker_ability == defender_ability
        or battle.ability_manager.is_change_blocked(ctx.defender)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="なかまづくり")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def なかまづくり_change_defender_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なかまづくりの効果: 相手の特性を使用者と同じ特性に書き換える。"""
    battle.change_ability(ctx.defender, ctx.attacker.ability.base_name)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### 失敗条件の詳細

- **なまけ個別チェック**: Wiki に「対象の特性がなまけか、上書きできない特性であるときは失敗する」と明記されている。なまけには `protected` フラグが付与されていないため（`data/ability.py` 確認済み）、`base_name == "なまけ"` の個別チェックが必要
- **使用者の uncopyable チェック**: `type_defs.py` の `AbilityFlag` に `"uncopyable"` の説明として「トレース・なかまづくり等でコピー/再現させない特性」と明記されている。使用者特性が uncopyable の場合は使用者自身がコピーできない特性を持っているため失敗する
- **同一特性チェック**: 使用者と対象の `base_name` が一致する場合は `change_ability` 内部で早期リターンされるが、wiki の仕様として「自分と相手が同じ特性である場合」は失敗とされているため、`can_apply` で明示的に失敗させる

### 特性変更後の挙動

- `battle.change_ability(defender, attacker_ability)` を呼ぶと内部で `ON_ABILITY_DISABLED` → ハンドラ解除 → 新特性設定 → ハンドラ登録 → `ON_ABILITY_ENABLED` が発火する。なかまづくりで付与された特性の場内発動（あめふらし等）は change_ability の ON_ABILITY_ENABLED で自動処理される
- 交代すると `base_ability_name` に戻る（`change_ability` は `base_name` を更新しない）ため、交代後は元の特性に戻る

### かたやぶりとの相互作用

- なかまづくりの失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100）はかたやぶりの貫通対象外である（かたやぶりは `mold_breaker_ignorable` フラグを持つ特性のみ無視する）
- そのため `protected` フラグを持つ特性を相手が持っていても、使用者がかたやぶりを持っていても失敗する

### おうごんのからだとの相互作用

- おうごんのからだは `protected` フラグを持っていないが `mold_breaker_ignorable` フラグを持つ
- `ON_TRY_MOVE_1` priority 110 でおうごんのからだによる変化技無効が処理されるため、ハンドラ側で追加ガードは不要
- かたやぶりを持つ使用者がなかまづくりを使う場合、おうごんのからだは貫通される（そもそもなかまづくりの失敗条件には関係しない）

### とくせいガードとの相互作用

- 対象がとくせいガードを持つ場合、`battle.ability_manager.is_change_blocked(ctx.defender)`
  が真になり `can_apply` で失敗する（`ON_CHECK_ABILITY_DISABLE` イベント経由で判定）。
  スキルスワップ・いえき等と共通のパターン。

### マジックコートによる反射

- wiki に「マジックコート ○」とあり、反射される。反射処理は `ON_CHECK_REFLECT` で manager 内部が処理するため、ハンドラ側での追加ガードは不要
- 反射されたとき（なかまづくり使用者の特性が反射先のマジックコート保持者の特性に書き換えられる）の失敗条件チェックは反射後のコンテキストで再評価される

### みがわり状態との相互作用

- wiki に「まもる ○」とあり、まもる状態で防がれる。「技の仕様」節にも「みがわり状態に防がれる」と明記されている。なかまづくりは `bypass_substitute` フラグを持たないため、他の特性変更技（シンプルビーム等）と同様に manager の通常フロー（`みがわり_immune`）でブロックされる

### ダイマックスしている対象への使用

- wiki に「対象がダイマックスしている場合は失敗する」と明記されている。ダイマックス中の失敗はダイマックス側のハンドラ（`ON_APPLY_MOVE` priority 30）で処理されるため、`can_apply` での追加ガードは不要

### スターモービルへの使用

- wiki に「対象がスターモービルである場合は失敗する」とあるが、ポケモンチャンピオンズシングルバトルの実装対象外であるため省略する
