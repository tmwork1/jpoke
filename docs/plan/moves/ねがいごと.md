# 実装計画書: ねがいごと

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 10 / 命中: なし（必中） / 対象: field（自陣営）
- 使用したターンの **次のターン終了時** に自分のHPが最大HPの **半分** 回復する
- **交代した場合** は同じポジションにいるポケモンが回復する（使用者でなくてよい）
- 同一陣営に既にねがいごとが設置されている場合は **失敗**（重複設置不可）
- ねがいごとフィールドは **カウント 2** で設置し、ON_TURN_END (priority=50) で tick_down される
  - カウント 2 → 1（1ターン目終了）: まだ回復しない
  - カウント 1 → 0（2ターン目終了）: `ON_FIELD_DEACTIVATE` が発火 → HP回復

## Priority 根拠

### ON_BEFORE_APPLY_MOVE (priority=100)

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` セクション（本実装では `ON_BEFORE_APPLY_MOVE` に対応）に以下の記載がある：

> | 100 | ねがいごと: 前ターン効果残り |

priority=100 を採用する。

### ON_STATUS_HIT (priority=100)

`docs/spec/turn.md` に `ON_STATUS_HIT` の直接記載はないが、類似するサイドフィールド設置ハンドラ（`おいかぜ_set_side_field`、`ひかりのかべ_set_side_field`、`しんぴのまもり_set_side_field` 等）はすべて priority=100（デフォルト）を使用している。ねがいごとも同パターンとする。

### ON_TURN_END (priority=50)

`docs/spec/turn.md` の `Event.ON_TURN_END` セクションに以下の記載がある：

> | 50 | ねがいごと |

priority=50 を明示する。`data/field.py` でも priority=50 が設定済みで一致している。

### ON_FIELD_DEACTIVATE (priority=100)

ねがいごとの `ON_FIELD_DEACTIVATE` ハンドラはカウントが 0 になった時に `tick_side_field` 内から発火される。priority はデフォルト値 100。`data/field.py` に既に登録済み。

## 実装箇所

### 1. `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: `にらみつける_modify_defender_stats` の直後、`ねをはる_apply_volatile_to_attacker` の前（「ね」行）。

```python
def ねがいごと_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねがいごとの失敗チェック: すでにねがいごとフィールドが有効なら失敗する。"""
    side = battle.get_side(ctx.attacker)
    if side.get("ねがいごと").is_active:
        battle.add_event_log(ctx.attacker, LogCode.MOVE_FAILED,
                             payload={"reason": "ねがいごと"})
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ねがいごと_set_side_field(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねがいごとの効果: 自陣営に「ねがいごと」を設置し、回復量を使用者の最大HPの半分に設定する。"""
    mon = ctx.attacker
    side = battle.get_side(mon)
    field = side.get("ねがいごと")
    side.activate("ねがいごと", 2)
    field.heal = mon.max_hp // 2
    return HandlerReturn(value=value)
```

### 2. `src/jpoke/data/move.py`

既存の `"ねがいごと"` エントリ（target="field", labels=["heal"] は維持）に handlers を追加する。

```python
"ねがいごと": MoveData(
    type="ノーマル",
    category="変化",
    pp=10,
    target="field",
    labels=["heal"],
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.ねがいごと_can_apply,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.ねがいごと_set_side_field,
        ),
    }
),
```

## 注意点・エッジケース

- **field.heal の設定タイミング**: `side.activate("ねがいごと", 2)` が `False` を返す場合（既にアクティブ）は `ON_BEFORE_APPLY_MOVE` で先に止まるため、`ON_STATUS_HIT` では `activate` が必ず `True` を返す。`field.heal` は `activate` の後で設定する
- **ctx.source の意味**: `ON_FIELD_DEACTIVATE` は ctx=None で emit されるため、`_build_context` が `registered_subject`（Player）から `battle.get_active(player)` を呼ぶ。つまり回復対象はそのポジションの**現在の場のポケモン**になる。これにより「交代した場合は同じ位置のポケモンが回復する」仕様を自動的に実現できる
- **heal 値のリセット**: `field.heal` は `Field.__init__` で `0` に初期化されているため、フィールド解除後の次回 activate 時にゴミ値が残らないよう、`side.activate` が成功したあとで `field.heal = mon.max_hp // 2` を設定する必要がある。`field.count` はリセットされるが `field.heal` は手動設定が必要
- **HP 満タン時の回復**: `battle.modify_hp(mon, v=value.heal)` は内部で `min(max_hp - hp, v)` により満タンを超えない。回復量が 0 になる（満タン）ケースも正常動作する
- **既存フィールドテスト**: `tests/test_field.py` の `test_ねがいごと_回復と解除` は手動で `side0={"ねがいごと": 2}` / `field.heal = 20` を設定しており、技使用経路のテストではない。計画書に含まれる技使用からのテストを別途追加すること
- **heal ラベルによるかいふくふうじ制限**: `labels=["heal"]` が設定されているため、「かいふくふうじ」状態の相手にはねがいごとが使えなくなる（move_executor 側でチェック済み）。ハンドラ側での追加ガードは不要
- **重複設置の失敗ログ**: `add_event_log(ctx.attacker, LogCode.MOVE_FAILED, ...)` を使用する。他の `_can_apply` ハンドラと同パターンにする
