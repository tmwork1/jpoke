# 実装計画書: ちからをすいとる

## 仕様要約

- タイプ: くさ / カテゴリ: 変化 / PP: 12 / 命中率: 100 / 対象: 1体選択 / 接触: なし / まもる: 防げる
- ラベル: `["heal"]`
- 相手の「こうげき」ランクを1段階下げ、下げる前の「こうげき」の**ランク補正込みの実数値**と
  同じ値だけ自分のHPを回復する（ドレイン技の一種として扱う）
- 特性などでランクが下がらない場合は、回復効果のみ発動する
- 相手の「こうげき」ランクがすでに -6（最低）の場合は技が失敗する（回復も発動しない）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手こうげきランクが -6 なら失敗判定 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手こうげきランク補正込み実数値分HPを回復し、相手こうげきランクを -1 |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「ちからをすいとる: こうげき-6」が priority 100 として明記されている（コード上のイベント名は `ON_BEFORE_APPLY_MOVE` に対応する）
- `ON_STATUS_HIT` priority 100（デフォルト）: 変化技のランク変更・回復効果は同テーブルでは未掲載だが、既存の同種ハンドラ（`かいでんぱ`・`あまえる`・`フェザーダンス` 等の `ON_STATUS_HIT` 実装）と同一パターンのため priority 100 を採用する

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「ち」行 — `ちいさくなる_apply` の後、`ちょうのまい_modify_attacker_stats` の前に挿入する。

```python
def ちからをすいとる_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ちからをすいとるの効果: 相手のこうげきランク補正込み実数値分HPを回復し、相手のこうげきを1段階下げる。

    特性などでランクが下がらない場合は回復効果のみ発動する。
    ドレイン技共通の ON_CALC_DRAIN（おおきなねっこ等）と reason="drain"
    （ヘドロえき等）を経由させる。
    """
    assert ctx.defender is not None
    # 相手のこうげきランク補正込みの実数値分だけ自分のHPを回復（ランク変更前に取得）
    recover_amount = ctx.defender.ranked_stats["atk"]
    recover_amount = battle.events.emit(Event.ON_CALC_DRAIN, ctx, recover_amount)
    battle.modify_hp(ctx.attacker, v=recover_amount, reason="drain")
    # 相手のこうげきを1段階下げる（失敗しても回復は発動済み）
    battle.modify_stats(ctx.defender, {"atk": -1}, source=ctx.attacker)
    return HandlerReturn(value=value)


def ちからをすいとる_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ちからをすいとるの失敗チェック: 相手のこうげきランクがすでに -6 なら失敗する。"""
    assert ctx.defender is not None
    if ctx.defender.rank["atk"] == -6:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="こうげき最低"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

- 回復量は相手の「こうげきランク補正込みの実数値」（`ctx.defender.ranked_stats["atk"]`）であり、
  ランク補正前の値（`ctx.defender.stats["atk"]`）ではない点に注意する
  （一次情報: 「回復量は、相手のランク補正込みの攻撃実数値となる」）
- `battle.modify_hp(ctx.attacker, v=recover_amount, reason="drain")` の `reason="drain"` は
  ヘドロえき特性（回復をダメージに変換）を正しく反応させるために必須。抜けると
  `ヘドロえき_reverse_drain` が `ctx.hp_change_reason != "drain"` で早期リターンし発動しない
- `Event.ON_CALC_DRAIN` を経由させることで、おおきなねっこ（回復量1.3倍）が
  ドレイン技共通の仕組みで自然に適用される
- `battle.modify_hp` で上限は内部でクランプされるため、満タン時の超過分はゼロになる
  （ハンドラ側で満タンチェック不要）
- 「特性などでランクが下がらない場合は回復効果のみ発動する」は `battle.modify_stats` の
  返り値を無視することで自然に実現される。失敗条件チェック（`-6` 判定）のみ
  `ON_BEFORE_APPLY_MOVE` で処理し、`ON_STATUS_HIT` では無条件に回復してからランク変更を試みる
- `ON_BEFORE_APPLY_MOVE` の失敗判定は「すでに -6 かどうか」のみ。あまのじゃく等で
  ランク変化の方向が反転する特性でも、この判定は常に `-6` 基準で行う（一次情報に明記）
- ミラーアーマー・あまのじゃくはいずれも `battle.modify_stats` 内部（`ON_BEFORE_MODIFY_STAT`）で
  汎用的に処理されるため、本ハンドラ側で個別対応は不要
- そうしょく・おうごんのからだ（くさタイプ技の吸収・無効化）は `ON_BEFORE_APPLY_MOVE` の
  特性側ハンドラ（`defender:self`）で汎用的に処理されるため、本ハンドラ側で個別対応は不要
- `かいふくふうじ` 状態での技選択禁止・使用前かいふくふうじ化による失敗は、
  `flags={"heal"}` により汎用処理（`かいふくふうじ_try_action`）で自動的にカバーされる
