# 実装計画書: ちからをすいとる

## 仕様要約

- タイプ: くさ / カテゴリ: 変化 / PP: 10 / 命中率: 100 / 対象: 1体選択 / 接触: なし / まもる: 防げる
- ラベル: `["heal"]`
- 相手の「こうげき」の**実数値**と同じ値だけ自分のHPを回復する
- 相手の「こうげき」ランクを1段階下げる
- 特性などでランクが下がらない場合は、回復効果のみ発動する
- 相手の「こうげき」ランクがすでに -6（最低）の場合は技が失敗する

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手こうげきランクが -6 なら失敗判定 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手こうげき実数値分HPを回復し、相手こうげきランクを -1 |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「ちからをすいとる: こうげき-6」が priority 100 として明記されている
- `ON_STATUS_HIT` priority 100（デフォルト）: 変化技のランク変更・回復効果は同テーブルでは未掲載だが、既存の同種ハンドラ（`かいでんぱ`・`あまえる`・`フェザーダンス` 等の `ON_STATUS_HIT` 実装）と同一パターンのため priority 100 を採用する

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「ち」行 — `ちいさくなる_apply` の後、`ちょうはつ_apply_volatile_to_defender` の前に挿入する。

```python
def ちからをすいとる_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ちからをすいとるの失敗チェック: 相手のこうげきランクがすでに -6 なら失敗する。"""
    if ctx.defender.rank["A"] <= -6:
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "ちからをすいとる_こうげき最低"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ちからをすいとる_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ちからをすいとるの効果: 相手のこうげき実数値分HPを回復し、相手のこうげきを1段階下げる。

    特性などでランクが下がらない場合は回復効果のみ発動する。
    """
    # 相手のこうげき実数値（ランク補正前）分だけ自分のHPを回復
    recover_amount = ctx.defender.stats["A"]
    battle.modify_hp(ctx.attacker, v=recover_amount)
    # 相手のこうげきを1段階下げる（失敗しても回復は発動済み）
    battle.modify_stats(ctx.defender, {"A": -1}, source=ctx.attacker)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

- 回復量は相手の「こうげき実数値」（`ctx.defender.stats["A"]`）であり、ランク補正後の値（`ranked_stats["A"]`）ではない点に注意する
- `battle.modify_hp(ctx.attacker, v=recover_amount)` で上限は `modify_hp` 内部でクランプされるため、満タン時の超過分はゼロになる（ハンドラ側で満タンチェック不要）
- 「特性などでランクが下がらない場合は回復効果のみ発動する」は `battle.modify_stats` の返り値を無視することで自然に実現される。失敗条件チェック（`-6` 判定）のみ `ON_BEFORE_APPLY_MOVE` で処理し、`ON_STATUS_HIT` では無条件に回復してからランク変更を試みる
- `ON_BEFORE_APPLY_MOVE` の失敗判定は「すでに -6 かどうか」のみ。ランクが下げられない特性（クリアボディ等）による無効化は `battle.modify_stats` 内部で処理されるため、ハンドラ側に特性チェックは不要
- `かいふくふうじ` 状態のとき、ラベル `heal` により回復効果が封じられる可能性がある。`battle.modify_hp` が `ON_MODIFY_HEAL` を発火して封じられる場合でも、ランク変更は継続して発動するかどうか仕様書に記載がない点は要確認
- `turn.md` の `ON_TRY_MOVE_2` priority 130「ランク補正下げ: すでに最低」は汎用の失敗チェックとして存在するが、`ちからをすいとる` は同時に回復効果も持つため、`ON_BEFORE_APPLY_MOVE` で独立した失敗チェックを実装することで意図通りに制御できる
