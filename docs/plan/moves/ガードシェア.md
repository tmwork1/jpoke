# 計画書: ガードシェア

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 10 / 命中: 100 / 対象: 相手
- 使用者と相手の「ぼうぎょ（B）」「とくぼう（D）」の**実数値**を平均化する
- ランク変化は一切行わない（実数値のみ書き換える）
- 平均化の計算式: `(使用者のB + 相手のB) // 2` を両者のBに、Dも同様
- まもる: × / マジックコート: × / 命中判定: あり（accuracy=100）

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT` の内部priority表は掲載されていない。既存の変化技ハンドラ（`いたみわけ_equalize_hp`・`おかたづけ_cleanup`・`りゅうのまい_modify_attacker_stats` 等）はすべて priority 100（`MoveHandler` デフォルト）を使用している。ガードシェアも同一パターンで priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「が」行 — 「か行」清音・濁音の順序により `かいでんぱ_modify_defender_stats`（かい）の前に挿入する。

実数値の書き換えは `mon._stats_manager._stats` リストの直接代入で行う（インデックス: B=2、D=4）。`stats` property は読み取り専用辞書を返すため代入不可。`modify_stats` は**ランク変化**用のAPIのため使用しない。

```python
def ガードシェア_equalize_stats(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """ガードシェア: 使用者と相手のぼうぎょ・とくぼうの実数値を平均化する。

    ランク変化は行わず、実数値のみを書き換える。
    平均は切り捨て（// 2）。
    """
    attacker = ctx.attacker
    defender = ctx.defender
    # B=インデックス2、D=インデックス4
    for idx in (2, 4):
        avg = (attacker._stats_manager._stats[idx] + defender._stats_manager._stats[idx]) // 2
        attacker._stats_manager._stats[idx] = avg
        defender._stats_manager._stats[idx] = avg
    return HandlerReturn(value=value)
```

### `src/jpoke/data/move.py`

既存の `"ガードシェア"` エントリに `handlers` を追加する。

```python
"ガードシェア": MoveData(
    type="エスパー",
    category="変化",
    pp=10,
    accuracy=100,
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.ガードシェア_equalize_stats,
        ),
    }
),
```

## 注意点・エッジケース

- **実数値変更 vs ランク変化の区別**: `battle.modify_stats()` はランク変化API（`status_manager.modify_stats` 委譲）であり、実数値には触れない。ガードシェアは実数値平均化のため `_stats_manager._stats` を直接書き換える必要がある
- **`stats` property の読み取り専用制約**: `mon.stats` は `_stats_manager.stats_dict`（dictコピー）を返すため、`mon.stats["B"] = avg` の代入は _stats_manager._stats に反映されない
- **HPへの非影響**: H=インデックス0 は対象外。誤って `_stats[0]` を変更しないよう注意する（`max_hp` ズレが生じる）
- **ランク非変化の確認**: `ON_BEFORE_MODIFY_STAT` や `ON_MODIFY_STAT` 系イベントは一切発火しない設計のため、クリアボディ・かがくへんかガス等の能力変化ガードは影響しない
- **パワーシェアとの類似**: パワーシェア（A・C実数値平均化）は同じパターンで実装できる。インデックスが A=1、C=3 になる点のみ異なる
- **ワンダールーム中の扱い**: ワンダールーム効果（B/D参照を入れ替え）は `ON_CALC_DEF_MODIFIER` ハンドラ内で対処されるため、ガードシェアハンドラ側では考慮不要
- **みがわり貫通の有無**: `ON_CHECK_SUBSTITUTE` で変化技のみがわり貫通チェックが入るが、ガードシェアが「みがわり貫通」かは仕様wiki不在のため実装時に要確認。貫通しない場合は `battle.events` の既定動作で弾かれる
