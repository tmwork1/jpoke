# 計画書: ヒートスタンプ（再レビュー）

## 仕様要約

* 自分（使用者）の体重と相手（対象）の体重の比率によって威力が変わる
  （`docs/spec/moves/ヒートスタンプ.md` 参照）。
* 比率 = 自分の体重 ÷ 相手の体重 から、40/60/80/100/120 の5段階で威力を決定する。
* ヘビーボンバーと全く同じ威力計算式を使用する（比率の算出方向が くさむすび/けたぐり とは
  逆で、自分の体重を基準にする点が異なる）。
* 相手が「ちいさくなる」状態のときは体重比率計算をスキップし、`minimize` ラベルの共通処理
  （必中・威力2倍）のみが適用される。

## 既存実装の確認結果

`src/jpoke/handlers/move_attack.py` の `ヒートスタンプ_calc_power` と共通関数
`_weight_ratio_to_power`（`ヘビーボンバー_calc_power` と共用）を確認した結果、仕様書の
威力段階・境界値と完全に一致しており、ちいさくなる状態のスキップ処理も正しい。

`src/jpoke/data/moves/move_ha.py` の `MoveData`（type/category/power/accuracy/flags/handlers）も
一次情報（wiki: タイプほのお、物理、命中100、直接攻撃あり、ちいさくなる対象への必中2倍）と
一致している。

**PPのみ不一致を検出**: 実装は `pp=10`（Gen9本家基準）になっていたが、
`src/jpoke/data/ps-champ-ja/moves.json` では `ヒートスタンプ	ほのお	物理	-	100	12	接○	守○` と
PP=12 が正であるため、`pp=10` は移行漏れのバグと判断し `pp=12` に修正する。

| ヒートスタンプ | `Event.ON_CALC_POWER_MODIFIER` | デフォルト（明示指定なし） | `attacker:self` / `defender:self`（`MoveHandler` デフォルト） | 自分と対象の体重比率から技の実威力を算出して返す |

## リーサル計算

`docs/progress/move.md` の該当行は「リーサル実装: n/a」「リーサルテスト: n/a」のため対象外。
理由: 威力は自分・相手双方の体重という固定パラメータにのみ依存し、相手のHP分布
`StateDist` を左右する要素ではないため、`lethal.py` 側の専用ハンドラは不要。

## テスト

既存の `tests/moves_attack/test_move_ha.py` に威力段階の境界を確認するテストが一通り揃っている
（比率2未満→40、2以上3未満→60、3以上4未満→80、4以上5未満→100、5以上→120、ちいさくなる中の
スキップ動作）。`tests/volatiles/test_volatile_satana.py` の `minimize_enhance_moves` にも
ヒートスタンプが含まれ、必中化の共通テストでカバーされている。今回のレビューで確認した範囲では
計算式・実装ともに正しいため、テストの追加は不要。
