# 計画書: スキルスワップ

## 仕様要約

- タイプ: エスパー / 変化技 / PP 10 / 命中: 必中（命中判定なし）/ 優先度: 0
- 効果: 使用者と対象の特性を互いに入れ替える
- まもる状態で防がれる
- 失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100）:
  - 使用者または対象の特性が `protected` フラグを持つ場合（かたやぶりでも失敗）
  - 対象がおうごんのからだを持つ場合は防がれる（使用者がかたやぶりなら成功）
  - 自分か対象のどちらかがとくせいガードを持っている場合
- 入れ替わった特性は交代すると元に戻る
- 対象がとくせいなし状態でも成功する
- 対象がみがわり状態でも成功する
- 入れ替え直後に新たに得た特性が発動する（第五世代以降）

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 節に以下の記載がある。

```
| 100 | スキルスワップ: 失敗条件 |
```

実装上は `ON_BEFORE_APPLY_MOVE` priority 100 に対応する（いえき・なりきり・シンプルビームが同じパターンで登録されていることを確認済み）。

`ON_STATUS_HIT` は priority 100 のまま（`MoveHandler` デフォルト）とする。これはシンプルビームを含む全変化技効果のパターンと一致する。

## 実装箇所

### `handlers/move_status.py` に追加する関数（五十音順: ス の位置）

```python
def スキルスワップ_can_apply(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """スキルスワップの失敗条件チェック。

    使用者または対象の特性が protected フラグを持つ場合は失敗する。
    かたやぶりを持っていても、この失敗条件は回避できない。
    """
    if (
        ctx.attacker.ability.has_flag("protected")
        or ctx.defender.ability.has_flag("protected")
    ):
        battle.add_event_log(ctx.attacker, LogCode.MOVE_FAILED,
                             payload={"reason": "スキルスワップ"})
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def スキルスワップ_swap_ability(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """スキルスワップの効果: 使用者と対象の特性を入れ替える。"""
    battle.ability_manager.swap_ability(ctx.attacker, ctx.defender)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### 失敗条件の対称性

- シンプルビームやいえきは「対象の特性が `protected`」のみチェックするが、スキルスワップは「使用者と対象どちらかの特性が `protected`」をチェックする点が異なる
- `protected` フラグを持つ代表的な特性: かがくへんかガス・ARシステム・アイスフェイス・イリュージョン・おもかげやどし・ぎょぐん・クォークチャージ・こだいかっせい・しれいとう・じんばいったい・スワームチェンジ・ぜったいねむり・テラスシェル・ばけのかわ・バトルスイッチ・はらぺこスイッチ・ふしぎなまもり・マイティチェンジ・マルチタイプ・リミットシールド
- かたやぶりの貫通: スキルスワップの失敗条件はかたやぶりを持っていても回避できないため、ハンドラ側で `ctx.attacker.ability.name == "かたやぶり"` の例外処理は不要

### おうごんのからだについて

- おうごんのからだは `protected` フラグを持っていない（かたやぶりで貫通可能）
- `ON_TRY_MOVE_1` の priority 110 でおうごんのからだの特性無効化が処理されるため、ハンドラ側で追加ガードは不要

### とくせいガードについて

- 自分か対象のどちらかがとくせいガード（アイテム）を持つ場合に失敗するが、これはアイテム側のハンドラで処理されるため、`スキルスワップ_can_apply` に追加実装は不要

### swap_ability の内部挙動

- `battle.ability_manager.swap_ability(mon1, mon2)` は内部で `ON_ABILITY_DISABLED` → ハンドラ解除 → ハンドラ再登録 → `ON_ABILITY_ENABLED` を発火するため、入れ替え直後に新特性が発動する挙動は自動的に再現される
- 交代時は `base_ability_name` に基づいて元の特性に戻るため、交代後はスワップ前の特性に戻ることが期待される

### 同じ特性どうしのスワップ

- 使用者と対象が同じ特性の場合でも第六世代以降は成功する（ポケモンチャンピオンズは第九世代ルールのため成功）
- `swap_ability` 内部では同名特性のスワップも問題なく処理できるため特別なガードは不要

### みがわり状態との相互作用

- 対象がみがわり状態でも成功する（Wikiに明記あり）
- `ON_APPLY_MOVE` の priority 30 でみがわりによる無効化チェックが行われるが、スキルスワップはその対象外のため manager 側で適切に処理される

### マジックコート・よこどり

- マジックコートによる反射は「×」（反射されない）
- よこどりによる奪取も「×」（奪われない）
- これらはマジックコート状態/よこどりのハンドラが `ON_TRY_MOVE_1` で処理するため、スキルスワップハンドラ側での追加ガードは不要
