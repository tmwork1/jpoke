# 計画書: スキルスワップ

## 仕様要約

- タイプ: エスパー / 変化技 / PP 12 / 命中 - (必中) / 対象: 相手1体 / 優先度: 0
- 効果: 使用者と対象の特性を互いに入れ替える
- まもる状態で防がれる（`unprotectable` フラグなし）
- 失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100）:
  - 使用者または対象の特性が `protected` フラグを持つ場合（かたやぶりでも失敗）
  - 使用者または対象がとくせいガードを持っている、またはその他の理由で
    特性変更がブロックされている場合（`is_change_blocked`）
  - 対象がおうごんのからだを持つ場合は防がれる（変化技共通の無効化処理。
    使用者がかたやぶりなら成功）
- 入れ替わった特性は交代すると元に戻る
- 対象がとくせいなし状態でも成功する
- 対象がみがわり状態でも成功する
- 入れ替え直後に新たに得た特性が発動する（第五世代以降、使用者側の新特性が先に発動）
- 詳細は `docs/spec/moves/スキルスワップ.md` を参照。
- `docs/spec/turn.md` 記載:
  - `Event.ON_APPLY_MOVE`（実装上 `ON_BEFORE_APPLY_MOVE`）priority 100:
    「スキルスワップ: 失敗条件」

## ハンドラ構成

| イベント | Priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | AttackContext（MoveHandler は subject_spec 不要） | 失敗条件チェック（使用者・対象どちらかの特性が `protected` または `is_change_blocked` なら stop_event） |

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 節に以下の記載がある。

```
| 100 | スキルスワップ: 失敗条件 |
```

実装上は `ON_BEFORE_APPLY_MOVE` priority 100 に対応する（いえき・なりきり・シンプルビームが同じパターンで登録されていることを確認済み）。

`ON_STATUS_HIT` は priority 100 のまま（`MoveHandler` デフォルト）とする。これはシンプルビームを含む全変化技効果のパターンと一致する。

## 実装内容（修正後）

### `handlers/move_status.py`（五十音順: ス の位置）

```python
def スキルスワップ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """スキルスワップの失敗条件チェック。

    使用者または対象の特性が protected フラグを持つ場合は失敗する。
    かたやぶりを持っていても、この失敗条件は回避できない。
    使用者または対象がとくせいガードを持ち特性の変更が防がれる場合も失敗する。

    例外: うのミサイルは上書き（いえき・なかまづくり等）には protected フラグで
    抵抗するが、SV Ver.3.0.0 以降はスキルスワップ/さまようたましいでの交換のみ
    可能になったため base_name で個別に除外する。

    かがくへんかガスは protected フラグを持たない（いえき/コアパニッシャーで
    とくせいなし状態になれるため）が、スキルスワップ/さまようたましいでの
    交換だけは対象外のため base_name で個別に判定する。
    """
    assert ctx.defender is not None
    if (
        (ctx.attacker.ability.has_flag("protected") and ctx.attacker.ability.base_name != "うのミサイル")
        or (ctx.defender.ability.has_flag("protected") and ctx.defender.ability.base_name != "うのミサイル")
        or ctx.attacker.ability.base_name == "かがくへんかガス"
        or ctx.defender.ability.base_name == "かがくへんかガス"
        or battle.ability_manager.is_change_blocked(ctx.attacker)
        or battle.ability_manager.is_change_blocked(ctx.defender)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="スキルスワップ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def スキルスワップ_swap_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """スキルスワップの効果: 使用者と対象の特性を入れ替える。"""
    assert ctx.defender is not None
    battle.ability_manager.swap_ability(ctx.attacker, ctx.defender)
    return HandlerReturn(value=value)
```

`スキルスワップ_can_apply` のかがくへんかガス個別判定は 2026-07-11 のかがくへんかガス
レビューで追加した（それ以外は既存実装のまま変更不要）。

## 注意点・エッジケース（review-test 向け）

### 失敗条件の対称性

- シンプルビームやいえきは「対象の特性が `protected`」のみチェックするが、スキルスワップは「使用者と対象どちらかの特性が `protected`」をチェックする点が異なる（使用者側の特性も書き換えられるため）
- `protected` フラグを持つ代表的な特性: かがくへんかガス・ARシステム・アイスフェイス・イリュージョン・おもかげやどし・ぎょぐん・クォークチャージ・こだいかっせい・しれいとう・じんばいったい・スワームチェンジ・ぜったいねむり・ダルマモード・テラスシェル・テラスチェンジ・ゼロフォーミング・ばけのかわ・バトルスイッチ・はらぺこスイッチ・ふしぎなまもり・マイティチェンジ・マルチタイプ・リミットシールド
  - うのミサイルは第八世代までは `protected` だったが、SV Ver.3.0.0以降（本プロジェクトが対象とする最新版）は入れ替え可能なため `protected` フラグを持たない（`docs/spec/abilities/うのミサイル.md` 参照。なりきりレビュー時に修正済み）
- かたやぶりの貫通: スキルスワップの失敗条件はかたやぶりを持っていても回避できないため、ハンドラ側で `ctx.attacker.ability.name == "かたやぶり"` の例外処理は不要

### おうごんのからだについて

- おうごんのからだは `protected` フラグを持っていない（かたやぶりで貫通可能）
- `Event.ON_BEFORE_APPLY_MOVE`（`おうごんのからだ_block_status_move`, priority 100,
  `subject_spec="defender:self"`）で変化技全般の無効化処理として実装されているため、
  スキルスワップ固有のハンドラ側追加ガードは不要

### とくせいガードについて

- 自分か対象のどちらかがとくせいガード（アイテム）を持つ場合に失敗する。
  `スキルスワップ_can_apply` は `battle.ability_manager.is_change_blocked(ctx.attacker)`
  / `is_change_blocked(ctx.defender)`（内部で `ON_CHECK_ABILITY_DISABLE` を発火し
  `とくせいガード_check_ability_disable` を評価する）で使用者・対象の両方を判定済み

### swap_ability の内部挙動

- `battle.ability_manager.swap_ability(mon1, mon2)` は先頭で `is_change_blocked` を
  再チェックしたうえで、内部で `ON_ABILITY_DISABLED` → ハンドラ解除 → ハンドラ再登録 →
  `ON_ABILITY_ENABLED` を発火するため、入れ替え直後に新特性が発動する挙動は
  自動的に再現される
- 新特性の発動順は `for mon, ability in ((mon1, ability2), (mon2, ability1))` の順で
  処理されるため、`mon1`（使用者=`ctx.attacker`）側の新特性が先に発動する
  （「第五世代以降ではすばやさに依らずスキルスワップ使用者側から先に発動する」という
  一次情報の記述と一致）
- 交代時は `base_ability_name` に基づいて元の特性に戻るため、交代後はスワップ前の特性に戻ることが期待される

### 同じ特性どうしのスワップ

- 使用者と対象が同じ特性の場合でも第六世代以降は成功する（ポケモンチャンピオンズは第九世代ルールのため成功）
- `swap_ability` 内部では同名特性のスワップも問題なく処理できるため特別なガードは不要

### みがわり状態との相互作用

- 対象がみがわり状態でも成功する（一次情報に明記あり）
- `flags={"bypass_substitute"}` によりみがわりを貫通する

### マジックコート・よこどり

- マジックコートによる反射は「×」（反射されない）。`flags` に `unreflectable` を
  追加することで対応する（`is_reflectable` プロパティが `category == "status" and
  target in ("foe", "foe_side") and not has_flag("unreflectable")` で判定するため、
  フラグを付けないと誤って反射されてしまう。修正前の実装ではこのフラグが欠落していた）
- よこどりによる奪取は対象外（Championsのルール上使用不可の技のため未実装）
