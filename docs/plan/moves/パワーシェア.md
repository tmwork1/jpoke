# 計画書: パワーシェア

## 仕様要約

- タイプ: エスパー / カテゴリ: 変化 / PP: 12 / 命中: 必中 / 対象: 相手
- 使用者と相手の「こうげき（A）」「とくこう（C）」の**実数値**を平均化する
- ランク変化は一切行わない（実数値のみ書き換える）
- 平均化の計算式: `(使用者のA + 相手のA) // 2` を両者のAに、Cも同様
- まもる: ○ / マジックコート: × / 命中判定: 必中（accuracy 欄なし）

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT` 内部の priority 表は掲載されていない。既存の変化技ハンドラ（`ガードシェア_equalize_stats`・`いたみわけ_equalize_hp`・`りゅうのまい_modify_attacker_stats` 等）はすべて priority 100（`MoveHandler` デフォルト）を使用している。パワーシェアも同一パターンで priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「は」行 — 既存の `はらだいこ_can_apply` / `はらだいこ_apply` の後、`はきだす` 系関数群の前後に配置する（「は」行内の五十音順: はきだす < はらだいこ < パワーシェア（ぱ → は行扱い））。実際には「ガ」行にある `ガードシェア_equalize_stats` の直後（「か」→「は」順だが、カタカナは「か行」に位置するため）、既存ファイル内の並びに従いカタカナ「パ」が「は行」相当として配置する。

実数値の書き換えは `mon._stats_manager.stats` リストへの直接代入で行う（インデックス: A=1、C=3）。`modify_stats` は**ランク変化**用 API のため使用しない。

```python
def パワーシェア_equalize_stats(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """パワーシェア: 使用者と相手のこうげき・とくこうの実数値を平均化する。

    ランク変化は行わず、実数値のみを書き換える。
    平均は切り捨て（// 2）。
    """
    attacker = ctx.attacker
    defender = ctx.defender
    # A=インデックス1、C=インデックス3
    for idx in (1, 3):
        avg = (attacker._stats_manager.stats[idx] + defender._stats_manager.stats[idx]) // 2
        attacker._stats_manager.stats[idx] = avg
        defender._stats_manager.stats[idx] = avg
    return HandlerReturn(value=value)
```

### `src/jpoke/data/move.py`

既存の `"パワーシェア"` エントリ（6161行付近）に `handlers` を追加する。

```python
"パワーシェア": MoveData(

    category="変化",
    pp=12,
    accuracy=100,
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(

```

## 注意点・エッジケース

- **実数値変更 vs ランク変化の区別**: `battle.modify_stats()` はランク変化 API（`status_manager.modify_stats` 委譲）であり、実数値には触れない。パワーシェアは実数値平均化のため `_stats_manager.stats` を直接書き換える必要がある
- **`stats` property の読み取り専用制約**: `mon.stats` は `_stats_manager.stats_dict`（dict コピー）を返すため、`mon.stats["A"] = avg` の代入は `_stats_manager.stats` に反映されない。`_stats_manager.stats` リストを直接参照して代入する
- **HPへの非影響**: H=インデックス0 は対象外。誤って `_stats[0]` を変更しないよう注意する（`max_hp` ズレが生じる）
- **ランク非変化の確認**: `ON_BEFORE_MODIFY_STAT` や `ON_MODIFY_STAT` 系イベントは一切発火しない設計のため、クリアボディ・かがくへんかガス等の能力変化ガードは影響しない
- **ガードシェアとの対称性**: ガードシェア（B=2・D=4）と全く同じパターン。インデックスのみ A=1・C=3 に変わる
- **ワンダールーム中の扱い**: ワンダールーム効果（B/D 参照入れ替え）は `ON_CALC_DEF_MODIFIER` 内で対処されるため、パワーシェアハンドラ側では考慮不要
- **みがわり貫通の有無**: 変化技のみがわりチェックは `ON_APPLY_MOVE` の `みがわりによる無効化` (priority 30) で処理されるため、ハンドラ側でガードは不要。みがわり状態の相手には技が届かない（貫通なし）
- **`_all_moves.md` の accuracy 欄**: 必中技のため `accuracy` フィールドは `100` で登録されているが、命中判定は内部で必中扱いとなる。既存のガードシェアエントリも `accuracy=100` で登録されており同一扱いとする
