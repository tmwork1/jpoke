# 計画書: おかたづけ

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 10 / 命中: なし / 対象: 自分
- 使用者を含む全ポケモンのみがわり状態を除去する（両陣営）
- 両陣営のサイドフィールドに設置されたトラップ（まきびし・どくびし・ステルスロック・ねばねばネット）をすべて除去する
- 自分のこうげきとすばやさを1段階ずつ上げる
- みがわりかトラップを1つでも除去した場合のみ「かたづけ おわり!」メッセージを出す
- みがわり・トラップが1つも存在しない場合でも、能力上昇は発生する
- まもる: × / マジックコート: × / 命中判定: ×

### wiki 仕様詳細（技の仕様節）

- 解除順序: みがわり（A→a→B→bの順）→ホスト側トラップ（まきびし→どくびし→ステルスロック→ねばねばネット）→相手側トラップの同順
- ランクがすでに最大のときも、みがわり・トラップの除去効果は発動する
- 姿を隠しているポケモンのみがわり状態も除去できる
- 使用者特性がすりぬけのときもみがわりを除去できる

`docs/spec/turn.md` には `ON_STATUS_HIT` の内部priority表がない。既存の変化技ハンドラ（`りゅうのまい_modify_attacker_stats`、`おいかぜ_set_side_field` など）がすべて priority 100（デフォルト）を使用している。おかたづけも同一パターンとして priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「お」行 — `おいかぜ_set_side_field` の後、`おたけび_modify_defender_stats` の前に挿入する。

```python
def おかたづけ_cleanup(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """おかたづけ: 両陣営のみがわり・トラップを除去し、こうげき・すばやさを1段階上げる。

    みがわりまたはトラップを1つでも除去した場合のみ除去メッセージを出力する。
    除去物がない場合でも能力上昇は発生する。
    """
    cleaned = False

    # 両陣営のみがわり状態を解除（battle.actives はシングルバトルでは2体）
    for mon in battle.actives:
        if mon.has_volatile("みがわり"):
            battle.volatile_manager.remove(mon, "みがわり")
            cleaned = True

    # 両陣営のトラップを解除
    trap_names: list[SideFieldName] = ["まきびし", "どくびし", "ステルスロック", "ねばねばネット"]
    for side in battle.side_managers:
        for trap in trap_names:
            if side.deactivate(trap):
                cleaned = True

    if cleaned:
        battle.add_event_log(ctx.attacker, LogCode.MOVE_CUSTOM,
                             payload={"message": "かたづけ おわり!"})

    # こうげき・すばやさを1段階上げる
    return modify_attacker_stats(battle, ctx, value, stats={"A": 1, "S": 1})
```

### `src/jpoke/data/move.py`

既存の `"おかたづけ"` エントリに `handlers` を追加する。

```python
"おかたづけ": MoveData(

    category="変化",
    pp=10,

        ),

```

## 注意点・エッジケース

- **「かたづけ おわり!」メッセージのLogCode**: `LogCode.MOVE_CUSTOM` が適切なコードか確認すること。既存の類似メッセージ出力パターンを参照して適切な `LogCode` に変更する
- **SideFieldName のインポート**: `trap_names` の型注釈に `SideFieldName` を使う場合は `from jpoke.types import SideFieldName` が必要。型注釈を省いてもよい
- **ランク上限でも除去は発動**: wiki仕様「こうげき・すばやさがすでに最大のときでも、みがわり・トラップの除去効果は発動する」に注意。`ON_TRY_MOVE_2` priority 130 で「ランク補正上げ: すでに最大」の失敗チェックが入る可能性がある。ただし、おかたづけはランク上昇「以外」の効果（除去）を持つため、ランクが最大でも技自体は成功とすべき。`ON_TRY_MOVE_2` のガードが除去効果より前に stop_event してしまわないか確認が必要
- **失敗条件なし**: まきびし等のチェック `side.activate` の失敗判定（`stop_event=True`）を他の設置技ハンドラでは使っているが、おかたづけは何もなくても成功するため、各 `deactivate` の戻り値は bool フラグのみに使い `stop_event` は使わない
- **両陣営への適用**: `battle.get_side(ctx.attacker)` だけでなく `battle.side_managers` を直接走査することで両サイドを確実に処理できる
- **こうそくスピン・きりばらい との違い**: こうそくスピン（攻撃技）は自陣トラップのみ除去。きりばらい（変化技）は壁系（対象側のみ）・トラップ（両陣営）・フィールドを除去し、回避率も-1する。きりばらいのトラップ/壁/フィールド除去は `きりばらい_defog` として実装済み（`handlers/move_status.py`）
- **みがわり除去ログ**: `battle.volatile_manager.remove` は内部で `LogCode.VOLATILE_REMOVED` を出力するため、個別ログを追加する必要はない
- **トラップ除去ログ**: `side.deactivate` は `_deactivate_field` 経由で `ON_FIELD_DEACTIVATE` を発火するため、フレームワーク側でログが出る想定。`cleaned` フラグのみ取得してメッセージ判定に使う
