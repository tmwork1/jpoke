# 実装計画: いびき

## 仕様サマリ（docs/spec/moves/いびき.md より）

- タイプ: ノーマル / 分類: 特殊 / 威力: 50 / 命中率: 100 / PP: 15 / 対象: 相手1体 / 優先度: 0 / 直接攻撃: ×
- 音のわざの1つ
- 自分が **ねむり状態**（`ぜったいねむり` によるゆめうつつ状態を含む）であるときのみ成功する。ねむり状態でない場合は失敗する
- 追加効果として30%の確率で相手をひるませる
- まもるで防がれる

## 現状の実装状況

| コンポーネント | 状況 |
|---|---|
| `data/move.py` の `いびき` エントリ | あり（`type`, `category`, `pp`, `power`, `accuracy`, `flags={"sound","secondary_effect"}`） |
| `handlers/move_attack.py` の `いびき_apply_flinch`（`ON_DAMAGE_HIT`） | あり（30%ひるみ、既存実装で問題なし） |
| `handlers/ailment.py` の `ねむり_check_action` | あり（いびき・ねごとはねむり中の行動不能チェックをパスする） |
| 「自分がねむり状態でないと失敗する」チェック | **未実装**（欠落） |

現在の `いびき` MoveData:

```python
"いびき": MoveData(
    type="ノーマル",
    category="special",
    pp=15,
    power=50,
    accuracy=100,
    flags={"sound", "secondary_effect"},
    handlers={
        Event.ON_DAMAGE_HIT: h.MoveHandler(
            ha.いびき_apply_flinch,
        )
    }
),
```

`ねむり_check_action`（`handlers/ailment.py`）は「ねむり状態のポケモンが行動しようとしたとき、いびき・ねごとだけは行動不能チェックを通過させる」処理のみを担当しており、
「ねむり状態でないポケモンがいびきを選んだ場合に失敗させる」処理は登録されていない。
そのため、現状は覚醒中のポケモンでも `いびき` を通常のダメージ技として使用できてしまう（一次情報の「自分がねむり状態であるときのみ成功する」という失敗条件が未実装）。

## ハンドラ構成（追加分）

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_TRY_MOVE_1` | 30 | `attacker:self` | 自分がねむり状態でない場合に失敗させる |

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_TRY_MOVE_1`（技使用可能かどうかの判定・相性判定より前）テーブルに以下の記述がある：

```
| 30 | いびき・ねごと: ねむり状態でない |
```

したがって `Event.ON_TRY_MOVE_1` / priority=30 を採用する。

注記: 既存の `ねごと`（`data/move.py`）は同じ失敗条件を `Event.ON_TRY_ACTION`（priority=30）に登録しているが、
turn.md 上でこの行は `Event.ON_TRY_MOVE_1` のテーブルに記載されている。本計画では turn.md の記載に忠実な
`Event.ON_TRY_MOVE_1` を採用する（ねごと側の既存実装の当否は本タスクの対象外）。

## 実装箇所

### handlers/move_attack.py への追加

挿入位置: 五十音順で `いのちのしずく` 系のハンドラ（存在する場合）の後、`いびき_apply_flinch` の前
（`python scripts/sort_handlers.py` で最終的な並びを整える）。

```python
def いびき_check_sleep(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """いびきの発動条件チェック: 自分がねむり状態（ゆめうつつ含む）でない場合に失敗させる。"""
    if not ctx.attacker.ailment.is_sleep:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload={"reason": "いびき_ねむり状態でない"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

- `ctx.attacker.ailment.is_sleep` を使うことで、通常の「ねむり」状態異常だけでなく `ぜったいねむり` 特性による
  「ゆめうつつ」状態異常（`data/ailment.py` で `is_sleep=True` 定義済み）のポケモンも成功扱いとなる。
  一次情報の「特性ぜったいねむりのポケモンが使用するときは常に成功する」を満たす。
- `ねごと_check_sleep`（`handlers/move_status.py`）は `has_ailment("ねむり")` のみで判定しており `ゆめうつつ` を
  考慮していないが、これは `ねごと` 側の既存実装であり本タスクでは変更しない。

### data/move.py の更新

`いびき` の `handlers` に `ON_TRY_MOVE_1` を追加する。

## 注意点・エッジケース（review-test 向け）

### ねむり状態回復とのタイミング

- `handlers/ailment.py` の `ねむり_check_action`（`ON_TRY_ACTION` priority=10）でねむりカウントを消費し、
  カウントが0になった場合はそのターンのうちに `ailment_manager.remove` される。
- `ON_TRY_ACTION` は `ON_TRY_MOVE_1` より前に評価されるため、「いびきを選択したターンに目が覚めた場合、
  いびきは失敗する」という一次情報の仕様（`docs/spec/ailments/ねむり.md` 参照）は、
  本チェック（`ON_TRY_MOVE_1` で `ailment.is_sleep` を見る）が `ねむり_check_action` 実行後の状態を見ることで
  自然に満たされる。

### ぜったいねむり特性との関係

- `ぜったいねむり` 特性のポケモンは常に「ゆめうつつ」状態異常（`is_sleep=True`）を持つため、`いびき` は常に成功する。

### 既存テストへの影響

- 既存テスト `test_いびき_ひるみが発動する`（`tests/moves_attack/test_move_a.py`）は `ailment0=("ねむり", 3)` で
  すでにねむり状態を付与しているため、本チェック追加後も成功しひるみが発動する（回帰なし）。
- 新規に「ねむり状態でない場合は失敗する」ケースのテストを追加する。
