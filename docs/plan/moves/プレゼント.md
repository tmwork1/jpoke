# 実装計画書: プレゼント

## 仕様要約

- タイプ: ノーマル / カテゴリ: 物理 / 威力: - / 命中率: 90 / PP: 15 / 対象: 1体選択
- 使用時に以下の確率でランダムにいずれかの効果が選ばれる。
  - 威力40の攻撃（40%）
  - 威力80の攻撃（30%）
  - 威力120の攻撃（10%）
  - 相手の最大HPの1/4を回復（20%）
- 回復が選ばれた場合も命中判定は行われる。相手のHPが満タンの場合は失敗する。
- 回復量の小数点以下は切り捨てる。

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| `data/move.py` エントリ | 実装済み（`power=0` のベース値） |
| `handlers/move_attack.py` 関数 | 実装済み（プレゼント_roll_outcome, プレゼント_check_heal_full, プレゼント_apply_heal） |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_TRY_MOVE_1` | 100（デフォルト） | `attacker:self`（デフォルト） | 乱数で `ctx.move.power` を 40/80/120 に設定するか、`0` のまま維持して回復ルートにする |
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 回復ルート時、相手のHPが満タンなら失敗させる |
| `ON_MODIFY_MOVE_DAMAGE` | 100（デフォルト） | `attacker:self`（デフォルト） | 回復ルート時、通常のダメージ計算結果を0に上書きし、相手のHPを1/4回復させる |

## Priority 根拠

- `ON_TRY_MOVE_1` は `docs/spec/turn.md` に掲載されているイベントだが、本ハンドラの役割（威力のランダム決定）に対応する行は明記されていない。既存の類似ハンドラ `なげつける_check_item`（アイテムに応じて `ctx.move.power` を設定する、priority=30）を参考にした結果、`なげつける` は「アイテム無効による失敗判定」も兼ねる priority=30 グループに属する一方、プレゼントの威力決定は失敗判定を伴わないため、他のON_TRY_MOVE_1ハンドラとの実行順が問題にならないデフォルト優先度(100)を採用した。
- `ON_BEFORE_APPLY_MOVE` / `ON_MODIFY_MOVE_DAMAGE` も turn.md に priority 表がないため、同種の回復判定ハンドラ（`いのちのしずく_heal` 等の HP満タン失敗チェック）に倣いデフォルト優先度を採用した。

## 実装詳細・エッジケース

- `category="physical"` のため、回復ルートでも `_execute_hit` は通常のダメージ計算フロー（`battle.roll_damage` → `ON_MODIFY_MOVE_DAMAGE`）を通過する。`プレゼント_apply_heal` が `damage` を `0` に上書きするため、実際のダメージは発生しない。
- 命中判定はカテゴリに関わらず技の `accuracy=90` に従って通常通り行われるため、「回復が選ばれた場合も命中判定は行われる」仕様を満たす。
- HP満タン失敗時は物理技カテゴリのため `move_executor.py` の自動MOVE_FAILEDログが出ないパスとなる（`ON_STATUS_HIT` 専用の自動ログは変化技カテゴリのみ）。そのため `プレゼント_check_heal_full` 内で明示的に `battle.add_event_log(..., LogCode.MOVE_FAILED, ...)` を呼び出している（`なげつける_check_item` と同様のパターン）。
- 既知の限界（対応不要と判断）: 回復ルートでも物理カテゴリのため急所判定自体は内部的に走るが、`ON_MODIFY_MOVE_DAMAGE` でダメージが0に上書きされるため実害はない（急所ログが表示され得る点のみ許容する軽微な仕様差）。
