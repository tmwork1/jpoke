# 実装計画: ねごと

## 仕様サマリ

- タイプ: ノーマル / 分類: 変化 / PP: 10 / 命中率: -（対象選択なし）
- 効果: 自分がねむり状態のときのみ成功する。自分の覚えている技の中から `non_negoto` ラベルを持たない技をランダムに1つ選び、その技を実行する
- 対象: 自分（技の効果はランダムに選ばれた技に依存）
- 直接攻撃: 選ばれた技依存
- まもる: 選ばれた技依存
- ゆびをふる: 出ない（第五世代以降）
- `non_negoto` ラベルを持つ技は選ばれない（きあいパンチ・くちばしキャノン・わるあがき など）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_TRY_ACTION` | 30 | `attacker:self` | ねむり状態でない場合に失敗させる |
| `ON_MODIFY_MOVE` | 100 | `attacker:self` | ねごとで出せる技がない場合はNoneを返して失敗させる。技があればランダム選択した技を ctx に返す |

## Priority 根拠

### ON_TRY_ACTION Priority 30

`docs/spec/turn.md` の `Event.ON_CHECK_ACTION` テーブルに以下の記述がある：

```
| 30 | いびき・ねごと: ねむり状態でない |
```

したがって priority=30 を採用する。

### ON_MODIFY_MOVE Priority 100（技の置き換え + 出せる技なし失敗）

`docs/spec/turn.md` の `Event.ON_CHECK_ACTION` テーブルに以下の記述がある：

```
| 150 | 他の技が出る技により技を置き換え |
| 160 | ねごとで出せる技が無いときは失敗 |
```

実装上、この2つの処理は `ON_MODIFY_MOVE` イベントで1つのハンドラとして実装する。理由は以下の通り：

- `ON_MODIFY_MOVE` は `ON_TRY_ACTION` より前に発火する（move_executor.py 行 245 vs 行 268）
- アンコール（`ON_MODIFY_MOVE` priority=200）が技置き換えに同イベントを使用しており、同パターンが適切
- 技置き換えは `ON_TRY_ACTION` で ctx.move を書き換えるより `ON_MODIFY_MOVE` の返り値として技を差し替える方が既存フレームワークの設計に合致する
- 「技なし失敗」は技選択不可を `None` 返却で表現できる（move_executor.py 行 246-247 で None チェック済み）

priority は既存の技ハンドラのデフォルト値 100 を採用する（アンコールの ON_MODIFY_MOVE は 200 であるが、ねごとはアンコールより後に評価される必要はなく、アンコール中はすでにアンコール優先で技が差し替わっているため問題なし）。

## 実装箇所

### handlers/move_status.py への追加

挿入位置: 五十音順で `ねばねばネット_*` と `ねむりごな_*` の間（ね→ねごと→ねばねば→ねむり）。正確には `ねをはる_*` の前、`ねばねばネット_*` の後。

```python
def ねごと_check_sleep(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねごとの発動条件チェック: ねむり状態でない場合に失敗させる。"""
    if not ctx.attacker.has_ailment("ねむり"):
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "ねごと_ねむり状態でない"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ねごと_select_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねごとで使う技をランダムに選ぶ。

    non_negoto ラベルを持たない技の中からランダムに選択する。
    選べる技がない場合は None を返して技の実行を中止する。
    """
    attacker = ctx.attacker
    candidates = [m for m in attacker.moves if not m.has_label("non_negoto")]
    if not candidates:
        battle.add_event_log(
            attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "ねごと_候補技なし"},
        )
        return HandlerReturn(value=None, stop_event=True)
    chosen = battle.random.choice(candidates)
    return HandlerReturn(value=chosen, stop_event=True)
```

## 注意点・エッジケース（review-test 向け）

### ねむり状態チェックの二重構造について

- `handlers/ailment.py` の `ねむり_check_action`（`ON_TRY_ACTION` priority=10）は「ねむり中のポケモンが行動しようとしたとき」のハンドラで、すでに `いびき` と `ねごと` を通過させている
- しかし「ねむり状態でないポケモンがねごとを使う場合」にはこのハンドラが登録されていないため、別途 priority=30 の失敗ハンドラが必要
- 結果として、ねむり状態でないときは priority=30 のハンドラが失敗させ、ねむり中は ailment の priority=10 が通過させた後、priority=30 のハンドラが（ねむり状態なので）スキップされて ON_MODIFY_MOVE に進む

### ON_MODIFY_MOVE で None を返した場合の挙動

- `move_executor.py` 行 245-247 にて `if move is None: return` が実装済み
- `ねごと_select_move` で候補がない場合 `value=None` を返すと、そのまま技の実行が中止される

### ねごとで選んだ技が「変化技」の場合

- 選ばれた技が変化技であれば `ON_STATUS_HIT` に進み通常の変化技フローで処理される
- `ちょうはつ` 状態中にねごとで変化技が選ばれた場合、`ちょうはつ_try_action`（`ON_TRY_ACTION` priority=200）がねごとの `ON_TRY_ACTION` (priority=30) の後に評価されるため、ちょうはつで技が封じられる可能性がある
  - 正確な挙動: turn.md の記述でちょうはつは「ON_CHECK_ACTION priority=100」に分類されており、ねごとの選択（priority=150）より前に評価されるべきである
  - ただし `ON_TRY_ACTION` と `ON_MODIFY_MOVE` の分離により、ちょうはつチェックの実質的な評価順は複雑になる。現状の実装ではちょうはつが priority=200 で `ON_TRY_ACTION` に登録されており、ねごとの置き換え後（ON_MODIFY_MOVE）の技カテゴリをチェックできない
  - この相互作用は現行フレームワークの制約上、深い実装変更が必要であり、初期実装では対象外とする

### PP の消費

- ねごとが成功した場合、ねごと自体の PP が消費される（move_executor.py の `_consume_pp` はON_TRY_ACTIONより前に実行）
- 選ばれた技の PP は消費されない

### アンコールとの相互作用

- `ねごと` は `non_encore` ラベルを持つためアンコールの対象外

### ちょうはつ状態との相互作用

- `ねごと` 自体は変化技のため、ちょうはつ状態で選択できない。ただしちょうはつは `ON_TRY_ACTION` で技の使用を封じ、ねごとも変化技であるため「ちょうはつ中はねごとを使えない」は turn.md どおりに機能する
- ねごとで選ばれた技に対するちょうはつ影響は前述の「注意点」を参照

### ぜったいねむり特性のポケモン

- `ぜったいねむり` 特性を持つポケモンは常にねむり状態であるため、ねごとが常に成功する。ailment.py の実装がどうあれ、ねごとの失敗チェックはねむり状態を確認するだけでよい

### ねごとで選ばれた技が「わるあがき」相当の場合

- `わるあがき` は `non_negoto` ラベルを持つため、ねごとで選ばれることはない

### 候補技がすべて non_negoto の場合

- 例: 覚えている技がすべて `non_negoto` ラベル付き（わるあがき、きあいパンチ等のみ）の場合、候補なしで失敗する
- 技スロットが4つとも `non_negoto` である実際の構成は稀だが、テストケースとして確認が必要
