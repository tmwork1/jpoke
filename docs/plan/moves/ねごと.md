# 実装計画: ねごと

## 仕様サマリ

`docs/spec/moves/ねごと.md` を参照。

- タイプ: ノーマル / 分類: 変化 / PP: 12（champions仕様） / 命中率: -（対象選択なし）
- 効果: 自分がねむり状態のときのみ成功する。自分の覚えている技の中から `non_negoto` フラグを
  持たない技をランダムに1つ選び、その技をそのまま実行する
- 対象: 自分（技の効果はランダムに選ばれた技に依存）
- 直接攻撃: 選ばれた技依存
- まもる: 選ばれた技依存
- ゆびをふる: 出ない（第五世代以降）
- `non_negoto` フラグを持つ技は選ばれない（きあいパンチ・くちばしキャノン・わるあがき・
  溜め技（ソーラービーム等）など）

## ハンドラ構成（実装済み）

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_TRY_MOVE_1` | 30 | `attacker:self` | ねむり状態でない場合に失敗させる |
| `ON_MODIFY_PP_CONSUMED` | デフォルト(100) | `attacker:self` | ねごとのサブ実行中は選ばれた技のPP消費を0にする |
| `ON_STATUS_HIT` | デフォルト(100) | `attacker:self` | 候補技をランダムに1つ選び、`battle.run_move` でそのまま実行する |

## Priority 根拠

### ON_TRY_MOVE_1 Priority 30（レビューで ON_TRY_ACTION から修正）

`docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` テーブルに以下の記述がある：

```
| 30 | いびき・ねごと: ねむり状態でない |
```

したがって `Event.ON_TRY_MOVE_1` priority=30 を採用する。
（`docs/spec/turn.md` には同内容の記述が古い `Event.ON_CHECK_ACTION` セクションにも残っているが、
このイベント名は現行コードベースの `Event` enum に存在しない旧称であり、実質的に
`ON_TRY_ACTION` + `ON_TRY_MOVE_1` に分割されている。姉妹技である `いびき` が
`Event.ON_TRY_MOVE_1` priority=30 で同じ判定を実装しているため、これに合わせる。）

`ON_TRY_ACTION` ではなく `ON_TRY_MOVE_1` を使う理由は、PP消費のタイミングと直結する：
`move_executor.run_move` は `ON_TRY_ACTION` の成否判定の直後に `_consume_pp` を実行し、
その後 `_execute_move`（内部で `ON_TRY_MOVE_1` を発火）に進む。
ねごとは「ねむり状態でないために失敗した場合でも自身のPPは消費される」という wiki 仕様が
あるため、ねむり判定は `_consume_pp` より後の `ON_TRY_MOVE_1` で行う必要がある。
`ON_TRY_ACTION` に置くと PP消費前に失敗するため、失敗時にPPが消費されない誤った挙動になる
（レビューで発見した回帰）。

### ON_STATUS_HIT / ON_MODIFY_PP_CONSUMED

status技の効果本体は `ON_STATUS_HIT` で実行するのが `handlers/move_status.py` 全体の標準パターン
（他の変化技ハンドラと同様）であり、優先度は明示的に競合する処理がないためデフォルト値を採用する。
`ON_MODIFY_PP_CONSUMED` も同様にデフォルト値で問題ない（ねごと自身のPP消費計算とは無関係に、
サブ実行中フラグの有無だけで判定するため他ハンドラとの前後関係に依存しない）。

初期計画では `ON_MODIFY_MOVE` で技を置き換える設計を検討したが、以下の理由から
`ON_TRY_MOVE_1`（失敗判定）+ `ON_STATUS_HIT`（技の選択・即時実行）+
`ON_MODIFY_PP_CONSUMED`（PP消費抑制）の3イベント構成を採用した：

- `ON_MODIFY_MOVE` で技を差し替える方式だと、差し替え後の技が `move_executor.run_move` の
  トップレベルフロー（PP消費・命中判定・ダメージ適用等）にそのまま乗ってしまい、
  「ねごと自身のPPだけを消費し、選ばれた技のPPは消費しない」という仕様を素直に表現できない
- `battle.run_move(attacker, chosen)` を `ON_STATUS_HIT` 内から直接呼び出すことで、
  選ばれた技を完全に独立したサブ実行として扱え、ねごと自身のPP消費・失敗ログ等と
  選ばれた技の処理を明確に分離できる

## 実装箇所

### handlers/move_status.py

```python
def ねごと_check_sleep(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねごとの発動条件チェック: ねむり状態でない場合に失敗させる。"""
    if not ctx.attacker.has_ailment("ねむり"):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ねごと_ねむり状態でない"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ねごと_select_and_execute(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねごとで選んだ技を実行する。

    non_negoto フラグを持たない技の中からランダムに選択し、そのままバトルで実行する。
    選ばれた技の PP は消費しない（ねごと自体のみ消費）。
    候補技がすべて non_negoto の場合は value=False を返して失敗する。

    ねごと自身の ON_TRY_MOVE_1（ねむりチェック）・ON_STATUS_HIT（本ハンドラ）は、
    選ばれた技の実行中は一時的に解除する（再帰・自己チェックの誤発火防止のため）。
    """
    attacker = ctx.attacker
    candidates = [m for m in attacker.moves if not m.has_flag("non_negoto")]
    if not candidates:
        battle.add_event_log(
            attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ねごと_候補技なし"),
        )
        return HandlerReturn(value=False, stop_event=True)
    chosen = battle.random.choice(candidates)
    attacker.sleep_talk_active = True
    suppressed_events = (Event.ON_TRY_MOVE_1, Event.ON_STATUS_HIT)
    handlers_data = ctx.move.data.handlers
    for event in suppressed_events:
        handler = handlers_data.get(event)
        if handler is None:
            continue
        for h in (handler if isinstance(handler, list) else [handler]):
            battle.events.off(event, h, ctx.attacker)
    try:
        battle.run_move(attacker, chosen)
    finally:
        for event in suppressed_events:
            handler = handlers_data.get(event)
            if handler is None:
                continue
            for h in (handler if isinstance(handler, list) else [handler]):
                battle.events.on(event, h, ctx.attacker)
        attacker.sleep_talk_active = False
    return HandlerReturn(value=value)


def ねごと_suppress_pp(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねごとのサブ実行中は選ばれた技のPP消費を0にする。"""
    if ctx.attacker.sleep_talk_active:
        return HandlerReturn(value=0, stop_event=True)
    return HandlerReturn(value=value)
```

### core/move_executor.py（コア側の修正）

`run_move` 内の「PPが0の技はわるあがきに置き換える」処理は、ねごとのサブ実行中は行わない：

```python
if move.pp == 0 and not attacker.sleep_talk_active:
    move = Move("わるあがき")
```

理由: ねごとで選ばれた技はPPを消費しないため、既に残りPPが0の技が選ばれても
そのまま成功する（wiki仕様どおり）。この置き換えを抑制しないと、
ねごとで選ばれた0PP技がわるあがきに化けてしまう。

### handlers/item.py（こだわり系アイテムとの相互作用）

`こだわり_lock_move`（`ON_MOVE_END` で発火）は `sleep_talk_active` の間はロックしない：

```python
def こだわり_lock_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    mon = ctx.attacker
    if mon.sleep_talk_active:
        return HandlerReturn(value=value)
    if not mon.has_volatile("こだわり"):
        battle.volatile_manager.apply(
            mon, "こだわり", source=mon, move_name=ctx.move.name
        )
    return HandlerReturn(value=value)
```

理由: `battle.run_move(attacker, chosen)` によるサブ実行でも `_execute_move` 内で
`ON_MOVE_END` が発火するため、何もガードしないと選ばれた技の名前でロックされてしまう
（第四世代までの仕様）。第五世代以降は「ねごと」自体でロックされる仕様のため、
サブ実行中の `ON_MOVE_END` では何もせず、`sleep_talk_active` が戻った後に発火する
ねごと自身の `ON_MOVE_END` でロックさせる。

### handlers/ailment.py（ねむりカウントの二重消費対策）

`ねむり_check_action`（`ON_TRY_ACTION`、ねむりのターンカウントを消費するハンドラ）は、
`battle.run_move(attacker, chosen)` によるサブ実行でも同じ攻撃者に対して再度 `ON_TRY_ACTION`
が発火するため、ガードなしでは1ターンでねむりのカウントが2回消費されてしまう
（レビューで手動検証: `ailment0=("ねむり", 3)` でねごとを1回使うと `count` が `3→1` になる回帰）。
`mon.sleep_talk_active` が真の間は `tick` を呼ばず素通りするよう修正した：

```python
def ねむり_check_action(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    mon = ctx.attacker
    if mon.sleep_talk_active:
        return HandlerReturn(value=True)
    battle.ailment_manager.tick(mon)
    if not mon.has_ailment("ねむり"):
        battle.ailment_manager.remove(mon)
        return HandlerReturn(value=True)
    if ctx.move.name in ["いびき", "ねごと"]:
        return HandlerReturn(value=True)
    battle.add_event_log(...)
    return HandlerReturn(value=False, stop_event=True)
```

### 溜め技への non_negoto フラグ付与（`data/moves/*.py`）

以下の実装済み溜め技（`ON_MOVE_CHARGE` を使い、攻撃者自身に溜め状態を付与して行動を
1ターン消費するタイプ）に `non_negoto` フラグを追加した：

あなをほる・エレクトロビーム・フリーズボルト・コールドフレア・ゴッドバード・
ゴーストダイブ・メテオビーム・シャドーダイブ・そらをとぶ・ソーラービーム・
ソーラーブレード・ダイビング・とびはねる

はめつのねがい・みらいよちも `ON_MOVE_CHARGE` を使うが、これらは「相手陣営に設置して
即座に行動終了する」技であり、攻撃者自身が2ターン目に改めて攻撃するタイプの溜め技では
ないため、ねごとで選ばれる対象から除外しない（wiki の「選ばれる技」一覧に明記あり）。

未実装の技（オウムがえし・さきどり・しぜんのちから・ねこのて・がまん・トラップシェル等）は
`non_negoto` フラグを付与できないため対象外。実装時に各技の担当レビューで追加すること。

## 注意点・エッジケース

### PP の消費

- ねごとの PP は `_consume_pp`（`ON_TRY_ACTION` 通過直後）で必ず1消費される。
  ねむり状態でないために `ON_TRY_MOVE_1` で失敗した場合も、その時点ではすでに
  PP消費が終わっているため、失敗時もねごと自身のPPは消費される（wiki仕様どおり）
- 選ばれた技の PP は消費されない（`ねごと_suppress_pp` により `ON_MODIFY_PP_CONSUMED` で 0 を返す）

### アンコールとの相互作用

- `ねごと` は `non_encore` フラグを持つためアンコールの対象外

### ちょうはつ状態との相互作用

- `ねごと` 自体は変化技のため、ちょうはつ状態で選択できない
- ねごとで選ばれた技に対するちょうはつの影響は、選ばれた技自身の `ON_TRY_ACTION` /
  `ON_TRY_MOVE_1` 等の通常フローに委ねられる（サブ実行として通常の `run_move` を通るため）

### ぜったいねむり特性のポケモン

- `ぜったいねむり` 特性を持つポケモンは常にねむり状態であるため、ねごとが常に成功する
- ailment.py の実装に関わらず、ねごとの失敗チェックはねむり状態を確認するだけでよい

### 「選ばれない技しか覚えていないためにねごとが失敗したときはこだわらない」について

- Gen5以降の細かい仕様だが、現行コアアーキテクチャでは `_execute_move` が
  `ON_STATUS_HIT` の成否に関わらず `ON_MOVE_END` を発火するため未対応
- 候補0件になるケース自体が非常に稀（`non_negoto` 技だけを4つ揃える必要がある）ため、
  初期実装では対象外とする
