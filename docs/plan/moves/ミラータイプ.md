# 実装計画書: ミラータイプ

参照: `docs/spec/moves/ミラータイプ.md`

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 16（`src/jpoke/data/ps-champ-ja/moves.json` 準拠） / 命中: なし / 対象: 1体選択
- 自分のタイプを対象の現在のタイプ（`defender.types`）と同じに変更する
- 対象がテラスタルしている場合はテラスタイプをコピーする
- 対象がステラタイプにテラスタルしている場合は元のタイプ（`data.types`）をコピーする
- 対象のタイプが完全に無い場合はノーマルタイプとして扱う
- **使用者がテラスタル中の場合は失敗する**
- 使用者のタイプが対象と同じ場合は失敗する（`docs/spec/turn.md` Priority 100 に明記）
- 使用者が交代またはひんしになった場合、タイプは元に戻る
- タイプ変更後は、もりののろい/ハロウィンによる自分の追加タイプ効果（added_types）はリセットされる
- まもるで防がれる（デフォルトの `target="foe"` のまま）
- マジックコートで跳ね返されない（`unreflectable` フラグ）
- みがわりを貫通する（`bypass_substitute` フラグ）

| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 使用者がテラスタル中、または使用者と対象のタイプが完全一致する場合に失敗 |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100（デフォルト）: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE`
  テーブルに「ミラータイプ: タイプ同じ」が priority 100 として明記されている。同種の
  `テクスチャー_can_apply` も明示的な priority 指定なし（デフォルト100）で実装されているため
  合わせる。
- `ON_STATUS_HIT` priority 100（デフォルト）: turn.md に記載なし。同種タイプ変更ハンドラ
  （テクスチャー・テクスチャー2）と同じデフォルト priority を採用する。

## タイプ置換の実装方針

`Pokemon.types` プロパティの優先順位は以下のとおりである:

1. テラスタル中: `[active_tera_type]`（ステラ時は `data.types`）
2. `move_override_types` が設定済み: `move_override_types`
3. `ability_override_type` が設定済み: `[ability_override_type]`
4. `volatile_override_type` が設定済み: `[volatile_override_type]`
5. それ以外: `data.types`
6. 上記に `added_types`（もりののろい・ハロウィン等）を追加し、`removed_types`
   （はねやすめ等、テラスタル中は無視）を除去する

ミラータイプは対象の「現在のタイプ」をそのままコピーする技であるため、**`defender.types`
プロパティをそのまま使う**（`defender.data.types` を直接参照しない）。これにより、対象の
テラスタル・特性によるタイプ変更・もりののろい/ハロウィンの追加タイプ・はねやすめによる
除去タイプが一括して考慮される。

コピーしたタイプは `attacker.move_override_types` に格納する（置換であり `added_types`
への追加ではない）。

`ability_override_type` は `Type | None`（単一タイプ）しか保持できないため、相手が2タイプの
場合に対応できない。そのため `Pokemon.move_override_types: list[Type] | None` を使用する
（`model/pokemon.py` に実装済み。ひんし・交代時に `None` へリセットされる）。

## 対象タイプの決定ロジック

```python
def ミラータイプ_取得_対象タイプ(defender: Pokemon) -> list[Type]:
    """ミラータイプでコピーする対象の現在のタイプを返す。

    defender.types はテラスタル・特性によるタイプ変更・もりののろい/ハロウィンによる
    追加タイプ・はねやすめ等による除去タイプをすべて反映済みのため、そのまま使用する。
    対象のタイプが完全に無くなっている場合はノーマルタイプとして扱う。
    """
    return list(defender.types) or ["ノーマル"]
```

## 失敗条件の判定ロジック

1. 使用者がテラスタル中の場合は失敗する（`attacker.terastallized`）。
2. 「タイプが同じ」の判定は、使用者の現在タイプ（sorted）と `ミラータイプ_取得_対象タイプ(defender)`
   （sorted）が一致する場合とする。

```python
sorted(attacker.types) == sorted(ミラータイプ_取得_対象タイプ(defender))
```

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「み」行 — `みちづれ_apply_volatile_to_attacker` の後、
`ミストフィールド_activate_terrain` の前。

```python
def ミラータイプ_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ミラータイプの効果: 使用者のタイプを対象の現在のタイプに置換する。"""
    attacker = ctx.attacker
    defender = ctx.defender
    attacker.move_override_types = ミラータイプ_取得_対象タイプ(defender)
    attacker.added_types = []
    return HandlerReturn(value=value)


def ミラータイプ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ミラータイプの失敗チェック。

    - 使用者がテラスタル中の場合は失敗する
    - 使用者と対象のタイプが一致する場合は失敗する
    """
    attacker = ctx.attacker
    defender = ctx.defender
    if attacker.terastallized:
        battle.add_event_log(
            attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ミラータイプ_テラスタル中"),
        )
        return HandlerReturn(value=False, stop_event=True)
    target_types = ミラータイプ_取得_対象タイプ(defender)
    if sorted(attacker.types) == sorted(target_types):
        battle.add_event_log(
            attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ミラータイプ_タイプ同じ"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ミラータイプ_取得_対象タイプ(defender: Pokemon) -> list[Type]:
    """ミラータイプでコピーする対象の現在のタイプを返す。"""
    return list(defender.types) or ["ノーマル"]
```

### `src/jpoke/data/moves/move_ma.py`

```python
"ミラータイプ": MoveData(
    type="ノーマル",
    category="status",
    pp=16,
    flags={"unreflectable", "bypass_substitute"},
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.ミラータイプ_can_apply,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.ミラータイプ_apply,
        ),
    }
),
```

- `pp=16`: `src/jpoke/data/ps-champ-ja/moves.json` 準拠（Wiki記載の第九世代本家PP15はChampionsの
  8/12/16/20統一ルールにより16へ変換される）。
- `flags={"unreflectable", "bypass_substitute"}`: 一次情報の「マジックコート: ×」
  「みがわり状態の対象に使用しても成功する」に対応する。
- `target` は指定不要（デフォルトの `"foe"` のままで「まもる: ○（防がれる）」に対応する）。
- `accuracy` は指定不要（必中）。

## 注意点・エッジケース

- **`move_override_types` フィールドの新設が必要**: `ability_override_type` は単一タイプのみ
  保持できないため、2タイプのポケモンをコピーできない。`model/pokemon.py` の変更が前提となる
  （実装済み）。
- **使用者がテラスタル中の場合**: 一次情報に「テラスタルしたポケモンが使用した場合、失敗する」と
  明記されているため、`attacker.terastallized` チェックを `can_apply` に追加する。
- **`ability_override_type` との共存**: `move_override_types` は `ability_override_type` より
  優先されるため、へんげんじざい・リベロ等でタイプが変わっている場合、ミラータイプ使用後は
  `move_override_types` が優先される。
- **added_types のリセット**: `added_types` を `[]` にリセットすることで、もりののろい・ハロウィン
  状態の追加タイプ効果が消える。テクスチャー仕様に準拠。
- **対象のタイプ取得は `defender.types` を使う**: `defender.data.types` ではなく
  `defender.types`（現在の実効タイプ）を使うことで、対象のテラスタル・特性変化・
  もりののろい/ハロウィンの追加タイプ・はねやすめの除去タイプが自動的に反映される。
- **対象のタイプが空の場合はノーマル扱い**: `defender.types` が空リストになるケース
  （純ひこうタイプが `はねやすめ` を使用した場合）は `["ノーマル"]` として扱う。
- **ミラータイプを受けた側の追加タイプ**: 使用者のみが変更対象であり、対象（defender）の
  タイプは変化しない。
- **turn.md の失敗判定位置**: `ON_BEFORE_APPLY_MOVE`（turn.md の `Event.ON_APPLY_MOVE` に相当）
  での失敗チェックは priority 100（デフォルト）で行う。
- **未実装のエッジケース**: `docs/spec/moves/ミラータイプ.md` の「対象範囲外・簡略化した点」を
  参照（もりののろい/ハロウィンのボラタイル状態そのもののコピー、対象・使用者双方が
  タイプ喪失状態にある複合ケース、イリュージョン、メガシンカ/フォルムチェンジとの相互作用）。
