# 実装計画書: ミラータイプ

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 15 / 命中: なし / 対象: 1体選択
- 自分のタイプを相手（対象）のタイプと同じに変更する
- 相手がテラスタルしている場合はテラスタイプをコピーする
- 相手がステラタイプにテラスタルしている場合は元のタイプ（`data.types`）をコピーする
- 使用者のタイプが対象と同じ場合は失敗する（`docs/spec/turn.md` Priority 100 に明記）
- 使用者が交代またはひんしになった場合、タイプは元に戻る
- タイプ変更後は、もりののろい/ハロウィンによる追加タイプ効果はリセットされる（テクスチャーと同仕様）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 使用者と対象のタイプが完全一致する場合に失敗 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 使用者のタイプを対象のタイプに置換する |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「ミラータイプ: タイプ同じ」が priority 100 として明記されている
- `ON_STATUS_HIT` priority 100（デフォルト）: turn.md に記載なし。同種タイプ変更ハンドラ（ハロウィン・もりののろい）と同じデフォルト priority を採用する

## タイプ置換の実装方針

`Pokemon.types` プロパティの優先順位は以下のとおりである:

1. テラスタル中: `[active_tera_type]`（ステラ時は `data.types`）
2. `ability_override_type` が設定済み: `[ability_override_type]`
3. それ以外: `data.types`

ミラータイプは「タイプを置換」する技であり、`added_types` による追加ではなく置換が必要である。

`ability_override_type` は `Type | None`（単一タイプ）しか保持できないため、相手が2タイプの場合に対応できない。

**推奨実装**: `Pokemon` に `move_override_types: list[Type] | None` フィールドを新設し、`types` プロパティでこれを参照する。

```python
# model/pokemon.py に追加するフィールド
self.move_override_types: list[Type] | None = None
```

```python
# reset_on_switch_out に追加
self.move_override_types = None
```

```python
# types プロパティの分岐に追加（ability_override_type の直前に挿入）
elif self.move_override_types is not None:
    base = self.move_override_types
```

`types` プロパティの判定順序は以下とする:

1. テラスタル中
2. **`move_override_types` が設定済み** ← 新規追加
3. `ability_override_type` が設定済み
4. `data.types`

`move_override_types` はひんし・交代時（`reset_on_switch_out`）に `None` へリセットされる。

## 対象タイプの決定ロジック

```python
def _get_mirror_type_target_types(defender) -> list[Type]:
    """ミラータイプで取得する対象のタイプを返す。

    - テラスタル中かつステラでない: [tera_type]
    - テラスタル中かつステラ: data.types
    - テラスタルなし: data.types
    """
    if defender.terastallized and defender.tera_type != "ステラ":
        return [defender.tera_type]
    return list(defender.data.types)
```

## 失敗条件の判定ロジック

「タイプが同じ」の判定は、使用者の現在タイプ（sorted）と対象から取得するタイプ（sorted）が一致する場合とする。

```python
# ミラータイプが有効かどうかの判定
sorted(attacker.types) == sorted(_get_mirror_type_target_types(defender))
```

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「み」行 — `みちづれ_apply_volatile_to_attacker` の後、`ミストフィールド_activate_terrain` の前に挿入する。

```python
def ミラータイプ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ミラータイプの失敗チェック: 使用者と対象のタイプが一致する場合は失敗する。"""
    attacker = ctx.attacker
    defender = ctx.defender
    if defender.terastallized and defender.tera_type != "ステラ":
        target_types = [defender.tera_type]
    else:
        target_types = list(defender.data.types)
    if sorted(attacker.types) == sorted(target_types):
        battle.add_event_log(
            attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "ミラータイプ_タイプ同じ"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ミラータイプ_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ミラータイプの効果: 使用者のタイプを対象のタイプに置換する。

    - テラスタル中かつステラでない: テラスタイプをコピー
    - テラスタル中かつステラ: 元のタイプ（data.types）をコピー
    - テラスタルなし: 元のタイプ（data.types）をコピー
    - タイプ置換後、もりののろい/ハロウィンによる added_types はリセットされる
    """
    attacker = ctx.attacker
    defender = ctx.defender
    if defender.terastallized and defender.tera_type != "ステラ":
        target_types = [defender.tera_type]
    else:
        target_types = list(defender.data.types)
    attacker.move_override_types = target_types
    # added_types（もりののろい・ハロウィン等の追加タイプ）をリセットする
    attacker.added_types = []
    return HandlerReturn(value=value)
```

## 注意点・エッジケース

- **`move_override_types` フィールドの新設が必要**: `ability_override_type` は単一タイプのみ保持できるため、2タイプのポケモンをコピーできない。`model/pokemon.py` の変更が前提となる
- **使用者がテラスタル中の場合**: 仕様書に明記がないが、第九世代のテクスチャーと同様に「テラスタル中は失敗する」可能性がある。`terastal.md` の「ミラータイプ: 元タイプをコピー」という記述から失敗しないと判断できるが、テスト時に確認すること
- **`ability_override_type` との共存**: `move_override_types` は `ability_override_type` より優先されるため、へんげんじざい・リベロ等でタイプが変わっている場合、ミラータイプ使用後は `move_override_types` が優先される。テスト時に確認すること
- **added_types のリセット**: `added_types` を `[]` にリセットすることで、もりののろい・ハロウィン状態の追加タイプ効果が消える。対応する volatile（もりののろい・ハロウィン等）が付いている場合でも、`added_types` を直接リセットする。テクスチャー仕様（「タイプが変化した後はタイプ2はなくなる。タイプが変化した後はもりののろい/ハロウィンの効果がなくなる。」）に準拠
- **ミラータイプを受けた側の追加タイプ**: 使用者のみが変更対象であり、対象（defender）のタイプは変化しない
- **turn.md の失敗判定位置**: `ON_BEFORE_APPLY_MOVE`（turn.md の `Event.ON_APPLY_MOVE` に相当）での失敗チェックは priority 100 で行う。みずびたし等の同類技（turn.md: 「みずびたし等: すでに該当タイプ」priority 100）と同じ扱い
- **accuracy の扱い**: `status_move.md` では命中率なし（必中）として記載されているため、`accuracy` は `data/move.py` に設定しない（`ON_ACCURACY_CHECK` を経由しない設計）
