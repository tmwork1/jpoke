# 計画書: ヘビーボンバー（再レビュー）

## 仕様要約

* 自分の体重と相手の体重の比率によって威力が変わる（`docs/spec/moves/ヘビーボンバー.md` 参照）。
  相手の体重が自分の1/5以下→120、1/4以下→100、1/3以下→80、1/2以下→60、それより重い→40。
* ヒートスタンプと全く同じ威力計算式を使用する（タイプがほのお→はがねに変わるのみ）。
* 相手が『ちいさくなる』状態のときは体重比計算をスキップし、`minimize` フラグの共通処理
  （必中・威力2倍）のみが適用される。

## 既存実装の確認結果

`src/jpoke/handlers/move_attack.py` の `ヘビーボンバー_calc_power` と共通関数
`_weight_ratio_to_power`（`ヒートスタンプ_calc_power` と共用）を確認した結果、仕様書の
威力段階・境界値と完全に一致しており、`src/jpoke/data/moves/move_ha.py` の `MoveData`
（type=はがね/category=physical/pp=12/power=1/accuracy=100/flags={"minimize","contact"}）
も一次情報（wiki: タイプはがね、物理、命中100、直接攻撃あり）および
`src/jpoke/data/ps-champ-ja/moves.json`（798行目、PP=12）と一致している。実装の修正は不要。

## ハンドラ構成（既存・変更なし）

| 技名 | イベント | priority | subject_spec | 役割 |
|------|----------|----------|--------------|------|
| ヘビーボンバー | `Event.ON_CALC_POWER_MODIFIER` | デフォルト（100） | `attacker:self` / `defender:self`（`MoveHandler` デフォルト） | 自分/相手の体重比から技の実威力を算出して返す |

* `Event.ON_CALC_POWER_MODIFIER` は `docs/spec/turn.md` に priority の記載がないため、
  既存の同種ハンドラ（`ヒートスタンプ_calc_power` ほか、`ON_CALC_POWER_MODIFIER` を使う
  全ての `MoveHandler`）が明示的な priority 指定なし＝デフォルト値（100）で登録されている
  慣例に合わせ、今回もデフォルトのままとする。

```python
def ヘビーボンバー_calc_power(battle: Battle, ctx: AttackContext, value: int) -> HandlerReturn:
    """ヘビーボンバー: 自分の体重 / 相手の体重 の比率で威力を決定する。

    相手がちいさくなる状態の場合は体重比率計算をスキップし、
    minimize ラベルの共通処理（威力2倍）のみが適用される。
    """
    if ctx.defender.has_volatile("ちいさくなる"):
        return HandlerReturn(value=value)
    return HandlerReturn(value=_weight_ratio_to_power(ctx.attacker.weight, ctx.defender.weight) * 4096)
```

```python

```

## リーサル計算

`docs/progress/move.md` の該当行は「リーサル実装: n/a」「リーサルテスト: n/a」のため対象外。
理由: 威力は自分（attacker）と相手（defender）の体重という固定パラメータにのみ依存し、
相手のHP分布 `StateDist` を左右する要素ではないため、`lethal.py` 側の専用ハンドラは不要。

## テスト

既存の `tests/moves_attack/test_move_ha.py` に以下のテストが揃っている。

* `test_ヘビーボンバー_体重比2未満のとき威力40`

境界値40/120とminimize時の挙動は確認済み。今回のレビューで確認した範囲では計算式・実装
ともに正しいため、テストの追加は不要。
