# 計画書: はねやすめ

## 仕様要約

- タイプ: ひこう / カテゴリ: 変化 / PP: 8 / 対象: 自分 / まもる: 無視
- 最大HPの1/2（切り捨て）を回復する
- 使用者がひこうタイプを持つ場合、そのターン中はひこうタイプを一時的に失う
- ターン終了時（priority 120）にひこうタイプを復帰する
- HPが満タンの場合は技が失敗し、回復もタイプ変化も発生しない
- テラスタル中はタイプ変化しない（テラスタイプがひこうであっても除去されない）
- ひこうタイプを持たないポケモンが使った場合はタイプ変化なし（HP回復のみ）
- `MoveData.target` を明示的に `"self"` にする必要がある
  （デフォルト値 `"foe"` のままだと `is_blocked_by_protect` / `is_reflectable` が
  誤って `True` を返し、相手の `まもる` に防がれたり `マジックコート` で跳ね返されたりする）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 | 登録場所 |
|---|---|---|---|---|
| `ON_STATUS_HIT` | 0 (標準) | `attacker:self` | HP満タン時の失敗チェック + 回復実行 + ひこうタイプ除去（volatile付与） | `data/moves/move_ha.py` の技ハンドラ |
| `ON_VOLATILE_START` | 0 (標準) | `source:self` | volatile付与時に `removed_types` へ「ひこう」を追加 | `data/volatile.py` の volatile ハンドラ |
| `ON_TURN_END` | 120 | `source:self` | ひこうタイプ復帰（`removed_types` から削除 + volatile除去） | `data/volatile.py` の volatile ハンドラ |

失敗チェックと回復・タイプ除去を単一の `ON_STATUS_HIT` ハンドラにまとめている
（`じこさいせい_heal_self` / `あさのひざし_heal_self` と同じパターン）。
`ON_BEFORE_APPLY_MOVE` で失敗判定を行う方式も検討したが、`move_executor.run_move` は
`ON_BEFORE_APPLY_MOVE` が `False` を返した場合に自動で `MOVE_FAILED` ログを出さないため
（`あくび_can_apply` のように失敗ハンドラ側で `add_event_log` を呼ぶ必要がある）、
`ON_STATUS_HIT` に統一した方がシンプルかつ失敗ログが自動化される。

## Priority 根拠

- `ON_TURN_END` priority 120: `docs/spec/turn.md` に「120 | はねやすめのひこうタイプ復帰」と明記されている
- `ON_STATUS_HIT` は `docs/spec/turn.md` に掲載がないイベント（技実行フロー内部の状態変化技専用イベント）。
  既存の同種ハンドラ（`じこさいせい_heal_self`, `あさのひざし_heal_self`）が標準 priority（指定なし）で
  登録されているため、それに合わせる。
- `ON_VOLATILE_START` も標準 priority（`ハロウィン_add_type` 等の既存タイプ変化volatileと同様）。

## モデル変更: `model/pokemon.py`

はねやすめのひこうタイプ除去を表現するために `removed_types` フィールドを追加済み。

```python
# __init__ 内（added_types と同じ箇所）
self.removed_types: list[Type] = []
```

```python
# reset_on_switch_out 内（added_types = [] と同じ箇所）
self.removed_types = []
```

```python
# types プロパティ内（既存の extra = [...] の後）
@property
def types(self) -> list[Type]:
    ...
    if not self.added_types:
        result = base
    else:
        extra = [t for t in self.added_types if t not in base]
        result = base + extra if extra else base
    # はねやすめ等によるタイプ除去を適用（テラスタル中は無視）
    if self.removed_types and not self.active_tera_type:
        result = [t for t in result if t not in self.removed_types]
    return result
```

注: `result` が空リストになるケース（ひこう単タイプ + はねやすめ）は空リストのまま返す。
`has_type` は `type_ in self.types` で動くため自動対応。

## 実装箇所

### `handlers/move_status.py`

「は」行の適切な位置に実装済み。

```python
def はねやすめ_heal_and_remove_flying(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """はねやすめの効果: 最大HPの1/2を回復し、ひこうタイプを一時的に除去する。

    HPが満タンの場合は失敗する。
    テラスタル中はタイプ除去を行わない。
    ひこうタイプを持たないポケモンはタイプ除去を行わない。
    """
    mon = ctx.attacker
    if mon.hp == mon.max_hp:
        return HandlerReturn(value=False, stop_event=True)
    battle.modify_hp(mon, r=1/2)
    # テラスタル中はタイプ変化しない
    if mon.active_tera_type:
        return HandlerReturn(value=value)
    # ひこうタイプを持つ場合のみ除去 volatile を付与
    if mon.has_type("ひこう"):
        battle.volatile_manager.apply(mon, "はねやすめ")
    return HandlerReturn(value=value)
```

### `handlers/volatile.py`

「は」行の適切な位置に実装済み。

```python
def はねやすめ_remove_flying(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """はねやすめ volatile 付与時: ひこうタイプを removed_types に追加する。"""
    mon = ctx.source
    if "ひこう" not in mon.removed_types:
        mon.removed_types.append("ひこう")
    return HandlerReturn(value=value)


def はねやすめ_restore_flying(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """はねやすめ volatile 解除時 (ターン終了): ひこうタイプを復帰する。"""
    mon = ctx.source
    if "ひこう" in mon.removed_types:
        mon.removed_types.remove("ひこう")
    battle.volatile_manager.remove(mon, "はねやすめ")
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### タイプ除去のスコープ

- `model/pokemon.py` の `types` プロパティに `removed_types` フィルタが実装済み。影響範囲が最も広い変更。
- `removed_types` は `reset_on_switch_out` でもクリアされる（場を離れた際に自動解除）。

### テラスタル中の挙動

- テラスタルでひこうタイプになっている場合: `mon.active_tera_type` が truthy なのでタイプ除去をスキップする（仕様通り）。
- ステラテラスタルの場合も `mon.active_tera_type == "ステラ"` は truthy なので除去はスキップされる。

### HP満タン失敗

- `ON_STATUS_HIT` の先頭でチェックし、`stop_event=True` で中断する。HP回復もタイプ除去も発生しない。
- `move_executor._execute_status_hit` が `ON_STATUS_HIT` の戻り値 `False` を検知して自動的に
  `MOVE_FAILED` ログを出す（ハンドラ側で個別にログを出す必要はない）。

### ひこうタイプを持たないポケモン

- `has_type("ひこう")` が False の場合は volatile を付与しない。HP回復のみ発動。

### ON_TURN_END の volatile 解除タイミング

- `VolatileHandler` の `ON_TURN_END` で `priority=120` を指定することで、ほろびのうた（120）と同タイミングになる。仕様（turn.md）通り。
- `はねやすめ_restore_flying` の中で `volatile_manager.remove` を直接呼ぶ。`tick_volatile` は使わない（ターンカウントではなくターン終了で即解除する仕組みのため）。

### まもる・マジックコートとの相互作用

- `MoveData.target="self"` を明示することで `Move.is_blocked_by_protect` / `Move.is_reflectable` が
  ともに `False` になり、相手の `まもる` に防がれず、`マジックコート`/`マジックミラー` で跳ね返されない。

### 交代時の自動解除

- `reset_on_switch_out` で `removed_types = []` がクリアされるため、はねやすめ使用ターン中に交代が発生した場合（さいはい等）でも次のポケモンは通常タイプになる。volatile 自体も交代で自動削除される（既存 volatile_manager の動作）。

### VolatileName / MoveName 型定義

- `"はねやすめ"` は `types/volatile.py` の `VolatileName` と `types/move.py` の `MoveName` に登録済み。
