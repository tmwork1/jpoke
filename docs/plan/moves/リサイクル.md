# 実装計画書: リサイクル

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 10 / 対象: 自分（target="self"）
- 自分が最後に使用した（失った）持ち物を手に入れる
- アイテムをすでに持っている場合は失敗する
- `last_lost_item_name` が空文字列（アイテムをまだ失っていない）場合は失敗する
- 取り戻したアイテムが即時発動条件を満たす場合はその場で発動する（`gain_item` 経由で自動処理）

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| `data/move.py` エントリ `"リサイクル"` | 登録済み（handlers 空） |
| `Pokemon.last_lost_item_name` | 実装済み（アイテム消失時に `item_manager` が自動セット） |
| `battle.gain_item(target, item_name)` | 実装済み（`item_manager.gain_item` への委譲） |
| `handlers/move_status.py` 関数 | 未実装 |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_APPLY_MOVE` | 100 | `attacker:self`（デフォルト） | 失敗条件チェック（アイテム所持中・未消失） |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | `last_lost_item_name` のアイテムを取り戻す |

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「リサイクル: 失敗条件」が priority 100 として明記されている。`ON_STATUS_HIT` は変化技の効果発動イベントであり、既存の他変化技ハンドラと同一の priority 100（デフォルト）を使用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: `リフレクター_set_side_field` の直前（「リ」行内の五十音順で「リサイクル」< 「リフレクター」）。

```python
def リサイクル_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """リサイクルの失敗条件チェック。

    以下のいずれかに該当する場合は失敗する:
    - 使用者がすでにアイテムを持っている
    - 使用者がまだアイテムを失ったことがない（last_lost_item_name が空）
    """
    mon = ctx.attacker
    if mon.has_item() or not mon.last_lost_item_name:
        battle.add_event_log(
            mon,
            LogCode.MOVE_FAILED,
            payload={"reason": "リサイクル"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def リサイクル_restore_item(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """リサイクルの効果: 最後に失ったアイテムを取り戻す。"""
    mon = ctx.attacker
    battle.gain_item(mon, mon.last_lost_item_name)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/move.py`

既存の `"リサイクル"` エントリを更新する。handlers を追加する。

```python
"リサイクル": MoveData(
    type="ノーマル",
    category="変化",
    pp=10,
    handlers={
        Event.ON_APPLY_MOVE: h.MoveHandler(
            hs.リサイクル_can_apply,
            priority=100,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.リサイクル_restore_item,
        ),
    }

```

## 注意点・エッジケース

- **アイテムを失ったことがないケース**: 初期からアイテムなしで出場したポケモンは `last_lost_item_name == ""` となるため失敗する。失敗条件チェックで `not mon.last_lost_item_name` を必ず確認すること。
- **アイテム所持中の失敗**: `item_manager.gain_item` 内部でも持っている場合は False を返すが、ログ出力のため `ON_APPLY_MOVE` ハンドラ側で先行チェックする。
- **`last_lost_item_name` の更新タイミング**: `item_manager._change_item` で `name=""` を渡した場合のみセットされる（トリック・すりかえ等でアイテムを交換した場合はセットされない点に注意。消失扱いでなければリサイクルは復元できない）。
- **取り戻したアイテムの即時発動**: `gain_item` -> `item_manager.gain_item` -> `_change_item` の中で `ON_ITEM_GAINED` が発火されるため、取り戻し後のきのみ発動等は自動処理される。リサイクル側で追加処理は不要。
- **マジックルーム**: マジックルーム状態中にリサイクルを使用した場合、アイテムは取り戻せるが効果は無効化される（マジックルームの処理側で対応済み）。
- **ものひろい特性との違い**: ものひろい特性は `ON_TURN_END` で発動するが、リサイクルは技使用時に即時発動する。内部ロジックは共通（`last_lost_item_name` + `gain_item`）。
