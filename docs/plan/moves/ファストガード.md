# 計画書: ファストガード

対象: Pokémon Champions シングルバトル
仕様書: `docs/spec/moves/ファストガード.md`（技本体）、`docs/spec/volatiles/ファストガード.md`（付与される揮発性状態）

## 仕様要約

- タイプ: かくとう / カテゴリ: 変化 / PP: 16（champions基準。docs/champions/move_list.txt 744行目） / 命中: なし（必中） / 対象: 自分 / 優先度: +3
- 使用ターン中、実効優先度+1以上の技（先制技）を防ぐ。優先度0以下の通常技は防げない点で `まもる` と異なる
- まもる系共通の連続使用失敗ルールが適用される（`まもる系_連続使用失敗チェック`、`Event.ON_TRY_MOVE_2` デフォルト priority）
- `"ファストガード"` volatile を自分に付与し、`ON_TRY_MOVE_1` で実効優先度を判定した上で攻撃を無効化する
- ターン終了時に volatile を解除する

## Priority 根拠

### `ON_TRY_MOVE_2`: priority 100（デフォルト）
`docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` 30番に「まもる系: 連続使用により失敗」と記載があるが、
既存の守る系技（まもる・みきり・こらえる・キングシールド・トーチカ等）がすべて `Event.ON_TRY_MOVE_2`
のデフォルト priority で実装済みのため、同一パターンに合わせる。理由は docs/spec/moves/まもる.md 参照。

### `ON_STATUS_HIT`: priority 100（デフォルト）
`docs/spec/turn.md` に `ON_STATUS_HIT` の内部 priority 表がない。既存の変化技ハンドラ（`まもる_apply`・
`トーチカ_apply` 等）がすべて priority 100（デフォルト）を使用しているため、同一パターンで 100 を採用する。

### `ON_TRY_MOVE_1`: priority 100
`docs/spec/turn.md` の `Event.ON_TRY_MOVE_1` セクションに明記：

```
| 100 | まもる・キングシールド等による無効化 |
```

既存の `まもる` / `キングシールド` / `トーチカ` volatile ハンドラがすべて priority 100 を明示しているため、同一の 100 を採用する。

### `ON_TURN_END`: priority 100（デフォルト）
`docs/spec/turn.md` に `ON_TURN_END` の「まもる系 volatile 解除」の記載はない。既存の `まもる_remove_volatile`・
`かえんのまもり_remove_volatile`・`キングシールド_remove_volatile`・`トーチカ_remove_volatile` 等が
すべてデフォルト priority を使用しているため、同一パターンで 100 を採用する。

### 4. `src/jpoke/handlers/volatile.py` — 優先度判定（レビューで修正）

`まもる`系他技と異なり、ファストガードは実効優先度+1以上の技のみをブロックする点が固有仕様。
技データの静的な `priority`（`ctx.move.priority`）ではなく、`speed_calculator.calc_move_priority()`
で算出した実効優先度を使う（`いたずらごころ`で動的に優先度が上がる変化技等を正しく判定するため）。

```python
def ファストガード_protect(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """ファストガードの保護判定。priority≥1の先制技のみブロックする。

    いたずらごころ・はやてのつばさ等による動的な優先度変化も考慮するため、
    技データの静的な priority ではなく `speed_calculator.calc_move_priority` で
    実効優先度を算出して判定する（`じょおうのいげん_block_priority` と同じパターン）。
    """
    effective_priority = battle.speed_calculator.calc_move_priority(ctx.attacker, ctx.move)
    if effective_priority < 1:
        return HandlerReturn(value=value)
    return _run_protect(battle, ctx, value)


def ファストガード_remove_volatile(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    return remove_volatile(battle, ctx, value, volatile="ファストガード")
```

## 注意点・エッジケース

### まもる系の連続使用失敗
`_run_protect` 自体は連続使用判定を行わない。連続使用失敗は技本体の `Event.ON_TRY_MOVE_2` に登録した
`まもる系_連続使用失敗チェック`（`executed_move` が `protect` フラグを持つ技かどうかで判定する簡略化
モデル）が担う。ファストガードもこのパターンに従う。

### 実効優先度の判定（レビューで修正した箇所）
`じょおうのいげん_block_priority`（`handlers/ability.py`）・`はやてがえし_can_apply`
（`handlers/move_attack.py`）と同様に、`battle.speed_calculator.calc_move_priority(ctx.attacker, ctx.move)`
を使って実効優先度を算出する。技データの静的な `move.priority` を直接参照すると、`いたずらごころ`
（変化技+1）・`はやてのつばさ`（攻撃技+1、条件付き）・`ヒーリングシフト` 等、特性による動的な
優先度変化が反映されず、本来防げるはずの技を防げなくなる不具合があったため修正した。

### ファストガード自身より速い先制技
ファストガードの優先度+3より高い優先度の技（現行技には存在しない）や、同優先度でファストガードの
使用者よりも素早いポケモンが使った同優先度技は、ターン順序上ファストガードの volatile 付与より前に
着弾するため、ハンドラ側で個別対応する必要はない（ターン順序の制御で自然に解決される）。

### フェイント等による解除
フェイント・ゴーストダイブ・シャドーダイブ・いじげんホール・いじげんラッシュは `_FAINT_REMOVE_VOLATILES`
（`handlers/move_attack.py`）に `"ファストガード"` を含めて実装済みで、ファストガード状態を取り除いて
ダメージを与える。

### ダイマックス技・Zワザは対象外
一次情報の「優先度が高いZワザ/ダイマックスわざを受けた場合、防げる変化技なら防ぐが、攻撃技なら
ダメージを1/4に抑えて受ける」は、本プロジェクトの対象範囲（第9世代・champions基準）に
ダイマックス・Zワザが存在しないため対応不要。

### 変化技への保護
ファストガードはデフォルト（`protect_non_attack=True`）で変化技も防ぐが、実効優先度+1以上でなければ
そもそも防御判定に入らない点が `まもる` と異なる。

## テスト方針

- priority=0 の通常技はファストガードで防げないこと（既存: `test_ファストガード_priority0技は通過する`）
- priority=1 の技（でんこうせっか）はファストガードで防げること（既存: `test_ファストガード_priority1技を防ぐ`）
- いたずらごころ等で動的に優先度が上がった変化技も防げること（レビューで追加:
  `test_ファストガード_いたずらごころで優先度が上がった変化技も防ぐ`）
- 2ターン連続使用すると2ターン目は失敗すること（既存: `test_ファストガード_連続使用で失敗する`）
- フェイントでファストガード状態が解除されダメージを受けること（既存:
  `test_フェイント_まもる系統の状態を解除して攻撃する`、`volatile_name="ファストガード"` パラメータ）
