# バリアーラッシュ 実装計画書

## 仕様要約（docs/spec/moves/バリアーラッシュ.md 参照）

* タイプ: エスパー / 分類: 物理 / 威力: 70（SV基準）→ 90（Champions基準） / 命中: 90 / PP: 10（SV基準）→ 12（Champions基準）
* 対象: 相手1体 / 優先度: 0 / 接触あり
* 効果: 100%の確率で**自分（攻撃側）**の『ぼうぎょ』ランクを1段階上げる（追加効果）。

## PP・威力（Champions基準）

`docs/champions/move_list.txt` 714行目に本技の項目が存在する。

```
バリアーラッシュ	エスパー	物理	90	90	12	接○	守○
```

SV基準の威力70・PP10から、Champions基準では威力90・PP12に変更されている
（`docs/champions/changes_from_sv.md` の「威力が変更されたわざ」「全技PPは8/12/16/20の4段階に統一」に該当）。
レビュー時点の実装は威力70・PP10のまま（Champions移行漏れ）だったため、90・12に修正する。

## 現状の実装状況

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_HIT` | 100（デフォルト） | attacker:self（`MoveHandler` 既定値） | 100%の確率で自分の『ぼうぎょ』ランクを1段階上げる |

`src/jpoke/handlers/move_attack.py` の実装（レビューにより関数名を修正）:

```python
def バリアーラッシュ_boost_attacker_B(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """バリアーラッシュ: 命中後、100%の確率で自分の『ぼうぎょ』ランクを1段階上げる。"""
    return modify_attacker_stats(battle, ctx, value, stats={"def": 1})
```

`modify_attacker_stats`（`handlers/move.py`）は `chance` 省略時に100%として扱い、
`battle.resolve_secondary_chance` を経由するため、ちからずくによる追加効果無効化（確率0化）は
汎用的に処理される。本技側に個別のガードは不要。

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_HIT` には「自分のランクを上げる追加効果」が priority=10 として
掲載されているが、同種の既存ハンドラ（グロウパンチ・コメットパンチ・メタルクロー等の
`_boost_attacker_X` 系ハンドラ）はいずれも `MoveHandler` のデフォルト値（priority=100）のまま
登録されており、明示的に priority=10 を指定している実装は存在しない。本技単独でこの慣例から
逸脱すると同種技との一貫性が崩れるため、既存の同種ハンドラに合わせてデフォルト値のまま登録する。

  旧名は「defender」を含んでおり、`グロウパンチ_boost_attacker_A` 等の命名規則

* ハンドラ参照名を `ha.バリアーラッシュ_boost_attacker_B` に更新した。

## 注意点・エッジケース（review-test 向け）

1. **効果の対象は自分（攻撃側）のぼうぎょ**であり、相手（防御側）のランクではない。
   一次情報（ポケモンWiki）の説明文「自分の防御を上げる」の通り。
2. **依頼文には「自分が壁（リフレクター等）を張っている場合に威力が上昇する技」と記載されていたが、
   一次情報（ポケモンWiki・Champions技一覧）を確認したところ誤りだった。** バリアーラッシュに
   壁関連の威力補正は存在せず、効果は「命中後100%の確率で自分のぼうぎょランクを1段階上げる」のみ。
   既存実装にも壁判定・威力補正のロジックは存在せず、追加しない。
3. **ちからずくとの相互作用**
   - `flags={"secondary_effect"}` が必要（実装済み）。ちからずく所持時は
     `resolve_secondary_chance` が確率を0にし、ぼうぎょランク上昇が発動しなくなる
     （威力1.3倍化は `ちからずく_boost` 側で処理）。
4. **直接攻撃**のため `contact` フラグを付与する（実装済み）。ぼうおん等の接触依存特性・アイテムの対象。
5. **テスト方針**
   - `battle.actives[0].rank["def"] == 1`（攻撃側自身のランク上昇）を確認する
     （100%発動のため乱数固定は不要）。既存テストで確認済み。

## リーサル計算

対象外（`docs/progress/move.md` の「リーサル実装」「リーサルテスト」は n/a）。
自分のぼうぎょランク変化はその場のダメージ・HPに影響しないため、同一パターンの他技
（グロウパンチ・コメットパンチ等）と同様にリーサル計算ハンドラの対象外とする。
