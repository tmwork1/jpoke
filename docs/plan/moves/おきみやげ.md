# 計画書: おきみやげ

## 仕様要約

- タイプ: あく / カテゴリ: 変化 / PP: 10 / 命中: 100 / 対象: 1体選択（相手）
- 使用者が自分のHPを0にしてひんしになり、相手のこうげき・とくこうをそれぞれ2段階下げる
- 第五世代以降: 命中しなかった場合・まもる/みがわりで防がれた場合は使用者はひんしにならない
- 第五世代以降: こうげき/とくこうランクが共に最低でも、ランク低下は失敗するが使用者はひんしになる
- クリアボディ等でランク低下が防がれた場合でも使用者はひんしになる
- まもる: ○（防がれる）/ マジックコート: ×（反射されない）/ よこどり: ×
- Zワザ版: 控えから出るポケモンのHPを全回復（ON_SWITCH_IN priority 100）

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT` 内部の priority 表がない。既存の変化技ハンドラ（`おいかぜ_set_side_field`、`おたけび_modify_defender_stats`、`いやしのねがい_faint_and_set_side_field` など）がすべて priority 100（デフォルト）を使用している。おきみやげも同一パターンとして priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「お」行 — `おかたづけ_cleanup` の後、`おたけび_modify_defender_stats` の前に挿入する。

```python
def おきみやげ_faint_and_modify_defender_stats(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """おきみやげ: 使用者をひんしにし、相手のこうげき・とくこうを2段階ずつ下げる。

    第五世代以降の仕様に従い、ON_STATUS_HIT が発火した時点（命中判定・まもる通過後）で
    ハンドラが呼ばれるため、ここでは無条件にひんし化とランク低下を実行する。
    ランク低下が防がれた場合でも使用者はひんしになる。
    """
    battle.modify_hp(ctx.attacker, -ctx.attacker.max_hp)
    modify_defender_stats(battle, ctx, value, stats={"A": -2, "C": -2})
    return HandlerReturn(value=value)
```

### `src/jpoke/data/move.py`

既存の `"おきみやげ"` エントリに `handlers` を追加する。

```python
"おきみやげ": MoveData(
    type="あく",
    category="変化",
    pp=10,
    accuracy=100,
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.おきみやげ_faint_and_modify_defender_stats,
        ),
    }
),
```

## 注意点・エッジケース

- **ひんし化のタイミング**: `ON_STATUS_HIT` は命中判定・まもる判定を通過した後に発火する。第五世代以降の「命中しなかった場合/まもる・みがわりで防がれた場合はひんしにならない」という仕様は、ON_STATUS_HIT が呼ばれない（= stop_event で止まる）ことで自然に実現される。ハンドラ側での追加ガードは不要
- **ランク最低でもひんし化する**: 第五世代以降「こうげき/とくこうランクが共に -6 でも使用者はひんしになる」。`modify_defender_stats` の戻り値に関わらず `battle.modify_hp` はすでに実行済みのため、ランク低下失敗の `stop_event=True` を返しても問題ない（modify_hp は先に呼ぶ）。ただし `modify_defender_stats` 自体が内部で stop_event を返す場合に備え、ランク低下処理の戻り値を最終的な HandlerReturn に反映させないこと
- **クリアボディ等でランク低下が防がれた場合も使用者はひんし**: wiki仕様「クリアボディ等により能力低下が無かった場合でも使用者はひんしになる」。`battle.modify_hp` を `modify_defender_stats` より先に呼ぶことで保証される
- **`modify_hp` には `-ctx.attacker.max_hp` を渡す**: いやしのねがい（`-mon.max_hp`）と同一パターン。`v=-ctx.attacker.hp`（現在HPを0にする）ではなくmax_hpを引く形にすることで、バトルシステム内部のひんし判定（`modify_hp` 内でクランプされる）に統一的に対処できる。いずれかの方法でよいが、既存の `いやしのねがい_faint_and_set_side_field` の書き方（`-mon.max_hp`）に合わせる
- **`target` フィールドのデフォルト**: MoveData の `target` デフォルトは `"foe"` であり、おきみやげは相手1体が対象のためデフォルトのまま変更不要
- **すてゼリフとの違い**: すてゼリフは使用者が「交代」するのに対し、おきみやげは「ひんし」になる点が異なる。すてゼリフは現在 data/move.py に handlers なし（未実装）であり、おきみやげは独立したハンドラとして実装する
- **Zおきみやげ**: ON_SWITCH_IN priority 100 で控えポケモンのHPを全回復する効果があるが、Zワザは現状対象外のためハンドラには含めない
- **ランク低下無効化（クリアボディ等）の priority**: `docs/spec/turn.md` ON_TRY_MOVE_2 の priority 140 でクリアボディ等によるランク低下無効が判定され、ON_STATUS_HIT より前のフェーズで止まる場合がある。ただし、おきみやげ判定テーブル（wiki）によれば「クリアボディ等でランク低下が防がれた場合も使用者はひんしになる」とある。これは ON_APPLY_MOVE 段階でのランク変化拒否（stop_event なし）を意味する可能性があるため、実装後にテストで動作確認が必要
