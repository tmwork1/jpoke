# 実装計画書: フラワーヒール

## 仕様要約

- タイプ: フェアリー / カテゴリ: 変化 / PP: 12（champions基準。move_list.txt に個別行はないため、
  原作PP 10 が同じ `いやしのはどう`（champions PP 12）に倣う） / 命中: なし（必中） /
  対象: 1体選択（相手）
- 相手（defender）の HP を最大 HP の **1/2** 回復する（端数**切り上げ**）
- 場が **グラスフィールド** の場合、回復量が最大 HP の **2732/4096（≒2/3）** になる
  （端数は五捨五超入 = `round_half_down`）
- `heal` フラグあり（`かいふくふうじ` の対象になる。ヒーリングシフトの優先度+3判定にも使われる）
- 相手の HP が満タンの場合は技が失敗する

| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | HP 満タン時失敗チェック → 相手（defender）の HP をグラスフィールド判定込みで回復 | `handlers/move_status.py` |

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` セクションに以下の記載がある：

> | 100 | いやしのはどう等: HP満タン |

このイベントはコード上の `ON_BEFORE_APPLY_MOVE` に対応する。同種の回復技（`いやしのはどう`・
`じこさいせい`・`なまける`・`いのちのしずく`・`こうごうせい`・`つきのひかり`・`あさのひざし`）は
すべて `ON_STATUS_HIT`（priority=100）内で HP 満タンチェックと回復処理を一括して行うパターンを
採用している。`フラワーヒール` も同パターンで実装し、priority はデフォルト（100）とする。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

```python
def フラワーヒール_heal_defender(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """フラワーヒール: 相手のHPを回復させる。HPが満タンの場合は失敗する。

    通常時は最大HPの1/2（端数切り上げ）、グラスフィールド中は最大HPの2732/4096（≒2/3、
    端数は五捨五超入）を回復する。
    """
    mon = ctx.defender
    if mon.hp == mon.max_hp:
        return HandlerReturn(value=False, stop_event=True)
    if battle.terrain.name == "グラスフィールド":
        heal = round_half_down(mon.max_hp * 2732 / 4096)
    else:
        heal = (mon.max_hp + 1) // 2
    battle.modify_hp(mon, v=heal)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ha.py`

```python
"フラワーヒール": MoveData(
    type="フェアリー",
    category="status",
    pp=12,
    flags={"heal"},
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.フラワーヒール_heal_defender,
        ),
    }
),
```

## 注意点・エッジケース

- **HP 満タン時の失敗**: `mon.hp == mon.max_hp` で判定し、`HandlerReturn(value=False, stop_event=True)`
  を返す。`move_executor.py` 側で `MOVE_FAILED` ログが記録されるため、ハンドラ内でのログ記録は不要
- **回復対象は defender（相手）**: `ctx.defender` を参照する
- **かいふくふうじとの相互作用**:
  - 使用者（attacker）が かいふくふうじ 状態の場合は `heal` フラグを持つ技として汎用処理
    （`かいふくふうじ_try_action`）で選択できない
  - 対象（defender）が かいふくふうじ 状態の場合は `ON_MODIFY_HEAL` の汎用処理
    （`かいふくふうじ_block_heal`）で回復量が 0 になる
  - いずれも本ハンドラ側での追加チェックは不要
- **回復量の端数処理**:
  - 通常時: 切り上げ。`(mon.max_hp + 1) // 2` で切り上げ除算を行う
  - グラスフィールド中: 五捨五超入（`round_half_down`）で丸める。`battle.modify_hp` の `r=` 引数は
    内部で `int()`（切り捨て）を使うため丸め方向が異なる。切り上げ・五捨五超入のいずれも本技では
    `v=` 引数に計算済みの値を渡す必要がある
- **グラスフィールドの接地判定**: 一次情報より、使用者・対象が地面についていなくてもグラスフィールドに
  よる回復量増加は適用される。`battle.terrain.name` の判定のみを行い、接地チェックは不要
- **ヒーリングシフトとの相互作用**: `heal` フラグを持つ技として汎用処理
  （`ヒーリングシフト_modify_priority`）で優先度+3される。本ハンドラ側での追加対応は不要
- **専用技**: キュワワー / リボンビー のみが習得する。専用技であること自体による技固有の実装は不要
  （ふういんされにくい等は使用可能ポケモンの絞り込みによる副次効果であり、本技のハンドラでは扱わない）
