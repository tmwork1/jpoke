# 実装計画書: コートチェンジ

## 仕様要約

- タイプ: ノーマル / 分類: 変化（`category="status"`） / 威力: - / PP: 10 / 対象: 全体の場（`target="field"`） / 優先度: 0 / 直接攻撃: ×
- 効果: お互いの陣営の「場の状態」を入れ替える
- 入れ替え対象（片方の陣営にのみ存在しうる状態）:
  - 壁: リフレクター / ひかりのかべ / オーロラベール（ひかりのねんどによる延長分もそのまま入れ替わる）
  - 設置技: まきびし / どくびし / ステルスロック / ねばねばネット
  - おいかぜ
  - しろいきり
  - しんぴのまもり
- 継続ターン数（まきびし等は層数）はそのまま保持して入れ替わる
- ファストガード / ワイドガード / トリックガードは対象外
- 単体対象の状態（いやしのねがい / みかづきのまい / みらいよち・はめつのねがいのみらいにこうげき状態 / ねがいごと / やどりぎのタネの回復先 / キョダイサジン等のバインド）は対象外
- 天候・フィールドなど全体の場の状態は対象外
- 入れ替える状態が無い場合は技自体が失敗する（このときも使用者の特性リベロ／へんげんじざいは発動しうる）
- おいかぜ等すばやさに関係する状態を入れ替えると、その瞬間から行動順に影響する
- コートチェンジでおいかぜ状態になっても特性かぜのり／ふうりょくでんきは発動しない
- 命中率欄は「100」だが、場を対象とした技のため入れ替える状態があれば必ず成功する（外れ判定は行われない）
- まもる／マジックコートでは防げない
- 詳細は `docs/spec/moves/コートチェンジ.md` を参照

## 重要な確認事項（世代差分・対象外要素の切り分け）

計画立案にあたり、関連する仕様書・既存実装を横断的に確認した結果、以下の点が判明した。実装時に過剰な作り込みをしないための根拠として明記する。

### 1. まけんき／かちき／ミラーアーマーとねばねばネットの「本来の使用者」情報について

`docs/spec/moves/コートチェンジ.md`（58〜59行目）には次のように書かれている。

> ねばねばネットを入れ替えた場合も本来の使用者の情報は記録されている。味方が敵側に撒いたねばねばネットがコートチェンジで味方側に来た場合、味方のまけんき／かちきのポケモンがこのねばねばネットに引っかかっても味方から能力を下げられたことになるためまけんき・かちきは発動しない。

しかし、より詳細な世代別記述を持つ個別特性仕様書を確認すると、これは **第八世代限定の挙動** であることが分かる。

- `docs/spec/abilities/まけんき.md` 30行目:
  > 第八世代のみ、味方が仕掛けたがコートチェンジで味方側に移ったねばねばネットの効果を受けた場合、味方からすばやさを下げられたことになりまけんきは発動しない。**第九世代ではこのときもまけんきは発動する。**
- `docs/spec/abilities/かちき.md` 29行目: 同内容（かちき版）
- `docs/spec/abilities/ミラーアーマー.md` 23〜27行目: 「使用者情報」がミラーアーマーの跳ね返し先に影響するのも第八世代限定の記述であり、「第九世代では、場に使用者がいたとしてもその素早さは下がらない」（ミラーアーマー自体がねばねばネットを無効化する）ため、コートチェンジによる使用者情報の話自体が第九世代では意味を持たない。

本プロジェクトの対象範囲はポケモンチャンピオンズ（第九世代／SVベース）のシングルバトルのみ（`CLAUDE.md`）であるため、**第八世代限定の「本来の使用者を記録する」実装は行わない**。

実際、既存実装 `handlers/field.py` の `ねばねばネット_reduce_spe` は次のようになっている。

```python
def ねばねばネット_reduce_spe(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    ...
    battle.modify_stats(ctx.source, {"spe": -1}, source=battle.foe(ctx.source))
    return HandlerReturn(value=value)
```

`source=battle.foe(ctx.source)` は「現在ねばねばネットを保持している陣営から見て、常に相手陣営が原因」という**動的な**判定であり、本来の使用者を一切記憶していない。これはコートチェンジで陣営を跨いでねばねばネットが移動した後でも「（元の使用者に関係なく）踏んだ側から見て相手が原因」として扱われることを意味し、**第九世代の挙動と完全に一致する**。

→ **結論: `ねばねばネット_reduce_spe` / `まけんき_boost_atk_on_stat_drop` / `かちき_boost_spa_on_stat_drop` / ミラーアーマー関連ハンドラは変更不要。Field に使用者（Pokemon）参照を追加する必要も無い。**
（Pokemon参照をFieldに持たせると、`Battle.__deepcopy__` のポケモン参照付け替え処理（`mon_map`）の対象外になり、コピー後に古い Pokemon を指したままになるバグを生む可能性があった。今回は不要と判明したため、このリスクごと回避できる。）

なお `docs/spec/moves/コートチェンジ.md` 自体の記述はGen8視点のまま残っており、`docs/spec/abilities/まけんき.md` 等と矛盾している。レビュー時に `コートチェンジ.md` へ「第九世代ではこの限りではない」旨の注記を追加することを推奨する（本計画書では計画書ファイルのみ出力するため、仕様書自体の修正は行っていない）。

### 2. ひかりのねんどによる壁延長について

`docs/spec/items/ひかりのねんど.md` 14行目により、延長効果は壁を張った時点で `count`（継続ターン数）に折り込み済みであることが確認できる（`SideFieldManager.apply()` が `ON_MODIFY_DURATION` を発火して `count` を決定している）。コートチェンジ側は `count` の値をそのまま交換するだけでよく、追加の実装は不要。
（「相手視点では5ターン表記のまま」という状況画面表示の仕様は、本シミュレータに状況画面という概念が無いため対象外）

### 3. ダイマックス／キョダイマックス関連の場（ひのうみ／にじ／しつげん／キョダイベンタツ等）

`src/jpoke/types/side_field.py`・`global_field.py` を確認した結果、これらはいずれも `SideFieldName` / `GlobalFieldName` に存在せず、本プロジェクトに未実装。CLAUDE.mdの対象範囲（チャンピオンズシングル）にダイマックス機構自体が存在しないため、**新規実装は行わず対象外として扱う**。同様に設置技側の「キョダイコウジン」も未実装のため対象外。

### 4. ファストガード／ワイドガード／トリックガード

`src/jpoke/types/volatile.py`・`move.py` を確認した結果、これらは `VolatileName`（ポケモン単位の揮発性状態）として実装されており、`SideFieldName` には含まれない。従って何も対応しなくても自動的に入れ替え対象外になる。

### 5. いやしのねがい／みかづきのまい／ねがいごと／みらいよち／はめつのねがい

これらは実装上は `SideFieldName`（`SideFieldManager` 管理下）だが、`docs/spec/moves/コートチェンジ.md` 47〜52行目および `docs/spec/fields/ねがいごと.md` 44行目で明示的に「単体の場に発生する状態」として対象外とされている。`SideFieldName` 型に含まれているからといって機械的に全種類を交換してはならない点に注意（後述の実装ではホワイトリスト方式で対応する）。

2. `SideFieldManager` クラス（`apply` メソッドの後）に `swap_fields` メソッドを追加する。

```python
    def swap_fields(self, other: "SideFieldManager", names: Iterable[SideFieldName]) -> bool:
        """指定したサイドフィールドの状態を他方のマネージャーと入れ替える（コートチェンジ用）。

        継続ターン数（まきびし等の層数を含む）をそのまま交換する。有効/無効が
        入れ替わったフィールドについてはハンドラの登録・解除のみ行い、
        ON_FIELD_ACTIVATE / ON_FIELD_DEACTIVATE イベントは発火しない
        （かぜのり・ふうりょくでんき等、フィールド「開始」を検知する特性が
        誤発動しないようにするため）。FIELD_STARTED / FIELD_ENDED のバトルログも
        出力しない（技の成功ログのみで表現する設計とする）。

        Args:
            other: 入れ替え相手のサイドフィールドマネージャー
            names: 入れ替え対象のフィールド名一覧
                （呼び出し側で「単体対象の状態」等の対象外フィールドを
                あらかじめ除外したリストを渡すこと）

        Returns:
            bool: 1つ以上のフィールドが実際に入れ替わった場合True
        """
        changed = False
        for name in names:
            field_self = self.get(name)
            field_other = other.get(name)
            active_self = field_self.is_active
            active_other = field_other.is_active
            if not active_self and not active_other:
                continue
            changed = True

            if active_self:
                for player in field_self.owners:
                    field_self.unregister_handlers(self._events, player)
            if active_other:
                for player in field_other.owners:
                    field_other.unregister_handlers(other._events, player)

            field_self.count, field_other.count = field_other.count, field_self.count

            if field_self.is_active:
                for player in field_self.owners:
                    field_self.register_handlers(self._events, player)
            if field_other.is_active:
                for player in field_other.owners:
                    field_other.register_handlers(other._events, player)
        return changed
```

`SideFieldName` は既にファイル冒頭で `from jpoke.types import GlobalFieldName, SideFieldName, WeatherName, TerrainName` としてインポート済みのため追加インポート不要。

```python
# コートチェンジで入れ替え対象となるサイドフィールド一覧。
# 壁3種・設置技4種（まきびし/どくびし/ステルスロック/ねばねばネット）・

_COURT_CHANGE_TARGET_FIELDS: tuple[SideFieldName, ...] = (
    "リフレクター", "ひかりのかべ", "オーロラベール",
    "まきびし", "どくびし", "ステルスロック", "ねばねばネット",
    "おいかぜ", "しろいきり", "しんぴのまもり",
)


def コートチェンジ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """コートチェンジの失敗チェック: 入れ替える場の状態がどちらの陣営にも無い場合は失敗する。"""
    attacker_side = battle.get_side(ctx.attacker)
    defender_side = battle.get_side(ctx.defender)
    has_target = any(
        attacker_side.get(name).is_active or defender_side.get(name).is_active
        for name in _COURT_CHANGE_TARGET_FIELDS
    )
    if not has_target:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="コートチェンジ_場の状態なし"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def コートチェンジ_swap_fields(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """コートチェンジの効果: お互いの場の状態（壁・設置技・おいかぜ・しろいきり・しんぴのまもり）を入れ替える。

    継続ターン数（ひかりのねんどによる延長・まきびし等の層数を含む）はそのまま
    入れ替わる。ON_FIELD_ACTIVATE/ON_FIELD_DEACTIVATE イベントは発火しないため、
    おいかぜが移動してもかぜのり・ふうりょくでんきは誤発動しない。
    """
    attacker_side = battle.get_side(ctx.attacker)
    defender_side = battle.get_side(ctx.defender)
    attacker_side.swap_fields(defender_side, _COURT_CHANGE_TARGET_FIELDS)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

- **命中率**: 仕様表は「100」だが `accuracy` フィールドは**設定しないこと**（`MoveData.accuracy: int | None = None` がデフォルト）。`move_executor.py` の `_check_hit` 呼び出し条件は `need_hit_check = ctx.move.accuracy is not None and ...` であり、`accuracy=None` なら命中判定自体がスキップされ必ず成功する。これは同じ `target="field"` の `フェアリーロック`（`accuracy` 未指定）と同一パターン。誤って `accuracy=100` を設定すると `_check_hit` 内の `assert ctx.defender is not None` 等を経由する通常の命中率判定ルートに乗ってしまい、外れ得る技になってしまうため厳禁。
- **まもる無効化**: `Move.is_blocked_by_protect` は `target == "foe"` の場合のみ True を返す実装のため、`target="field"` にするだけで自動的にまもるで防げない技になる。追加フラグ不要。
- **マジックコート無効化**: `Move.is_reflectable` も `target in ("foe", "foe_side")` の場合のみ True になる実装のため、`target="field"` にするだけで自動的に反射対象外になる。追加フラグ不要。
- **失敗時のリベロ／へんげんじざい**: `docs/spec/turn.md` の `Event.ON_MOVE_CHARGE`（priority=100「へんげんじざい・リベロの発動」）は `ON_BEFORE_APPLY_MOVE`（本技の失敗判定）より前のフェーズで実行されるため、`コートチェンジ_can_apply` が技を失敗させても、タイプ変化自体は既に完了している。ハンドラ側で追加のガードは不要。ただしテストでは「場の状態が無い状態でコートチェンジ（リベロ所持）を使用 → 技は失敗するがタイプはノーマルになる」ことを確認するとよい。
- **入れ替え対象のホワイトリスト方式**: `SideFieldName` 型には `いやしのねがい/みかづきのまい/ねがいごと/みらいよち/はめつのねがい` も含まれるが、これらは `_COURT_CHANGE_TARGET_FIELDS` に**含めていない**。将来 `SideFieldName` に新しい名前を追加する際、うっかり全件走査に変更しないよう注意（例えば `for name in get_args(SideFieldName)` のような実装は誤り）。
- **ねばねばネットの「本来の使用者」情報**: 前述の「重要な確認事項」の通り、Gen9では追跡不要と判明したため実装しない。レビュー時に誤って `docs/spec/moves/コートチェンジ.md` の58〜59行目（Gen8視点の記述）を根拠に実装を追加しないよう注意。
- **おいかぜ交換時の行動順反映**: `おいかぜ_boost_spe` は `DomainEvent.ON_CALC_SPEED` で毎回動的に判定するため、`count` の交換のみで自動的に該当ターンの素早さ計算に反映される。追加対応不要。
- **ひかりのねんど延長分の保持**: `count` をそのまま交換するだけで延長ターン数ごと移動する。追加実装不要（「重要な確認事項」参照）。
- **まきびし／どくびしの層数**: `count` は「残りターン」ではなく「層数」を表す（`activate()` の実装より）。`swap_fields` は count を単純に交換するだけなので、層数もそのまま正しく入れ替わる。
- **ハンドラの再登録順序について**: 既に両陣営で有効な同名フィールド（例: 両者ともリフレクター展開中）を入れ替える場合も `swap_fields` は一旦 unregister → register を行う。`docs/spec/turn.md` の「同一Priority値の複数の処理は、実行順序が不定である」という規則により、この再登録による順序変化は仕様上問題にならない。
- **ダブルバトル関連要素は対象外**: ひのうみ/にじ/しつげん/キョダイベンタツ等のキョダイマックス由来の場は本プロジェクト（チャンピオンズシングル、ダイマックス機構自体が存在しない）では未実装であり、本計画でも新規実装しない。
- **ファストガード／ワイドガード／トリックガード**: `VolatileName` として実装されているため、`SideFieldManager` を扱う本実装の対象に含まれず、自動的に対象外となる。
- **`docs/progress/move.md` の更新（実装後、impl エージェントが対応）**: 現在の行 `コートチェンジ	-	-	x	-	n/a	n/a	-	変化	ノーマル	-	-	100	defender	？` は「仕様書」列・「計画書」列を `x` に、PP列を `100→10`、対象列を `defender→field`、効果列を実際の効果テキストに更新する必要がある。
- **テスト方針の参考**: 少なくとも以下のケースを想定（テスト作成はreview/testエージェントの担当）。
  - 自陣営のみに壁が張られている状態でコートチェンジ→相手陣営に壁が移り、自陣営の壁は消える
  - 両陣営に異なる継続ターンの壁がある状態でコートチェンジ→ターン数ごと入れ替わる
  - 自陣営のねがいごとと相手陣営のリフレクターが同時にある状態でコートチェンジ→リフレクターのみ入れ替わり、ねがいごとはそのまま残る
  - どちらの陣営にも対象フィールドが無い状態でコートチェンジ→技失敗（`MOVE_FAILED` ログ、リベロ所持なら発動はする）
  - コートチェンジでおいかぜが自陣営に移った直後のターンで行動順に反映されること、かつ移動元・移動先どちらのポケモンもかぜのり／ふうりょくでんきが発動しないこと
  - 味方が設置したねばねばネットがコートチェンジで味方側に来た場合でも、Gen9仕様としてまけんき／かちきは通常通り発動すること（`battle.foe(ctx.source)` による動的判定のまま変更していないことの回帰確認）
