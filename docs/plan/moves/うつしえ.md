# 計画書: うつしえ

## 仕様要約

- タイプ: ノーマル / 変化技 / PP 10 / 命中率 100
- 効果: 使用者の特性を、対象（相手）の現在の特性と同じにする
  （シングルバトルのみが対象のため `なりきり` と実質同一効果）
- 失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100）:
  - 対象の特性が `uncopyable` フラグを持つ場合
  - 使用者自身の特性が `protected` フラグを持つ場合
  - 使用者と対象の特性がすでに同じ場合
  - 使用者の特性変更が `is_change_blocked`（とくせいガード等）で防がれる場合
- まもる: 効果を受ける（`unprotectable` フラグ）
- マジックコート・マジックミラー: 反射されない（`unreflectable` フラグ、本レビューで新設）
- みがわり: 防がれる（既存の汎用判定。特別なフラグ不要）
- 詳細は `docs/spec/moves/うつしえ.md` を参照

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 節に以下の記載がある。

```
| 100 | なりきり: 失敗条件 |
```

実装上は `ON_BEFORE_APPLY_MOVE` priority 100 に対応する。これは `なりきり`・
`シンプルビーム`・`なかまづくり`・`なやみのタネ`・`スキルスワップ` と同じパターン
（いずれも `docs/spec/turn.md` で priority 100 に記載）。`うつしえ` は turn.md に
個別記載がないが、同種の「特性変更技」の失敗条件チェックとして同じ priority を採用する。

## 実装箇所

### `handlers/move_status.py` に追加する関数（五十音順: う たう と うらみ の間）

```python
def うつしえ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """うつしえの失敗条件チェック。

    以下の場合は失敗する:
    - 対象の特性が uncopyable フラグを持つ（コピーできない特性）
    - 使用者の特性が protected フラグを持つ（上書きできない特性）
    - 使用者と対象の特性がすでに同じである
    - 使用者の特性変更がとくせいガード等で防がれる
    """
    assert ctx.defender is not None
    if (
        ctx.defender.ability.has_flag("uncopyable")
        or ctx.attacker.ability.has_flag("protected")
        or ctx.attacker.ability.base_name == ctx.defender.ability.base_name
        or battle.ability_manager.is_change_blocked(ctx.attacker)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="うつしえ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def うつしえ_change_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """うつしえの効果: 使用者の特性を対象の特性と同じにする。

    対象がとくせいなし状態の場合は無効化された特性名（空文字列）を、
    かがくへんかガス等それ以外の理由で抑制されている場合は元の特性名を
    コピーする（`_ability_name_to_copy` 参照）。
    """
    assert ctx.defender is not None
    battle.change_ability(ctx.attacker, _ability_name_to_copy(ctx.defender))
    return HandlerReturn(value=value)
```

### `model/move.py` に `is_reflectable` プロパティを追加

`is_blocked_by_protect` と対になるプロパティとして追加し、マジックコート・
マジックミラー両方のハンドラから共通で参照する。

```python
@property
def is_reflectable(self) -> bool:
    """技がマジックコート・マジックミラーで跳ね返されるかどうかを判定する。"""
    return (
        self.category == "status"
        and self.target in ("foe", "foe_side")
        and not self.has_flag("unreflectable")
    )
```

### `handlers/volatile.py` の `マジックコート_reflect` を簡略化

def マジックコート_reflect(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:

def マジックミラー_reflect(battle: Battle, ctx: AttackContext, value: bool) -> HandlerReturn:
    """マジックミラー特性: 反射対象の変化技を跳ね返す。"""
    value = ctx.move.is_reflectable
    if value:
        _announce_ability_triggered(battle, ctx.defender)
    return HandlerReturn(value=value)

## 注意点・エッジケース（review-test 向け）

### `なりきり` との違い

- `なりきり` は現状、失敗条件に「使用者自身の特性がprotected」「使用者と対象の特性が
  同じ」を含めていない（`docs/plan/moves/なりきり.md` 参照）。うつしえは Wiki に
  明記されているためこれらを実装するが、`なりきり` 側の修正は本レビューの対象外。

### コピー元の特性名

- `_ability_name_to_copy(ctx.defender)` を参照する（`なりきり` と共通のヘルパー、
  `handlers/move_status.py` に定義）。相手がとくせいなし状態
  （`has_volatile("とくせいなし")`）なら空文字列（`ability.name`）を、
  かがくへんかガス等それ以外の理由で特性が抑制されている場合は本来の特性名
  （`ability.base_name`）をコピーする
  （2026-07-11 のかがくへんかガスレビューで `ability.name` 直接参照から修正。
  かがくへんかガスは特性そのものを消去せず効果の発動のみを抑制するため）。
- 同一性判定（`すでに対象と同じ特性`）は `base_name` 同士を比較する
  （`なかまづくり_can_apply` と同じ方式。かがくへんかガス等による無効化状態に
  左右されない、特性そのものの同一性を見るため）。

### とくせいガードとの関係

- `is_change_blocked` は使用者（attacker）側にのみ適用する。対象（defender）が
  とくせいガードを持っていても、それによって「対象の特性がコピーされること」自体は
  防がれない（`docs/spec/items/とくせいガード.md` 参照）。

### おうごんのからだとの関係

- 対象の特性がおうごんのからだの場合、うつしえ側の失敗条件チェックとは無関係に、
  おうごんのからだ自身の `ON_BEFORE_APPLY_MOVE` ハンドラ
  （`おうごんのからだ_block_status_move`）が変化技全般をブロックするため、
  追加の実装は不要（`docs/spec/abilities/おうごんのからだ.md` 参照）。

### unreflectable フラグの影響範囲

- `is_reflectable` プロパティ新設に伴い `マジックコート_reflect` /
  `マジックミラー_reflect` の判定ロジックを統一した。既存のどの技も
  `unreflectable` フラグを持たないため、うつしえ以外の挙動に変化はない
  （回帰確認は `tests/volatiles/test_volatile_hamayarawa.py` と
  `tests/abilities/test_ability_ma.py` の既存テストで担保する）。

### ゆびをふるとの関係

- Wiki には「ゆびをふるに選ばれない」と明記されているが、`ゆびをふる` 自体が
  本プロジェクトで未実装（`data/moves/move_ya.py` で `handlers={}` のTODO状態）
  のため、対応する除外フラグやリストは存在しない。`ゆびをふる` を実装する際に
  併せて対応すること。
