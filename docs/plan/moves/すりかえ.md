# すりかえ 実装計画書

## 仕様要約（docs/spec/moves/すりかえ.md 参照）

* タイプ: あく / 分類: 変化 / 威力: - / 命中: 100 / PP: 12
* 対象: 相手1体 / 優先度: 0 / 接触なし
* 効果: 自分と相手の持ち物を交換する。両者とも道具を持っていない場合は失敗する。
* トリックとタイプ以外完全に同一の効果を持つため、共通ハンドラ `すりかえ_swap_items` を共有する。

## 現状の実装状況

`src/jpoke/data/moves/move_sa.py` に登録済み。

```python
"すりかえ": MoveData(
    type="あく",
    category="status",
    pp=12,
    accuracy=100,
    flags={"unreflectable"},
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.すりかえ_swap_items,
        )
    }
),
```

## ハンドラ構成

| Event | subject_spec | 関数 | 役割 |
|---|---|---|---|
| `ON_STATUS_HIT` | `attacker:self` （デフォルト） | `すりかえ_swap_items` | `battle.item_manager.swap_items()` を呼び出し、両者の持ち物を交換する |

`swap_items()` 内部で以下を判定する（`core/item_manager.py`）:

1. 両者とも道具を持っていない場合は失敗（早期リターン）。
2. 上記を満たせば、両者に対して `can_change_item()`（`ON_CHECK_ITEM_CHANGE` イベント）で
   交換可否を判定する。
   * ねんちゃく・マルチタイプ／ARシステム（プレート・メモリ以外）・こだいかっせい/クォークチャージ
     （ブーストエナジー）・だいこんごうだま等の専用道具ガード・オーガポンのめん・メガストーン等は、
     すべてこの `ON_CHECK_ITEM_CHANGE` イベント経由で既存の特性・アイテムハンドラ側が実装済みであり、
     本技側の実装は不要。
   * みがわり・すりぬけの相互作用は、`core/move_executor.py` の `check_hit_substitute`（技の
     `target == "foe"` かつ `bypass_substitute` フラグなしの場合に `ON_CHECK_HIT_SUBSTITUTE` を
     発火）と特性すりぬけ側の実装で共通対応済みであり、本技側の実装は不要。
3. 交換可能なら両者の道具を入れ替える（`track_loss=False` のため、渡した道具はリサイクル等の
   復元対象にならない）。

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT`（変化技の効果発動イベント）が掲載されていないため、
CLAUDE.md の指示に従い既存の同種ハンドラを参照して決定する。トリック（`data/moves/move_ta.py`）が
同一の `すりかえ_swap_items` ハンドラを `Event.ON_STATUS_HIT` に明示的な priority 指定なし
（デフォルト値100）で登録しており、本技もこれに合わせてデフォルト優先度を使用する。
すりかえ・トリックとも単一の効果ハンドラしか持たない変化技であり、他ハンドラとの実行順序の
競合が発生しないため、デフォルト優先度で仕様を満たす。

## 実装箇所

### handlers/move_status.py

既存の `すりかえ_swap_items` で完結しており、追加実装は不要。

```python
def すりかえ_swap_items(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """すりかえ・トリックのアイテム交換効果。"""
    success = battle.item_manager.swap_items()
    return HandlerReturn(value=success)
```

### data/moves/move_sa.py（本レビューで修正）

* `pp` を `10` → `12` に修正（`docs/champions/move_list.txt` の値に合わせる。第6世代以降の
  本家PPは10だが、チャンピオンズでは全技PPが8/12/16/20の4段階に統一されているため）。
* `flags={"unreflectable"}` を追加。`Move.is_reflectable` は `category == "status"` かつ
  `target in ("foe", "foe_side")` の変化技をデフォルトでマジックコート反射対象とするが、
  すりかえ・トリックは仕様上マジックコートで反射されない例外技のため、フラグで明示的に除外する。

### handlers/item.py（本レビューで修正）

`こだわり_lock_move`（こだわり系アイテムの `ON_MOVE_END` ハンドラ）が、すりかえ・トリックの
使用によって新たにこだわり系アイテムを入手した場合でも無条件にロックしてしまう不具合を修正した。
すりかえ・トリック自身の使用時は、ロックを付与せず、既存のロックがあれば解除するよう分岐を追加した
（第五世代以降の仕様: 「使用者がこだわり系アイテムを手にした場合、次のターンは別の技を選ぶことができる。
すでにトリックでこだわっていたときも、こだわりは解除される」）。

```python
def こだわり_lock_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    mon = ctx.attacker
    if ctx.move.name in ("トリック", "すりかえ"):
        if mon.has_volatile("こだわり"):
            battle.volatile_manager.remove(mon, "こだわり")
        return HandlerReturn(value=value)
    if not mon.has_volatile("こだわり"):
        battle.volatile_manager.apply(
            mon, "こだわり", source=mon, move_name=ctx.move.name
        )
    return HandlerReturn(value=value)
```

なお、どろぼう・ほしがるにも同種の「奪った道具がこだわり系アイテムでもロックされない」仕様が
`docs/spec/moves/どろぼう.md` に明記されているが、この不具合修正は対象外（別途どろぼう側の
レビューで対応する）。

## 注意点・エッジケース（review-test 向け）

1. **両者とも道具を持っていない場合** → 失敗。`swap_items()` の早期リターンで対応済み。
2. **片方のみ道具を持っている場合** → 成功（持っていない側に道具が渡る）。
3. **対象がみがわり状態** → 技がみがわりに防がれるため失敗。特性すりぬけなら貫通する
   （共通インフラで対応済み）。
4. **対象の特性がねんちゃく** → 交換失敗（`ねんちゃく_prevent_item_change` で対応済み）。
5. **専用道具の組み合わせ（ディアルガ/だいこんごうだま等）** → 交換失敗（各アイテムの
   `ON_CHECK_ITEM_CHANGE` ハンドラで対応済み）。
6. **マジックコート・マジックミラーで跳ね返らない** → 本レビューで `unreflectable` フラグを
   追加して対応。
7. **こだわり系アイテムを入手してもロックされない** → 本レビューで `こだわり_lock_move` を
   修正して対応。
8. **リサイクル・しゅうかく・ものひろいとの相互作用** → `swap_items()` が `track_loss=False`
   で `_change_item` を呼ぶため、渡した道具は復元・拾得対象にならない（既存実装で対応済み）。
