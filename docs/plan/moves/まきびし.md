# 実装計画書: まきびし

## 仕様要約

詳細は `docs/spec/side_fields/まきびし.md` を参照。

- タイプ: じめん / カテゴリ: 変化 / PP: 20 / 対象: 相手の場（target="foe_side"）
- 使用後、相手はポケモンを交代するたびに出てきたポケモンが最大HPの 1/8 のダメージを受ける
- 2回使用すると最大HPの 1/6、3回使用すると最大HPの 1/4 のダメージになる（最大 3 層）
- ひこうタイプ・特性「ふゆう」等、地面にいない扱いのポケモンには無効（詳細は仕様書参照）
- 「どくタイプ」による無効はない（どくびしと異なり、まきびしにタイプ免疫は存在しない）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手陣営サイドフィールドに「まきびし」を 1 層追加 |

## Priority 根拠

`Event.ON_STATUS_HIT` は `docs/spec/turn.md` の優先度テーブルに未掲載のイベントのため、
既存の同種ハンドラを参照して決定する。`どくびし_set_field` / `ステルスロック_set_field` /
`ねばねばネット` 等、他の設置技ハンドラもすべて priority を明示指定せずデフォルト値（100）の
ままで登録されているため、まきびしも同様にデフォルト priority（100）・`subject_spec` 省略
（デフォルトの `attacker:self`）で登録する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「ま」行 — `ほたるび_modify_attacker_stats` より後、`まるくなる_apply` の前に挿入する。

`ON_STATUS_HIT` は攻撃フローの一部として発火するため、`ctx` は `AttackContext`
（`attacker`/`defender` ロール）を使う。`どくびし_set_field` / `ステルスロック_set_field`
と同じパターン。

```python
def まきびし_set_field(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """まきびし: 相手陣営に「まきびし」を1層設置する（最大3層）。"""
    side = battle.get_side(ctx.defender)
    if not side.activate("まきびし", 1):
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ma.py`

既存の `"まきびし"` エントリに handlers を追加する。

```python
"まきびし": MoveData(
    type="じめん",
    category="status",
    pp=20,
    target="foe_side",
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.まきびし_set_field,
        ),
    }
),
```

## 注意点・エッジケース

- `side.activate("まきびし", 1)` は重複設置（層数加算）を `StackableFieldManager.activate()` が管理する
  （`core/field_manager.py`）。`field.data.max_count`（3）に達している場合は False を返し、技は失敗する
- `ステルスロック` は max_count=1（単層フィールド）のため count=1 が「設置済み判定」として機能するが、
  まきびしは `max_count=3` の多層フィールドとして `data/field/side_field.py` に登録済み
- ひこうタイプ・ふゆう特性・どく無効（存在しない）等の判定はサイドフィールドの `ON_SWITCH_IN`
  ハンドラ（`まきびし_damage`、`is_hazard_immune` / `is_floating` クエリ経由）側で処理されるため、
  技ハンドラ側でのガードは不要
- ダメージ量（1/8 → 1/6 → 1/4）の段階制御もサイドフィールドハンドラ側の責務
