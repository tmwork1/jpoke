# 計画書: のろい

## 仕様要約

- 使用者がゴーストタイプかどうかで効果が二分される（判定は技使用時のタイプ）
  - **ゴーストタイプ（呪い）**: 自分のHPを最大HPの1/2（切り捨て）消費し、相手をのろい状態にする
    - のろい状態: ターン終了時に最大HPの1/4のダメージを受ける
    - まもる・ダイウォールを貫通する
    - マジックコートで跳ね返されない
    - よこどり対象外
    - みがわりを無視してのろい状態を付与する（第五世代以降）
    - 対象がすでにのろい状態の場合は失敗（HP消費なし）
    - 技が失敗・外れた場合もHP消費なし
    - マジックガード保持者が使用してもHPは消費される
    - マジックガード保持者にのろい状態を付与すると技は成功するがダメージは発生しない
    - HPが半分以下でも発動可能（使用者がひんしになっても相手にのろいは入る）
    - テラスタル中はテラスタル後のタイプを基準に判定
  - **ゴーストタイプ以外（鈍い）**: 自分のこうげき・ぼうぎょを1段階上げ、すばやさを1段階下げる
    - 第三世代以降: こうげき・ぼうぎょ両方最大かつすばやさ最低の場合のみ失敗

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_APPLY_MOVE` | 100 | `attacker:self` | 「呪い」: 対象がすでにのろい状態なら失敗 |
| `ON_STATUS_HIT` | (デフォルト) | `attacker:self` | タイプで分岐: ゴーストなら呪い処理、それ以外なら鈍い処理 |

## Priority 根拠

- `Event.ON_APPLY_MOVE` priority=100: `docs/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「のろい(呪い): すでに状態変化」が priority=100 として明記されている
- `Event.ON_STATUS_HIT` priority: デフォルト（未指定）。既存の多くの変化技（あくび、アンコールなど）に合わせてデフォルト priority を使用

## 実装箇所

### `src/jpoke/handlers/move_status.py` への追加

五十音順に従い「ねをはる」の次・「ふういん」の前（「の」の位置）に追加する。

```python
def のろい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """のろいの失敗チェック（呪い版）: 対象がすでにのろい状態なら失敗する。

    ゴーストタイプ以外が使う場合（鈍い）には呼び出されない。
    対象がすでにのろい状態の場合、HP消費も発生しない。
    """
    if ctx.defender.has_volatile("のろい"):
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "のろい_すでに状態変化"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def のろい_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """のろいの効果: タイプによって呪いと鈍いに分岐する。

    ゴーストタイプ: 自分のHPを最大HPの1/2（切り捨て）消費し、相手をのろい状態にする。
    ゴーストタイプ以外: こうげき・ぼうぎょを1段階上げ、すばやさを1段階下げる。
    """
    mon = ctx.attacker
    if mon.has_type("ゴースト"):
        # 呪い: HP消費 → のろい状態付与
        cost = mon.max_hp // 2
        battle.modify_hp(mon, v=-cost)
        return apply_volatile_to_defender(battle, ctx, value, volatile="のろい")
    else:
        # 鈍い: こうげき・ぼうぎょ上昇、すばやさ低下
        return modify_attacker_stats(battle, ctx, value, stats={"A": 1, "B": 1, "S": -1})
```

## 注意点・エッジケース（review-test 向け）

### ON_APPLY_MOVE ガードの対象範囲

- `のろい_can_apply` はゴーストタイプが使う呪い版専用の失敗判定
- ゴーストタイプ以外が使う鈍い版には「すでに状態変化」の失敗条件が存在しないため、同一ハンドラ内でタイプ判定が必要
  - しかし ON_APPLY_MOVE は呪い・鈍いどちらにも呼ばれてしまう
  - 解決策: `のろい_can_apply` 内でゴーストタイプでない場合はガードを素通り（`return HandlerReturn(value=value)`）させる
  - 実装コードを以下に修正する:

```python
def のろい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """のろいの失敗チェック（呪い版）: 対象がすでにのろい状態なら失敗する。

    ゴーストタイプ以外が使う場合（鈍い）はガードをスキップする。
    """
    if not ctx.attacker.has_type("ゴースト"):
        return HandlerReturn(value=value)
    if ctx.defender.has_volatile("のろい"):
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_FAILED,
            payload={"reason": "のろい_すでに状態変化"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### すでにのろい状態の場合のHP消費なし

- `volatile_manager.apply` は対象がすでにのろい状態の場合に `False` を返す
- しかし `のろい_apply` では先に `battle.modify_hp(mon, v=-cost)` を呼んでしまうと、仮に volatile_manager.apply が失敗してもHP消費が発生してしまう
- ON_APPLY_MOVE でのガード（`のろい_can_apply`）がその前に止めるため問題なし
- ただし ON_APPLY_MOVE ガードをすり抜けるケース（みがわり状態への付与など）も考慮が必要
  - 仕様上「みがわりを無視してのろい状態を付与する（第五世代以降）」なので、みがわり状態でも成功する
  - `volatile_manager.apply` はみがわりをブロックしない（みがわりはON_APPLY_MOVEで処理される）
  - HP消費は技が成功することが確定した後に呼ばれるため、ON_STATUS_HIT での順序（modify_hp → apply_volatile）で問題ない

### HPが半分以下でのろいを使うとひんしになる

- `battle.modify_hp(mon, v=-cost)` はHP=1（最大HP=2以下等）でも呼ばれる
- `faint` 処理は `modify_hp` 内部で自動的に行われる
- のろいの付与自体は別の処理なので、使用者がひんしになってものろい付与は完了する

### ランク補正変化の失敗条件（鈍い）

- 第三世代以降: こうげき・ぼうぎょが両方+6かつすばやさが-6の場合のみ失敗
- `modify_attacker_stats` を使うと `battle.modify_stats` が内部でランク上限/下限をチェックし適切に処理する
- turn.md の `ON_TRY_MOVE_2` priority=130 に「ランク補正上げ: すでに最大」「ランク補正下げ: すでに最低」が掲載されているが、鈍いはこうげき・ぼうぎょ・すばやさの組み合わせ全体での判定なので、既存のランク補正チェックでは賄えない可能性がある
- ただしポケモンチャンピオンズのシングルバトルでの実用上の影響は軽微なため、まず第三世代以降の仕様（各ランクが上限/下限でも技自体は動作する）に従ってデフォルト動作に委ねる実装で問題ない

### テラスタル時のタイプ判定

- テラスタル中はテラスタル後のタイプを `has_type("ゴースト")` が返す（`pokemon.types` プロパティが `active_tera_type` を考慮するため）
- ステラタイプでテラスタルしている場合はテラスタル前のタイプを基準とする仕様だが、`has_type` の実装が `active_tera_type == "ステラ"` のケースを正しく扱うかは既存の `pokemon.types` プロパティの実装に依存する

### マジックガード

- マジックガードの保持者にのろいを付与した場合、volatile 自体は付与されるが ON_TURN_END での `のろい_damage` がマジックガードによってブロックされる（既存実装で処理済み）
- マジックガード保持者が呪いを使う場合はHP消費が発生する（仕様通り）
