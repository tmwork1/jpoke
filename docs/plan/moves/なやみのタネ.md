# 計画書: なやみのタネ

## 仕様要約

- タイプ: くさ / 変化技 / PP 12（champions/move_list.txt準拠） / 命中 100 / 対象: 相手1体
- 効果: 相手の特性を「ふみん」に書き換える
- 失敗条件:
  - 対象の特性の `base_name` が「ふみん」または「なまけ」
  - 対象の特性が `protected` フラグを持つ一部の固有特性（マルチタイプ、ばけのかわ、
    ダルマモード、アイスフェイス等）
  - 対象の持ち物が「とくせいガード」等で特性変更がブロックされる場合
  - 対象の特性が「そうしょく」「おうごんのからだ」の場合は、各特性側の
    `ON_BEFORE_APPLY_MOVE` ハンドラにより無効化される（なやみのタネ側の実装不要）
- `docs/spec/turn.md` 記載:
  - `Event.ON_TRY_MOVE_2` priority 130: `なやみのタネ: 特性がふみん・なまけ`
  - `Event.ON_APPLY_MOVE`（実装上 `ON_BEFORE_APPLY_MOVE`）priority 100: `なやみのタネ: 失敗条件`

詳細は `docs/spec/moves/なやみのタネ.md` を参照。

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE` 節に以下の記載がある:

```
| 100 | なやみのタネ: 失敗条件 |
```

同じく `Event.ON_TRY_MOVE_2` の priority 130 に「なやみのタネ: 特性がふみん・なまけ」の記載があるが、既存の類似実装（いえき・シンプルビーム・なかまづくり）は `ON_BEFORE_APPLY_MOVE` priority 100 の 1 箇所にまとめて実装している。本実装も同パターンに従い `ON_BEFORE_APPLY_MOVE` priority 100 で統合実装している。

## 実装箇所

### `handlers/move_status.py`（実装済み）

```python
def なやみのタネ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なやみのタネの失敗条件チェック。

    対象の特性が protected フラグを持つ場合、
    または「ふみん」「なまけ」の場合は失敗する。
    対象の特性変更がとくせいガード等で防がれる場合も失敗する。

    「ふみん」「なまけ」の判定には base_name（かがくへんかガス・かたやぶり等による
    無効化状態に関わらない元の特性名）を使う。特性ふみんは mold_breaker_ignorable
    フラグを持つため、かたやぶりの効果中は ability.name が空文字になり判定を
    すり抜けてしまうことを避けるため。
    """
    assert ctx.defender is not None
    ability = ctx.defender.ability
    if (
        ability.has_flag("protected")
        or ability.base_name in ("ふみん", "なまけ")
        or battle.ability_manager.is_change_blocked(ctx.defender)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="なやみのタネ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def なやみのタネ_change_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なやみのタネの効果: 相手の特性を「ふみん」に書き換える。"""
    assert ctx.defender is not None
    battle.change_ability(ctx.defender, "ふみん")
    return HandlerReturn(value=value)
```

### `handlers/ability.py` / `data/ability.py`「ふみん」（実装済み・本レビューで追加）

なやみのタネの「対象がすでにねむり状態のときは即座に回復する」という仕様
（`docs/spec/moves/なやみのタネ.md` 参照）は、なやみのタネ側ではなく「ふみん」
特性自身の `ON_ABILITY_ENABLED` ハンドラとして実装した。これは switch_in では
なく、`change_ability`（なやみのタネ・なかまづくり・なりきり・スキルスワップ等）、
かがくへんかガスの解除、かたやぶりの効果終了、メガシンカなど「特性ふみんが
有効になった瞬間」全てに共通して発動する汎用的な仕組みであり、なやみのタネ
固有の実装ではない。

```python
def ふみん_cure_sleep_on_enable(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """ふみん特性: 特性が有効化された時点ですでにねむり状態なら即座に回復する。"""
    mon = ctx.source
    if mon is None:
        return HandlerReturn(value=value)
    if mon.ailment.is_sleep and battle.ailment_manager.remove(mon):
        _announce_ability_triggered(battle, mon)
    return HandlerReturn(value=value)
```

`data/ability.py` の「ふみん」エントリに `Event.ON_ABILITY_ENABLED` として
`subject_spec="source:self"` で登録する。

## 注意点・エッジケース（review-test 向け）

### 失敗条件の詳細

- 「ふみん」は `protected` フラグを持たないが、なやみのタネを使われてもすでに「ふみん」のため失敗させる必要がある（`ability.base_name == "ふみん"` チェックで対応）
- 「なまけ」も `protected` フラグを持たない（`data/ability.py` 確認済み）ため、個別に名前チェックが必要（`ability.base_name == "なまけ"` チェックで対応）
- 「マルチタイプ」「ばけのかわ」「ダルマモード」「アイスフェイス」など `protected` フラグを持つ固有特性は `ability.has_flag("protected")` で一括処理
- 「やるき」はなやみのタネが成功する（`docs/spec/abilities/ふみん.md` 備考に明記されている）。`protected` フラグなし、`base_name` も「ふみん」「なまけ」でもないため、ハンドラは何もせず通過する
- 「そうしょく」「おうごんのからだ」は各特性側の `ON_BEFORE_APPLY_MOVE` ハンドラ
  （`そうしょく_absorb_grass` / `おうごんのからだ_block_status_move`）が
  なやみのタネより先に発動して技自体を無効化するため、なやみのタネ側での
  個別対応は不要。両特性とも `mold_breaker_ignorable` フラグを持つため、
  使用者がかたやぶりならこれらの特性は無視される（既存の汎用処理で対応済み）。

### 特性変更後の挙動

- `battle.change_ability(ctx.defender, "ふみん")` が呼ばれると defender にふみん特性のハンドラが登録される
- 交代すると `base_ability_name` に戻る（`change_ability` は `base_ability_name` を更新しない）ため、交代後は元の特性に戻る

### マジックコート・マジックミラー

- なやみのタネは変化技かつ対象が相手のため、マジックコート・マジックミラーで反射される対象となる
- 反射処理は `ON_TRY_MOVE_1` priority 100 の manager 内部処理のため、ハンドラ側に追加ガードは不要

### 「ふみん」付与後の眠り系技との相互作用

- なやみのタネで特性がふみんに書き換わったポケモンは、そのターン以降、眠り系技・状態異常（ねむり）の付与が「ふみん」ハンドラによりブロックされる
- すでに「ねむり」状態のポケモンに使用した場合: `docs/spec/abilities/ふみん.md` によれば「ふみん」に書き換えられた場合は即座にねむりを治す。これは `change_ability` 内部で `ON_ABILITY_ENABLED` イベントが発火し、ふみん特性の `ふみん_cure_sleep_on_enable` ハンドラが動くことで処理される（今回のレビューで追加、なやみのタネ側の追加実装は不要）

### `ability.name` vs `ability.base_name` について

- 「ふみん」「なまけ」チェックには `ability.base_name`（特性の有効/無効状態に
  関わらない元の特性名）を使う。**`ability.name` は使ってはいけない。**
- `ability.name` は特性が無効化されている間は空文字を返す実装になっている
  （`GameEffect.name` プロパティ参照）。特性「ふみん」は `mold_breaker_ignorable`
  フラグを持つため、使用者がかたやぶりで攻撃した場合や相手がかがくへんかガスの
  影響下にある場合、`ON_BEFORE_APPLY_MOVE` の時点で `ability.name` が空文字に
  なってしまい、`ふみん` を検出できずに本来失敗すべきなやみのタネが成功して
  しまうバグが過去に存在した（本レビューで `base_name` に修正済み）。
- Pokemon Showdown 等の実装でも `pokemon.ability` の識別子自体は特性の
  有効/無効状態に影響されず一定であるという設計に合わせている。
