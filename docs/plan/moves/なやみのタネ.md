# 計画書: なやみのタネ

## 仕様要約

- タイプ: くさ / 変化技 / PP 10 / 命中 100 / 対象: 相手1体
- 効果: 相手の特性を「ふみん」に書き換える
- 失敗条件:
  - 対象の特性が「ふみん」（特性名が `protected` フラグを持つか、直接ふみん判定）
  - 対象の特性が「なまけ」（なまけは `protected` フラグを持たないため、名前で個別チェックが必要）
  - 対象の特性が「バトルスイッチ」など `protected` フラグを持つ一部の固有特性
- `docs/spec/turn.md` 記載:
  - `Event.ON_TRY_MOVE_2` priority 130: `なやみのタネ: 特性がふみん・なまけ`
  - `Event.ON_APPLY_MOVE`（実装上 `ON_BEFORE_APPLY_MOVE`）priority 100: `なやみのタネ: 失敗条件`

## 実装箇所

### `handlers/move_status.py` に追加する関数（五十音順: な の位置）

```python
def なやみのタネ_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なやみのタネの失敗条件チェック。

    対象の特性が protected フラグを持つ場合、
    または「ふみん」「なまけ」の場合は失敗する。
    """
    ability = ctx.defender.ability
    if (ability.has_flag("protected")
            or ability.name == "ふみん"
            or ability.name == "なまけ"):
        battle.add_event_log(ctx.attacker, LogCode.MOVE_FAILED,
                             payload={"reason": "なやみのタネ"})
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def なやみのタネ_change_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """なやみのタネの効果: 相手の特性を「ふみん」に書き換える。"""
    battle.change_ability(ctx.defender, "ふみん")
    return HandlerReturn(value=value)
```

### `data/move.py` の「なやみのタネ」エントリを更新（五十音順: な 行）

```python
"なやみのタネ": MoveData(
    type="くさ",
    category="変化",
    pp=10,
    accuracy=100,
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.なやみのタネ_can_apply,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.なやみのタネ_change_ability,
        ),
    }
),
```

## 注意点・エッジケース（review-test 向け）

### 失敗条件の詳細

- 「ふみん」は `protected` フラグを持たないが、なやみのタネを使われてもすでに「ふみん」のため失敗させる必要がある（`ability.name == "ふみん"` チェックで対応）
- 「なまけ」も `protected` フラグを持たない（`data/ability.py` 確認済み）ため、個別に名前チェックが必要（`ability.name == "なまけ"` チェックで対応）
- 「バトルスイッチ」など `protected` フラグを持つ固有特性は `ability.has_flag("protected")` で一括処理
- 「やるき」はなやみのタネが成功する（`docs/spec/abilities/ふみん.md` 備考に明記されている）。`protected` フラグなし、名前も「ふみん」「なまけ」でもないため、ハンドラは何もせず通過する

### 特性変更後の挙動

- `battle.change_ability(ctx.defender, "ふみん")` が呼ばれると defender にふみん特性のハンドラが登録される
- 交代すると `base_ability_name` に戻る（`change_ability` は `base_ability_name` を更新しない）ため、交代後は元の特性に戻る

### マジックコート・マジックミラー

- なやみのタネは変化技かつ対象が相手のため、マジックコート・マジックミラーで反射される対象となる
- 反射処理は `ON_TRY_MOVE_1` priority 100 の manager 内部処理のため、ハンドラ側に追加ガードは不要

### 「ふみん」付与後の眠り系技との相互作用

- なやみのタネで特性がふみんに書き換わったポケモンは、そのターン以降、眠り系技・状態異常（ねむり）の付与が「ふみん」ハンドラによりブロックされる
- すでに「ねむり」状態のポケモンに使用した場合: `docs/spec/abilities/ふみん.md` によれば「ふみん」に書き換えられた場合は即座にねむりを治す。これは `change_ability` 内部で `ON_ABILITY_ENABLED` イベントが発火し、ふみんのねむり回復ハンドラが動くことで処理される（ハンドラ側で追加処理は不要）

### `ability.name` vs `ability.base_name` について

- 「ふみん」「なまけ」チェックには `ability.name`（現在の特性名）を使う
- かがくへんかガスで特性が無効化されている場合も `ability.name` は変わらないため、判定は正常に動作する（いえき等と同じ挙動）
