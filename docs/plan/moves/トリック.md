# トリック 実装計画書

## 仕様要約（docs/spec/moves/トリック.md 参照）

* タイプ: エスパー / 分類: 変化 / 威力: - / 命中: 100 / PP: 12
* 対象: 相手1体 / 優先度: 0 / 接触なし
* 効果: 自分と相手の持ち物を交換する。両者とも道具を持っていない場合は失敗する。
* すりかえとタイプ以外完全に同一の効果を持つため、共通ハンドラ `すりかえ_swap_items` /
  `すりかえ_release_choice_lock` を共有する。

## 現状の実装状況

`src/jpoke/data/moves/move_ta.py` に登録済み。

```python
"トリック": MoveData(
    type="エスパー",
    category="status",
    pp=12,
    accuracy=100,
    flags={"unreflectable", "non_copycat"},
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.すりかえ_swap_items,
        ),
        # こだわり系アイテムの ON_MOVE_END ハンドラ（デフォルト優先度100）より
        # 後に発動させ、自身の効果で入手したこだわり系アイテムによるロックを解除する。
        Event.ON_MOVE_END: h.MoveHandler(
            hs.すりかえ_release_choice_lock,
            priority=110,
        ),
    }
),
```

すりかえ側（`src/jpoke/data/moves/move_sa.py`）と完全に同一のハンドラ構成であり、
実装済み。本レビューでの修正は不要と判断した（下記「ハンドラ構成」「注意点」を参照して
仕様との整合を確認済み）。

## ハンドラ構成

| Event | priority | subject_spec | 関数 | 役割 |
|---|---|---|---|---|
| `ON_STATUS_HIT` | デフォルト（100） | `attacker:self` （デフォルト） | `すりかえ_swap_items` | `battle.item_manager.swap_items()` を呼び出し、両者の持ち物を交換する |
| `ON_MOVE_END` | 110 | `attacker:self` （デフォルト） | `すりかえ_release_choice_lock` | 自身の効果で新たに入手したこだわり系アイテムによるロックを解除する（既存ロックがあれば解除） |

`swap_items()` 内部で以下を判定する（`core/item_manager.py`）:

1. 両者とも道具を持っていない場合は失敗（早期リターン）。
2. 上記を満たせば、両者に対して `can_change_item()`（`ON_CHECK_ITEM_CHANGE` イベント）で
   交換可否を判定する。
   * ねんちゃく・マルチタイプ／ARシステム（プレート・メモリ以外）・こだいかっせい/クォークチャージ
     （ブーストエナジー）・だいこんごうだま等の専用道具ガード・オーガポンのめん・メガストーン等は、
     すべてこの `ON_CHECK_ITEM_CHANGE` イベント経由で既存の特性・アイテムハンドラ側が実装済みであり、
     本技側の実装は不要。
   * みがわり・すりぬけの相互作用は、`core/move_executor.py` の `check_hit_substitute`（技の
     `target == "foe"` かつ `bypass_substitute` フラグなしの場合に `ON_CHECK_HIT_SUBSTITUTE` を
     発火）と特性すりぬけ側の実装で共通対応済みであり、本技側の実装は不要。
3. 交換可能なら両者の道具を入れ替える（`track_loss=False` のため、渡した道具はリサイクル等の
   復元対象にならない）。

`ON_MOVE_END` の `すりかえ_release_choice_lock` は、こだわり系アイテムの `ON_MOVE_END`
ハンドラ（`こだわり_lock_move`、デフォルト優先度100）より後に発動するよう priority=110 を
指定し、自身の効果で新規にこだわり系アイテムを入手した場合でもロックが付与されないよう、
または既存のロックがあれば確実に解除するようにしている。

## Priority 根拠

`docs/spec/turn.md` には `ON_STATUS_HIT`（変化技の効果発動イベント）・`ON_MOVE_END` の
詳細な相対順位が掲載されていないため、CLAUDE.md の指示に従い既存の同種ハンドラを参照して
決定する。すりかえ（`data/moves/move_sa.py`）が同一のハンドラ構成（`ON_STATUS_HIT` は
デフォルト優先度100、`ON_MOVE_END` は優先度110）を採用しており、本技もこれに合わせている。
`ON_MOVE_END` の優先度110は、こだわり系アイテムの `ON_MOVE_END` ハンドラ（デフォルト優先度
100）より確実に後に実行してロック解除を成立させるための根拠がある。

## 実装箇所

### handlers/move_status.py

既存の `すりかえ_swap_items` / `すりかえ_release_choice_lock` で完結しており、追加実装は不要。

```python
def すりかえ_release_choice_lock(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """すりかえ・トリックの使用後、こだわりロックを解除する。"""
    mon = ctx.attacker
    if mon.has_volatile("こだわり"):
        battle.volatile_manager.remove(mon, "こだわり")
    return HandlerReturn(value=value)


def すりかえ_swap_items(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """すりかえ・トリックのアイテム交換効果。"""
    success = battle.item_manager.swap_items()
    return HandlerReturn(value=success)
```

### data/moves/move_ta.py

既存実装のとおりで修正不要。`flags={"unreflectable", "non_copycat"}` により、
マジックコート・マジックミラーで反射されない仕様と、まねっこでコピーできない仕様の
両方に対応済み。

## 注意点・エッジケース（review-test 向け）

1. **両者とも道具を持っていない場合** → 失敗。`swap_items()` の早期リターンで対応済み。
2. **片方のみ道具を持っている場合** → 成功（持っていない側に道具が渡る）。
3. **対象がみがわり状態** → 技がみがわりに防がれるため失敗。特性すりぬけなら貫通する
   （共通インフラで対応済み）。
4. **対象の特性がねんちゃく** → 交換失敗（`ねんちゃく_prevent_item_change` で対応済み）。
5. **専用道具の組み合わせ（ディアルガ/だいこんごうだま等）** → 交換失敗（各アイテムの
   `ON_CHECK_ITEM_CHANGE` ハンドラで対応済み）。
6. **マジックコート・マジックミラーで跳ね返らない** → `unreflectable` フラグで対応済み。
7. **こだわり系アイテムを入手してもロックされない** → `すりかえ_release_choice_lock` で対応済み。
8. **既にこだわりでロックされていた場合も解除される** → 同上。
9. **まねっこでコピーできない** → `non_copycat` フラグで対応済み。
10. **リサイクル・しゅうかく・ものひろいとの相互作用** → `swap_items()` が `track_loss=False`
    で `_change_item` を呼ぶため、渡した道具は復元・拾得対象にならない（既存実装で対応済み）。
