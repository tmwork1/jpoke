# ころがる 実装計画書

## 仕様要約（docs/spec/moves/ころがる.md 参照）

* タイプ: いわ / 分類: 物理 / 威力: 30（命中するたびに2倍） / 命中: 90 / PP: 20
* 対象: 相手1体 / 優先度: 0 / 接触あり
* 効果: 外れるまで最大5ターンの間連続で攻撃する強制行動技。命中するたびに威力が2倍になり、
  外れる・失敗する・5ターン経過すると強制行動状態が終了する。

`まるくなる` 状態による威力2倍効果（`まるくなる_boost_power`）はすでに実装済みで、
`ころがる`/`アイスボール` の名前で判定しているため、本計画の変更を必要としない。

## 実装方針

`あばれる`（Thrash系）・`あなをほる`（Dig）と同じ「forced 揮発性状態 + `ON_MODIFY_COMMAND_OPTIONS` で
`force_command` を使い行動を固定する」パターンを踏襲する。ただし、あばれるはヒット有無に関わらず
ターン数で強制終了するのに対し、ころがるは「外れたら即終了」「命中回数に応じて威力が指数的に増加する」
という点で異なるため、専用の揮発性状態 `ころがる` を新設する。

### 揮発性状態 `ころがる` の設計

* `Volatile.count` フィールドを「これまでの連続命中回数」として利用する（`あばれる` の
  ターン残数カウントダウンとは逆に、命中のたびに加算していく）。
* `count == 1`: 1回命中した状態（次の攻撃威力は30×2^1=60）。
* `count` が5に達したら（5回目の命中が発生したら）揮発性状態を解除し、強制行動を終了する。
* 命中しなかった場合（外れ・まもる・タイプ無効・状態異常等での不発）は、ターン終了時に
  `Pokemon.failed_or_immobile_last_turn` を見て揮発性状態を解除する。

## ハンドラ構成

| 対象 | イベント | ハンドラ | subject_spec | priority | 説明 |
|------|---------|---------|--------------|----------|------|
| Move | ON_HIT | ころがる_apply | attacker:self | 100（デフォルト） | 命中時、揮発性状態がなければ付与（count=1）。あれば count を+1し、5に達したら解除する |
| Volatile | ON_MODIFY_COMMAND_OPTIONS | force_command（既存共通関数） | source:self | 100（デフォルト） | 行動選択を `ころがる` に固定する（あばれる・あなをほると共通） |
| Volatile | ON_CALC_POWER_MODIFIER | ころがる_boost_power | attacker:self | 100（デフォルト） | `value *= 2 ** count` で威力を指数的に増加させる |
| Volatile | ON_TURN_END | ころがる_check_interrupt | source:self | 150 | このターンの行動が失敗していた場合（`failed_or_immobile_last_turn`）、揮発性状態を解除する |

### priority の根拠

* `ON_HIT`: docs/spec/turn.md の Event.ON_HIT 節に本効果と同種の処理は明記されていないため、
  既存の同種ハンドラ（`あばれる_apply` 等、追加効果のない汎用フックは全てデフォルト優先度=100で登録）に
  合わせてデフォルト値を採用する。
* `ON_MODIFY_COMMAND_OPTIONS`: `あばれる`・`あなをほる` の `force_command` 登録がいずれもデフォルト優先度=100
  であるため、それに合わせる。
* `ON_CALC_POWER_MODIFIER`: docs/spec/turn.md に本イベントの行はないため、既存の同種ハンドラ
  （`まるくなる_boost_power`、`あなをほる_boost_power`）のデフォルト優先度=100に合わせる。
* `ON_TURN_END`: docs/spec/turn.md の Event.ON_TURN_END 節で、`さわぐ`・`ねむりによるあばれる中断`・
  `スロースタート`・`ナイトメア` 等の「多ターン行動技の状態管理」がpriority=150に分類されているため、
  同じ分類として priority=150 を採用する。

### subject_spec の根拠

* Move側ハンドラ（`ON_HIT`）は攻撃フェーズ（`AttackContext`）なので `attacker:self`（デフォルト）を使用する。
* Volatile側ハンドラのうち `ON_MODIFY_COMMAND_OPTIONS`・`ON_TURN_END` は非攻撃フェーズ（`EventContext`）の
  イベントであり、揮発性状態の保持者自身に発動するため `source:self` を使用する
  （`あばれる`・`あなをほる`・`アクアリング` 等の既存パターンと同一）。
* `ON_CALC_POWER_MODIFIER` は攻撃フェーズのイベントであり、揮発性状態を持つ攻撃側自身の威力に影響するため
  `attacker:self` を使用する（`まるくなる_boost_power`・`あなをほる_boost_power` と同一パターン）。

## 実装箇所

### src/jpoke/handlers/move_attack.py

```python
def ころがる_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ころがる: 命中時に揮発性状態を更新する。

    初回命中時は揮発性状態（count=1）を付与して強制行動を開始する。
    継続中の命中の場合は count を+1し、5回目の命中で強制行動を終了する。
    """
    attacker = ctx.attacker
    if attacker.has_volatile("ころがる"):
        volatile = attacker.volatiles["ころがる"]
        volatile.count += 1
        if volatile.count >= 5:
            battle.volatile_manager.remove(attacker, "ころがる")
    else:
        battle.volatile_manager.apply(
            attacker, "ころがる", count=1, source=attacker, move_name=ctx.move.name
        )
    return HandlerReturn(value=value)
```

### src/jpoke/handlers/volatile.py

```python
def ころがる_boost_power(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ころがる状態: これまでの連続命中回数に応じて威力を2^count倍にする（30→60→120→240→480）。"""
    volatile = ctx.attacker.volatiles.get("ころがる")
    if volatile is not None:
        value *= 2 ** volatile.count
    return HandlerReturn(value=value)


def ころがる_check_interrupt(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """ころがる状態: このターンの行動が失敗していた場合、強制行動を中断する。

    外れ・まもるによる無効化・状態異常等での不発など、いずれの場合も
    Pokemon.failed_or_immobile_last_turn が True になるため、これを判定基準にする。
    """
    mon = ctx.source
    if mon.failed_or_immobile_last_turn:
        return remove_volatile(battle, ctx, value, volatile="ころがる", reason="ミス・行動失敗")
    return HandlerReturn(value=value)
```

### src/jpoke/data/ability.py（おやこあい）

現状の `おやこあい_modify_hit_count` は `がむしゃら` のみを連続攻撃化の対象外としており、
`ころがる`/`アイスボール` が除外されていない（`docs/spec/abilities/おやこあい.md` に明記されている
「ころがる/アイスボールは連続攻撃にならない」という仕様と不整合）。本計画で以下のように修正する。

```python
def おやこあい_modify_hit_count(battle: Battle, ctx: AttackContext, value: int) -> HandlerReturn:
    """おやこあい特性: 単発攻撃技を2ヒット化する。

    がむしゃらはHPの変動有無に関わらずいかなる状況でも連続攻撃にならないため対象外とする。
    ころがる/アイスボールも連続攻撃にならない（数ターン継続する強制行動技のため）。
    """
    if ctx.move.name in ("がむしゃら", "ころがる", "アイスボール"):
        return HandlerReturn(value=value)
    if ctx.move.is_attack and ctx.move.max_hits == 1:
        value = 2
    return HandlerReturn(value=value)
```

この修正がないと、おやこあい所持ポケモンが `ころがる` を使った際に1ターンで2回 `ON_HIT`（＝2回の
命中カウント加算）が発生し、威力上昇のペースが本来より速くなってしまう不整合が生じるため、
本技の正しい実装に必須の修正として扱う。

## 実装しないスコープ（今回は対象外）

* `ねごと` から出した場合の単発化: `ねごと` 自体が未実装のため対象外。今後 `ねごと` を実装する際に、
  `attacker.sleep_talk_active` を `ころがる_apply` でチェックし、True の場合は揮発性状態を作らず
  単発攻撃として扱う対応が必要になる（実装メモとして残す）。
* `さきどり`（Me First）からコピーした場合の威力補正: `さきどり` 自体が未実装のため対象外。
* `フリーフォール`（そらとぶ返し的な技）に連れ去られた場合の扱い: 未実装のため対象外。
* PP消費を0にする専用ハンドラ: `Command.FORCED` 経由の継続行動では新しい `Move` インスタンスが
  生成されるため、既存の枠組みで自然にPP消費が発生しない。専用実装は不要。
