# 計画書: かなしばり

参照仕様書: `docs/spec/moves/かなしばり.md`, `docs/spec/volatiles/かなしばり.md`

## 仕様要約

- タイプ: ノーマル / カテゴリ: 変化 / PP: 20 / 命中: 100 / 対象: 相手
- 相手が「直前に使用した技」を 4 ターン使用不能にする（第五世代以降）
- まだ PP を消費した行動をしていない相手に使うと失敗する（`executed_move is None` で判定）
- 相手がすでにかなしばり状態の場合は失敗する（`volatile_manager.apply` が False を返す）
- わるあがきに対して使うと失敗する
- 特性アロマベールに守られている相手には効果がない（`アロマベール_prevent_volatile` が `ON_BEFORE_APPLY_VOLATILE` で無効化）
- みがわり状態の相手にも有効
- まもる: 防がれる / マジックコート: 跳ね返される

## ハンドラ構成

| イベント | priority | subject_spec | 役割 | 実装場所 |
|----------|----------|--------------|------|----------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self` | 失敗条件チェック（`executed_move` 未設定 / わるあがき） | `handlers/move_status.py` の `かなしばり_can_apply` |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self` | 相手へかなしばり volatile 付与（count=4、move_name=相手の直前技名） | `handlers/move_status.py` の `かなしばり_apply` |

volatile 側（付与後の行動制限・ターン消費）はすでに `data/volatile.py` + `handlers/volatile.py` に実装済み。

## Priority 根拠

`docs/spec/turn.md` の `Event.ON_APPLY_MOVE`（実装上の `Event.ON_BEFORE_APPLY_MOVE`）テーブルに
「かなしばり: 失敗条件」が priority 100 として明記されている。
既存の同種変化技（`アンコール_can_apply`、`いばる_can_apply`、`うらみ_can_apply` 等）もすべて
「失敗条件チェックを `ON_BEFORE_APPLY_MOVE`、効果適用を `ON_STATUS_HIT`」の 2 イベント構成
（いずれも `MoveHandler` デフォルト priority 100）を採用しており、かなしばりも同一パターンに揃える。

`docs/spec/turn.md` の `Event.ON_CHECK_ACTION`（実装上の `ON_TRY_ACTION`）では「100: かなしばりで技が出せない」と明記されており、volatile 側のブロックが priority 100 であることも整合している。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

```python
def かなしばり_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """かなしばりの失敗条件を判定する。

    - 相手がまだ技を使っていない（executed_move が None）場合は失敗する
    - わるあがきに対して使うと失敗する
    """
    move = ctx.defender.executed_move
    if not move or move.name == "わるあがき":
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="かなしばり")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def かなしばり_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """かなしばりの効果: 相手に「かなしばり」状態を付与する（4 ターン）。"""
    move = ctx.defender.executed_move
    return apply_volatile_to_defender(
        battle, ctx, value, volatile="かなしばり", count=4, move_name=move.name
    )
```

### `src/jpoke/data/moves/move_ka.py`

"かなしばり": MoveData(

## 注意点・エッジケース

- **`executed_move` の判定**: `executed_move` は `Pokemon.reset()` で None に戻される。交代直後に場に出たばかりのポケモン（`executed_move is None`）に使うと失敗する。アンコール（`アンコール_can_apply`）と同一パターン。
- **わるあがきへの失敗**: `executed_move.name == "わるあがき"` でガードする。第五世代以降の仕様に準拠。
- **すでにかなしばり状態の場合の失敗**: `volatile_manager.apply` が既に volatile がある場合 False を返すため、ハンドラ側での重複チェックは不要。`apply_volatile_to_defender` ヘルパーはその返り値をそのまま `HandlerReturn.value` に渡すため、自動的に失敗ログが出る。
- **アロマベール無効化**: `アロマベール_prevent_volatile`（`handlers/ability.py`）が `Event.ON_BEFORE_APPLY_VOLATILE` で「かなしばり」を含む対象リストを直接ブロックするため、かなしばりハンドラ側には記述しない。
- **のろわれボディ・メンタルハーブ**: それぞれ `handlers/ability.py` の `のろわれボディ_maybe_disable_move`、`handlers/item.py` の `メンタルハーブ_cure_mental_volatile` が対応済み。かなしばり側の実装変更は不要。
- **失敗条件チェックと効果適用のイベント分離**: `can_apply`（`ON_BEFORE_APPLY_MOVE`）と `apply`（`ON_STATUS_HIT`）を別イベント・別 `MoveHandler` に分けることで、`アンコール` / `いばる` / `うらみ` 等の同種ハンドラと構成を揃える。`can_apply` が `stop_event=True` を返すと `ON_STATUS_HIT` 自体が発火しないため、`apply` 側の失敗を待たずに早期リターンできる。
- **move_name の保存**: `apply_volatile_to_defender` の `**kwargs` として `move_name=move.name` を渡す。`Volatile.__init__` が `move_name` 引数を受け取り `self.move_name` に格納する。volatile 側のハンドラ（`かなしばり_try_action` 等）は `mon.volatiles["かなしばり"].move_name` でアクセスする。
- **PP が 0 の技に対しての挙動**: wiki によると第五世代以降は PP が 0 の技にも成功する。`executed_move` に PP=0 の技が入っていても失敗させないため、PP チェックは行わない。
