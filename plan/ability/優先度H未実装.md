## 結論

優先度Hの未実装9件は、個別実装より先に以下の3バッチで進める（かわりもの はイリュージョン同様に実装保留）。

1. 強天候と天候無効化の基盤整備
2. ダメージ適用前と追加効果処理の基盤整備
3. 専用種族のタイプ・フォルム連動整備

この順にする理由は、後続の特性が先行バッチの共通仕組みに依存しているため。特に ちからずく / てんのめぐみ は Event.ON_MOVE_SECONDARY が未実装のままでは表層実装になりやすく、マルチタイプ / ARシステム は Pokemon.types の TODO と道具保護仕様を先に整理しないと破綻しやすい。

## 共通事前整備

### 1. 天候効果の共通判定を切り出す
- 背景: 現在は [src/jpoke/handlers/ability.py](src/jpoke/handlers/ability.py) の しゅうかく 補助関数内で ノーてんき / エアロック を個別判定している。
- 対応: `battle.weather.name` を直接見る箇所から、天候名取得と「天候効果が有効か」を分離するヘルパーへ寄せる。
- 想定変更先: `src/jpoke/handlers/common.py`, `src/jpoke/handlers/ability.py`, `src/jpoke/handlers/field.py`, `src/jpoke/handlers/move.py`
- 目的: 強天候、および エアロック / ノーてんき の抑制を一箇所で扱う。

### 2. 追加効果処理フックを実装する
- 背景: [src/jpoke/enums/event.py](src/jpoke/enums/event.py) に Event.ON_MOVE_SECONDARY が予約済みだが、emit 側・handler 側が未実装。
- 対応: 追加効果の確率決定と発動抑制を、威力計算や ON_HIT 本体効果と分離する。
- 想定変更先: `src/jpoke/core/move_executor.py`, `src/jpoke/enums/event.py`, `src/jpoke/handlers/ability.py`, `src/jpoke/handlers/item.py`
- 目的: ちからずく と てんのめぐみ を相互作用込みで実装できる形にする。

### 3. 専用道具によるタイプ・フォルム反映の共通化
- 背景: [src/jpoke/model/pokemon.py](src/jpoke/model/pokemon.py) の `types` には「アルセウスのタイプ変化は未実装」の TODO が残っている。
- 対応: プレート / メモリから現在タイプとフォルムを求める専用ヘルパーを追加し、テラスタルと両立する優先順位を決める。
- 想定変更先: `src/jpoke/model/pokemon.py`, `src/jpoke/handlers/ability.py`, `src/jpoke/core/item_manager.py`
- 目的: マルチタイプ と ARシステム の重複実装を防ぐ。

## 実装バッチ

## バッチ1: 強天候と天候無効化

### 対象
- おわりのだいち
- はじまりのうみ
- エアロック
- ノーてんき

### 実装方針
- おわりのだいち / はじまりのうみ は、ひでり / あめふらし と同じ `Event.ON_SWITCH_IN` 起点で実装する。
- ただし通常天候ではなく「強天候」を Weather 側で表現し、水技失敗 / 炎技失敗を field ハンドラまたは move 実行直前で判定する。
- エアロック / ノーてんき は常在特性として「天候そのものは残すが、天候効果だけ無効化する」判定を共通ヘルパー化する。
- 既存の ようりょくそ / すいすい / しゅうかく / ぼうふう / ふぶき なども新ヘルパー経由へ寄せる。

### 変更候補
- `src/jpoke/data/ability.py`
- `src/jpoke/data/field.py`
- `src/jpoke/handlers/common.py`
- `src/jpoke/handlers/ability.py`
- `src/jpoke/handlers/field.py`
- `src/jpoke/handlers/move.py`
- `tests/test_ability.py`
- `tests/test_global_field.py`
- `tests/test_move.py`

### 個別メモ

#### おわりのだいち
- 入場時に おおひでり を起動する。
- おおひでり 中は みずタイプ攻撃技を失敗扱いにする。
- 交代・瀕死・とくせいなし化で解除されることを確認する。
- はじまりのうみ との競合は「後発が残る」を最優先で固定する。

#### はじまりのうみ
- 入場時に おおあめ を起動する。
- おおあめ 中は ほのおタイプ攻撃技を失敗扱いにする。
- おわりのだいち と同じ解除・競合ルールで対称実装にする。
- ゲンシ系の「通常天候を経由しない」は、現行実装にゲンシカイキ経路が無ければ TODO として切り分ける。

#### エアロック / ノーてんき
- 仕様は同一なので同一バッチで実装する。
- 天候ダメージ、威力補正、素早さ補正、命中補正、回復率補正を全て抑制対象に含める。
- 強天候の「技失敗」まで抑制するかは最初に明文化する。仕様書どおりなら失敗条件と無効判定の順序確認が必要。
- 既存の局所判定を残すと将来漏れやすいので、ヘルパー一本化を優先する。

### テスト優先度
1. 強天候の入場時発動
2. 強天候中の対応タイプ技失敗
3. エアロック / ノーてんき 下で既存の天候依存能力が止まる
4. 強天候と天候無効化の同居時に状態だけ残り効果が止まる

## バッチ2: ダメージ適用前と追加効果

### 対象
- がんじょう
- ちからずく
- てんのめぐみ

### 実装方針
- がんじょう は `Event.ON_BEFORE_DAMAGE_APPLY` を使い、HP満タン時の致死ダメージを 1 残しへ補正する。
- 一撃必殺技無効は `move.ohko_check_immune` 近傍の判定へ Ability 側条件を差し込む。
- ちからずく / てんのめぐみ は、追加効果を MoveData の個別 ON_HIT 実装から分離しない限り整合を保ちにくい。
- 追加効果の「確率変更」「発動無効化」「本体効果は維持」「いのちのたま反動なし」「かいがらのすず回復なし」を同じ文脈情報で扱う。

### 変更候補
- `src/jpoke/enums/event.py`
- `src/jpoke/core/move_executor.py`
- `src/jpoke/data/ability.py`
- `src/jpoke/handlers/ability.py`
- `src/jpoke/data/item.py`
- `src/jpoke/handlers/item.py`
- `src/jpoke/data/move.py`
- `tests/test_ability.py`
- `tests/test_item.py`
- `tests/test_move.py`
- `tests/test_ohko.py`

### 個別メモ

#### がんじょう
- HP満タン時の被ダメージ致死を HP1 に補正する。
- 一撃必殺技も無効化する。
- きあいのタスキ より先に発動することを確認する。
- 連続技は各ヒットごとに判定し、1発目でHPが減った後は2発目以降で発動しない。
- こんらん自傷には有効、反動・みちづれ・ほろびのうた には無効を切り分ける。

#### ちからずく
- 対象は「追加効果つき攻撃技」に限定する。
- 威力 1.3 倍は攻撃補正ではなく威力補正に寄せる。
- 追加効果だけ無効化し、吸収・反動・優先度・タイプ変化など本体効果は残す。
- ちからずく 適用技では いのちのたま 反動と かいがらのすず 回復も止めるため、ctx に「この技は Sheer Force 対象だった」印を持たせる設計が必要。

#### てんのめぐみ
- 追加効果確率を 2 倍、上限 100% とする。
- 固定発動の効果や自己強化本体効果は対象外にする。
- 多段技は各ヒットの追加効果判定に個別適用する。
- ちからずく と同じ追加効果フック上に載せて、確率変更と無効化が競合しない順序を決める。

### テスト優先度
1. がんじょう の致死回避と OHKO 無効
2. ちからずく の威力上昇と追加効果抑制
3. ちからずく + いのちのたま / かいがらのすず の相互作用
4. てんのめぐみ の確率倍化と上限 100%
5. ちからずく と てんのめぐみ の相互作用順序

## バッチ3: 専用道具連動のタイプ・フォルム

### 対象
- マルチタイプ
- ARシステム

### 実装方針
- どちらも「専用道具でタイプとフォルムが変わる」「特性を書き換えできない」「道具を奪えない」という共通構造を持つ。
- Ability の一時状態ではなく、Pokemon 側の現在タイプ解決ロジックへ組み込む。
- フォルム変更は `Pokemon.set_form()` を利用しつつ、入場時と道具変更時の再評価箇所を決める。
- 専用道具保護は `ItemManager.can_change_item()` に種族 + 特性 + 道具条件を追加するのが第一候補。

### 変更候補
- `src/jpoke/model/pokemon.py`
- `src/jpoke/data/ability.py`
- `src/jpoke/handlers/ability.py`
- `src/jpoke/core/item_manager.py`
- `src/jpoke/data/move.py`
- `tests/test_ability.py`
- `tests/test_item.py`
- `tests/test_move.py`

### 個別メモ

#### マルチタイプ
- アルセウス専用。プレートに応じてタイプとフォルムを更新する。
- かがくへんかガス や かたやぶり でも無効化されないため、`uncopyable` に加えて「undeniable 相当」の扱いを検討する。
- プレート所持中の道具交換・奪取失敗を ItemManager で止める。
- さばきのつぶて のタイプ変更まで同時に扱うか、別タスクに切るかを着手前に決める。

#### ARシステム
- シルヴァディ専用。メモリに応じてタイプとフォルムを更新する。
- マルチタイプ と同じ基盤を使い、対応表だけ差し替える。
- メモリ保護と マルチアタック のタイプ変更を同一タイミングで評価する。
- マジックルーム / さしおさえ 中でも本体タイプは維持、マルチアタック だけノーマルに戻る点を分離する。

### テスト優先度
1. タイプとフォルムの切替
2. 専用道具の奪取不可
3. マジックルーム / さしおさえ 中の分離挙動
4. 専用技のタイプ変化

## バッチ4: 変身処理と かわりもの

### 対象
- かわりもの

### 実装方針
- まず技 へんしん の共通変身処理を実装し、その上に `Event.ON_SWITCH_IN` の自動発動として かわりもの を載せる。
- コピー対象はタイプ、技、ランク補正、見た目、HP以外の実数値、特性などを段階的に同期する。
- 対象制限は最初から全部盛りにせず、シングル前提で「正面に相手がいる」「みがわりでない」「uncopyable 特性でない」から固める。
- 変身後に得た特性の入場時効果再発動が必要かを、トレースの再 emit 実装を参考に整理する。

### 変更候補
- `src/jpoke/data/move.py`
- `src/jpoke/handlers/move.py`
- `src/jpoke/data/ability.py`
- `src/jpoke/handlers/ability.py`
- `src/jpoke/model/pokemon.py`
- `tests/test_move.py`
- `tests/test_ability.py`

### 個別メモ

#### かわりもの
- 入場時自動発動で、正面の相手へ変身する。
- HPと持ち物は据え置き、技は PP 5 固定を基本にコピーする。
- この特性自体はコピー不可なので、`uncopyable` フラグ付与が前提。
- かがくへんかガス 同時入場、みがわり、相手不在などの非発動条件を最低限の回帰として先に固定する。

### テスト優先度
1. 入場時変身
2. HP・持ち物据え置き
3. 技とタイプのコピー
4. 非発動条件
5. コピー不能特性との相互作用

## 推奨実装順

1. バッチ1の共通ヘルパーだけ先行実装する
2. おわりのだいち / はじまりのうみ を対で実装する
3. エアロック / ノーてんき を対で実装し、既存の天候依存特性テストを回帰させる
4. がんじょう を単独実装してダメージ適用前フックを固める
5. Event.ON_MOVE_SECONDARY を実装して ちからずく / てんのめぐみ を対で実装する
6. マルチタイプ / ARシステム を同じ共通表現で実装する
7. へんしん 共通処理を実装してから かわりもの を載せる

## リスクと未解決論点

- 強天候を Weather 名だけで増やすと既存の通常天候前提ロジックが漏れる。抑制判定ヘルパー先行が必要。
- Event.ON_MOVE_SECONDARY は予約だけ存在するため、ここを飛ばすと ちからずく が MoveData 個別実装を壊しやすい。
- マルチタイプ / ARシステム は「特性が無効化されない」と「専用道具が守られる」が別問題なので、1関数で抱え込まない。
- かわりもの は仕様の裾野が広い。まずシングル対戦の正面変身だけを完成条件に置き、レイド・ダブル固有挙動は後続タスクに切る。
- 進捗更新時は progress/ability.md の優先度H行と本計画書の対象10件が一致していることを同時確認する。
