# バトルスイッチ 実装計画

## スコープ
- 優先度: H
- 仕様書: `spec/ability/バトルスイッチ.md`
- 関連仕様: `spec/turn.md`, `spec/volatile/キングシールド.md`
- 実装方針: ギルガルドのフォルム状態をバトル中に明示管理し、行動直前にフォルム更新を反映する。

## 実装ゴール（完了定義）
- シールドフォルム時に攻撃技を選んだ場合、技解決前にブレードフォルムへ変化する。
- ブレードフォルム時にキングシールドを選んだ場合、技解決前にシールドフォルムへ変化する。
- フォルム変更後の実数値（A/B/C/D）でダメージ計算・被ダメ計算される。
- 交代時にシールドフォルムへ戻る。
- 実装済みフラグ `unreproducible` / `protected` の意味を崩さない。

## 現状整理
- `src/jpoke/data/ability.py` の `バトルスイッチ` は `flags` のみ定義され、ハンドラ未登録。
- `src/jpoke/model/pokemon.py` は種族データ `self.data` を参照して実数値を算出するが、フォルム切替 API は未実装。
- `src/jpoke/data/pokedex.py` は `name` をキーに `pokedex` を構築しており、同名複数フォルムを直接引く導線がない。
- `spec/turn.md` ではバトルスイッチの発動を技処理前段で扱う前提になっている。

## 仕様反映ポイント
- 発動イベントは `Event.ON_CHECK_ACTION` を採用し、行動可否判定中にフォルムを更新する。
- 判定条件は以下に限定する。
  - 使用者が `バトルスイッチ` を所持し、特性が有効。
  - 使用者がギルガルド。
  - 変化先が現在フォルムと異なる。
  - シールド -> ブレード: 選択技が攻撃技。
  - ブレード -> シールド: 選択技がキングシールド。
- 非発動条件は仕様書どおりに固定する。
  - シールド時の変化技。
  - ブレード時のキングシールド以外。
- 交代時はシールドへ戻す。実装簡素化のため `ON_SWITCH_OUT` と `ON_SWITCH_IN` の両方でシールド固定を許容する。

## 変更対象（ファイル単位）
- `src/jpoke/data/pokedex.py`
  - 同名複数フォルムへアクセスできる索引を追加する（例: `(name, form_id)` で `PokemonData` を引ける辞書）。
  - 既存 `pokedex[name]` の互換は維持する。
- `src/jpoke/model/pokemon.py`
  - フォルム切替メソッド（例: `set_form(form_id: int, keep_damage: bool = True)`）を追加する。
  - フォルム切替時に `self.data` を差し替え、`update_stats(keep_damage=True)` を呼ぶ。
  - フォルム未変更時は no-op にして無駄な再計算を避ける。
- `src/jpoke/handlers/ability.py`
  - バトルスイッチ用ハンドラを追加する。
  - `ON_CHECK_ACTION` でフォルム変化を処理する本体。
  - `ON_SWITCH_OUT`（必要なら `ON_SWITCH_IN` も）でシールドへ戻す補助処理。
  - 発動ログを `LogCode.ABILITY_TRIGGERED` で記録する。
- `src/jpoke/data/ability.py`
  - `バトルスイッチ` にハンドラ登録を追加する。
  - `Event.ON_CHECK_ACTION`（必要に応じて `Event.ON_SWITCH_OUT` / `Event.ON_SWITCH_IN`）へ接続する。
- `tests/test_ability.py`
  - バトルスイッチ専用テストを追加する。

## 実装ステップ
1. 図鑑データアクセス拡張
- `pokedex` 初期化時にフォルム索引を構築し、ギルガルドの form_id 0/1 を引ける状態にする。
2. Pokemon フォルム更新 API 追加
- `Pokemon.set_form()` を実装し、`form_id` 指定で `self.data` を切替。
- 実数値再計算は `keep_damage=True` をデフォルトにする。
3. バトルスイッチハンドラ実装
- `ON_CHECK_ACTION` で技情報から遷移先を判定。
- 遷移が必要な場合のみ `set_form()` 実行と発動ログ出力。
- 交代時はシールドへ戻す処理を接続。
4. Ability データ登録
- `バトルスイッチ` にイベントハンドラを登録。
5. テスト追加と回帰確認
- フォルム遷移、非遷移、交代リセット、キングシールド連携を確認する。

## テスト計画
- 発動ケース: シールドで攻撃技を使用するとブレードへ遷移する。
- 発動ケース: ブレードでキングシールドを使用するとシールドへ遷移する。
- 非発動ケース: シールドで変化技を使っても遷移しない。
- 非発動ケース: ブレードでまもるを使っても遷移しない。
- リセットケース: 交代後にシールドへ戻る。
- 数値反映ケース: フォルム遷移後の攻撃実数値差がダメージへ反映される。
- 相互作用ケース: `protected` 特性としてガス系無効化処理で挙動が崩れない。

## リスクと対策
- リスク: 図鑑の同名キー運用により、初期フォルムが意図せずブレード固定になる。
  - 対策: form_id 指定アクセスを新設し、バトル開始時に明示的にシールドへ合わせる。
- リスク: フォルム切替時のステータス再計算でHPが不正に変化する。
  - 対策: `update_stats(keep_damage=True)` を経由して被ダメージ量を維持する。
- リスク: 発動タイミングが遅く、旧フォルム実数値でダメージ計算される。
  - 対策: `ON_CHECK_ACTION` の段階で遷移し、以後の計算が新フォルム値を参照することをテストで固定する。
