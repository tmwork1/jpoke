# ハンドラ共通化・partial 化メモ

更新日: 2026-05-04

## 目的

- src/jpoke/handlers 配下で、名前付き関数を維持したまま実装量を減らせる箇所を整理する。
- lambda 置換ではなく、共通関数化と partial 化の候補を優先度つきで残す。
- 安易な抽象化で可読性や仕様追跡性を落とさない。

## 方針

- data 側は宣言的にし、handlers 側に共通ロジックを寄せる。
- handlers 側では、匿名関数ではなく名前付き共通関数と薄いラッパーを優先する。
- 条件分岐やログ出力や副作用が微妙に違う群は、無理に 1 つへ畳み込まない。

## 優先度 High

### 1. 砂ダメージ無効群

対象:

- src/jpoke/handlers/ability.py の すなかき_ignore_sandstorm_damage
- src/jpoke/handlers/ability.py の すながくれ_ignore_sandstorm_damage
- src/jpoke/handlers/ability.py の すなのちから_ignore_sandstorm_damage
- src/jpoke/handlers/ability.py の ぼうじん_ignore_sandstorm_damage

現状:

- いずれも hp_change_reason が sandstorm_damage のときに value=0 と stop_event=True を返すだけの同型処理。

提案:

- common に damage reason ベースの無効化関数を追加する。
- 例: ignore_damage_by_reason(battle, ctx, value, *, reason)
- data 側では partial で reason を束縛して登録する。

メモ:

- user memory の通り、hp_change_reason ベースのダメージ無効化は既存実装パターンになっている。
- sandstorm_damage 専用に切るより、 burn や poison などにも転用できる API 名のほうがよい。
- stop_event=True を落とすと挙動が変わるため必須。

## 優先度 Medium

### 2. 状態異常防止群

対象:

- src/jpoke/handlers/ability.py の めんえき_prevent_poison
- src/jpoke/handlers/ability.py の ふみん_prevent_sleep
- src/jpoke/handlers/ability.py の やるき_prevent_sleep
- src/jpoke/handlers/ability.py の じゅうなん_prevent_paralysis
- src/jpoke/handlers/ability.py の みずのベール_prevent_burn
- src/jpoke/handlers/ability.py の マグマのよろい_prevent_freeze

現状:

- 特定 ailment 名を弾いて value を空文字にし stop_event=True を返す構造がほぼ共通。
- 本来は能力発動ログも含めて共通処理として扱うべき群。

提案:

- ability.py 内部ヘルパー、または common に ailment 防止関数を追加する。
- 例: prevent_ailment_by_name(battle, ctx, value, *, ailment_names, ability_name)
- 防止判定、log 出力、value の空文字化、stop_event=True を 1 つの共通関数へ寄せる。

メモ:

- 統一時は「防止成功時のログ書き込み」を API の標準挙動として持たせる。
- 防止対象が揮発状態か状態異常かでイベントが異なるため、 volatile 防止と同じ抽象にはしない。

### 3. protect 系の派生処理

対象:

- src/jpoke/handlers/volatile.py の まもる_protect
- src/jpoke/handlers/volatile.py の かえんのまもり_protect
- src/jpoke/handlers/volatile.py の キングシールド_protect
- src/jpoke/handlers/volatile.py の スレッドトラップ_protect
- src/jpoke/handlers/volatile.py の トーチカ_protect

現状:

- 保護成功判定、成功ログ、成功時に value=False と stop_event=True を返す骨格が共通。
- ただし かえんのまもり は変化技を防げず、他の protect 系と入口条件が異なる。
- それ以外の差分は接触時の追加効果が中心。

提案:

- volatile.py に protect 共通関数を置き、接触時効果だけを注入する。
- 例: protect_with_contact_effect(battle, ctx, value, *, on_contact=None, protect_non_attack=True)
- partial より、名前付き共通関数 + 小さな効果関数の組み合わせが向く。

メモ:

- かえんのまもり を同じ共通関数に載せる場合は、変化技を防がない分岐を明示的に引数化する必要がある。
- move_executor.is_contact(ctx) を呼ぶ境界は残す。

### 4. 特定タイプ・ラベル依存の火力補正群

対象例:

- src/jpoke/handlers/ability.py の いわはこび_modify_atk
- src/jpoke/handlers/ability.py の はがねつかい_modify_atk
- src/jpoke/handlers/ability.py の はがねのせいしん_modify_power
- src/jpoke/handlers/ability.py の りゅうのあぎと_modify_atk
- src/jpoke/handlers/ability.py の てつのこぶし_modify_power
- src/jpoke/handlers/ability.py の メガランチャー_modify_power

現状:

- move.type または move label を見て fixed modifier を掛ける形が複数ある。
- ただし ON_CALC_ATK_MODIFIER と ON_CALC_POWER_MODIFIER が混在し、除外条件の有無も揺れる。

提案:

- ability.py の内部共通関数として move property 条件の補正関数を用意する。
- 例: _modify_by_move_condition(battle, ctx, value, *, move_type=None, move_label=None, modifier)
- data 側で partial を使うのではなく、handlers 側で別名や partial を作る形が安全。

メモ:

- イベント種が違うため、汎用化しても data 側の記述量はそこまで減らない。
- ごりむちゅう や スロースタート のような除外条件つきは、この群へ入れない。

## data 側 partial の見直し候補

前提:

- data 側で partial を使うこと自体は問題ではない。
- 見直し対象は、「定数束縛」ではなく「小さな実装 DSL」になっている箇所。

### A. move.py の common.modify_stat / modify_stats 多引数 partial 群

対象例:

- src/jpoke/data/move.py の アームハンマー
- src/jpoke/data/move.py の アイアンテール
- src/jpoke/data/move.py の ばかぢから系の self debuff
- src/jpoke/data/move.py の 各種 追加効果付き能力変化技

現状:

- stat, v, chance, target_spec, source_spec を data 側で同時に束縛している箇所が多い。
- subject_spec と合わせて読む必要があり、1 つの handler 定義に role 解決の文脈が分散している。

なぜよくないか:

- 宣言より「引数で組んだ小さなロジック」に近くなっている。
- 特に target_spec と source_spec が両方入ると、data 側だけで意味を追うコストが高い。
- 似た partial が大量に並ぶため、技ごとの差分より引数ノイズが目立つ。

見直し方針:

- handlers/move.py 側に、self lower speed、foe lower defense、self raise stat など頻出パターンの名前付き関数または別名を置く。
- data 側は「その技が何を起こすか」が読める粒度まで戻す。

優先度:

- High

### B. move.py の common.apply_volatile / apply_ailment で count や source_spec まで束縛している群

対象例:

- src/jpoke/data/move.py の アンコール
- src/jpoke/data/move.py の いちゃもん
- src/jpoke/data/move.py の ロックオン
- src/jpoke/data/move.py の もうどく付与系

現状:

- volatile 名や ailment 名の束縛までは自然だが、count や source_spec まで重なると data 側の説明量が増える。

なぜよくないか:

- 付与対象、起点、継続ターン数が handler 呼び出し側に散って、仕様名より設定値の羅列に見えやすい。
- 同系統の技が増えるほど、partial の引数差分レビューが主作業になる。

見直し方針:

- 汎用 apply_volatile / apply_ailment は残しつつ、よく出る構成は handlers/move.py に薄い命名ラッパーを置く。
- 特に count 付き揮発状態は、付与関数を名前付きに寄せたほうが仕様追跡しやすい。

優先度:

- Medium

### C. ability.py の common.modify_stat をそのまま特性挙動に使っている箇所

対象例:

- src/jpoke/data/ability.py の いかく

現状:

- partial(common.modify_stat, stat="A", v=-1, target_spec="source:foe", source_spec="source:self") のように、特性固有の意味を汎用 API の引数列で表現している。

なぜよくないか:

- 単発では読めるが、特性名と処理意図の対応が handlers 側に残らない。
- 後からログや例外条件を足したくなったときに data 側 partial から剥がすことになる。

見直し方針:

- 特性固有の意味が強いものは handlers/ability.py に名前付き関数を置く。
- data 側 partial は、天候起動や terrain 起動のような純粋な定数束縛に留める。

優先度:

- Medium

### D. toggle や追加日数のようなフラグ引数を含む partial

対象例:

- src/jpoke/data/move.py の トリックルーム / マジックルーム / ワンダールーム
- src/jpoke/data/item.py の さらさらいわ

現状:

- activate_global_field(..., toggle=True) や resolve_field_count(..., additonal_count=3) のように、動作モードを切り替えるフラグを data 側で直接渡している。

なぜよくないか:

- 真偽値フラグは partial だけ見ても意味が伝わりにくい。
- API 側の振る舞い変更が data 定義の読みやすさへ直結しやすい。

見直し方針:

- toggle 系は専用の activate_or_toggle_global_field のような名前へ分ける余地がある。
- 延長系も resolve_field_count の汎用 API で十分かを再確認する。

優先度:

- Low

### 現状のままでよい partial

対象例:

- src/jpoke/data/ability.py の 天候起動 / terrain 起動
- src/jpoke/data/field.py の tick_global_field / tick_side_field
- src/jpoke/data/volatile.py の tick_volatile / remove_volatile
- src/jpoke/data/item.py の modify_power_by_type / modify_super_effective_damage

理由:

- 名前、対象、倍率のような定数束縛に留まっており、宣言として読みやすい。
- handlers 側の共通関数名だけで挙動がほぼ説明できる。

## まとめないほうがよい箇所

- 条件差分が大きいピンチ系補正群
- ログ出力や副作用順序が仕様そのものになっている関数
- 特性名がそのまま仕様の見出しになる短い関数
- handlers 側の名前付き関数を lambda に置き換える案全般

## 既存の良い参考例

- src/jpoke/handlers/ability.py の _handle_type_absorb 系
- src/jpoke/handlers/ability.py の _skin_modify_move_type / _skin_modify_power 系
- src/jpoke/handlers/item.py の modify_power_by_type 系

## 実施順

1. 砂ダメージ無効群を共通化する。
2. 状態異常防止群をログ込みの共通 API として整理する。
3. protect 系を共通骨格 + 差分効果に寄せる。
4. 火力補正群は削減量より読みにくさが勝たないかを見て採用判断する。
5. data 側 partial は、多引数の modify_stat 系から見直す。
