# コンテキスト伝搬導入計画

更新日: 2026-05-03

## 背景

`check_def_ability_enabled` は現在、常に `ON_CHECK_DEF_ABILITY_ENABLED` を発火するため、
技文脈のない状態異常/揮発付与イベントでも「かたやぶり系」の防御側特性無視が混入しうる。

特に以下の経路で `BattleContext(move=None)` が生成されるため、
「技による攻防瞬間のみ無視」の仕様とズレる。

- `AilmentManager.apply` の `ON_BEFORE_APPLY_AILMENT` 発火
- `VolatileManager.apply` の `ON_BEFORE_APPLY_VOLATILE` 発火

## 調査結果（変更対象の全体像）

### 1. 防御側特性有効判定の中核

- `src/jpoke/core/context.py`
  - `check_def_ability_enabled`

### 2. 状態異常付与導線

- `src/jpoke/core/pokemon_state.py`
  - `AilmentManager.apply`（`BattleContext(target=mon, source=source)` を新規生成）
- `src/jpoke/handlers/common.py`
  - `apply_ailment`（`battle.ailment_manager.apply(...)` 呼び出し）
- `src/jpoke/handlers/field.py`
  - `どくびし_poison`（直接 `battle.ailment_manager.apply`）
- `src/jpoke/handlers/volatile.py`
  - `ねむけ_tick`（直接 `battle.ailment_manager.apply`）

### 3. 揮発付与導線

- `src/jpoke/core/pokemon_state.py`
  - `VolatileManager.apply`（`BattleContext(target=mon, source=source)` を新規生成）
- `src/jpoke/handlers/common.py`
  - `apply_volatile`（`battle.volatile_manager.apply(...)` 呼び出し）
- `src/jpoke/handlers/volatile.py`
  - `volatile_manager.apply` 直接呼び出しが複数（`かくれる`, `こんらん`, `さわぐ`, `さわがしい`, `みがわり` など）

### 4. テスト影響

- `tests/test_ability.py`
  - リーフガード系テストで `battle.ailment_manager.apply(..., source=attacker)` を直接呼ぶケースあり
  - 技由来を再現したいテストは、`origin_ctx` 付き呼び出しへ変更が必要

## 必須変更

### A. BattleContext に伝搬ヘルパーを追加

対象:
- `src/jpoke/core/context.py`

内容:
1. 元 `BattleContext` から必要要素を引き継いで派生インスタンスを作るメソッドを追加する。
2. 既定で引き継ぐ候補:
   - `source`, `target`, `move`, `field`, `hit_index`, `hit_count`
3. 既定で引き継がない候補（都度明示）:
   - `hp_change`, `hp_change_reason`, `stat_change_reason`, `fainted`, `def_ability_enabled`
4. 呼び出し側で上書き可能にする（`target=...`, `source=...` 等）。

備考:
- move を明示的に `None` にしたいケースに備え、
  「未指定」と「`None` 指定」を区別できる API 設計にする。

### B. check_def_ability_enabled の move ガード

対象:
- `src/jpoke/core/context.py`

内容:
1. `self.move is None` の場合は即 `True` を返す。
2. move がある場合のみ `ON_CHECK_DEF_ABILITY_ENABLED` を発火する。

効果:
- 技文脈なしイベントでの誤無効化を即時防止。

### C. AilmentManager.apply に origin context を受け取らせる

対象:
- `src/jpoke/core/pokemon_state.py`

内容:
1. `AilmentManager.apply` に `origin_ctx: BattleContext | None = None` を追加。
2. `ON_BEFORE_APPLY_AILMENT` 発火時の context 構築を次に変更:
   - `origin_ctx` がある: `origin_ctx` 派生 + `target/source` を実適用対象に補正
   - `origin_ctx` がない: 既存どおり新規 `BattleContext(target=..., source=...)`

### D. 共通ヘルパーから origin context を伝搬

対象:
- `src/jpoke/handlers/common.py`

内容:
1. `apply_ailment` で `battle.ailment_manager.apply(..., origin_ctx=ctx)` を渡す。

効果:
- 技・接触・保護貫通などの文脈を維持したまま状態異常付与判定が可能になる。

## 推奨変更（同時導入推奨）

### E. VolatileManager.apply にも origin context を導入

対象:
- `src/jpoke/core/pokemon_state.py`
- `src/jpoke/handlers/common.py`
- `src/jpoke/handlers/volatile.py`

内容:
1. `VolatileManager.apply` に `origin_ctx: BattleContext | None = None` を追加。
2. `ON_BEFORE_APPLY_VOLATILE` 発火 context を Ailment と同様に派生可能にする。
3. `common.apply_volatile` から `origin_ctx=ctx` を渡す。
4. `handlers/volatile.py` の直接 `volatile_manager.apply` 呼び出しは以下方針で整理:
   - move 由来: `origin_ctx=ctx` を渡す
   - move 非由来: 渡さない

理由:
- 将来、揮発状態でも防御側特性無視や move 条件依存が追加された際の再発防止。

### F. 既存デバッグ出力の除去

対象:
- `src/jpoke/handlers/ability.py`

内容:
- `リーフガード_prevent_ailment` の `print(...)` を削除。

## 導線カバレッジ判定（追記）

「文脈を継ぐべき導線」を実装時に取りこぼさないため、
`BattleContext` 生成/派生に関わる主要導線を以下で固定する。

### 継承必須（今回の改修対象）

1. `AilmentManager.apply` の `ON_BEFORE_APPLY_AILMENT` 発火
  - ファイル: `src/jpoke/core/pokemon_state.py`
  - 理由: 技由来の状態異常で move 文脈を落とすと、かたやぶり系の判定が不正確になる。

2. `common.apply_ailment` -> `AilmentManager.apply` の呼び出し
  - ファイル: `src/jpoke/handlers/common.py`
  - 理由: 汎用導線であり、ここを繋がないとほぼ全技由来状態異常の文脈が失われる。

3. （推奨）`VolatileManager.apply` の `ON_BEFORE_APPLY_VOLATILE` 発火
  - ファイル: `src/jpoke/core/pokemon_state.py`
  - 理由: 現時点の必須不具合対象ではないが、同型バグの再発防止に有効。

4. （推奨）`common.apply_volatile` -> `VolatileManager.apply` の呼び出し
  - ファイル: `src/jpoke/handlers/common.py`
  - 理由: 揮発付与の標準導線。将来の move 依存判定に備えて文脈を保持する。

### 条件付き継承（個別判定）

1. `handlers/volatile.py` の `volatile_manager.apply` 直接呼び出し
  - move 由来（例: `かくれる`, `さわぐ`, `みがわり`）: `origin_ctx=ctx` を渡す
  - 非 move 由来（例: 継続効果や単純内部処理）: 渡さない

2. `handlers/field.py` の `どくびし_poison` による `ailment_manager.apply`
  - 非 move 由来のため継承不要。

3. `handlers/volatile.py` の `ねむけ_tick` による `ailment_manager.apply`
  - 非 move 由来のため継承不要。

### 継承不要（今回の対象外として固定）

1. ターン/交代/管理系の新規イベント context 生成
  - 例: `switch.py`, `command_manager.py`, `ability_manager.py`, `speed.py`, `field_manager.py`
  - 理由: そもそも独立した新規イベント文脈であり、上流文脈を持ち込むと副作用リスクが高い。

2. ダメージ計算の独立 context 生成
  - 例: `damage.py` の計算用 `BattleContext(...)`
  - 理由: 単発計算に必要な入力を明示で与えており、派生化の利益が小さい。

### 既存の擬似派生生成（要整理）

1. `ctx.__class__(...)` を使う箇所
  - ファイル: `src/jpoke/handlers/ability.py`
  - 対応方針: 直ちに必須ではないが、`BattleContext` に派生メソッド導入後は統一のため置換候補にする。
  - 理由: 仕様意図が読み取りにくく、将来の `BattleContext` 拡張時に取りこぼしやすい。

### カバレッジ判定結論

- 現計画は主要導線をカバーしている。
- ただし、上記の「条件付き継承/継承不要/擬似派生」の明文化がないと実装時に判断がぶれるため、
  この節を基準としてレビューする。

## テスト改修項目

対象:
- `tests/test_ability.py`

必須観点:
1. リーフガード:
   - はれ中、技文脈なし（直接 apply）では防げる。
   - はれ中、かたやぶり技文脈ありでは防げない。
   - はれ中、かたやぶり対面でも非技経路（どくびし踏み等）では防げる。
2. 既存の「技由来」を直接 `AilmentManager.apply` で疑似再現しているテストは、
   `origin_ctx` を与えるか、`run_move` ベースへ置換する。

推奨追加:
- `VolatileManager.apply` でも `origin_ctx` あり/なしの分岐テストを追加（将来退行防止）。

## 変更順序（実装手順）

1. `BattleContext` に派生ヘルパー追加
2. `check_def_ability_enabled` に move ガード追加
3. `AilmentManager.apply` へ `origin_ctx` 導入
4. `common.apply_ailment` から伝搬
5. （推奨）`VolatileManager.apply` と `common.apply_volatile` へ同様導入
6. 直接呼び出し箇所（`handlers/volatile.py`）の `origin_ctx` 付与有無を整理
7. テスト更新・追加
8. テスト実行（少なくとも `tests/test_ability.py`）

## 受け入れ条件

1. かたやぶり対面でも、技文脈なしの状態異常付与経路ではリーフガードが有効。
2. かたやぶり技由来の状態異常はリーフガードを貫通。
3. 既存の かたやぶり系ダメージ補正テストが退行しない。
4. `print` などデバッグ出力が残らない。
