# サンダーダイブ（Bolt Beak…ではなく Supercell Slam）仕様書

参照: .internal/wiki/moves/サンダーダイブ.html（ポケモンWiki）

## 基本情報

* タイプ: でんき
* 分類: 物理
* 威力: 100
* 命中: 95
* PP: 15（Gen9本家の値。本プロジェクトは champions 基準のため `src/jpoke/data/ps-champ-ja/moves.json` の
  値（16）を正とする。全技PPが8/12/16/20の4段階に統一されている `.internal/champions/changes_from_sv.md` に基づく）
* 対象: 相手1体
* 優先度: 0
* 接触: あり（直接攻撃）

## 効果

* 失敗すると、自分の最大HPの1/2のダメージを受ける（小数点以下切り捨て、最低1）。
  タイプとPPを除けば「とびげり」と同様の効果を持つ（wiki記載）。
  * 相性やとくせい（ちくでん・でんきエンジン・ひらいしん等ででんき技が無効化された場合）により
    技が無効化された場合も、外れたときと同量の反動を受ける（wiki記載）。
  * まもる等により防がれた場合も同様に反動を受ける（かかとおとし・とびげり・とびひざげりと同じ失敗反動仕様）。
  * 相手の場に対象がすでに全員ひんしで技自体が不発になった場合は反動を受けない。
  * 状態異常・状態変化（まひ・こおり・ねむり・ひるみ・こんらん自傷等）により技を使用できなかった場合は
    反動を受けない。
* ちいさくなる状態の相手には必ず命中し、ダメージが2倍になる。ただし、姿を隠している相手（そらをとぶ等）には
  当たらない。
* とびひざげりと異なり、じゅうりょく状態でも使用できる（`gravity_restricted` フラグの対象外）。

## 判定フラグ

* 急所: 通常判定
* 命中判定: あり（命中率95）
* 追加効果: なし
* まもる: 防がれる（防がれた場合も反動を受ける）
* おうじゃのしるし: 発動する（第五世代以降）
* マジックコート: 反射不可（変化技ではないため対象外）
* よこどり: 対象外（変化技ではないため）

## 特性との相互作用

* いしあたま: 失敗反動ダメージは防げない。本プロジェクトでは `reason="fixed_recoil"` として実装しており、
  いしあたまが判定するのは `reason == "recoil"` のみのため防げない仕様と一致する。
* マジックガード: マジックガード所持時にサンダーダイブが外れても反動ダメージを受けない（間接ダメージとして
  扱われるため）。本プロジェクトでは `reason="fixed_recoil"` とすることでマジックガードの無効化対象にしている
  （`handlers/ability.py` の `マジックガード_ignore_damage` は `self_cost` を除外リストに含めているため、
  `self_cost` のままではマジックガードで防げないバグになる。とびげり・とびひざげりのレビューで判明した
  既知パターン）。
* すてみ: 「失敗すると自分がダメージを受ける技」グループとしてすてみの威力1.2倍の対象。本プロジェクトでは
  `recoil` フラグを持たせることですてみの対象とする。
* ちからずく: 追加効果を持たないため、ちからずくの影響（威力1.3倍・追加効果無効化）は受けない。

## 他の技との相互作用

* ちいさくなる: `minimize` フラグにより、相手がちいさくなる状態のとき必中・威力2倍になる
  （`handlers/volatile.py` の `ちいさくなる_boost_power` / `ちいさくなる_guaranteed_hit`）。

## 実装上の制約（既知の限界）

* 本プロジェクトの現状実装では、失敗反動ダメージは `Event.ON_MISS`（命中判定失敗時）でのみ発火する。
  タイプ相性による無効化や「まもる」による防御成功時、でんきタイプ無効化特性（ちくでん・でんきエンジン・
  ひらいしん）による無効化時は `move_executor.py` が `ON_TRY_MOVE_1`/`ON_BEFORE_APPLY_MOVE` の時点で
  処理を打ち切るため、現状は反動ダメージが発生しない。
  これは サンダーダイブ に限らず、同種の失敗反動技グループ（かかとおとし・とびげり・とびひざげり）に
  共通する既知の実装制約であり、`.internal/plan/moves/失敗反動技.md` に記載がある。
  本仕様書は正式な仕様（まもる・タイプ無効化時も反動を受ける）を記載しているが、
  この制約を解消するには `move_executor.py` の共通フロー変更が必要であり、本レビューの対象範囲を超えるため
  現状の実装（ON_MISSのみ対応）を維持する。
