# 計画書: ほおばる

## 仕様要約（詳細は `.internal/spec/moves/ほおばる.md` 参照）

- タイプ: ノーマル / カテゴリ: 変化 / PP: 12（`src/jpoke/data/ps-champ-ja/moves.json` 準拠） /
  命中: なし（必中） / 対象: 自分
- 自分が持っているきのみを即座に強制消費して効果を発動させ、さらに自分の防御ランクを2段階上げる
- きのみを持っていない場合は技が失敗する（`ON_TRY_MOVE_1` priority=30 の失敗条件）
- 防御ランクがすでに上限の場合も技が失敗する（`ON_TRY_MOVE_2` priority=130 の失敗条件）
- きのみの効果は残りHP量に関係なく発動する（HP閾値チェックを無視）

### あまのじゃく持ちのポケモンとの相互作用（仕様書より）

- 防御上昇効果が下降に反転するため、防御ランクが+6のときも技は成功してきのみを消費する
- 防御ランクが-6のときに使うと失敗してきのみを消費しない
- → `ON_TRY_MOVE_2` の失敗チェックは `mon.ability.name == "あまのじゃく"` かどうかで判定方向を
  切り替える必要がある（`battle.modify_stats` 自体はあまのじゃくを自動的に処理するが、
  技全体の成否を決める `ON_TRY_MOVE_2` の事前チェックは別途あまのじゃくを考慮しないと
  正しい方向で判定できない）

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「ほ」行 — `ほたるび_modify_attacker_stats` の前に挿入する。

```python
def ほおばる_check_defense_max(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ほおばるの失敗条件: ぼうぎょランクがすでに上限の場合に失敗させる。

    通常はぼうぎょランク+6のときに失敗する。あまのじゃく持ちのポケモンは
    ぼうぎょの上昇効果が下降に反転するため、ぼうぎょランク-6のときに失敗する
    （.internal/spec/abilities/あまのじゃく.md）。
    このチェックは battle.modify_stats の内部（ON_TRY_MOVE_2 の後）で
    行われるわけではないため、ここで明示的にガードする。
    """
    mon = ctx.attacker
    if mon.ability.name == "あまのじゃく":
        maxed = mon.rank["def"] <= -6
    else:
        maxed = mon.rank["def"] >= 6
    if maxed:
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ほおばる_ぼうぎょ最大")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ほおばる_check_has_berry(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ほおばるの失敗条件: きのみを持っていない場合に失敗させる。"""
    mon = ctx.attacker
    if not mon.item.is_berry():
        battle.add_event_log(mon, LogCode.MOVE_FAILED,
                             payload=FailureLogPayload(move=ctx.move.name, display_reason="ほおばる_きのみなし"))
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def ほおばる_consume_berry_and_boost(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ほおばるの効果: 自分のきのみを強制消費して効果を発動し、ぼうぎょを2段階上げる。

    battle.item_manager.force_trigger_berry により HP 閾値を無視してきのみ効果を発動し消費する。
    その後ぼうぎょを 2 段階上げる。
    """
    mon = ctx.attacker
    battle.item_manager.force_trigger_berry(mon)
    # ぼうぎょを2段階上げる
    return modify_attacker_stats(battle, ctx, value, stats={"def": 2})
```

## 注意点・エッジケース

### きのみ効果の強制発動方法

きのみの効果（例: オボンのみのHP回復、アッキのみのぼうぎょ+1、ラムのみの状態異常回復）を
HP閾値・状態異常有無等の通常条件を無視して即座に発動させるため、共通ヘルパー
`battle.item_manager.force_trigger_berry(mon)` を使う（`src/jpoke/core/item_manager.py`）。

発火順:
1. `Event.ON_HP_CHANGED`（`value=mon.max_hp`）: HP閾値ベースのきのみ（オボンのみ等）を対象にする。
   `value >= mon.max_hp` を「強制発動」の合図として各きのみハンドラ側（`src/jpoke/handlers/item.py`）
   が解釈し、HP閾値チェックと `is_nervous`（きんちょうかん）チェックの両方をバイパスする。
2. `Event.ON_FORCE_BERRY_TRIGGER`: `ON_HP_CHANGED` に登録されていないきのみ（状態異常治療きのみ等）
   を発動する専用イベント。
3. いずれの発火でも消費されなければ `consume_item` で明示的に消費する。

`おちゃかい`（`.internal/plan/moves/おちゃかい.md` 相当のロジック、`おちゃかい_force_consume_berries`）
でも同じヘルパーを使っており、実装方針は統一されている。

### ゲップフラグ

`battle.item_manager.consume_item()` は消費時に無条件で `mon.ate_berry = True` を設定するため、
`force_trigger_berry` 経由の消費でも自動的にゲップの使用条件を満たす。ほおばる側での追加実装は
不要。

### ON_TRY_MOVE_2 でのぼうぎょ最大チェックの必要性

`.internal/spec/turn.md` に「ほおばる: ぼうぎょ最大」が priority 130 で `ON_TRY_MOVE_2` に明記されている。
`modify_attacker_stats` 内の `battle.modify_stats` がぼうぎょランクを上げようとする際にランク上限で
不発になった場合、それは技の失敗ではなく「ランク変化が起きないが技は成功」扱いになってしまう。
そのため、技全体を失敗させるには `ON_TRY_MOVE_2` での明示的なチェックが必要。

### あまのじゃく持ちとの相互作用

`.internal/spec/abilities/あまのじゃく.md` によれば：
- あまのじゃく持ちは「ぼうぎょ最大（+6）のときも成功してきのみを消費する」
- あまのじゃく持ちは「ぼうぎょ最低（-6）のときに失敗してきのみを消費しない」

`battle.modify_stats` にはあまのじゃく補正が `ON_BEFORE_MODIFY_STAT` を通じて内部で入るため、
実際のランク変化そのものはあまのじゃく側で自然に処理される。ただし `ON_TRY_MOVE_2` の
事前失敗チェックは `battle.modify_stats` を経由しないため、`ほおばる_check_defense_max` 内で
`mon.ability.name == "あまのじゃく"` を明示的に判定し、判定方向（+6 か -6 か）を切り替える。

### ほおぶくろ・はんすう・じゅくせい特性との相互作用

これらは各特性のハンドラが `ON_HP_CHANGED` / `ON_FORCE_BERRY_TRIGGER` 等の正規イベントを通じて
発動するため、`force_trigger_berry` が正しく発火していれば自動的に連動する。ほおぶくろ自体は
本プロジェクトでは未実装（`AbilityData()` の空データ）のため、実際の連動は確認できない。

### しゅうかく・ものひろいによるきのみ復活後のほおばる再使用

各特性ハンドラが管理するため、ほおばる側での追加実装は不要。

### ランク上昇ピンチきのみがぼうぎょ最大時に発動した場合（リュガのみ）

`battle.modify_stats` の内部でランク上限チェックが行われるため、ほおばる側での追加実装は不要。

### ぶきよう・マジックルームとの相互作用（既知の制限）

仕様書に記載の通り、本プロジェクトの `Item.enabled` ゲート機構により、ぶきよう/マジックルーム下では
`item.name` が空文字列扱いとなり `is_berry()` が False を返すため、この状況下ではきのみの消費・
発動自体がスキップされてしまう。これは `おちゃかい` と共通の既知の未対応エッジケースであり、
本計画では踏み込んだ修正を行わない。
