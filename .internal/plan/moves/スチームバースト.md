# スチームバースト 実装計画書

参照: .internal/spec/moves/スチームバースト.md

## 現状（レビュー前）

* `src/jpoke/data/moves/move_sa.py` の `"スチームバースト"` エントリは以下の通り実装済み。
  - type="みず", category="special", pp=**5**, power=110, accuracy=95
  - flags={"secondary_effect", "thaw"}
  - `Event.ON_DAMAGE_HIT`: `スチームバースト_apply_burn_to_defender`（30%でやけど付与）
* `flags` に `"thaw"` が設定されているため、`こおり_cure_by_thaw_move`
  （`src/jpoke/handlers/ailment.py`, `Event.ON_DAMAGE_HIT`, defender:self）により、
  相手がこおり状態のときにこの技のダメージを受けるとこおりが解除される効果は既に
  実装済み。このハンドラは技のタイプが『ほのお』でない場合、使用者の特性が『ちからずく』
  であれば発動しない、という分岐も既に実装済み（`.internal/plan/moves/ねっとう.md` の
  レビューで対応済み）。スチームバーストはタイプ=みずのため、この既存の共通実装が
  そのまま正しく適用される。**この部分は追加実装不要**。
* 一方、仕様書に記載した「使用者がこおり状態の場合、こおり状態を治してから攻撃する」効果は
  **未実装だった**。`"thaw"` フラグは現状 `こおり_cure_by_thaw_move`（被弾側＝defender）にのみ
  使われており、使用者自身の無条件解凍処理は個別実装が必要
  （かえんぐるま・かえんボール・せいなるほのお・もえつきる・ねっさのだいち・ねっとう・
  シャカシャカほうは実装済みだが、スチームバーストは未実装だった）。
* PP=5 は Gen9本家基準の値。`src/jpoke/data/ps-champ-ja/moves.json` にはスチームバーストの行が
  存在しない（ボルケニオンの専用技であるため、championsの技一覧データに収録漏れが
  あると考えられる）。ただし `.internal/champions/changes_from_sv.md` に明記の通り
  「すべての技のPPが8・12・16・20の4段階に統一」されているのがchampionsの仕様であり、
  例外は記載されていない。同ファイル内でGen9のPPが5だった他の特殊技を確認すると、
  だいもんじ（460行目）・ハイドロポンプ（668行目）・れんごく（960行目）・
  リーフストーム（934行目）・りゅうせいぐん（942行目）はいずれもchampions PP=8に
  統一されている。この一貫したパターンから、直接の記載はなくてもスチームバーストの
  champions PPは8であると推定するのが妥当。**PP=5は移行漏れのバグであり8に修正する**。

## 追加実装

### 1. PP修正

`src/jpoke/data/moves/move_sa.py` の `"スチームバースト"` エントリの `pp=5` を `pp=8` に修正する
（`src/jpoke/data/ps-champ-ja/moves.json` に直接の記載はないが、同じ元PP=5の特殊技が軒並みPP=8に
統一されているパターンに基づく推定値。根拠は上記「現状」節および
.internal/spec/moves/スチームバースト.md を参照）。

### 2. 使用者自身の無条件解凍ハンドラ

`src/jpoke/handlers/move_attack.py` に `スチームバースト_thaw_attacker` を追加する。
`ねっとう_thaw_attacker` / `シャカシャカほう_thaw_attacker`
（同ファイル内、Event.ON_TRY_ACTION, priority=5）と全く同じパターンで実装する。

```python
def スチームバースト_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """スチームバースト: こおり状態でも使用可能にし、こおりを解凍する。

    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

`src/jpoke/data/moves/move_sa.py` の `"スチームバースト"` エントリに以下を追加する。

```python
Event.ON_TRY_ACTION: h.MoveHandler(
    ha.スチームバースト_thaw_attacker,
    priority=5,
),
```

#### priority の根拠

* `.internal/spec/turn.md` の `Event.ON_CHECK_ACTION`（コード上は `Event.ON_TRY_ACTION`）節には
  `10 | こおりの回復判定` と `170 | こおりを回復するわざでこおり回復` の2行がある。
* しかし `170` は `10`（`こおり_action`）より後に実行される（`_sort_handlers` は priority 昇順）。
  `こおり_action` は priority=10 で通常の25%解凍抽選を行い、解凍しなければ
  `stop_event=True` で即座にイベントを打ち切るため、priority=170 のハンドラは
  「25%抽選で解凍しなかった場合」には実行されない。これでは「無条件解凍」という
  仕様（かえんぐるま・ねっとう等と同様）を満たせない。
* 既存の同種実装（かえんぐるま・かえんボール・せいなるほのお・もえつきる・ねっさのだいち・
  ねっとう・シャカシャカほう）は全て priority=5（`こおり_action` の10より前）で登録されて
  おり、`こおり_action` が発火する前に ailment を除去することで `_check_handler_validity` の
  `ailment` ソースチェック（`subject.ailment.is_active`）により `こおり_action` 自体が
  スキップされる、という設計になっている。これに倣う。

### やけど付与とこおり解除の実行順序について

`ねっとう` のレビュー（`.internal/plan/moves/ねっとう.md`）で確認済みの通り、`Event.ON_DAMAGE_HIT`
上でやけど付与ハンドラ（move側登録）がこおり解除ハンドラ（ailment側登録、
`こおり_cure_by_thaw_move`）より先に実行される登録順序になっており、
「相手がこおり状態のときにやけどを付与しようとしても、こおりが治る前なので状態異常が
埋まっており付与に失敗し、その後こおりが解除される」という仕様通りの挙動になる。
スチームバーストも同じ実装パターン（`スチームバースト_apply_burn_to_defender` が
`Event.ON_DAMAGE_HIT` に登録）のため、priority の明示的な変更は不要。
