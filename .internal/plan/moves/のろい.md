# 計画書: のろい

## 仕様要約

- 使用者がゴーストタイプかどうかで効果が二分される（判定は技使用時のタイプ）
  - **ゴーストタイプ（呪い）**: 自分のHPを最大HPの1/2（切り捨て）消費し、相手をのろい状態にする
    - のろい状態: ターン終了時に最大HPの1/4のダメージを受ける
    - まもる・ダイウォールを貫通する
    - マジックコートで跳ね返されない
    - よこどり対象外（本プロジェクト未実装のため対象外）
    - みがわりを無視してのろい状態を付与する（第五世代以降）
    - 対象がすでにのろい状態の場合は失敗（HP消費なし）
    - 技が失敗・外れた場合もHP消費なし
    - マジックガード保持者が使用してもHPは消費される（`reason="self_cost"` で防げないHP消費として扱う）
    - マジックガード保持者にのろい状態を付与すると技は成功するがダメージは発生しない
    - HPが半分以下でも発動可能（使用者がひんしになっても相手にのろいは入る）
    - テラスタル中はテラスタル後のタイプを基準に判定（ステラタイプはテラスタル前のタイプ基準）
  - **ゴーストタイプ以外（鈍い）**: 自分のこうげき・ぼうぎょを1段階上げ、すばやさを1段階下げる
    - 第三世代以降: こうげき・ぼうぎょ両方最大かつすばやさ最低の場合のみ失敗

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|---|---|---|---|
| `ON_BEFORE_APPLY_MOVE` | 100 | `attacker:self` | 「呪い」: 対象がすでにのろい状態なら失敗（ゴーストタイプ以外はガードをスキップ） |
| `ON_STATUS_HIT` | (デフォルト=100) | `attacker:self` | タイプで分岐: ゴーストなら呪い処理、それ以外なら鈍い処理 |

## Priority 根拠

- `Event.ON_BEFORE_APPLY_MOVE` priority=100: `.internal/spec/turn.md` の `## Event.ON_APPLY_MOVE` テーブル（実装上のイベント名は `ON_BEFORE_APPLY_MOVE`）に「みずびたし等: すでに該当タイプ」等と同種の「すでに状態変化」系失敗判定が priority=100 として明記されている。同じ性質の「すでにのろい状態」判定もこれに準ずる
- `Event.ON_STATUS_HIT` priority: デフォルト（未指定、`MoveHandler` のデフォルト値100）。既存の多くの変化技（あくび、アンコールなど）に合わせてデフォルト priority を使用

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順に従い「のみこむ」の次・「はいすいのじん」の前（「のろ」の位置）に実装済み。

```python
def のろい_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """のろいの効果: タイプによって呪いと鈍いに分岐する。

    ゴーストタイプ: 自分のHPを最大HPの1/2（切り捨て）消費し、相手をのろい状態にする。
    ゴーストタイプ以外: こうげき・ぼうぎょを1段階上げ、すばやさを1段階下げる。
    """
    mon = ctx.attacker
    if mon.has_type("ゴースト"):
        # 呪い: HP消費 → のろい状態付与
        # reason="self_cost": マジックガードでも防げない自己HP消費として扱う
        cost = mon.max_hp // 2
        battle.modify_hp(mon, v=-cost, reason="self_cost", source=mon)
        return apply_volatile_to_defender(battle, ctx, value, volatile="のろい")
    else:
        # 鈍い: こうげき・ぼうぎょ上昇、すばやさ低下
        return modify_attacker_stats(battle, ctx, value, stats={"atk": 1, "def": 1, "spe": -1})


def のろい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """のろいの失敗チェック（呪い版）: 対象がすでにのろい状態なら失敗する。

    ゴーストタイプ以外が使う場合（鈍い）はガードをスキップする。
    """
    if not ctx.attacker.has_type("ゴースト"):
        return HandlerReturn(value=value)
    if ctx.defender.has_volatile("のろい"):
        battle.add_event_log(
            ctx.attacker,
            LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="のろい_すでに状態変化"),
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース

### `unprotectable` / `unreflectable` / `bypass_substitute` フラグが必須

- `Move.is_blocked_by_protect` は `target == "foe" and not has_flag("unprotectable")` で判定される。
  「のろい」の `target` はデフォルトの `"foe"` のままのため、`unprotectable` フラグがないと
  まもる状態の相手に阻まれてしまう（仕様の「まもる: ×」に反する）
- `Move.is_reflectable` は `category == "status" and target in ("foe", "foe_side") and not has_flag("unreflectable")`
  で判定される。「のろい」は `category="status"` のため、`unreflectable` フラグがないとマジックコートで
  跳ね返されてしまう（仕様の「マジックコート: ×」に反する）
- `MoveExecutor.check_hit_substitute` は `target == "foe" and not has_flag("bypass_substitute")` の場合に
  みがわりへの命中判定を行う。`bypass_substitute` フラグがないと第五世代以降の仕様
  （「みがわりを無視してのろい状態にする」）に反してみがわりに防がれてしまう
- これら3つのフラグは呪い・鈍い両方の分岐に対して一律で適用されるが、仕様の「判定」節が
  ゴーストタイプ・ゴーストタイプ以外を区別せず「まもる: ×」「マジックコート状態によって跳ね返されない」と
  記載しているため、両分岐に共通するフラグとして問題ない

### `reason="self_cost"` が必須（マジックガード対応）

- `battle.modify_hp` は `v < 0` の場合、常に `Event.ON_MODIFY_NON_MOVE_DAMAGE` を発火する
  （`status_manager.modify_hp` 参照）。マジックガードの `マジックガード_ignore_damage` ハンドラは
  `hp_change_reason` が `{"move_damage", "pain_split", "self_attack", "self_cost", "perish"}` の
  いずれでもない場合にダメージを無効化する
- そのため `reason` を指定せずに `battle.modify_hp(mon, v=-cost)` を呼ぶと、マジックガード保持者の
  自己HP消費が誤って無効化されてしまう（仕様の「特性マジックガードのポケモンが使用したときもHPが減る」に反する）
- `reason="self_cost"` を指定することでマジックガードでも防げない扱いになる
  （`handlers/move_attack.py` の `かかとおとし_crash` 等、同様のパターンが多数存在する）

### PP は champions の値（12）を使用する

- `src/jpoke/data/ps-champ-ja/moves.json` の「のろい」の行は PP=12
- 本家Gen9のPP=10ではなく、Pokémon Champions（`.internal/champions/changes_from_sv.md` によるPP統一）の
  値である12を使用する

### ON_BEFORE_APPLY_MOVE ガードの対象範囲

- `のろい_can_apply` はゴーストタイプが使う呪い版専用の失敗判定
- ゴーストタイプ以外が使う鈍い版には「すでに状態変化」の失敗条件が存在しないため、同一ハンドラ内で
  タイプ判定が必要
  - しかし ON_BEFORE_APPLY_MOVE は呪い・鈍いどちらにも呼ばれてしまう
  - 解決策: `のろい_can_apply` 内でゴーストタイプでない場合はガードを素通り
    （`return HandlerReturn(value=value)`）させる

### すでにのろい状態の場合のHP消費なし

- `volatile_manager.apply` は対象がすでにのろい状態の場合に `False` を返す
- しかし `のろい_apply` では先に `battle.modify_hp(...)` を呼んでしまうと、仮に
  `volatile_manager.apply` が失敗してもHP消費が発生してしまう
- ON_BEFORE_APPLY_MOVE でのガード（`のろい_can_apply`）がその前に止めるため問題ない
- ただし ON_BEFORE_APPLY_MOVE ガードをすり抜けるケース（みがわり状態への付与など）も考慮が必要
  - 仕様上「みがわりを無視してのろい状態を付与する（第五世代以降）」なので、みがわり状態でも成功する
  - `volatile_manager.apply` はみがわりをブロックしない（みがわりは `bypass_substitute` フラグで
    ON_TRY_MOVE 系のヒット判定自体をスキップさせる）
  - HP消費は技が成功することが確定した後に呼ばれるため、ON_STATUS_HIT での順序
    （modify_hp → apply_volatile）で問題ない

### HPが半分以下でのろいを使うとひんしになる

- `battle.modify_hp(mon, v=-cost, ...)` はHP=1（最大HP=2以下等）でも呼ばれる
- `faint` 処理は `modify_hp` 内部で自動的に行われる
- のろいの付与自体は別の処理なので、使用者がひんしになってものろい付与は完了する

### ランク補正変化の失敗条件（鈍い）

- 第三世代以降: こうげき・ぼうぎょが両方+6かつすばやさが-6の場合のみ失敗
- `modify_attacker_stats` を使うと `battle.modify_stats` が内部でランク上限/下限をチェックし適切に処理する
- turn.md の `ON_TRY_MOVE_2` priority=130 に「ランク補正上げ: すでに最大」「ランク補正下げ: すでに最低」が
  掲載されているが、鈍いはこうげき・ぼうぎょ・すばやさの組み合わせ全体での判定なので、既存のランク補正
  チェックでは賄えない可能性がある
- ただしポケモンチャンピオンズのシングルバトルでの実用上の影響は軽微なため、第三世代以降の仕様
  （各ランクが上限/下限でも技自体は動作する）に従ってデフォルト動作に委ねる実装で問題ない

### テラスタル時のタイプ判定

- テラスタル中はテラスタル後のタイプを `has_type("ゴースト")` が返す（`pokemon.types` プロパティが
  `active_tera_type` を考慮するため）
- ステラタイプでテラスタルしている場合はテラスタル前のタイプを基準とする仕様だが、`pokemon.types` は
  `active_tera_type == "ステラ"` のとき `self.data.types`（テラスタル前のタイプ）を返す実装になっており、
  既存実装のままで仕様通り動作する

### マジックガード

- マジックガードの保持者にのろいを付与した場合、volatile 自体は付与されるが ON_TURN_END での
  `のろい_damage` がマジックガードによってブロックされる（既存実装で処理済み、`reason` 未指定のため
  `ON_MODIFY_NON_MOVE_DAMAGE` で通常通りブロックされる）
- マジックガード保持者が呪いを使う場合はHP消費が発生する（`reason="self_cost"` によりブロックされない）

## リーサル計算

- `.internal/progress/move.md` の「のろい」行は「リーサル実装」「リーサルテスト」ともに `n/a`
  （攻撃側の自己HP消費は攻撃側自身の生存判定に関わるため、防御側の被弾リーサル計算の対象外。
  `いたみわけ` `じばく` `だいばくはつ` `みちづれ` 等の自己HP変動系の技と同様のパターン）
- のろい状態（volatile）自体のターン終了時ダメージは `handlers/lethal.py` の `のろい_damage` として
  実装済み（`data/volatile.py` の `lethal_handlers` に登録済み）
