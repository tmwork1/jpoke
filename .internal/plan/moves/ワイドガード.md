# 計画書: ワイドガード

対象: Pokémon Champions シングルバトル
仕様書: `.internal/spec/moves/ワイドガード.md`

## 仕様要約

- タイプ: いわ / カテゴリ: 変化 / PP: 12（champions基準。src/jpoke/data/ps-champ-ja/moves.json 970行目） /
  命中: なし（必中） / 対象: 味方の場 / 優先度: +3
- 使用ターン中、自分（ダブル・トリプルでは味方全体）を「複数のポケモンが対象になる技」から守る
- 一次情報によれば、シングルバトルでも技の対象範囲区分（スプレッド技かどうか）によって
  防御対象が決まり、実際に複数体へ命中したかどうかは関係ない

## 実装方針（初回レビューでの結論・履歴）

初回実装時点では、本プロジェクトの `MoveData.target` はダブル・トリプルバトルにおける
「相手全体」「自分以外全体」等のスプレッド対象区分をモデル化していなかった
（`MoveTarget` は `"foe"` / `"foe_side"` / `"own_side"` / `"self"` / `"field"` のみ）。
この区分を再現するには技データ全体（いわなだれ・じしん・ハイパーボイス等、実機で
ダブル時に相手全体を攻撃する技すべて）にスプレッド判定用フラグを追加する必要があり、
技システム全体に影響する大規模な変更になるため、初回レビュー（ワイドガード単体）の
スコープ外とし、ワイドガードを **防御効果を持たないデータ登録のみの技**
として扱っていた（`handlers={}`）。優先度+3による先制効果のみが機能する状態
（既存の `ハッピータイム`（`target="own_side"`、`handlers={}`）と同じパターン）。

将来的にスプレッド技の対象範囲区分を実装する場合の対応として、以下4点を挙げていた
（下記「実装状況（spreadフラグ実装, 2026年7月）」で全て実施済み）:

1. `MoveFlag` に `"spread"`（あるいは`MoveTarget`に`"all_foes"`等）を追加する
2. いわなだれ・じしん・マグニチュード・ハイパーボイス・ふぶき等、実機でダブル時に
   相手全体を攻撃する技すべてにフラグを付与する（src/jpoke/data/ps-champ-ja/moves.jsonの
   「対象」列が「相手全体」「自分以外全体」になっている技が該当）
3. ワイドガードに `Event.ON_TRY_MOVE_1` ハンドラ（`defender:self`想定、`まもる`と
   同様の`_run_protect`風の骨格）を追加し、上記フラグを持つ技をブロックする
4. ドラゴンアロー（`unprotectable`相当の除外）、ワイドフォース・テラクラスター
   （範囲拡張時のみ防御対象になる条件付き）などの例外を個別対応する

## 実装状況（spreadフラグ実装, 2026年7月）

上記4点を実施し、ワイドガードは**実際に防御効果を持つ技**になった。

| 項目 | 状態 |
|------|------|
| `types/literals.py` の `MoveFlag` | `"spread"` を追加 |
| `data/moves/*.py` の対象38技 | `src/jpoke/data/ps-champ-ja/moves.json` の `target` が
  `allAdjacentFoes`（22技）/`allAdjacent`（16技）の技全てに `"spread"` フラグを機械的に付与
  （手動漏れがあった`だくりゅう`も含め最終的に38技） |

### ドラゴンアロー・ワイドフォース・テラクラスターの扱い（wiki照合結果）

`.internal/wiki/moves/ワイドガード.html`（技の仕様節）で以下を確認した。計画書の
当初想定と一致しており、個別の除外コードは不要だった:

* 「ドラゴンアローは複数のポケモンが対象になった場合でもワイドガードで防ぐことが
  できない。」→ `src/jpoke/data/ps-champ-ja/moves.json` のドラゴンアローの `target`
  は `normal`（相手1体、Showdown用語）であり、機械的な付与ルール
  （`allAdjacentFoes`/`allAdjacent`のみ付与）に従うと自然に対象外になる。
* 「ワイドフォース/テラクラスターは範囲が全体攻撃になった場合のみ防ぐことが
  できる。」→ 両技とも `target` は `normal`（テラクラスターは
  moves.json未収録のためjpoke側デフォルト）。本プロジェクトはシングルバトル専用
  のため範囲拡張が起こる状況（ダブルバトルのサイコフィールド中等）自体が
  存在せず、常に非スプレッド技として扱われる。これは実機のシングルバトルでの
  挙動とも一致する。
* 他に「複数体を攻撃できるがワイドガードで防げない/防げる」特殊技が
  `.internal/wiki/moves/` 配下に見当たらなかった（フェイント等はワイドガードを
  解除する側の技であり、ワイドガードで防がれる側ではない）。

## `target` フィールド修正の根拠（レビューで発見・修正）

修正前は `target` フィールドが未指定のため `MoveData` のデフォルト値 `"foe"`
（相手1体を対象とする技）になっていた。これは以下の点で誤り・潜在バグだった:

* `Move.is_reflectable`（`model/move.py`）は `category == "status" and target in
  ("foe", "foe_side")` を満たすとマジックコート・マジックミラーで反射可能と判定する。
  `target="foe"`のままだと、ワイドガードが相手のマジックコートで跳ね返される
  誤った挙動になる（一次情報「マジックコート: ×」と矛盾）。
* `Move.is_blocked_by_protect`（`model/move.py`）は `target == "foe"` を満たすと
  まもる等で防がれる対象と判定する。`target="foe"`のままだと、相手のまもるに
  よってワイドガードの使用自体がブロックされる誤った挙動になる
  （一次情報「まもる: ×」と矛盾。ワイドガードは相手を対象としない技のため、
  相手のまもるに関係なく成功するべき）。
* `target="own_side"` に修正することで、上記いずれの判定も `target` が
  `"foe"`/`"foe_side"` に一致しなくなり、意図通り「反射不可・妨害不可」になる。
  `own_side` を採用したのは、champions記載「対象: 味方の場」および一次情報
  「範囲: 味方の場」に一致し、かつ既存の同種技（`リフレクター`・`おいかぜ`・
  `ハッピータイム`等）が`target="own_side"`を使っているパターンに合わせるため。

## テスト方針

詳細は `.internal/spec/moves/ワイドガード.md` の「テスト方針」節を参照。要約:

- `"spread"`フラグを持たない技は防げないこと／持つ技（いわなだれ・じしん）は防げること
- 例外技（ドラゴンアロー・ワイドフォース）が防げないこと
- `"protect"`フラグによる連続使用失敗チェック、フェイントによる解除
- ふかしのこぶしの接触技（ワイドブレイカー）が貫通すること
- `target="own_side"`の担保: 相手のマジックコートで跳ね返されない・相手のまもるに
  関係なく成功する
