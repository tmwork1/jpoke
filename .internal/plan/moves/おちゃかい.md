# 計画書: おちゃかい

## 仕様要約

`.internal/spec/moves/おちゃかい.md` を参照。

- タイプ: ノーマル / 分類: 変化 / PP: 12（`src/jpoke/data/ps-champ-ja/moves.json` L152 準拠） /
  命中: なし（必中） / 対象: 自分含む場の全ポケモン（実装上は `target="self"`）
- 専用技（ポットデス）
- 場にいるポケモン全員（自分・相手）が持っているきのみを、発動条件を無視して強制消費させ、
  その効果を受けさせる
- きのみを消費できるポケモンがいない場合も技自体は成功扱い
- 消費順: 使用者 → 使用者の味方 → 相手左 → 相手右（シングルでは使用者 → 相手の1体）
- 姿を隠しているポケモン（そらをとぶ・あなをほる・ダイビング）はきのみを消費しない
- みがわり状態のポケモンはきのみを消費する
- まもる・トリックガードを無視する
- ねんちゃく・きんちょうかんの影響下でもきのみを消費する（既存実装のロジックにより自然に成立）
- ぶきよう・マジックルーム状態のポケモンも本来は消費・効果発動するはずだが、
  本プロジェクトの `Item.enabled` ゲート機構により実装上は対象外（既知の制限、詳細は仕様書参照）

## 進捗表の訂正

`.internal/progress/move.md` はレビュー前、おちゃかい行の全列を「ダブル専用」としていた。
しかし本技は「使用者＋相手」の2体構成で完結するシングルバトルでも実際に機能する
（相手のきのみを強制的に消費させる／自分のきのみも消費される、という実利がある）。
`てだすけ`・`このゆびとまれ`・`いかりのこな`・`おさきにどうぞ` 等の
「味方の存在が前提で、味方がいないシングルでは効果が成立しない技」とは性質が異なる。

さらに `src/jpoke/core/item_manager.py` の `force_trigger_berry()` および
`src/jpoke/enums/event.py` の `ON_FORCE_BERRY_TRIGGER` のコメントには、
「ほおばる・おちゃかい等できのみを強制消費するとき専用」と明記されており、
フレームワーク側は当初からおちゃかいの実装を前提に設計されている。

以上より、今回のレビューで「ダブル専用」の分類自体が誤りだったと判断し、実装対象に切り替える。

## ハンドラ構成

| イベント | priority | subject_spec | 役割 | 実装場所 |
|----------|----------|--------------|------|----------|
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self` | 場の全ポケモンのきのみを強制消費し効果を発動する | `handlers/move_status.py` |

## Priority 根拠

`.internal/spec/turn.md` には `ON_STATUS_HIT` の内部 priority 表がない。既存の変化技ハンドラ
（`おきみやげ_apply`、`おかたづけ_cleanup` など）がすべて priority 100（デフォルト）を
使用している。おちゃかいも同一パターンとして priority 100 を採用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

挿入位置: 五十音順「お」行 — `おきみやげ_apply` の後、`おたけび_modify_defender_stats` の前
（`おちゃかい` は50音順で「おきみやげ」「おさきにどうぞ」より後、「おたけび」より前）。

```python
def おちゃかい_force_consume_berries(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """おちゃかい: 場の全ポケモン（使用者→相手の順）が持っているきのみを、
    発動条件を無視して強制消費し、その効果を受けさせる。

    姿を隠しているポケモン（そらをとぶ・あなをほる・ダイビング）は消費しない。
    きのみを消費できるポケモンが1体もいなくても技自体は成功する
    （失敗判定は行わない）。
    """
    hidden_volatiles = ("そらをとぶ", "あなをほる", "ダイビング")
    attacker = ctx.attacker
    targets = [attacker] + [mon for mon in battle.actives if mon is not attacker]

    for mon in targets:
        if any(mon.has_volatile(v) for v in hidden_volatiles):
            continue
        if not mon.item.is_berry():
            continue
        battle.item_manager.force_trigger_berry(mon)

    return HandlerReturn(value=value)
```

`battle.item_manager.force_trigger_berry(mon)` は `ほおばる_consume_berry_and_boost` と
同一の既存API（HP閾値チェックを無視してきのみ効果を強制発動し消費する）を再利用する。
新規APIの追加は不要。

### `src/jpoke/data/moves/move_a.py`

既存の `"おちゃかい"` エントリを修正する（PP修正・target追加・handlers登録）。

```python
"おちゃかい": MoveData(
    type="ノーマル",
    category="status",
    pp=12,
    target="self",
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.おちゃかい_force_consume_berries,
        ),
    }
),
```

`target="self"` を指定する理由:
- `Move.is_blocked_by_protect` は `target == "foe"` の場合のみ True になるため、
  `target="self"` にすることで「まもるで防がれない」（仕様通り）を自動的に満たす。
- `Move.is_reflectable` も `target in ("foe", "foe_side")` の場合のみ True になるため、
  同様に「マジックコートで反射されない」を自動的に満たす。
- `MoveExecutor.check_hit_substitute` も `target != "foe"` の場合は False を返すため、
  みがわり関連の分岐にも影響しない（本ハンドラは `battle.actives` を直接走査するため
  そもそも `ctx.defender` 経由の判定を使わない）。
- 同様に「自分＋場全体」に効果が及ぶ `おかたづけ` も `target="self"` を採用しており、
  本プロジェクトにおける確立された実装パターンと一致する。

## 注意点・エッジケース

### 姿を隠しているポケモンのスキップ

`VolatileName` に定義されている隠れ状態は `あなをほる`・`そらをとぶ`・`ダイビング` の3種類
（`みずのなか` という名前の volatile は本プロジェクトに存在しない。旧版計画書の誤り）。
`mon.has_volatile(...)` で判定する。

### ぶきよう・マジックルーム状態のポケモン

wiki仕様では「ぶきよう/マジックルームでも消費・効果発動する」とあるが、本プロジェクトの
`GameEffect.name` プロパティは `enabled=False` のとき空文字列を返すため、
`mon.item.is_berry()`（内部で `self.name.endswith("のみ")` を参照）が False になり、
消費自体がスキップされる。この制限は `ほおばる` にも共通する既存の未対応エッジケースであり、
本レビューでは対応を見送る（`.internal/spec/moves/おちゃかい.md` に明記済み）。

### ねんちゃく・きんちょうかん

- ねんちゃく: `battle.item_manager.consume_item()` は `remove_item(mon, source=mon)` を
  呼び出す。`ねんちゃく_prevent_item_change` は `ctx.source != ctx.target` の場合のみ
  阻止するため、自分自身の消費（source=target=mon）では阻止されない。追加対応不要。
- きんちょうかん: `ON_CHECK_NERVOUS` イベントを経由しないアイテム消費処理には影響しない。
  追加対応不要。

### 対象指定（target）

`target="self"` を採用する（詳細は「実装箇所」節を参照）。`MoveTarget` 型に「自分＋相手」を
表す専用の値は存在しないが、`おかたづけ` の前例と同じ設計判断のため新規の型追加は行わない。

### 失敗条件なし

きのみを持つポケモンが0体でも技は成功扱いになるため、`ON_BEFORE_APPLY_MOVE` や
`ON_TRY_MOVE_*` での失敗判定ハンドラは不要。

### メッセージ演出

「おちゃかいをして みんなで きのみを 食べた!」「しかし 何も 起こらなかった!」の専用テキストは
本プロジェクトの実装方針（技専用の演出テキストは実装しない。`おかたづけ` の「かたづけ おわり!」も
未実装）に合わせて実装しない。

### リーサル計算

本技はダメージを与えないため `.internal/progress/move.md` の「リーサル実装」「リーサルテスト」列は
`n/a` とする（対象外）。
