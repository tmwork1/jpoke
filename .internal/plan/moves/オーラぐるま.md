# オーラぐるま 実装計画

参照: .internal/spec/moves/オーラぐるま.md

## 現状（レビュー時点）

`src/jpoke/data/moves/move_a.py` に `MoveData` は登録済みだが、以下の問題がある。

* `pp=10` になっている（champions版の正しい値は `src/jpoke/data/ps-champ-ja/moves.json` 136行目より 12）。
* タイプ変更用の `Event.ON_MODIFY_MOVE_TYPE` ハンドラ（`オーラぐるま_check_move_type`）のみが
  登録されており、追加効果（自分のすばやさ+1、100%）を処理する `Event.ON_HIT` ハンドラが
  実装されていない。
* `flags` に `"secondary_effect"` が設定されておらず、ちからずく（`ちからずく_boost` /
  `ちからずく_disable_secondary_effect`）の対象として扱われない。
* `src/jpoke/handlers/ability.py` に `はらぺこスイッチ_modify_move_type` という未使用の関数が
  存在する（`data/ability.py` の `はらぺこスイッチ` エントリに登録されておらず、呼び出し元もない
  デッドコード）。オーラぐるまのタイプ変更は既に move 側の `オーラぐるま_check_move_type` で
  完結しており、この重複関数は削除する。

## 対応内容

### 1. `data/moves/move_a.py`

* `pp` を `10` → `12` に修正する。
* `flags={"secondary_effect"}` を追加する。
* `Event.ON_HIT: h.MoveHandler(ha.オーラぐるま_boost_attacker_spe)` を追加する
  （`Event.ON_MODIFY_MOVE_TYPE` の既存ハンドラはそのまま維持する）。

### 2. `handlers/move_attack.py`

`オーラぐるま_check_move_type` の直前に、以下を追加する
（五十音順は `オーラぐるま_check_move_type` より前になるため、この位置に挿入する）。

```python
def オーラぐるま_boost_attacker_spe(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    return modify_attacker_stats(battle, ctx, value, stats={"spe": 1})
```

既存の `オーラウイング_boost_attacker_spe`（同ファイル内）と全く同じパターンを踏襲する。

### 3. `handlers/ability.py`

未使用の `はらぺこスイッチ_modify_move_type` 関数を削除する（デッドコード整理）。

## ハンドラ構成

| 技名 | イベント | priority | subject_spec | 役割 |
|---|---|---|---|---|
| オーラぐるま | ON_MODIFY_MOVE_TYPE | 100（デフォルト） | attacker:self | フォルムに応じたタイプ判定（既存実装、変更なし） |
| オーラぐるま | ON_HIT | 100（デフォルト） | attacker:self | S+1（100%、ちからずく対象） |

## Priority 根拠

* `Event.ON_MODIFY_MOVE_TYPE` は `.internal/spec/turn.md` に priority 一覧の掲載がないイベントである。
  既存実装 `そうでん_move_type`（`docs/jpoke/handlers/volatile.py`）が
  「めざめるダンス等、使用者のタイプを参照して技タイプを決定する効果よりも後に評価し、
  でんきタイプへの変換を確実に優先させる」ため priority=110 を使っている。
  オーラぐるま自身のタイプ判定ハンドラは既存実装のままデフォルト priority（未指定）を使う。
  そうでん状態時はより高い priority=110 のハンドラが後から評価され でんきタイプに上書きされるため、
  そうでんとの相互作用は現状の実装のままで正しく動作する（本レビューでの変更不要）。
* `Event.ON_HIT` は `.internal/spec/turn.md` の該当セクションに「自分のランクを上げる追加効果」が
  priority=10 と記載されている。ただし既存の同種実装（`オーラウイング_boost_attacker_spe`,
  `くさわけ_boost_attacker_spe`, `こうそくスピン_boost_attacker_spe` 等）はいずれも
  `MoveHandler` のデフォルト priority（未指定＝100）で登録されており、
  turn.md の priority=10 は ON_HIT イベント内での処理グループの優先度を示すものであるため、
  既存パターンに合わせてデフォルト priority のまま登録する。

## ちからずくとの相互作用

* `flags={"secondary_effect"}` を追加することで、既存の汎用処理
  `ちからずく_boost`（ON_CALC_POWER_MODIFIER, 威力1.3倍）と
  `ちからずく_disable_secondary_effect`（追加効果確率を0にする）が自動的に適用される。
* タイプ変更（フォルムによる でんき/あく の切り替え）は追加効果とは独立した
  `ON_MODIFY_MOVE_TYPE` ハンドラで処理されるため、ちからずくの影響を受けず、
  一次情報の「モルペコのフォルムでタイプが変わる効果は発動する」という記述と一致する。
  この部分は既存実装のままで正しく、変更不要。

## リーサル計算

`.internal/progress/move.md` の該当行「リーサル実装」列は `n/a`。
自分のすばやさランクを上げる効果であり、相手へのダメージ確定計算（HP分布）には影響しないため対象外。

## テスト

`tests/moves_attack/test_move_a.py` に以下を追加する。

* まんぷくもようのモルペコが使用した場合、でんきタイプの技として扱われること。
* はらぺこもようのモルペコが使用した場合、あくタイプの技として扱われること。
* 命中時、自分の『すばやさ』ランクが1段階上がること（100%）。
* 特性『ちからずく』を持つ場合、威力が上がり、すばやさランクは上がらないこと
  （タイプ変更は引き続き発動すること）。
