## 仕様要約

- タイプ: ノーマル / 分類: 変化 / PP: 12（`src/jpoke/data/ps-champ-ja/moves.json` 基準。
  本家SVは10だがチャンピオンズは全技PPを8/12/16/20に統一） / 命中: なし / 優先度: 0 / 対象: 自分
- 効果: 使用者の **最大HPの1/4（小数点以下切り捨て）** を消費し、消費量と同じHPを持つ
  「みがわり」状態になる。技が成功すると自分の **バインド状態を解除** する（第三世代以降）。
- 失敗条件:
  1. すでにみがわり状態
  2. 使用者の残りHPが最大HPの1/4以下（`hp <= max_hp // 4`）
- `src/jpoke/data/ps-champ-ja/moves.json` で「守×」と明記されており、まもるで無効化されない
  （自分対象の変化技のためまもる判定自体が発生しない）。

## 現状の実装状況（レビュー時点）

| 対象 | 状態 |
|------|------|
| `data/moves/move_ma.py` "みがわり" エントリ | 実装済みだが `target="self"` 欠落・`pp=10`（誤り）→ **本レビューで修正** |
| `handlers/move_status.py` みがわり_check | 実装済み（失敗条件判定は仕様どおり） |
| `handlers/move_status.py` みがわり_apply | 実装済みだが **HP消費とバインド解除が欠落** → **本レビューで修正** |
| `data/volatile.py` "みがわり" エントリ（防御・貫通判定） | 実装済み（本計画の対象外。`.internal/spec/volatiles/みがわり.md` 参照） |

### 発見した不具合と修正内容

1. **`target="self"` 欠落（重大）**
   - `MoveData.target` はデフォルト `"foe"` のため、指定漏れにより `is_reflectable`
     （`category == "status" and target in ("foe", "foe_side")`）が誤って `True` になり、
     マジックコート／マジックミラーで跳ね返される扱いになっていた。
   - 同様に `is_blocked_by_protect`（`target == "foe"`）も誤って `True` になり、
     相手がまもる状態のときにみがわりが失敗する不具合があった（本来、自分対象の技は
     相手のまもるの影響を受けない）。
   - `target="self"` を追加して修正。
2. **PP誤り**: `pp=10`（本家SV基準）になっていたが `src/jpoke/data/ps-champ-ja/moves.json` の
   `pp=12` に修正。
3. **HP消費の欠落（重大）**: `みがわり_apply` が `battle.modify_hp` を呼んでおらず、
   使用者のHPが一切減らない状態でみがわりが生成されていた。`battle.modify_hp(mon, -migawari_hp)`
   を追加。
4. **バインド解除の欠落**: 仕様上、みがわり成功時に自分のバインド状態を解除する
   （第三世代以降）が未実装だった。`battle.volatile_manager.remove(mon, "バインド")` を追加。

## ハンドラ構成

| イベント | priority | subject_spec | 登録先ハンドラ関数 | 役割 |
|---------|----------|--------------|--------------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100 | `attacker:self` | `みがわり_check` | 失敗条件チェック |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self` | `みがわり_apply` | HP消費・みがわり生成・バインド解除 |

---

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority=100 は `.internal/spec/turn.md` の `ON_APPLY_MOVE` セクションにある
  「みがわり: すでに状態変化」「みがわり: HP不足」の行に対応する（両方 priority=100）。
- `ON_STATUS_HIT` はデフォルト priority=100。変化技命中後のメイン効果は同様の既存ハンドラ
  （`しっぽきり_apply` 等）もすべて priority=100 が標準（`.internal/spec/turn.md` に本イベントは
  未掲載のため既存の同種ハンドラを根拠とする）。

## 実装箇所

### `src/jpoke/data/moves/move_ma.py`

```python
"みがわり": MoveData(
    type="ノーマル",
    category="status",
    pp=12,
    target="self",
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.みがわり_check,
            priority=100,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.みがわり_apply,
        )
    }
),
```

### `src/jpoke/handlers/move_status.py`

```python
def みがわり_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """みがわりの効果を発動する。

    最大HPの1/4（切り捨て）を消費し、消費量と同じHPのみがわりを生成する。
    技の成功に伴い、自分のバインド状態を解除する（第三世代以降の仕様）。
    """
    mon = ctx.attacker
    migawari_hp = mon.max_hp // 4
    battle.modify_hp(mon, -migawari_hp)
    battle.volatile_manager.apply(mon, "みがわり", hp=migawari_hp)
    battle.volatile_manager.remove(mon, "バインド")
    return HandlerReturn(value=value)


def みがわり_check(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """みがわりが使用可能かを判定する。"""
    mon = ctx.attacker
    if (
        mon.has_volatile("みがわり")
        or mon.hp <= mon.max_hp // 4
    ):
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="みがわり")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

（`みがわり_check` は既存実装のままで仕様どおりのため変更なし）

## 注意点・エッジケース（review-test 向け）

1. **HP消費と生成HPは常に同量**
   - 消費量・みがわりHPともに `max_hp // 4`（切り捨て）。しっぽきりとは消費量の端数処理が異なる
     （`.internal/spec/moves/しっぽきり.md` 参照）。
2. **失敗条件の閾値**
   - `hp <= max_hp // 4` で失敗（ちょうど1/4のときも失敗）。
3. **すでにみがわり状態の優先失敗**
   - `みがわり_check` は `or` 条件で両方の失敗理由をまとめて判定しているが、
     いずれの理由でも `stop_event=True` で技が失敗するだけであり、
     ゲーム内メッセージの厳密な出し分け（Wikiでは「すでに出ている」が優先表示）は
     本プロジェクトのログ実装では区別していない（`display_reason="みがわり"` 固定）。
4. **バインド解除は成功時のみ**
   - 失敗した場合はバインドを解除しない（`みがわり_check` で失敗すれば `みがわり_apply` 自体が
     呼ばれないため自然に満たされる）。
5. **まもる・マジックコートの影響を受けない**
   - `target="self"` により `Move.is_blocked_by_protect` / `Move.is_reflectable` が
     いずれも `False` になるため、ハンドラ側で追加ガードは不要。
6. **しっぽきり・バトンタッチとの関係**
   - `みがわり_apply` はみがわり状態を新規生成するのみ。既存みがわりの引き継ぎ処理は
     `しっぽきり_apply` / バトンタッチ側の責務（`.internal/spec/moves/しっぽきり.md` 参照）。
7. **PP**
   - `src/jpoke/data/ps-champ-ja/moves.json` 基準で PP=12（本家SVの10から変更されている）。
