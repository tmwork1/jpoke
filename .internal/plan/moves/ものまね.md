# 計画書: ものまね

## 仕様要約

* ノーマルタイプ / 変化 / 威力- / 命中-（必中） / PP10（Championsのmove_list.txtに記載が無いため
  Gen9本家基準を採用） / 対象: 相手1体 / 優先度0 / 接触なし
* 相手が直前に使用しPPを消費した技をコピーし、ものまね自身の代わりに一時的に覚える。PPはコピー元
  技の初期最大PPになる。交代/ひんし/バトル終了で「ものまね」に戻る。
* 詳細は `.internal/spec/moves/ものまね.md` を参照。

## 実装保留の判断

以下の理由により、現行の `Pokemon` モデル・イベント登録機構だけでは完結せず、新規のモデル変更を
要するため実装を見送る。

1. **技スロットの動的差し替え**: `Pokemon.moves` は `set_moves()` で構築後、各要素の `Move`
   オブジェクトの種類（技名）が固定される前提で扱われている。「特定インデックスの技だけを別の
   技に差し替え、対応するイベントハンドラを登録し直す」機構が存在しない。まねっこ
   （`まねっこ_execute`、`src/jpoke/handlers/move_status.py`）は `battle.run_move()` でその場限り
   の実行に留め、技スロット自体は変更しないため流用できない。
2. **復元タイミング**: `Pokemon.reset_on_switch_out()`（`src/jpoke/model/pokemon.py`）には技リスト
   の復元処理が無く、交代時にものまねへ戻す仕組みを追加する必要がある。ひんし時・バトル終了時も
   同様の考慮が要る。
3. **コピー技のPP管理**: へんしんは全コピー技のPPを一律5にする単純な規則だが、ものまねはコピーした
   1技の初期最大PPをそのまま使う。`Move` はコンストラクタで `self.pp = self.data.pp` を設定する
   仕様（`src/jpoke/model/move.py`）のため、差し替えた技用に新しい `Move` インスタンスを生成すれば
   初期PPは自然に得られるが、生成後の `modify_pp` 等の既存API・ログ・PPアップ等との整合を検証する
   必要がある。
4. **対象の「直前にPPを消費した技」の取得**: `Pokemon.executed_move` が候補だが、「PPを消費した」
   という条件（まもるで防がれた場合はPP自体は通常通り消費される点に注意）と、場に出てから一度も
   技を使っていない場合の判定を正しく区別できるか要検証。

上記はへんしん（種族・実数値ステータス・特性・能力ランクの丸ごとコピー、`.internal/plan/moves/へんしん.md`
参照）ほど大規模ではないものの、新規のモデル変更を要する点は同種であるため、前例（へんしん・
スケッチ・ゆびをふる。いずれも `.internal/progress/move.md` で実装保留として扱われている技コピー系の技）
と同様に、本技も実装保留として扱う。

## data/moves/move_ma.py の修正内容

```python
"ものまね": MoveData(
    type="ノーマル",
    category="status",
    pp=10,  # champions/move_list.txtに記載なし。Gen9本家基準の値をそのまま採用
            # （おしゃべり・テラクラスター・ダイマックスほう等、Championsに記載のない技と同様）。
    flags={"non_encore", "non_negoto", "non_copycat", "unreflectable"},
    # 実装保留: 相手が直前に使用した技を技スロットごと一時的にコピーし、交代/ひんし/
    # バトル終了で元に戻す技コピー機構が必要なため対応を見送る。
    # 詳細は .internal/plan/moves/ものまね.md 参照（前例: へんしん・スケッチ・ゆびをふる）。
    handlers={},
),
```

* `pp`: 変更なし（10）。Championsの4段階PP統一の対象外（move_list.txtに記載が無いため）。
* `flags`: `non_copycat`・`unreflectable` を追加する。従来は `non_encore`・`non_negoto` のみで、
  「まねっこでコピーできない」「マジックコートで反射されない」という仕様（
  `.internal/spec/moves/ものまね.md` の技の仕様節）が反映されていなかった。
  * `non_copycat`: まねっこ側の実装（`まねっこ_can_use`、`src/jpoke/handlers/move_status.py`）が
    `copied_move.has_flag("non_copycat")` を見て失敗判定するため、本フラグを付与することで
    まねっこがものまねをコピーできない挙動が正しく反映される。
* `handlers`: 実装保留のため `{}` のまま維持する。

## 将来実装時の設計メモ

* コピー対象の取得には `Pokemon.executed_move` を基本としつつ、「PPを消費した技」の判定条件を
  別途詰める必要がある。
* 技スロットの差し替えは、アンコール（`Volatile` に `move_name` を保持する既存パターン。
  `src/jpoke/handlers/move_status.py` の `アンコール_apply`、`src/jpoke/handlers/volatile.py` の
  `アンコール_modify_move` 等）を参考に、「ものまね」volatileへ
  `{original_index: int, copied_move_name: MoveName}` のような情報を保持し、
  `reset_on_switch_out` 相当のタイミング（および瀕死・バトル終了時）で元の技へ復元する設計が
  有力。
* コピー不可技のリストは `_INSTRUCT_BLOCKED_MOVES`（`src/jpoke/handlers/move_status.py`）と同様の
  frozenset方式で個別に管理する。`non_copycat` フラグはまねっこ用途と完全には一致しないため
  （例: まもるはものまねでコピー可能だがまねっこでは不可）、ものまね専用の
  `_MIMIC_BLOCKED_MOVES` のような別セットを用意すること。
* イベントハンドラの登録/解除は `まねっこ_execute` の `unregister_handlers`/`register_handlers`
  パターンを参考にする。ただし、まねっこは技を一時実行した直後に元へ戻すのに対し、ものまねは
  差し替えた技を複数ターンにわたって保持し続ける点が異なるため、単純な流用はできない。
