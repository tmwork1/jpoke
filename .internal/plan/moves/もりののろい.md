# 実装計画書: もりののろい

## 仕様要約

- タイプ: くさ / カテゴリ: 変化 / PP: 20 / 命中: 100 / 対象: 1 体選択
- 相手の「タイプ」に「くさ」タイプを追加する（既存のタイプを置き換えるのではなく複合タイプに追加する）
- 以下の場合は失敗する
  - 相手がすでにくさタイプを持っている（複合タイプの一部として持っている場合を含む）
  - 相手がテラスタルしている
  - （`.internal/spec/turn.md` の `ON_APPLY_MOVE` priority 100 に「ハロウィン等: すでに該当タイプ保持」として明記）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手がすでにくさタイプ／テラスタル中なら失敗 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手に「もりののろい」volatile を付与 |
| `ON_VOLATILE_START` | 100（デフォルト） | `source:self` | `added_types` にくさタイプを追加 |
| `ON_VOLATILE_END` | 100（デフォルト） | `source:self` | `added_types` からくさタイプを除去 |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `.internal/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「ハロウィン等: すでに該当タイプ保持」が priority 100 として明記されている
  （テラスタルの追加失敗条件も同一関数内で判定するため同じ priority でよい）
- `ON_STATUS_HIT` priority 100（デフォルト）: ハロウィンと同じパターン
- `ON_VOLATILE_START` / `ON_VOLATILE_END` priority 100（デフォルト）: ハロウィン volatile と同じパターン

## タイプ追加の実装方針

もりののろいは「タイプを追加」する技であり、`Pokemon.added_types` への append で実現する
（みずびたし・まほうのこなの「タイプを完全に置き換える」技とは異なり `volatile_override_type` は使わない）。

1. volatile の `ON_VOLATILE_START` で `added_types` にくさタイプを追加する（実装済み）
2. volatile の `ON_VOLATILE_END` で `added_types` からくさタイプを除去する（実装済み）
3. 失敗判定は「すでにくさタイプを持っている（`has_type("くさ")`）」に加えて「テラスタル中」を
   `ON_BEFORE_APPLY_MOVE` で明示的にチェックする（本レビューでテラスタル判定を追加）

## 実装箇所

### 1. `src/jpoke/handlers/move_status.py`

```python
def もりののろい_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """もりののろいの効果: 相手にもりののろい状態を付与してくさタイプを追加する。"""
    return apply_volatile_to_defender(battle, ctx, value, volatile="もりののろい")


def もりののろい_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """もりののろいの使用条件チェック: 相手がすでにくさタイプ、またはテラスタル中の場合は失敗する。"""
    defender = ctx.defender
    if defender.has_type("くさ") or defender.terastallized:
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="もりののろい")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### 3. `src/jpoke/handlers/volatile.py`（登録済み・変更なし）

```python
def もりののろい_add_type(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """もりののろい付与時: くさタイプを added_types に追加する。"""
    if "くさ" not in ctx.source.added_types:
        ctx.source.added_types.append("くさ")
    return HandlerReturn(value=value)


def もりののろい_remove_type(battle: Battle, ctx: EventContext, value: Any) -> HandlerReturn:
    """もりののろい解除時: added_types からくさタイプを除去する。"""
    if "くさ" in ctx.source.added_types:
        ctx.source.added_types.remove("くさ")
    return HandlerReturn(value=value)
```

## 注意点・エッジケース

- **テラスタル中の相手は `ON_BEFORE_APPLY_MOVE` の時点で明示的に失敗させる**。wiki に「テラスタルした相手に
  使用したときは失敗する」と明記されているため、volatile 自体を付与しない（みずびたし・まほうのこなと同じ扱い）。
  修正前の実装は `has_type("くさ")` のみでテラスタル判定が漏れており、テラスタル中の相手（くさタイプ以外に
  テラスタルした場合）に対して誤って volatile が付与され `added_types` にくさタイプが積まれてしまうバグが
  あったため、本レビューで `defender.terastallized` チェックを追加した
- `has_type("くさ")` は複合タイプの判定にも対応しているため、既にくさタイプを含む複合タイプの相手には
  正しく失敗する
- 交代・ひんし・戦闘終了で volatile がリセットされるため、くさタイプの追加効果も消える
  （`reset_on_switch_out` 等の既存リセット処理で対応済み）
- ハロウィン（ゴーストタイプを追加する技）とは互いに独立した volatile として実装されており、
  一方を受けたときにもう一方が自動的に解除されるという wiki の記述までは再現していない
  （ポケモンチャンピオンズのシングルバトルでは影響が乏しいため簡略化。詳細は仕様書参照）
- `もりののろい` は「は」行の `ハロウィン` と同じパターンのため、実装の整合性は `ハロウィン` 側の
  実装（`data/moves/move_ha.py`, `data/volatile.py`, `handlers/move_status.py`, `handlers/volatile.py`）と
  比較して確認すること
