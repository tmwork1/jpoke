# 実装計画書: みをけずる

## 仕様要約（.internal/spec/moves/みをけずる.md 参照）

- タイプ: ノーマル / カテゴリ: 変化 / PP: 12（champions基準） / 対象: 自分（`MoveData.target="self"` を明示）
- 自分の最大HPの半分（切り捨て）を消費して、こうげき・とくこう・すばやさランクを2段階ずつ上げる
- 失敗条件:
  - 現在HPが最大HPの半分以下（`hp <= max_hp // 2`）の場合
  - こうげき・とくこう・すばやさが全て+6まで上がっており、ランクが1つも変化しない場合
    （このときHPは消費されない）
- あまのじゃく特性の場合はランク変化の符号が反転する（`battle.modify_stats` が汎用的に処理するため
  本技側での特別対応は不要）。

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| `data/move.py`（`data/moves/move_ma.py`） エントリ | 登録済み（`pp=10` → 12へ修正が必要） |
| `handlers/move_status.py` 関数 | 実装済み（`みをけずる_can_apply` / `みをけずる_apply`）。
`みをけずる_apply` がランク変化の有無に関わらず無条件にHPを消費しているバグを修正する。 |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | HPが最大HPの半分以下ならば失敗判定 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | こうげき・とくこう・すばやさを+2し、
実際にいずれかのランクが変化した場合のみHPを最大HPの半分（切り捨て）消費 |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `.internal/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに
  「はらだいこ: 失敗条件」等、同種の自己強化技のHP判定が priority 100 として明記されている
  （実装上のイベント定数は `Event.ON_BEFORE_APPLY_MOVE`）。同じ枠組みで扱う。
- `ON_STATUS_HIT` priority 100（デフォルト）: turn.md に個別記載なし。同種の自己強化技
  （`つるぎのまい`・`はらだいこ` 等）と同じデフォルト優先度パターン。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

```python
def みをけずる_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """みをけずるの効果: こうげき・とくこう・すばやさを2段階ずつ上げる。
    いずれかのランクが実際に変化した場合のみ、最大HPの半分（切り捨て）を消費する。
    3ランクすべてが上限で変化しなかった場合はHPを消費せず、技自体も失敗扱いになる
    （modify_attacker_stats が空の変化辞書を返すため、フレームワーク側で自動的に
    MOVE_FAILEDログが出力される）。"""
    result = modify_attacker_stats(battle, ctx, value, stats={"atk": 2, "spa": 2, "spe": 2})
    if result.value:
        battle.modify_hp(ctx.attacker, r=-0.5)
    return result


def みをけずる_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """みをけずるの使用条件チェック: HPが最大HPの半分以下ならば失敗する。"""
    mon = ctx.attacker
    if mon.hp <= mon.max_hp // 2:
        battle.add_event_log(
            mon, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="みをけずる")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ma.py`

```python
"みをけずる": MoveData(
    type="ノーマル",
    category="status",
    pp=12,
    target="self",
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.みをけずる_can_apply,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.みをけずる_apply,
        ),
    }
),
```

## 注意点・エッジケース

- `modify_attacker_stats`（`handlers/move.py`）は ON_STATUS_HIT の場合、`battle.modify_stats` が返す
  「実際に変化したランクの辞書」をそのまま `HandlerReturn.value` として返す。この辞書が空
  （＝1つもランクが変化しなかった）場合、フレームワーク側（`move_executor._execute_status_hit`）が
  自動的に `MOVE_FAILED` ログを出力し `move_success = False` にする。`みをけずる_apply` は
  この `result.value` の真偽で分岐し、空でない場合のみHPを消費する。
- `battle.modify_hp(mon, r=-0.5)` は `int(max_hp * -0.5)` により端数切り捨てのマイナス方向計算となり、
  `-(max_hp // 2)` と同じ結果になる。
- `battle.modify_hp` に `reason` を渡さない（デフォルト `""`）ため、`hits_taken` は増加しない。
  これによりダメおしの威力2倍判定に影響しない（`status_manager.modify_hp` は
  `reason in ("recoil", "fixed_recoil")` のときのみ `hits_taken` を加算する）。
- PP は champions 基準で 12（`src/jpoke/data/ps-champ-ja/moves.json` には本技のエントリがないため、
  Gen9本家PP10の他の変化技が軒並み12へ変換されている規則性から類推）。
  Gen9本家基準の10ではない。**この誤りが過去の実装に存在したため、今回のレビューで12に修正した。**
- `MoveData.target` は既存実装で `"self"` が設定済み。まもる・マジックコートの誤爆を防ぐために必須。
