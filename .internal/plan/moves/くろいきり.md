# くろいきり 計画書

参照仕様書: `.internal/spec/moves/くろいきり.md`

## 仕様要約

- こおりタイプの変化技（PP: 20、命中: ―、優先度: 0）
- 対象: 全体の場（`target="field"`）
- **場にいるすべてのポケモンの能力ランクの変化を±0にリセットする**
- しろいきり状態でもランクリセットは防げない（wiki の「技の仕様」より）
- まもる・マジックコート・よこどり・オウムがえしの影響を受けない
- あまのじゃく／たんじゅんの影響を受けない。結果的なランク変化検知系の特性・持ち物
  （まけんき／かちき／びんじょう／ものまねハーブ／しっとのほのお／うっぷんばらし）も
  発動しない

## ハンドラ構成

| イベント | Priority | subject_spec | 役割 |
|---|---|---|---|
| `Event.ON_STATUS_HIT` | 100（デフォルト） | `"attacker:self"` | `battle.actives` で場の全ポケモンを列挙し、各ポケモンの `rank` を±0にリセットする |

## Priority 根拠

`.internal/spec/turn.md` に `ON_STATUS_HIT` のイベント定義は記載されていないが、`move_executor.py` の内部実装で発火されるイベントである。既存の変化技ハンドラ（例: `くろいまなざし`、`えんまく`、フィールド系）はすべて priority=100（`MoveHandler` のデフォルト）で実装されている。くろいきりも同様に priority=100 を使用する。

## 実装箇所（実装済み）

### `src/jpoke/handlers/move_status.py`

`くすぐる_modify_defender_stats` と `くろいまなざし_apply` の間に配置済み。

```python
def くろいきり_reset_all_ranks(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """くろいきりの効果: 場にいる全ポケモンの能力ランクを±0にリセットする。

    しろいきり状態でも防げない（ON_BEFORE_MODIFY_STAT を経由しない直接リセット）。
    """
    for mon in battle.actives:
        changed = {s: v for s, v in mon.rank.items() if v != 0}
        if changed:
            for s in changed:
                mon.rank[s] = 0
            battle.add_event_log(
                mon, LogCode.STAT_CHANGED,
                payload=StatChangePayload(
                    stats={s: -v for s, v in changed.items()}, display_reason="くろいきり"
                ),
            )
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ka.py`

```python
    "くろいきり": MoveData(

        category="status",
        pp=20,
        target="field",

        }

```

## 注意点・エッジケース（review-test 向け）

1. **しろいきり貫通**: `battle.modify_stats()` を使うと内部で `ON_BEFORE_MODIFY_STAT` が発火し、しろいきり状態によるランク低下防止ハンドラに引っかかる可能性がある。くろいきりはしろいきりを貫通してリセットするため、`pokemon.rank` を直接操作する。（参考: `item.py` の しろいハーブ実装でも `mon.rank[s] = 0` の直接代入が使われている）

2. **ログの `payload`**: `StatChangePayload` の `stats` は実際に変化した差分（例: atk が +2→0 なら `{"atk": -2}`）を記録するのが既存実装の慣例（`クリアスモッグ_reset_defender_rank` と同一パターン）。`display_reason="くろいきり"` で識別可能にする。ランクがすでに 0 のポケモンはログを出さない。

3. **`battle.actives` の列挙順**: シングルバトルでは攻撃側・防御側の 2 体が対象になる。両者のランクをリセットすることを確認するテストが必要。

4. **自分自身のランクもリセット**: くろいきり使用者自身のランクも対象（仕様: 「自分を含めた、場にいるポケモン全員のランク補正を±0にする」）。`battle.actives` には使用者も含まれるため、別扱い不要。

5. **ランクがすべて 0 の場合**: 変化なしなので `STAT_CHANGED` ログは出さない（技自体は「成功」扱いで `HandlerReturn(value=value)` を返す）。技が完全に失敗（False 返却）になるかどうかは仕様に記載がないため、既存の天候変更技（すでに同じ天気の場合の失敗判定）とは異なり、くろいきりはリセット対象がなくても失敗しないと判断する。

6. **クリアスモッグとの違い**: クリアスモッグは攻撃技（ダメージ+ダメージを与えた相手のみのランクリセット）で `ON_DAMAGE_HIT` 後に発動するが、くろいきりは純粋な変化技（ダメージなし・場にいる全ポケモンが対象）。実装フローが異なる。

7. **`accuracy`・`evasion` もリセット対象**: `Stat = Literal["hp", "atk", "def", "spa", "spd", "spe", "accuracy", "evasion"]` のうち `hp` はランク辞書に含まれているが、通常 0 のまま変化しない。`mon.rank.items()` で全キーを処理するため問題ない。なお、`hp` ランクを直接操作しても実害はない。

8. **あまのじゃく／たんじゅん・ランク変化検知系特性の非発動**: `battle.modify_stats()` を経由しないため、あまのじゃく／たんじゅんによる反転が発生せず、まけんき／かちき／びんじょう／ものまねハーブ／しっとのほのお／うっぷんばらしといった「ランク変化検知」系の特性・持ち物も発動しない（実装上自然に満たされる）。
