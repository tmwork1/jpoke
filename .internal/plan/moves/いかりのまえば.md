# いかりのまえば 実装計画書

## 仕様要約（.internal/spec/moves/いかりのまえば.md 参照）

* タイプ: ノーマル / 分類: 物理 / 威力: -（固定ダメージ） / 命中: 90 / PP: 10
* 対象: 相手1体 / 優先度: 0 / 接触あり
* 効果: 相手の残りHP×1/2（切り捨て）の固定ダメージ。相手の残りHPが1の場合は1ダメージ。

## 現状の実装状況

`src/jpoke/data/move.py` に既に登録済み。

```python
"いかりのまえば": MoveData(
    type="ノーマル",
    category="physical",
    pp=10,
    power=0,
    accuracy=90,
    flags={"contact", "fixed_damage"},
    handlers={
        Event.ON_MODIFY_MOVE_DAMAGE: h.MoveHandler(
            ha.half_damage,
            subject_spec="attacker:self",
            priority=15,
        )
    }
),
```

`handlers/move_attack.py` の `half_damage`（既存の共通ハンドラ。カタストロフィと共用）:

```python
def half_damage(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """対象の現在HPの半分を与える固定ダメージを計算する。"""
    return HandlerReturn(value=max(1, ctx.defender.hp // 2))
```

### 今回のレビューで修正した点

1. **`power` フィールドの明示**: 従来 `power` が未指定（`None`）だった。
   同じ `fixed_damage` 技であるカタストロフィ・ちきゅうなげ・ナイトヘッドは
   いずれも `power=0` を明示しており、表記を統一するため `power=0` を追加した。
   （`damage.py` の `calc_damages` は `if not move.power: return [0]` のため、
   `None` と `0` で挙動に差はない。純粋に一貫性のための修正。）

2. **`priority=15` の追加（重要・挙動修正）**: デフォルトの `priority=100` のままだと、
   `みがわり`（`ON_MODIFY_MOVE_DAMAGE` priority=20, `data/volatile.py`）や
   `こらえる`（`ON_MODIFY_MOVE_DAMAGE` priority=95, `data/volatile.py`）よりも
   **後に** `half_damage` が実行されてしまい、以下の実害を確認した。

   - みがわり状態の相手に使用すると、`みがわり_block_damage`（priority=20）が
     先に「入力値0」を見て「ダメージ0」と判断し、みがわりのHPを消費しない。
     その後 `half_damage`（priority=100）が本体HPの半分を再計算し、
     その値がそのまま本体の実HPに適用されてしまう
     （みがわりが存在するのに本体が直接ダメージを受ける、という誤動作）。
   - こらえる状態で残りHP1のときに使用すると、`こらえる_endure`（priority=95）が
     先に「入力値0」を見て何もせず、その後 `half_damage`（priority=100）が
     1ダメージを計算してそのまま適用し、こらえるが機能せず瀕死になる。

   これは「固定ダメージ技は通常のダメージ計算式（`calc_damages`）を経由せず、
   本来ダメージ計算前に確定しているはずの基礎ダメージ値を
   `ON_MODIFY_MOVE_DAMAGE` ハンドラ内で初めて算出する」という実装方式に起因する。
   そのため `half_damage` は「基礎ダメージの確定」という役割を持ち、
   みがわり・こらえる等の「確定済みダメージへの修正」ハンドラより
   **前に**実行される必要がある。

   `.internal/spec/turn.md` の `ON_MODIFY_DAMAGE` 節にはこのケース（固定ダメージ技の
   基礎ダメージ確定）は掲載されていないため、既存の同種ハンドラを参照して
   priority を決定した。
   - `みがわり`: priority=20（`data/volatile.py`）
   - `こらえる`: priority=95（`data/volatile.py`）
   - `ばけのかわ`: priority=10（`data/ability.py`、同じく「確定済み値を
     無視して上書きする」ハンドラ。ただしミミッキュはゴーストタイプのため
     ノーマル技のいかりのまえばとは実際には競合しない）
   - 一撃必殺技（`ohko_damage`）: priority=90（`data/move.py`）。こちらは
     みがわりとの競合が未修正のまま残っている可能性があるが、本タスクの対象外。

   以上を踏まえ、`みがわり`(20) より前・`ばけのかわ`(10) と競合しない値として
   `priority=15` を採用した。これにより、
   - みがわり状態の相手に使用時: 本体HP基準の正しい半減ダメージが
     `half_damage`(15) で確定 → `みがわり_block_damage`(20) が
     みがわりのHPを上限として肩代わりし、超過分は本体に持ち越されない
     （本体HPは変化しない）。
   - こらえる状態で使用時: 正しいダメージが先に確定 → `こらえる_endure`(95) が
     瀕死を防ぎ1HP残す。
   という、本来の挙動に修正された。
   （がんじょう・きあいのタスキは priority=100 だが、いかりのまえばの
   ダメージは常に現在HPの半分以下であり満タンHPから瀕死にはならないため、
   この2つとの実害のある競合は生じない。）

## ハンドラ構成

| Event | 対象 | subject_spec | priority | 関数 |
|---|---|---|---|---|
| ON_MODIFY_MOVE_DAMAGE | いかりのまえば | attacker:self | 15 | `ha.half_damage` |

## Priority 根拠

`.internal/spec/turn.md` に `ON_MODIFY_MOVE_DAMAGE`（固定ダメージ技のダメージ確定）は
個別掲載されていない未掲載イベントのため、既存の同種ハンドラ
（`みがわり`=20, `こらえる`=95, `ばけのかわ`=10, `ohko_damage`=90）を参照し、
`みがわり`より前に基礎ダメージを確定させる必要から `priority=15` とした。

### data/move.py

`power=0` を追加し、`ON_MODIFY_MOVE_DAMAGE` ハンドラに `priority=15` を追加。

## 注意点・エッジケース（review-test 向け）

1. **ゴーストタイプへの無効化**
   - `_check_hit_by_type`（`move_executor.py`）が `calc_def_type_modifier` を
     参照して別途無効化判定を行うため、`fixed_damage` フラグや `power=0` とは
     独立して正しく無効化される。実装済み・修正不要。

2. **みがわり・こらえるとの相互作用**
   - 上記の priority 修正により正しく動作することをテストで確認済み。

3. **がんじょう・きあいのタスキ**
   - いかりのまえばは現在HPの半分以下しかダメージを与えられないため、
     満タンHPから瀕死になることはなく、実害のある競合は発生しない。
