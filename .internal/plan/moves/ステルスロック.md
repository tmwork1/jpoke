# 実装計画書: ステルスロック

## 仕様要約

詳細は `.internal/spec/moves/ステルスロック.md`（技本体）・
`.internal/spec/fields/ステルスロック.md`（ステルスロック状態そのもの）を参照。

- タイプ: いわ / カテゴリ: 変化 / PP: 20 / 対象: 相手の場（target="foe_side"）
- 使用後、相手の場を『ステルスロック』状態にする（永続、最大1層）
- すでに設置済みの場合はこの技自体が失敗する
- 状態そのもの（ダメージ計算・免疫・解除条件）はサイドフィールド側の責務

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| サイドフィールド `ステルスロック` | 実装済み（`data/field/side_field.py` に登録済み。ダメージ処理・免疫判定込み） |
| `data/moves/move_sa.py` エントリ | 実装済み（target="foe_side"、`ステルスロック_set_field` を登録済み） |
| `handlers/move_status.py` 関数 | 実装済み（`ステルスロック_set_field`） |

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手陣営サイドフィールドに「ステルスロック」を設置 |

## Priority 根拠

`Event.ON_STATUS_HIT` は `.internal/spec/turn.md` の優先度テーブルに未掲載のイベントのため、
既存の同種ハンドラを参照して決定する。`どくびし_set_field` / `まきびし_set_field` /
`ねばねばネット_set_side_field` 等、他の設置技ハンドラもすべて priority を明示指定せずデフォルト値
（100）のままで登録されているため、ステルスロックも同様にデフォルト priority（100）・
`subject_spec` 省略（デフォルトの `attacker:self`）で登録する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「す」行 — `すてゼリフ_modify_defender_stats_and_pivot` より後、
`すなあつめ_heal_self` の前に挿入する（実装済み）。

`ON_STATUS_HIT` は攻撃フローの一部として発火するため、`ctx` は `AttackContext`
（`attacker`/`defender` ロール）を使う。`どくびし_set_field` / `まきびし_set_field`
と同じパターン。

```python
def ステルスロック_set_field(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ステルスロック: 相手側のフィールドにステルスロックを設置する。"""
    side = battle.get_side(ctx.defender)
    if not side.activate("ステルスロック", 1):
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_sa.py`

既存の `"ステルスロック"` エントリに handlers を追加する（実装済み）。

```python
"ステルスロック": MoveData(
    type="いわ",
    category="status",
    pp=20,
    target="foe_side",
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.ステルスロック_set_field,
        )
    }
),
```

### `src/jpoke/data/field/side_field.py`

サイドフィールド側のダメージ処理は既存実装済み。詳細は `.internal/plan/fields/ステルスロック.md` を参照。

## 注意点・エッジケース

- `side.activate("ステルスロック", 1)` は `max_count` 未指定（デフォルト1）のため、
  すでに設置済みの場合は `False` を返し技自体が失敗する
  （`まきびし`・`どくびし`のような多層フィールドとは異なる）
- 免疫（あつぞこブーツ）・無効化（マジックガード）・タイプ相性ダメージ計算の判定は
  サイドフィールドの `ON_SWITCH_IN` ハンドラ（`ステルスロック_damage`）側で処理されるため、
  技ハンドラ側でのガードは不要
- `Move.is_reflectable` が `target in ("foe", "foe_side")` を含むため、マジックコートで
  跳ね返された場合は使用者側の場に設置される（追加の実装は不要、既存ロジックで対応済み）
