# ふいうち 実装計画書

## 仕様要約

- タイプ: あく、分類: 物理、威力: 70、命中: 100、PP: 5、優先度: +1
- **発動条件**: 対象がそのターンに「ダメージを与える攻撃技（物理または特殊）」を選択していること
- **失敗条件（以下のいずれか）**:
  - 対象がそのターンに変化技を選択している
  - 対象が技以外の行動（交代・テラスタル・どうぐ使用など）を選択している
  - 対象がすでに行動済み（先行）でコマンドが消費されている
  - 対象が行動不能（まひ・こおり・ひるみ等）で攻撃技が実行されなかった場合（別の仕組みで処理済みのため本ハンドラでは不要）
- 通常の攻撃技と同様のダメージ計算（STAB・タイプ相性・急所・天候・テラスタル等の補正あり）

## 参照した類似実装

### はやてがえし（`handlers/move_attack.py`）

はやてがえしも「対象が特定技を選択していること」を `ON_TRY_MOVE_1` で検査する同種のハンドラである。
`_はやてがえし_can_apply` は以下のロジックを持つ：

1. `not def_state.reserved_commands` で対象が行動済みか確認（`next_command` は空リスト時に例外を投げるため、`reserved_commands` を直接確認する）
2. `defender_move.is_attack` で攻撃技かどうか確認
3. 技の優先度チェック（はやてがえし固有のロジック）

ふいうちでは 1 と 2 だけが必要で、3 は不要。

## 実装箇所

### `src/jpoke/handlers/move_attack.py` に追加する関数

五十音順（「ふ」の位置: びりびりちくちく〜フェイタルクロー の間）に配置する。

```python
def _ふいうち系_can_apply(battle: Battle, ctx: AttackContext) -> bool:
    """ふいうち・じんらいの発動条件を判定する。

    対象が未行動かつ攻撃技を選択している時のみ成功する。
    """
    def_player = battle.get_player(ctx.defender)
    def_state = battle.player_states[def_player]

    # 対象がすでに行動済み（コマンドが消費済み）なら失敗。
    if not def_state.command_reserved():
        return False

    defender_command = def_state.next_command
    defender_move = battle.command_to_move(def_player, defender_command)

    # 変化技（攻撃技でない）を選択している場合は失敗。
    if not defender_move.is_attack:
        return False

    return True


def ふいうち_try_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ふいうちの発動条件を判定する。

    対象が未行動かつ攻撃技を選択している時のみ成功する。
    交代・どうぐ使用など技以外の行動を選択している場合も失敗する。
    """
    if not _ふいうち系_can_apply(battle, ctx):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="ふいうち")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

注: `_ふいうち系_can_apply` は、同一ロジックを持つ「じんらい」（タケルライコ専用技）と共有するヘルパー関数として実装済み（詳細は `.internal/plan/moves/じんらい.md` を参照）。

## 注意点・エッジケース（review-test 向け）

### コマンド判定ロジック

- `not def_state.command_reserved()` は「対象がターン中にすでに行動し、コマンドが `pop_command()` で消費された状態」を意味する
- `next_command` プロパティは `reserved_commands` が空のとき `ValueError` を投げるため、先に `command_reserved()` での空チェックが必須
- 交代コマンド（`SWITCH_x`）は `_run_switch_phase`（技発動フェーズより前）で `pop_command()` 済みのため、対象が交代を選択した場合は本ハンドラの評価時点で `command_reserved()` が `False` になり、正しく失敗判定される
- テラスタルコマンド（`TERASTAL_x`）はテラスタルの発動と技の使用がセットになった単一コマンドであり、`_run_terastal_phase` ではテラスタル状態への変更のみ行い `pop_command()` はされない。そのため `command_to_move` は `command.index` からテラスタルとともに選択された実際の技を正しく解決できる（`resolve_move_from_command` に `is_terastal` 用の特殊分岐は無く、通常の技コマンドと同様に `attacker.moves[command.index]` を返す）
- `defender_move.is_attack` は `category in ["physical", "special"]` と等価。変化技選択時は `is_attack=False` となる。わるあがき（`Command.STRUGGLE` → 内部で `category="physical"` の専用 Move を返す）は `is_attack=True` となり成功しうる
- こんらん自傷（内部技 `"_こんらん"`）は対象が選択可能なコマンドではなく、対象の実際の行動タイミングで発生する内部的な代替効果のため、`next_command`/`command_to_move` の解決には影響しない。ふいうちは優先度+1で対象より先に評価されるため、対象がこんらん自傷を起こすかどうかが決まる前に判定が行われる

### 状態異常との相互作用

- 「対象がねむり・こおり状態で攻撃技を選択している」場合 → `reserved_commands` にコマンドが残っており `is_attack=True` → ふいうち**成功**（仕様通り）
- 「対象がまひ・ひるみで行動不能になった場合」 → これらは `ON_CHECK_ACTION` で処理され、行動がスキップされるが `reserved_commands` のコマンドはその時点で消費済み（`pop_command()` はターン処理内で実行） → ふいうち発動タイミングが先かどうかで結果が変わる。ふいうちは優先度+1のため、同優先度の通常技より先に行動する。行動不能で相手の技が実行されなかったかどうかは本ハンドラでは判定しない（仕様書2.2節「行動不能状態で攻撃技が実行されない場合は失敗」）。

  → この仕様を厳密に実装するには追加の状態フラグが必要になる可能性があるが、まずは「コマンドが残っている & 攻撃技を選択している」のみで実装し、行動不能ケースはテストで確認する

### 優先度処理

- ふいうちは優先度+1のため、通常技（優先度0）使用者よりも先に行動する
- 同じ優先度の技を相手が使用した場合、素早さ判定で先行すれば相手のコマンドは `reserved_commands` に残るためふいうちは成功する。後攻の場合は相手のコマンドが消費済みとなり失敗する

### まもる・みがわりとの関係

- まもる系はふいうちを防ぐことができる（`ON_TRY_MOVE_1` priority=100 で処理）
- みがわりはダメージを吸収する（`ON_APPLY_MOVE` で処理）
- これらはハンドラ側でガードする必要はなく、既存のフレームワークで処理される

### 五十音順配置

`move_attack.py` での関数配置位置は「ピヨピヨパンチ」〜「フェイタルクロー」の間（実装済み: `_ふいうち系_can_apply` / `ふいうち_try_move`）。共有ヘルパー `_ふいうち系_can_apply` は「じんらい」からも呼び出される（`.internal/plan/moves/じんらい.md` 参照）。
