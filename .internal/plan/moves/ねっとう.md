# ねっとう 実装計画書

参照: .internal/spec/moves/ねっとう.md

## 現状（レビュー前）

* `src/jpoke/data/move.py` の `"ねっとう"` エントリは以下の通り実装済み。
  - type="みず", category="special", pp=15, power=80, accuracy=100
  - flags={"secondary_effect", "thaw"}
  - `Event.ON_DAMAGE_HIT`: `ねっとう_apply_burn_to_defender`（30%でやけど付与）
* 上記は仕様と一致しており修正不要。
* また `flags` に `"thaw"` が設定されているため、`こおり_cure_by_thaw_move`
  （`Event.ON_DAMAGE_HIT`, defender:self）により、相手がこおり状態のときに
  この技のダメージを受けるとこおりが解除される効果は既に実装済み。
* 一方、仕様書に記載した「使用者がこおり状態の場合、こおり状態を治してから攻撃する」効果は
  **未実装だった**。`"thaw"` フラグは現状 `こおり_cure_by_thaw_move`（被弾側＝defender）にのみ
  使われており、使用者自身の無条件解凍処理は個別実装が必要
  （かえんぐるま・かえんボール・せいなるほのお・もえつきる・ねっさのだいちは実装済みだが、
  ねっとうは未実装だった）。
* さらに、`こおり_cure_by_thaw_move` は技のタイプに関わらず無条件でこおりを解除しており、
  `.internal/spec/abilities/ちからずく.md`（13行目・39行目）が明記する「ねっとう（第六世代以降）
  で相手のこおりを治す効果はちからずくで無効化される」という仕様が**未実装だった**。

## 追加実装

### 1. 使用者自身の無条件解凍ハンドラ

`src/jpoke/handlers/move_attack.py` に `ねっとう_thaw_attacker` を追加する。
`かえんぐるま_thaw_attacker` / `ねっさのだいち_thaw_attacker`
（同ファイル内、Event.ON_TRY_ACTION, priority=5）と全く同じパターンで実装する。

```python
def ねっとう_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ねっとう: こおり状態でも使用可能にし、こおりを解凍する。

    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

`src/jpoke/data/move.py` の `"ねっとう"` エントリに以下を追加する。

```python
Event.ON_TRY_ACTION: h.MoveHandler(
    ha.ねっとう_thaw_attacker,
    priority=5,
),
```

#### priority の根拠

* `.internal/spec/turn.md` の `Event.ON_CHECK_ACTION`（コード上は `Event.ON_TRY_ACTION`）節には
  `10 | こおりの回復判定` と `170 | こおりを回復するわざでこおり回復` の2行がある。
* しかし `170` は `10`（`こおり_action`）より後に実行される（`_sort_handlers` は priority 昇順）。
  `こおり_action` は priority=10 で通常の25%解凍抽選を行い、解凍しなければ
  `stop_event=True` で即座にイベントを打ち切るため、priority=170 のハンドラは
  「25%抽選で解凍しなかった場合」には実行されない。これでは「無条件解凍」という
  仕様（かえんぐるま・ねっさのだいち等と同様）を満たせない。
* 既存の同種実装 `かえんぐるま_thaw_attacker` / `ねっさのだいち_thaw_attacker` は
  priority=5（`こおり_action` の10より前）で登録されており、`こおり_action` が発火する前に
  ailment を除去することで `_check_handler_validity` の `ailment` ソースチェック
  （`subject.ailment.is_active`）により `こおり_action` 自体がスキップされる、という設計に
  なっている。実際に
  `tests/moves_attack/test_move_ka.py::test_かえんぐるま_こおり状態で使うと解凍されて攻撃できる`
  でこの動作が検証済みであり、正しく機能する実装パターンであることを確認した。
* 以上より、`.internal/spec/turn.md` に記載の priority=170 は本ケースでは機能しないため、
  既存の同種ハンドラ（かえんぐるま・ねっさのだいち）に倣い priority=5 を採用する。

### 2. ちからずくによる被弾側解凍効果の無効化

`src/jpoke/handlers/ailment.py` の `こおり_cure_by_thaw_move` を修正し、以下の条件を追加する。

* 技のタイプが『ほのお』でない `"thaw"` フラグ技（シャカシャカほう・スチームバースト・
  ねっさのだいち・ねっとう・ハイドロスチーム等）を、使用者がちからずくで使用した場合、
  被弾側のこおりを解除しない。
* 技のタイプが『ほのお』の場合（かえんぐるま・かえんほうしゃ・だいもんじ等）は、タイプ由来の
  効果であり追加効果に該当しないため、ちからずくの影響を受けず従来通り無条件で解除する。
* りんぷんはこの効果を防がない（`.internal/spec/abilities/りんぷん.md` 33行目）ため、りんぷんに
  関する分岐は追加しない。

```python
def こおり_cure_by_thaw_move(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """thawラベルを持つ技でダメージを受けたら解凍する。

    ほのおタイプ技による解凍はタイプ由来の効果であり追加効果に該当しないため、
    ちからずくの影響を受けない。一方、シャカシャカほう/スチームバースト/
    ねっさのだいち/ねっとう（第六世代以降）/ハイドロスチーム等、ほのお以外の
    タイプでこの効果を持つ技は追加効果として扱われるため、使用者がちからずくの
    場合はこの効果が発動しない（りんぷんの影響は受けない）。
    """
    if not ctx.move.has_flag("thaw"):
        return HandlerReturn(value=value)
    if (
        ctx.move.type != "ほのお"
        and ctx.attacker.ability.name == "ちからずく"
    ):
        return HandlerReturn(value=value)
    battle.ailment_manager.remove(ctx.defender)
    return HandlerReturn(value=value)
```

この修正は `"thaw"` フラグを持つ全ての技（かえんぐるま・かえんほうしゃ・かえんボール・
クロスフレイム・シャカシャカほう・スチームバースト・せいなるほのお・だいもんじ・
ねっさのだいち・ねっとう・ハイドロスチーム・フレアドライブ・めらめらバーン）に影響するが、
ほのおタイプの技は分岐条件から除外されるため挙動は変わらない。ほのお以外の技のうち
ねっとう以外（シャカシャカほう・スチームバースト・ねっさのだいち・ハイドロスチーム）も
本来同じ仕様のはずだが、今回のレビュー対象はねっとうのみのため、これらの技の仕様書・
計画書・テストの追補は別途のレビュー対象とする。

### やけど付与とこおり解除の実行順序について

`.internal/spec/turn.md` の `Event.ON_DAMAGE`（コード上は `Event.ON_DAMAGE_HIT`）節では
「追加効果（特殊なもの除く）」が priority=10、「ねっとう等受けてこおり回復」が priority=100 と
記載されている。実装では両ハンドラとも明示的な priority を設定しておらず、デフォルトの
priority=100 で登録されているが、実際に動作を検証したところ、やけど付与ハンドラ
（`ねっとう_apply_burn_to_defender`、move 側で登録）が、こおり解除ハンドラ
（`こおり_cure_by_thaw_move`、ailment 側で登録）より先に実行されており、
「相手がこおり状態のときにやけどを付与しようとしても、こおりが治る前なので状態異常が
埋まっており付与に失敗し、その後こおりが解除される」という仕様通りの挙動になっている
ことを確認した（`tests/moves_attack/test_move_na.py::test_ねっとう_こおり状態の相手に命中しても同一ヒットではやけどにならない`
で検証）。優先度を明示的に設定しなくても現状の登録順序で正しい挙動が得られているため、
本レビューでは priority の明示的な変更は行わない。
