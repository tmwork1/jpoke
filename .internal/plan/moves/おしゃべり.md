# おしゃべり 実装計画書

## 仕様要約（.internal/spec/moves/おしゃべり.md 参照）

* タイプ: ひこう / 分類: 特殊 / 威力: 65 / 命中: 100 / PP: 20
* 対象: 相手1体 / 優先度: 0 / 接触なし / 音のわざ
* 効果: ダメージを与えた後、100%の確率で相手をこんらん状態にする。

## 現状の実装状況

`src/jpoke/data/move.py` に登録済み。

```python
"おしゃべり": MoveData(
    type="ひこう",
    category="special",
    pp=20,
    power=65,
    accuracy=100,
    flags={"sound", "secondary_effect"},
    handlers={
        Event.ON_DAMAGE_HIT: h.MoveHandler(
            ha.おしゃべり_apply_confusion,
        )
    }
),
```

`src/jpoke/handlers/move_attack.py` に登録済み。

```python
def おしゃべり_apply_confusion(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """おしゃべりの追加効果: 相手をこんらん状態にする（確率100%）。"""
    return apply_confusion_to_defender(battle, ctx, value)
```

`apply_confusion_to_defender`（`handlers/move.py`）は `chance` 省略時デフォルト1.0で、
内部で `battle.resolve_secondary_chance(ctx, chance)` を通すためちからずく（`secondary_effect` フラグ）
の影響も汎用的に処理される。100%技であるため専用の確率チェックは不要。

## ハンドラ構成

| 技名 | event | priority | subject_spec | 役割 |
|---|---|---|---|---|
| おしゃべり | ON_DAMAGE_HIT | 100（デフォルト） | attacker:self（デフォルト） | 確率1.0でこんらん付与 |

## Priority 根拠

`Event.ON_DAMAGE_HIT` は `.internal/spec/turn.md` に独立したセクションが存在しない
（ダメージ計算内部の未掲載イベント）。
既存の同種ハンドラ（`かかとおとし_apply_confusion` / `サイケこうせん_apply_confusion_to_defender` /
`ばくれつパンチ_apply_confusion_to_defender` など、こんらん付与を行う攻撃技全て）が
`Event.ON_DAMAGE_HIT` を priority のデフォルト値（100）のまま使用しているため、
本技も同一パターンを採用する。

`ON_HIT`（みがわりに被弾しても発動）ではなく `ON_DAMAGE_HIT`（ダメージ適用後）を使うのは、
みがわりに当たった場合はこんらんを防御側本体に付与すべきでないため
（既存の全こんらん付与技が `ON_DAMAGE_HIT` に統一されている）。

## 注意点・エッジケース（review-test 向け）

1. **PPの誤り**
   - 従来 `pp=15` と誤って登録されていた。一次情報では PP20 のため修正済み。

2. **ちからずく（Sheer Force）との相互作用**
   - `flags={"secondary_effect"}` により威力1.3倍・こんらん無効化が汎用ハンドラで処理される。
     本技固有の対応は不要。

3. **オウムがえし・さきどり・まねっこ・ねこのて・ゆびをふるとの相互作用**
   - いずれも未実装のため、本技側での対応は不要。将来実装時は仕様書の記載を参照すること
     （特にオウムがえしは第六世代以降でコピー可能になる点に注意）。

4. **ねごととの相互作用**
   - おしゃべりは `ねごと` の「選ばれない技」リストに含まれない
     （`ねごと.html` 一次情報で確認済み）。`non_negoto` フラグは付与しない。

5. **テスト方針**
   - 確率100%のため `fix_random` 等の乱数固定は不要。`t.run_move` 実行後に
     `battle.actives[1].has_volatile("こんらん")` を確認するだけで十分
     （既存テスト `test_おしゃべり_こんらんが発動する` のパターンを踏襲）。
