# クロスフレイム 実装計画書

参照: .internal/spec/moves/クロスフレイム.md

## 現状

* `src/jpoke/data/moves/move_ka.py` の `"クロスフレイム"` エントリは以下の通り実装済みだった。
  - type="ほのお", category="special", pp=5, power=100, accuracy=100
  - flags={"thaw"}
  - handlers={}（追加効果なし、と記載）
* 上記のうち type・category・power・accuracy・flags は仕様と一致しており修正不要。
  pp は5→8に修正が必要（`.internal/spec/moves/クロスフレイム.md` の PP 節参照）。
* 仕様書に記載した「使用者がこおり状態の場合、こおり状態を治してから攻撃する」効果は
  未実装だった。`flags` の `"thaw"` は現状 `こおり_cure_by_thaw_move`（`Event.ON_DAMAGE_HIT`,
  被弾側＝defender）にのみ使われており、使用者自身の無条件解凍処理は個別実装が必要
  （`.internal/spec/moves/かえんぐるま.md` と同じ課題）。
* デュアルクロス（クロスサンダー直後に使うと威力2倍）も未実装だった。発動条件に味方限定の
  制限はなく、シングルバトルでも相手が同ターン内に先にクロスサンダーを命中させていれば
  成立するため、対象外にはできない（`.internal/spec/moves/クロスフレイム.md` 参照）。
  クロスサンダー側のレビューで対になる実装（`クロスサンダー_calc_power` /
  `クロスサンダー_record_hit`、`battle.fusion_bolt_used_turn` / `fusion_flare_used_turn`）が
  既に追加されているため、本レビューではクロスフレイム側の対になるハンドラを追加する。

## 追加実装

### ハンドラ

`src/jpoke/handlers/move_attack.py` に `クロスフレイム_thaw_attacker` を追加する
（`クロスポイズン_apply_poison_to_defender` の直前に配置。`もえつきる_thaw_attacker` /
`かえんぐるま_thaw_attacker` と全く同じパターン）。

```python
def クロスフレイム_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """クロスフレイム: こおり状態でも使用可能にし、こおりを解凍する。

    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

### PP修正

`src/jpoke/data/moves/move_ka.py` の `"クロスフレイム"` エントリの `pp` を 5 から 8 に修正する。

### 登録

`src/jpoke/data/moves/move_ka.py` の `"クロスフレイム"` エントリの `handlers` に以下を追加する。

```python
Event.ON_TRY_ACTION: h.MoveHandler(
    ha.クロスフレイム_thaw_attacker,
    priority=5,
),
```

### priority の根拠

* `.internal/spec/moves/かえんぐるま.md`（先行レビュー済み）に記載の通り、`.internal/spec/turn.md` の
  `Event.ON_CHECK_ACTION`（コード上は `Event.ON_TRY_ACTION`）節の priority=170
  （こおりを回復するわざでこおり回復）は、priority=10 の `こおり_action` が25%抽選で
  解凍しなかった場合に `stop_event=True` で即座にイベントを打ち切ってしまうため機能しない。
* 既存の同種実装（`もえつきる_thaw_attacker`, `かえんぐるま_thaw_attacker`）はいずれも
  priority=5（`こおり_action` の10より前）で登録されており、`こおり_action` が発火する前に
  ailment を除去することで、`_check_handler_validity` の `ailment` ソースチェック
  （`subject.ailment.is_active`）により `こおり_action` 自体がスキップされる設計になっている。
  `tests/moves_attack/test_move_ka.py::test_かえんぐるま_こおり状態で使うと解凍されて攻撃できる`
  等でこの動作パターンが検証済みのため、同じ priority=5 を採用する。

### デュアルクロス（威力2倍）について

`src/jpoke/handlers/move_attack.py` に以下を追加する（クロスサンダー側のレビューで実装済みの
`クロスサンダー_calc_power` / `クロスサンダー_record_hit` と対になる）。

```python
def クロスフレイム_calc_power(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """クロスフレイム: 同じターン中に直前でクロスサンダーが命中していた場合、威力が2倍になる。"""
    if battle.fusion_bolt_used_turn == battle.turn:
        value = apply_fixed_modifier(value, 8192)
    return HandlerReturn(value=value)


def クロスフレイム_record_hit(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """クロスフレイム: 命中したターンを記録する（クロスサンダーとの威力2倍判定用）。"""
    battle.fusion_flare_used_turn = battle.turn
    return HandlerReturn(value=value)
```

`"クロスフレイム"` エントリの `handlers` に `Event.ON_CALC_POWER_MODIFIER` /
`Event.ON_HIT`（いずれも `subject_spec="attacker:self"`）で登録する。
