# 実装計画書: グラススライダー

## 仕様要約（.internal/spec/moves/グラススライダー.md 参照）

- タイプ: くさ / 分類: 物理 / 威力: 55 / 命中: 100 / PP: 20 / 対象: 相手1体
- 優先度: 使用者が「グラスフィールド」の効果を受けている（グラスフィールド展開中 かつ
  使用者が接地している）とき +1、それ以外は 0
- 追加効果なし

## 現状の実装状況（レビュー時点）

| 項目 | 状態 |
|------|------|
| `data/moves/move_ka.py` エントリ | 実装済みだが `priority=1` が固定値になっており誤り |
| グラスフィールドによる動的優先度上昇 | 未実装 |
| テスト | 未実装 |

### 発見したバグ

`MoveData(priority=1)` と静的に設定されていたため、グラスフィールドの有無・使用者の
接地状態に関わらず常に優先度+1として扱われていた。これにより:

- グラスフィールドが展開されていないターンでも先制技として扱われる（行動順が誤る）。
- `damage.py` `_can_apply_terastal_power_floor` が `move.priority >= 1` を除外条件と
  しているため、テラスタル一致時の威力60底上げが常に適用されなくなる
  （本家仕様では本技は例外的に底上げ対象になるべき）。

## ハンドラ構成

技自身に優先度ハンドラを持たせる方式は使えない。理由は
`.internal/spec/moves/グラススライダー.md` の「優先度上昇の実装箇所」節を参照:
`SpeedCalculator.calc_move_priority` はターン開始時、技の実行前に発火するため、
`move_executor` が技実行時にのみ `register_handlers` する技側のハンドラでは
間に合わない。地形「グラスフィールド」側のハンドラとして実装する
（トリックルームの `DomainEvent.ON_CHECK_SPEED_REVERSE` と同一パターン）。

| 対象 | イベント | priority | subject_spec | 役割 |
|------|----------|----------|--------------|------|
| グラスフィールド | `DomainEvent.ON_MODIFY_MOVE_PRIORITY` | 100（デフォルト） | `attacker:self` | グラススライダー使用時、使用者が接地していれば優先度+1 |

### Priority 根拠

`DomainEvent.ON_MODIFY_MOVE_PRIORITY` は `.internal/spec/turn.md` に掲載されていない
（行動順決定に関与する `DomainEvent` は同ドキュメントの対象外）。既存の同種ハンドラ
`いたずらごころ_modify_move_priority`・`はやてのつばさ_modify_priority`
（いずれも `AbilityHandler` の priority デフォルト値=100を使用）に合わせ、
デフォルト値をそのまま採用する。

## 実装箇所

### `src/jpoke/handlers/field.py`

`グラスフィールド_power_modifier` の直後に追加する。

```python
def グラスフィールド_boost_move_priority(battle: Battle, ctx: AttackContext, value: int) -> HandlerReturn:
    """グラスフィールド: 接地している使用者のグラススライダーの優先度を+1する。"""
    if (
        ctx.move.name == "グラススライダー"
        and not battle.query.is_floating(ctx.attacker)
    ):
        value += 1
    return HandlerReturn(value=value)
```

### `src/jpoke/data/field/terrain.py`

`"グラスフィールド"` エントリの `handlers` に追加する（`DomainEvent` を新規 import する
必要がある）。

```python
DomainEvent.ON_MODIFY_MOVE_PRIORITY: h.FieldHandler(
    h.グラスフィールド_boost_move_priority,
    subject_spec="attacker:self",
),
```

### `src/jpoke/data/moves/move_ka.py`

`"グラススライダー"` エントリから `priority=1` を削除する（デフォルト値0を使用）。

```python
"グラススライダー": MoveData(
    type="くさ",
    category="physical",
    pp=20,
    power=55,
    accuracy=100,
    flags={"contact"},
    handlers={},  # 追加効果なし。優先度補正はグラスフィールド側で処理する（グラスフィールド_boost_move_priority）
),
```

## 注意点・エッジケース

- 接地判定は `battle.query.is_floating(ctx.attacker)` の否定で行う（グラスフィールドの
  威力補正・回復ハンドラと同じ判定方法、`グラスフィールド_power_modifier` /
  `グラスフィールド_heal` を参照）。
- サイコフィールドの先制技無効化・ファストガード等は `ctx.move.priority`
  （基本優先度・常に0）を見るため、グラスフィールドによる動的な優先度上昇には反応しない。
  既存の同種の限界（いたずらごころ等）と同じであり本レビューでは対応しない
  （詳細は仕様書の「既知の制限」節）。
- テラスタルの威力60底上げ補正の除外判定も基本優先度を参照するため、
  `priority=1` を削除したことで自然に「本技は先制技扱いされず底上げ対象になる」という
  本家仕様と一致する挙動になる（追加のコード変更は不要）。
