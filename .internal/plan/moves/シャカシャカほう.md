# シャカシャカほう 実装計画書

参照: .internal/spec/moves/シャカシャカほう.md

## 現状（レビュー前）

* `src/jpoke/data/moves/move_sa.py` の `"シャカシャカほう"` エントリは以下の通り実装済み。
  - type="くさ", category="special", pp=**15**, power=80, accuracy=90
  - flags={"heal", "secondary_effect", "thaw"}
  - `Event.ON_HIT`: `シャカシャカほう_drain`（priority=20、`_drain_hp` heal_ratio=0.5）
  - `Event.ON_DAMAGE_HIT`: `シャカシャカほう_apply_burn_to_defender`（20%でやけど付与）
* PP=15 は Gen9本家基準の値であり、`src/jpoke/data/ps-champ-ja/moves.json`（342行目）の
  champions基準値=16と食い違っている。champions側はPPを8/12/16/20の4段階に統一している
  ため、**PP=15は移行漏れのバグ**であり16に修正する。
* `flags` に `"thaw"` が設定されているため、`こおり_cure_by_thaw_move`
  （`src/jpoke/handlers/ailment.py`, `Event.ON_DAMAGE_HIT`, defender:self）により、
  相手がこおり状態のときにこの技のダメージを受けるとこおりが解除される効果は既に
  実装済み。このハンドラは技のタイプが『ほのお』でない場合、使用者の特性が『ちからずく』
  であれば発動しない、という分岐も既に実装済み（`.internal/plan/moves/ねっとう.md` の
  レビューで対応済み）。シャカシャカほうはタイプ=くさのため、この既存の共通実装が
  そのまま正しく適用される。**この部分は追加実装不要**。
* 一方、仕様書に記載した「使用者がこおり状態の場合、こおり状態を治してから攻撃する」効果は
  **未実装だった**。`"thaw"` フラグは現状 `こおり_cure_by_thaw_move`（被弾側＝defender）にのみ
  使われており、使用者自身の無条件解凍処理は個別実装が必要
  （かえんぐるま・かえんボール・せいなるほのお・もえつきる・ねっさのだいち・ねっとうは
  実装済みだが、シャカシャカほうは未実装だった）。

## 追加実装

### 1. PP修正

`src/jpoke/data/moves/move_sa.py` の `"シャカシャカほう"` エントリの `pp=15` を `pp=16` に修正する
（`src/jpoke/data/ps-champ-ja/moves.json` 342行目に合わせる）。

### 2. 使用者自身の無条件解凍ハンドラ

`src/jpoke/handlers/move_attack.py` に `シャカシャカほう_thaw_attacker` を追加する。
`ねっとう_thaw_attacker` / `ねっさのだいち_thaw_attacker`
（同ファイル内、Event.ON_TRY_ACTION, priority=5）と全く同じパターンで実装する。

```python
def シャカシャカほう_thaw_attacker(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """シャカシャカほう: こおり状態でも使用可能にし、こおりを解凍する。

    こおり_action (priority=10) より先に発火させる (priority=5) ことで、
    ailment が除去された状態で こおり_action の validity check が走り、
    こおり_action がスキップされる。
    """
    mon = ctx.attacker
    if mon.ailment.name == "こおり":
        battle.ailment_manager.remove(mon)
    return HandlerReturn(value=value)
```

`src/jpoke/data/moves/move_sa.py` の `"シャカシャカほう"` エントリに以下を追加する。

```python
Event.ON_TRY_ACTION: h.MoveHandler(
    ha.シャカシャカほう_thaw_attacker,
    priority=5,
),
```

#### priority の根拠

* `.internal/spec/turn.md` には `Event.ON_TRY_ACTION`（コード上の名称）に対応する行が無い
  （ダメージ計算内部等ではないが、move_executor.py 内の行動実行可否判定という技実行フロー
  内部のステップであり、turn.md には未掲載）。
* CLAUDE.md の指示に従い、既存の同種ハンドラを参照して priority を決定する。
  `.internal/plan/moves/ねっとう.md`「priority の根拠」節で検証済みの通り、`こおり_action`
  （priority=10）は25%抽選で解凍しなければ `stop_event=True` で即座にイベントを打ち切るため、
  無条件解凍ハンドラは `こおり_action` より前（priority=5）で実行し、`ailment` を先に除去して
  `こおり_action` 自体をスキップさせる必要がある。
* 既存の同種実装（かえんぐるま・ねっさのだいち・ねっとう）は全て priority=5 で登録されており、
  これに倣う。

### やけど付与とこおり解除の実行順序について

`ねっとう` のレビュー（`.internal/plan/moves/ねっとう.md`）で確認済みの通り、`Event.ON_DAMAGE_HIT`
上でやけど付与ハンドラ（move側登録）がこおり解除ハンドラ（ailment側登録、
`こおり_cure_by_thaw_move`）より先に実行される登録順序になっており、
「相手がこおり状態のときにやけどを付与しようとしても、こおりが治る前なので状態異常が
埋まっており付与に失敗し、その後こおりが解除される」という仕様通りの挙動になる。
シャカシャカほうも同じ実装パターン（`シャカシャカほう_apply_burn_to_defender` が
`Event.ON_DAMAGE_HIT` に登録）のため、priority の明示的な変更は不要。
