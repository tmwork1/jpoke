# 計画書: デカハンマー

## 仕様要約

- 物理・はがねタイプ、威力160、命中100、PP5、直接攻撃なし
- **2回連続で選択できない**: 自分が直前にデカハンマーのPPを消費していた場合、次のターンのコマンド選択でデカハンマーを選べなくなる
- PPを消費したが相手がまもる状態で防がれた場合・外れた場合も次のターン選択不可（PP消費が判定の鍵）
- 状態異常・状態変化で行動できなかった場合（PP未消費）は次のターンも選択可能
- 選択画面を経由せずに発動した場合（ねごと・さいはい等）は2回連続使用が可能
- デカハンマーしか選べる技がない状態では、次のターンは「わるあがき」を使用する

## Priority 根拠

### `ON_TRY_MOVE_1` priority=50

`.internal/spec/turn.md` の `Event.ON_TRY_MOVE_1` セクションに以下の記載がある：

```
| 50 | エコーボイス: 次ターン威力上昇 |
| 50 | いかり: いかり状態になる |
```

デカハンマーの揮発付与は「PP消費確定後に次ターンの制限を設定する」処理であり、上記と同種の「技使用後の自身状態変化」に分類される。priority=50 を使用する。

### `ON_MODIFY_COMMAND_OPTIONS` priority=100（デフォルト）

`いちゃもん`・`こだわり` 等の既存コマンド制限ハンドラがすべてデフォルト priority=100 を使用しているため、これに合わせる。

### `ON_TURN_END` priority=100（デフォルト）

`アンコール` の tick（priority=110）より前に解除することで問題は生じない。デフォルト priority=100 を使用する。

## count=2 の根拠

揮発状態は ON_TRY_MOVE_1（ターンT中）で付与される。ON_TURN_END（ターンT終了時）で tick され count は 2→1 になる。次ターンT+1 の ON_MODIFY_COMMAND_OPTIONS 発火時点で count=1（揮発あり）なので制限が有効。ターンT+1 の ON_TURN_END で tick され count=0 → volatile_manager が自動 remove。ターンT+2 以降は制限なし。

## 注意点・エッジケース（review-test 向け）

- **count=2 の意図**: 揮発付与はターンT中の ON_TRY_MOVE_1 で行われ、その後 ON_TURN_END で count が 1 つ減る。次のターンT+1 の選択肢制限時には count=1（揮発あり）で制限が有効になる。count=1 で付与するとターンT の ON_TURN_END で即座に解除されてしまう
- **PP未消費時は付与されない**: `ON_TRY_ACTION` が失敗（状態異常等）するとPP消費もなく ON_TRY_MOVE_1 が発火しないため、揮発状態は付与されない（仕様通り）
- **まもるで防がれた場合**: PP消費後に ON_TRY_MOVE_1 が発火するため揮発が付与される。次ターン選択不可（仕様通り）
- **外れた場合**: 命中判定が ON_TRY_MOVE_1 より後のため、揮発付与後に外れる。次ターン選択不可（仕様通り）
- **交代時のリセット**: 揮発状態は交代で消えるため、使用後に交代した場合は次の登場時にデカハンマーを選べる（仕様通り）
- **さいはい・ねごとによる2回使用**: これらの技はコマンド選択を経由しないため ON_MODIFY_COMMAND_OPTIONS は発火しない。制限を受けずに使用可能（仕様通り）
- **ブラッドムーンとの相互作用**: ブラッドムーンの使用後に付与される揮発状態は `"ブラッドムーン"` として別途実装する想定。デカハンマー揮発とブラッドムーン揮発は独立しているため、デカハンマーの次にブラッドムーンを選ぶことは可能（仕様通り）
- **ON_MODIFY_COMMAND_OPTIONS の subject_spec**: `command_manager.py` が `EventContext(source=active)` で発火するため `"source:self"` を使う
- **ON_TRY_MOVE_1 の subject_spec**: `AttackContext` のため `"attacker:self"` を使う
- **デカハンマーしか技が無い場合**: コマンド選択時に全技が除外され、わるあがきが自動選択される。この動作はゲームエンジン側（command_manager）の責任であり、ハンドラ側の対処は不要
