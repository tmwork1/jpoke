# 実装計画: ロックオン

仕様書: .internal/spec/moves/ロックオン.md

## 仕様サマリ

- タイプ: ノーマル / 分類: 変化 / PP: 8（champions基準） / 命中率: -（必中） / 対象: 相手1体
- 効果: 自分をロックオン状態にする（第五世代以降。効果詳細は
  `.internal/spec/volatiles/ロックオン.md` の揮発性状態を参照）。
- まもる: ○（防がれる） / マジックコート: ×（反射されない） / 直接攻撃: ×
- 効果の付与対象は**自分（attacker）**。技の選択対象（相手）ではない点に注意。

## 対象イベント

`ON_STATUS_HIT` — 変化技命中時に発火。他の揮発性状態を付与する変化技（かえんのまもり・
こらえる・まもる 等）と同パターン。

## Priority

`.internal/spec/turn.md` に `ON_STATUS_HIT` の記載なし。
既存の `かえんのまもり_apply` 等が `MoveHandler(func)` のみ（priority=100デフォルト）で
実装されているため、同様に priority=100（デフォルト）を採用する。

## 実装手順

### handlers/move_status.py

`ロックオン_apply` 関数を追加する（既存実装あり、`apply_volatile_to_attacker` を使う
点に注意）。

```python
def ロックオン_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """ロックオンの効果: 自分をロックオン状態にし、次のターンの自分の技を相手に必中させる。"""
    return apply_volatile_to_attacker(battle, ctx, value, volatile="ロックオン", count=2)
```

- `apply_volatile_to_defender` を使うと相手に揮発状態が付与されてしまい仕様と矛盾する
  （Gen5以降は自分に付与するのが正）。
- `count=2` は「使用したターン」+「次のターン（必中を発揮するターン）」の2ターンで
  `.internal/spec/volatiles/ロックオン.md` の持続期間仕様と一致させるため。
- すでにロックオン状態の場合の失敗判定は `VolatileManager.apply` が同名状態の
  重複付与を自動的に拒否するため、ハンドラ側で個別に実装する必要はない。

配置位置: `りゅうのまい_*` 関数の直後、`ロックカット_*` 関数の直前（五十音順）。

### data/moves/move_ra.py

`ロックオン` の MoveData を以下のように登録する（PP・flags に注意）。

```python
"ロックオン": MoveData(
    type="ノーマル",
    category="status",
    pp=8,
    # マジックコートで跳ね返されない
    flags={"unreflectable"},
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.ロックオン_apply,
        ),
    }
),
```

## テスト方針

- `t.run_move` でロックオンを使用し、使用者（attacker）に「ロックオン」揮発状態が
  付与されることを確認する（相手ではない点を明示的に確認）。
- 揮発状態自体の必中効果・ターン経過解除・交代解除は
  `tests/volatiles/test_volatile_hamayarawa.py` で既にテスト済みのため重複実装しない。
