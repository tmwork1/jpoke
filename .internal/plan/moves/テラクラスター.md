# 計画書: テラクラスター

## 仕様要約

- ノーマルタイプ・特殊・威力120・命中100・PP8（champions基準の推定値。詳細は下記「PPについて」を参照）
- テラパゴス専用技。直接攻撃ではない。
- 使用者が **ステラテラスタル状態**（`active_tera_type == "ステラ"`）のとき、以下の変化が発生する。
  - 技タイプが **ステラタイプ** に変化する。
  - 自身の補正込みステータスが **こうげき > とくこう** の場合、技分類が **物理技** に変化する
    （同値・とくこうが高い場合は特殊技のまま）。
  - 技の対象が「相手1体」から「相手全体」に変わる（本プロジェクトはシングルバトル専用のため
    実質的な効果なし。詳細は「対象範囲変化について」を参照）。
- 威力はステラテラスタル時も変化しない（120固定）。テラバーストと異なり威力補正・命中後の
  能力ランクダウンは発生しない。
- ステラテラスタルしていないテラパゴス（通常フォルム／テラスタルフォルムでステラ以外の
  テラスタイプを選んだ場合）が使用した場合は、タイプ・分類とも変化しない
  （テラパゴスの仕様上、通常テラスタルすると強制的にステラフォルム・ステラタイプになるため、
  実戦では「テラスタルしているがステラでない」状況は発生しない）。
- まねっこでコピー不可（`non_copycat` フラグ）。ものまね・スケッチ・ゆびをふるでも選択・コピー
  不可だが、これらの技自体が本プロジェクトで未実装（`実装保留`／`TODO`）のため、対応フラグは
  存在せず実装不要。

## PPについて

- `src/jpoke/data/ps-champ-ja/moves.json` にテラクラスター（および専用技元のテラパゴスの記載）が
  存在しない。理由不明だが、championsの技リストのスクレイピング元から漏れている可能性が高い。
- `.internal/champions/changes_from_sv.md` に「すべての技のPPが8・12・16・20の4段階に統一。
  move_list.txtを参照」とあり、全技が対象と明記されている。
- move_list.txt 内で本家PP5の技を調査したところ、以下いずれも champions PP=8 に統一されている
  （例外なし）。
  - アーマーキャノン（本家PP5 → champions PP8）
  - アイアンローラー（本家PP5 → champions PP8）
  - はかいこうせん（本家PP5 → champions PP8）
  - しんそく（本家PP5 → champions PP8）
  - じばく／だいばくはつ（本家PP5 → champions PP8）
- テラクラスターの本家PPは5のため、上記の一貫した変換規則に従い championsでも PP8 と推定し、
  実装値をPP8とする（従来の実装値は本家基準のPP5のままだった。バグとして修正）。
- 例外として「まもる」は本家PP10だが championsではPP8に個別調整されている（10まんばりき等の
  本家PP10技は通常PP12になるため、まもるは意図的な特別ナーフと考えられる）。テラクラスターに
  ついても同様の意図的な特別調整が入っている可能性はゼロではないが、確証がないため、確認できた
  規則性（本家PP5→champions PP8、例外なし）を優先して採用する。
- 将来 move_list.txt にテラクラスターの記載が追加された場合は、その値を正として再修正すること。

## 対象範囲変化について

- 本家ではステラフォルムのテラパゴスが使用すると対象が「相手全体」に変わり、ワイドガードで
  防がれるようになる。
- 本プロジェクトはポケモンチャンピオンズ **シングルバトル専用**（`CLAUDE.md`）であり、対戦相手は
  常に1体のみ。複数対象（スプレッド）技の概念自体が実装されていない
  （`data/moves/move_wa.py` の `"ワイドガード"` エントリのコメントに
  「シングルバトルでは効果なし（複数対象技が存在しない）」と明記されている）。
- したがって対象範囲変化は本プロジェクトの戦闘結果に一切影響しない。実装は不要。

## Priority 根拠

`ON_MODIFY_MOVE_TYPE` と `ON_MODIFY_MOVE_CATEGORY` は `.internal/spec/turn.md` に個別行の記載がない
（ダメージ計算内部のイベント）。

既存の同種ハンドラ（`テラバースト_modify_move_type` / `テラバースト_modify_move_category`、
`フォトンゲイザー_modify_move_category`）がいずれも `MoveHandler` のデフォルト priority=100 を
使用しているため、テラクラスターも同様とする。

## 実装箇所

### `src/jpoke/handlers/move_attack.py`

```python
def テラクラスター_modify_move_category(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テラクラスター: ステラテラスタル時、補正込みAがCより高い場合は物理技として計算する。"""
    mon = ctx.attacker
    if mon.active_tera_type == "ステラ":
        atk = mon.ranked_stats["atk"]
        spa = mon.ranked_stats["spa"]
        if atk > spa:
            value = "physical"
    return HandlerReturn(value=value)


def テラクラスター_modify_move_type(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テラクラスター: ステラテラスタル時、技タイプがステラタイプに変化する。"""
    if ctx.attacker.active_tera_type == "ステラ":
        value = "ステラ"
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### テラバーストとの相違点

- テラバーストは「テラスタルしていれば（テラスタイプが何であれ）タイプが変化し、ステラの
  ときのみ威力100・命中後スタットダウン」という2段階の分岐を持つ。
- テラクラスターは「ステラテラスタルのときのみ」タイプ・分類が変化し、威力変化・命中後の
  能力ランクダウンは一切ない、というシンプルな仕様。`mon.terastallized` ではなく
  `mon.active_tera_type == "ステラ"` を直接判定条件にすることで実装できる
  （`active_tera_type` はテラスタルしていない場合 `""` を返すため、この1条件で
  「ステラテラスタル済み」を判定できる）。

### 分類判定のタイミング

- `ranked_stats` はランク補正込みの実数値。テラバースト・フォトンゲイザーと同じ判定ロジック。

### コピー・選択制限

- `non_copycat` フラグにより `まねっこ`（Copycat）でのコピーを防止。
- `ものまね` / `スケッチ` / `ゆびをふる`（Metronome）は本プロジェクトで未実装のため、
  対応するフラグ（`non_sketch` 等）は存在せず、実装不要。将来実装される際は
  `.internal/spec/moves/テラクラスター.md` の記載を参照して除外リストに加えること。

### リーサル計算

- `.internal/progress/move.md` の「リーサル実装」列は `n/a`。通常のダメージ技であり、タイプ・分類の
  変化は `ON_MODIFY_MOVE_TYPE` / `ON_MODIFY_MOVE_CATEGORY` を通じて通常のダメージ計算経路に
  自然に反映されるため、専用のリーサルハンドラは不要。
