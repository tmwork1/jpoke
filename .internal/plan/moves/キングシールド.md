# 計画書: キングシールド

対象: Pokémon Champions シングルバトル
仕様書: `.internal/spec/moves/キングシールド.md`（技本体）、
`.internal/spec/volatiles/キングシールド.md`（付与される揮発性状態）

## 概要

キングシールドは既に実装済み。本計画書は実装の意図・設計根拠を記録する
（レビュー時点で PP に誤りがあったため修正した）。

## 技データ (`data/moves/move_ka.py`)

```python
"キングシールド": MoveData(
    type="はがね",
    category="status",
    pp=8,
    priority=4,
    target="self",
    flags={"protect"},
    handlers={
        Event.ON_TRY_MOVE_2: h.MoveHandler(
            hs.まもる系_連続使用失敗チェック,
        ),
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.キングシールド_apply,
        ),
    }
),
```

- `pp=8`: champions基準（`src/jpoke/data/ps-champ-ja/moves.json` 220行目
  「キングシールド はがね 変化 - - 8 接× 守×」）。Gen9本家は10だが、
  champions は全技PPを8/12/16/20の4段階に統一しているため8を採用する
  （`.internal/champions/changes_from_sv.md` 参照）。レビューでpp=10→8に修正した
  （champions移行漏れのバグだった）。
- `flags={"protect"}`: 「まもる系技」共通の連続使用失敗判定（後述）で使用するラベル。
  この `protect` フラグは `まねっこ_can_use`（`handlers/move_status.py`）でも
  参照され、守る系技を一律まねっこのコピー対象から除外する。まもるとは異なり
  `non_copycat` フラグは不要（`protect` フラグのみで両方の判定を兼ねる）。

## ハンドラ構成 (`handlers/move_status.py`)

```python
def キングシールド_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    return apply_volatile_to_attacker(battle, ctx, value, volatile="キングシールド")


def まもる系_連続使用失敗チェック(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """守る系技の連続使用失敗チェック（まもる・みきり・こらえる・キングシールド等で共有）。"""
    ...
```

- `キングシールド_apply`: `ON_STATUS_HIT` で「キングシールド」揮発状態を自分に付与する。
  まもる・みきり等と同一のヘルパー呼び出しパターン。
- `まもる系_連続使用失敗チェック`: まもる・みきり・こらえる・キングシールド・ニードルガード・
  トーチカ・スレッドトラップ・かえんのまもりで共有される既存関数（`.internal/plan/moves/まもる.md`
  参照）。キングシールド専用の実装は不要。

## キングシールド volatile (`data/volatile.py` / `handlers/volatile.py`)

キングシールド volatile 自体の設計（`ON_TRY_MOVE_1` priority=100 での攻撃技無効化・
直接攻撃者の攻撃ランクダウン、`ON_TURN_END` での解除、`_run_protect` の共通骨格）は
`.internal/spec/volatiles/キングシールド.md`・`.internal/plan/volatiles/キングシールド.md`（既存レビュー済み）
を参照。本計画書では技側（`キングシールド` MoveData・`キングシールド_apply`）のみを扱う。

## Priority 根拠

### `Event.ON_STATUS_HIT`: priority 100（デフォルト）

`.internal/spec/turn.md` に `ON_STATUS_HIT` の個別 priority 表がない。既存の変化技ハンドラ
（`まもる_apply`・`みきり_apply`・`こらえる_apply` 等）がすべて priority 100（デフォルト）を
使用しているため、同一パターンで 100 を採用する。

### `Event.ON_TRY_MOVE_2`: priority 100（デフォルト）

`.internal/spec/turn.md` の `Event.ON_TRY_MOVE_1` 30番に「まもる系: 連続使用により失敗」と
明記されているが、既存実装（まもる・みきり・キングシールド・こらえる等）はすべて
`Event.ON_TRY_MOVE_2` のデフォルト priority（100）で実装済みであり、このパターンは
`.internal/spec/moves/まもる.md`・`.internal/spec/moves/こらえる.md`（レビュー済み）でも踏襲することが
確認されている。本レビューでも既存パターンを維持する。

## スコープ外（既知の限界）

まもる系共通の限界（連続使用による失敗率の確率モデル未実装、`executed_move` の
行動スキップ時の非リセット等）は `.internal/spec/moves/まもる.md` を参照。キングシールド固有の
追加スコープ外事項はない。

ギルガルドのフォルムチェンジ（バトルスイッチ）は特性側で実装済み（対象外ではなく別ファイルで管理）。
詳細は `.internal/spec/abilities/バトルスイッチ.md` 参照。

## テスト方針

- キングシールドを使うとキングシールド状態が付与されること（既存:
  `test_まもる系_キングシールドとニードルガードでも相互に失敗する`）
- キングシールドを2ターン連続使用すると2ターン目は失敗すること・異なる守る系技を挟んでも
  連続使用扱いになること（既存: まもる系共通テスト、`test_まもる系_キングシールドとニードルガードでも
  相互に失敗する` 等）
- 攻撃技を防ぐこと・変化技は防がないこと・ターン終了で解除されること（既存:
  `tests/volatiles/test_volatile_hamayarawa.py` のまもる系共通パラメトリックテスト、
  `tests/volatiles/test_volatile_ka.py` の `test_キングシールド_*` 系）
- 直接攻撃を防ぐと相手の攻撃ランクが下がること・非接触では下がらないこと（既存:
  `tests/volatiles/test_volatile_ka.py` の `test_キングシールド_接触技で攻撃を下げる`・
  `test_キングシールド_非接触では攻撃が下がらない`）
