# 計画書: かえんのまもり

## 仕様要約

- タイプ: ほのお / カテゴリ: 変化 / PP: 10 / 命中: なし（必中） / 対象: 自分 / 優先度: +4
- 使用ターン中、相手の**攻撃技のみ**を防ぐ（変化技は防げない）
- 直接攻撃で攻撃してきた相手をやけど状態にする
- まもる系共通の連続使用失敗ルールが適用される
- **かえんのまもり固有**: そのターンの最後に行動する場合は失敗する（守る対象がないため）
- ウガツホムラの専用技
- 詳細仕様は `.internal/spec/volatiles/かえんのまもり.md` を参照

## Priority 根拠

### `ON_TRY_MOVE_2`: まもる系連続使用失敗チェック・最後の行動チェック（既存パターンを踏襲）

`.internal/spec/turn.md` の `Event.ON_TRY_MOVE_1` priority 30 に「まもる系: 連続使用により失敗」と
明記されているが、本実装ではまもる・キングシールド・トーチカ・ニードルガード・スレッドトラップ・
ファストガードを含む protect 系技すべてが `Event.ON_TRY_MOVE_2`（デフォルト priority=100）で
この失敗チェックを登録しており、turn.md の記載と実装イベントが一致していない（本レビュー時点で
判明した既存の齟齬。protect 系全体に及ぶため本レビューの対象外として現状維持する）。

「そのターンの最後の行動なら失敗する」チェックはかえんのまもり固有の失敗条件であり、
turn.md に対応する記載はない。上記の既存パターン（まもる系連続使用失敗チェックと同じ
`ON_TRY_MOVE_2`）に合わせて登録し、`かえんのまもり` の `handlers` 辞書内で
`まもる系_連続使用失敗チェック` の後に並べる（同一 Event キーに対してリスト形式で複数
`MoveHandler` を登録する）。

### `ON_STATUS_HIT` / `ON_TRY_MOVE_1`（volatile側）/ `ON_TURN_END`: 既存踏襲

まもる・キングシールド・トーチカ等の同系統技と同一パターン（`ON_STATUS_HIT` で volatile 付与、
`ON_TRY_MOVE_1` priority=100 で volatile 側の無効化処理、`ON_TURN_END` デフォルト priority で
解除）を踏襲する。詳細根拠は `.internal/spec/volatiles/かえんのまもり.md` を参照。

## 実装箇所（本レビューでの追加分）

### `src/jpoke/handlers/move_status.py`

`かえんのまもり_apply` の直後に追加。

```python
def かえんのまもり_check(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """かえんのまもりの効果: そのターンの最後に行動する場合は失敗する。

    かえんのまもりはそのターンに自分より後に行動する相手の技から身を守るための
    技であるため、自分がそのターンの最後に行動する場合は守る対象がなく失敗する
    （シングルバトル想定）。
    失敗時は executed_move を None にリセットし、まもる系の連続使用カウントも
    リセットする（連続使用扱いにしない）。
    """
    attacker_player = battle.get_player(ctx.attacker)
    if battle.query.is_second_actor(attacker_player):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="かえんのまもり_最終行動")
        )
        ctx.attacker.executed_move = None
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

`battle.query.is_second_actor()` は既存の `しっぺがえし_double_power_when_second` 等で使われている
1vs1想定のユーティリティをそのまま利用する。

### `src/jpoke/data/moves/move_ka.py`

`"かえんのまもり"` の `Event.ON_TRY_MOVE_2` をリスト化し、`かえんのまもり_check` を追加する。

"かえんのまもり": MoveData(

## 注意点・エッジケース

### `is_second_actor` はターン全体を進める `battle.step()` 経由でのみ機能する

`battle.turn_controller.action_order` はターン開始時の行動順解決（`_resolve_action_order`）で
構築されるため、`battle.run_move()` を直接呼び出す単体技実行ヘルパー（`t.run_move`）では
`action_order` が空のままとなり `is_second_actor()` は `None`（Falsy）を返す。そのため
「最後の行動なら失敗する」挙動を検証するテストは `battle.step()` でターン全体を進める必要がある
（`しっぺがえし` のテストと同じ制約）。

### `reserve_command` + 素の `battle.step()` の多ターンテストにおける注意

`t.reserve_command()` で予約したコマンドと、`battle.step()` が内部で自動解決するコマンドは
別々に `state.reserved_commands` へ追記される仕組みになっており、1ターンで消費しきれない
場合に次のターンへ意図しないコマンドが混入することがある。複数ターンにわたって特定の技を
明示的に選ばせたい場合は `battle.step({player0: Command.MOVE_x, player1: Command.MOVE_y})`
の形で明示的にコマンド辞書を渡す（本レビューで追加したテストを参照）。

### 失敗時の `executed_move` リセット

まもる系の連続使用失敗チェックと同様、「そのターンの最後の行動」による失敗も
`executed_move = None` にリセットする。これにより、次ターンに自分が先に行動できれば
連続使用扱いにならず成功する（ポケモンWikiの「かえんのまもり状態」節にある
「失敗した場合は成功率がリセットされる」という一般ルールに従う）。

### 既存の限界（対象外）

- まもる系の連続使用失敗チェックは確率的な逓減（1/3ずつ）ではなく、簡略化された
  決定的な実装（直前に protect 技が成功していれば必ず失敗）になっている。これは
  protect 系全体に共通する既存の設計であり、本レビューでは変更しない
- フェイント/ゴーストダイブ/シャドーダイブ/いじげんホール/いじげんラッシュのうち、
  ゴーストダイブ/シャドーダイブは `unprotectable` フラグのみで守りを無視しており、
  かえんのまもり volatile を明示的に解除しない（フェイント等は解除する）。
  シングルバトルでは同ターン中の挙動に影響しないため実害はないが、それらの技を
  受けた次のターンを「連続使用と見なさない」という仕様までは再現できていない。
  ゴーストダイブ/シャドーダイブ自体の実装修正は当該技のレビュー対象とし、
  本レビューでは対象外とする
