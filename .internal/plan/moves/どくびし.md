# 実装計画書: どくびし

## 仕様要約

- タイプ: どく / カテゴリ: 変化 / PP: 20 / 対象: 相手の場（target="foe_side"）
- 使用後、相手はポケモンを交代するたびに出てきたポケモンが「どく」状態になる
- 2 回使用すると「もうどく」状態になる（最大 2 層）
- どく・はがね・ひこうタイプ、特性「ふゆう」のポケモンには無効
- どくタイプのポケモンが出ると効果が消える

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 相手陣営サイドフィールドに「どくびし」を 1 層追加 |

## Priority 根拠

`.internal/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「設置技: すでに最大」が priority 100 として記載されている。実際の失敗判定（最大層数超過）は `ON_STATUS_HIT` ハンドラ内の `side.activate()` の戻り値で行う。これは `ステルスロック_set_field` および `まきびし_set_field` と同一パターンであり、これらの既存実装済み設置技と処理タイミングを揃えるためにこの構成を採用する。

## 実装箇所（実装済み）

### `src/jpoke/handlers/move_status.py`

五十音順の位置: `どくのこな_apply_ailment_to_defender` の後、`ドわすれ_modify_attacker_stats` の前に実装済み。

```python
def どくびし_set_field(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """どくびし: 相手陣営に「どくびし」を1層設置する（最大2層）。"""
    side = battle.get_side(ctx.defender)
    if not side.activate("どくびし", 1):
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ta.py`

```python
"どくびし": MoveData(
    type="どく",
    category="status",
    pp=20,
    target="foe_side",
    handlers={
        Event.ON_STATUS_HIT: h.MoveHandler(
            hs.どくびし_set_field,
        ),
    }
),
```

## 注意点・エッジケース

- `side.activate("どくびし", 1)` が最大 2 層を超えた場合に False を返すことを、サイドフィールドの実装で確認すること
- 1 層時はどく、2 層時はもうどくの判定はサイドフィールドの `ON_SWITCH_IN` ハンドラ側で処理されるため、技ハンドラ側でのガードは不要
- どくタイプが出た場合の層消去もサイドフィールドハンドラ側の責務
- どく・はがね・ひこうタイプ、ふゆう特性への無効化もサイドフィールドハンドラ側で処理
