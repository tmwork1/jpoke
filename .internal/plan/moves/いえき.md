# 計画書: いえき

## 仕様要約

- 技「いえき（Gastro Acid）」は相手の特性を「とくせいなし」volatile で無効化する変化技
- 効果が発動している間、相手は特性の効果を一切受けなくなる
- 「とくせいなし」volatile は交代・戦闘不能で解除される（ターン終了でのカウントダウンはない）
- 以下の「protected」フラグを持つ特性を持つポケモンに使用した場合は失敗する
  - ARシステム / アイスフェイス / うのミサイル / おもかげやどし / ぎょぐん / じんばいったい
  - スワームチェンジ / ぜったいねむり / ダルマモード / テラスチェンジ / バトルスイッチ
  - ばけのかわ / マイティチェンジ / マルチタイプ / リミットシールド
- 対象の持ち物が「とくせいガード」である場合も失敗する
  （`battle.ability_manager.is_change_blocked(ctx.defender)` で判定。
  シンプルビーム・なりきり・スキルスワップ等と同一パターン）
- 対象がすでに「とくせいなし」状態の場合も失敗する（`volatile_manager.apply` 内部で処理）

## 実装箇所

### `src/jpoke/handlers/move_status.py`

アンコール（ア）の次、いとをはく（い）の前の位置（五十音順）に実装済み。

```python
def いえき_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """いえきの効果: 相手に「とくせいなし」状態を付与する。"""
    return apply_volatile_to_defender(battle, ctx, value, volatile="とくせいなし")


def いえき_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """いえきの失敗条件を判定する。

    対象の特性が protected フラグを持つ場合、
    または対象の特性変更がとくせいガード等で防がれる場合は失敗する。
    """
    if (
        ctx.defender.ability.has_flag("protected")
        or battle.ability_manager.is_change_blocked(ctx.defender)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="いえき")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

- **「とくせいなし」二重付与の防止**: `volatile_manager.apply` の内部で `target.has_volatile(name)` をチェックしており、すでに「とくせいなし」状態の場合は自動的に `False` を返す。ハンドラ側の重複チェックは不要
- **「protected」フラグ対象**: `ctx.defender.ability.has_flag("protected")` で判定する。フラグは `AbilityData.flags` に定義されており、`Ability.has_flag()` で参照できる
- **とくせいガードとの相互作用**: `is_change_blocked` によるチェックが無いと、とくせいガード所持者に対しても「とくせいなし」volatile 自体は付与されてしまい、`ON_VOLATILE_START` の `add_ability_disabled_reason` 呼び出しだけが（とくせいガードの `ON_CHECK_ABILITY_DISABLE` ハンドラにより）ブロックされる状態になる。この場合、特性は無効化されないにもかかわらず「とくせいなし」volatile が残るため、以降のいえき使用（例: 道具を失った後の再使用）が `has_volatile` チェックにより誤って失敗し続けるバグになる。`いえき_can_apply` で `is_change_blocked` を明示チェックし、失敗させることで volatile 自体を付与しないようにする
- **特性が既に無効化されている場合**: 相手がかがくへんかガス等ですでに特性無効状態でも、いえきの失敗条件は「protected フラグ」「とくせいガード」「すでにとくせいなし」の3つのみ
- **交代による解除**: 「とくせいなし」volatile の解除は交代処理で行われる（volatile_manager が交代時に全 volatile を除去するため、ハンドラ側での実装は不要）
- **かたやぶり等との相互作用**: `protected` フラグを持つ特性はかたやぶりでも上書きできない（`.internal/spec/abilities/かたやぶり.md` に記載）。いえきも同様に失敗する
- **テストケース**:
  - 通常特性を持つ相手にいえき → 成功し「とくせいなし」状態になる
  - protected フラグ持ち（例: アイスフェイス）にいえき → 失敗する
  - とくせいガード所持者にいえき → 失敗し「とくせいなし」状態にならない
  - すでに「とくせいなし」状態の相手にいえき → 失敗する（volatile_manager 内部で弾かれる）
  - いえき後に交代した相手の特性が復活していることを確認
