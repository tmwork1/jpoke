# 実装計画書: リサイクル

## 仕様要約

参照: .internal/spec/moves/リサイクル.md

- タイプ: ノーマル / カテゴリ: 変化 / PP: 12（src/jpoke/data/ps-champ-ja/moves.json 準拠。
  本家Gen9基準は10だが、チャンピオンズは全技PPを8/12/16/20の4段階に統一しているため12を採用）
  / 対象: 自分（target="self"）
- 自分が最後に使用した（失った）持ち物を手に入れる
- アイテムをすでに持っている場合は失敗する
- `last_lost_item_name` が空文字列（アイテムをまだ失っていない）場合は失敗する
- 取り戻したアイテムが即時発動条件を満たす場合はその場で発動する（`gain_item` 経由で自動処理）

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100 | `attacker:self`（デフォルト） | 失敗条件チェック（アイテム所持中・未消失） |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | `last_lost_item_name` のアイテムを取り戻す |

## Priority 根拠

`.internal/spec/turn.md` の `Event.ON_APPLY_MOVE` の節に「リサイクル: 失敗条件」が priority 100 として明記されている
（同節の見出しは `ON_APPLY_MOVE` だが、実装上の enum 名は `Event.ON_BEFORE_APPLY_MOVE` であり、
なかまづくり・いえき・なりきり等の同節の他ハンドラもすべて `ON_BEFORE_APPLY_MOVE` に登録されている）。
`ON_STATUS_HIT` は変化技の効果発動イベントであり、既存の他変化技ハンドラと同一の priority 100（デフォルト）を使用する。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: `リフレクター_set_side_field` の直前（「リ」行内の五十音順で「リサイクル」< 「リフレクター」）。

```python
def リサイクル_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """リサイクルの失敗条件チェック。

    以下のいずれかに該当する場合は失敗する:
    - 使用者がすでにアイテムを持っている
    - 使用者がまだアイテムを失ったことがない（last_lost_item_name が空）
    """
    mon = ctx.attacker
    if mon.has_item() or not mon.last_lost_item_name:
        battle.add_event_log(
            mon,
            LogCode.MOVE_FAILED,
            payload={"reason": "リサイクル"},
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def リサイクル_restore_item(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """リサイクルの効果: 最後に失ったアイテムを取り戻す。"""
    mon = ctx.attacker
    battle.item_manager.gain_item(mon, mon.last_lost_item_name)
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ra.py`

既存の `"リサイクル"` エントリを更新する。handlers を追加する。

```python
"リサイクル": MoveData(
    type="ノーマル",
    category="status",
    pp=12,
    handlers={
        Event.ON_BEFORE_APPLY_MOVE: h.MoveHandler(
            hs.リサイクル_can_apply,
            priority=100,

        ),

```

## 注意点・エッジケース

- **アイテムを失ったことがないケース**: 初期からアイテムなしで出場したポケモンは `last_lost_item_name == ""` となるため失敗する。失敗条件チェックで `not mon.last_lost_item_name` を必ず確認すること。
- **アイテム所持中の失敗**: `item_manager.gain_item` 内部でも持っている場合は False を返すが、ログ出力のため `ON_BEFORE_APPLY_MOVE` ハンドラ側で先行チェックする。
- **`last_lost_item_name` の更新タイミング**: `item_manager._change_item` で `name=""` を渡した場合、かつ `track_loss=True`（デフォルト）のときのみセットされる。以下のケースでは `track_loss=False` を渡して意図的にトラッキングを止めている（本レビューで発見・修正）:
  - `swap_items`（トリック・すりかえ・どろぼう・ほしがる等）: 道具は場に存在したまま相手に渡るため。
  - `はたきおとす_remove_item` / `やきつくす_burn_item` / `ふしょくガス_remove_item`: 道具は場に存在したまま消滅する扱いのため。
  - `ふうせん_pop_on_hit`（`_announce_and_consume_item(..., track_loss=False)` 経由）: 攻撃を受けて割れたふうせんは復元対象にならないため（なげつけるで消費した場合は別経路のため対象になる）。
  - `むしくい_steal_and_use_berry`（`force_trigger_berry(attacker, track_loss=False)` 経由）: 相手から奪って消費したきのみは、自分の持ち物を失った扱いにならないため。
  - `force_trigger_berry` は個々のきのみハンドラを経由して `consume_item` を呼ぶため、`track_loss=False` 指定時は発動前後で `last_lost_item_name` を退避・復元することで記録を打ち消している（個々のきのみハンドラを変更せずに済ませるため）。
- **取り戻したアイテムの即時発動**: `item_manager.gain_item` -> `_change_item` の中で `ON_ITEM_GAINED` が発火されるため、取り戻し後のきのみ発動等は自動処理される。リサイクル側で追加処理は不要。
- **マジックルーム**: マジックルーム状態中にリサイクルを使用した場合、アイテムは取り戻せるが効果は無効化される（マジックルームの処理側で対応済み）。
- **ものひろい特性との違い**: ものひろい特性は `ON_TURN_END` で発動するが、リサイクルは技使用時に即時発動する。内部ロジックは共通（`last_lost_item_name` + `gain_item`）。
