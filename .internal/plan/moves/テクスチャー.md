# 実装計画書: テクスチャー

## 仕様要約

- タイプ: ノーマル / 分類: 変化 / 威力: - / 命中率: - / PP: 30 / 対象: 自分 / 優先度: 0 / 直接攻撃: ×
- 自分の覚えている技のうち、一番上の技（`moves[0]`）と同じタイプに自分のタイプを変更する（第六世代以降の仕様。本プロジェクトの対象世代はSVベースのためこの仕様を採用する）
- 一番上の技が「選ばれない技」（自分の現在のタイプのいずれかと同じタイプ）の場合は失敗する。第五世代までと異なり、次の技へのフォールバックは行わない
- タイプ判定には「選択画面に表示されている本来のタイプ」（技の基本データ上のタイプ）を用いる。そうでん状態や技のタイプを変える特性の影響、めざめるパワー等のタイプが変わる技の実際の発動タイプは考慮しない
- タイプ変更後はタイプ2がなくなる（単一タイプになる）
- タイプ変更後はもりののろい/ハロウィンによって追加されていたタイプの効果が消える
- 自分がテラスタルしているときは失敗する
- 交代・ひんし・戦闘終了でタイプは元に戻る
- 判定: 急所×、命中判定×（必中）、追加効果×、まもる×（自分対象のため防がれない）、おうじゃのしるし×、マジックコート×（自分対象のため跳ね返されない）、よこどり○（第五世代以降）

## 現状の実装状況

| 項目 | 状態 |
|------|------|
| `data/moves/move_ta.py` エントリ | 登録済み（`handlers={}`, `target` 未指定＝デフォルトの `"foe"` のまま） |
| `handlers/move_status.py` 関数 | 未実装 |

**バグ**: `target` が未指定のため `MoveData.target` のデフォルト値 `"foe"` が使われている。テクスチャーは自分対象の技（`target="self"`）であるべきで、この誤りにより `Move.is_blocked_by_protect`（`target == "foe"`）や `Move.is_reflectable`（`category == "status" and target in ("foe", "foe_side")`）が誤って `True` を返し、まもる・マジックコートに阻害されてしまう状態になっている。`target="self"` を明示的に設定する。

## ハンドラ構成

| イベント | priority | subject_spec | 役割 |
|----------|----------|--------------|------|
| `ON_BEFORE_APPLY_MOVE` | 100（デフォルト） | `attacker:self`（デフォルト） | テラスタル中・一番上の技が「選ばれない技」の場合に失敗 |
| `ON_STATUS_HIT` | 100（デフォルト） | `attacker:self`（デフォルト） | 一番上の技と同じタイプに自分のタイプを変更する |

## Priority 根拠

- `ON_BEFORE_APPLY_MOVE` priority 100: `.internal/spec/turn.md` の `Event.ON_APPLY_MOVE` テーブルに「テクスチャー: タイプ同じ」が priority 100 として明記されている。テラスタル失敗チェックも同じ関数内で扱うため同一 priority とする
- `ON_STATUS_HIT` priority 100（デフォルト）: turn.md に記載なし。ミラータイプ・ハロウィン・もりののろい等の同種タイプ変更ハンドラと同じデフォルト priority を採用する

## タイプ変更の実装方針

ミラータイプ（`.internal/plan/moves/ミラータイプ.md`）で新設済みの `Pokemon.move_override_types: list[Type] | None` をそのまま利用する（`types` プロパティの優先順位: テラスタル中 > `move_override_types` > `ability_override_type` > `volatile_override_type` > `data.types`）。

### 対象タイプの決定ロジック

```python
top_type = attacker.moves[0].data.type
```

- `moves[0]` は使用者が覚えている技リストの先頭（＝チーム編成順の一番上）
- `.data.type` を用いることで、そうでん状態や技のタイプを変える特性、めざめるパワー等の実行時タイプ変化の影響を受けない「本来のタイプ」を参照する

### 失敗条件の判定ロジック

```python
attacker.terastallized  # テラスタル中は失敗
top_type in attacker.types  # 一番上の技が自分の現在のタイプ（複数タイプ・added_types含む）のいずれかと同じなら失敗
```

`attacker.types` は `added_types`（もりののろい・ハロウィンの追加タイプ）を含めた現在のタイプを返すため、追加タイプも自動的に「所有するタイプ」として扱われる（仕様どおり）。

## 実装箇所

### `src/jpoke/handlers/move_status.py`

五十音順の挿入位置: 「て」行 — `つるぎのまい_modify_attacker_stats` の後、`てっぺき_modify_attacker_stats` の前に挿入する。

```python
def テクスチャー_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テクスチャーの失敗チェック。

    - 自分がテラスタルしているときは失敗する。
    - 自分が覚えている技のうち一番上の技（moves[0]）のタイプが、自分の現在のタイプ
      （added_types含む）のいずれかと同じ場合は失敗する。第六世代以降の仕様であり、
      次の技へのフォールバックは行わない。
    """
    attacker = ctx.attacker
    if attacker.terastallized:
        battle.add_event_log(
            attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="テクスチャー_テラスタル")
        )
        return HandlerReturn(value=False, stop_event=True)
    top_type = attacker.moves[0].data.type
    if top_type in attacker.types:
        battle.add_event_log(
            attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="テクスチャー_タイプ同じ")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def テクスチャー_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """テクスチャーの効果: 自分が覚えている技のうち一番上の技と同じタイプに
    自分のタイプを変更する（単一タイプになる）。

    タイプ変更後は、もりののろい・ハロウィンによる追加タイプ（added_types）はリセットされる。
    """
    attacker = ctx.attacker
    top_type = attacker.moves[0].data.type
    attacker.move_override_types = [top_type]
    attacker.added_types = []
    return HandlerReturn(value=value)
```

### `src/jpoke/data/moves/move_ta.py`

既存の `"テクスチャー"` エントリを修正する（`target="self"` を追加し、ハンドラを登録する）:

```python
"テクスチャー": MoveData(

    category="status",
    pp=30,
    target="self",

```

## 注意点・エッジケース

- **`target` フィールドのバグ修正**: 既存実装は `target` 未指定のためデフォルト値 `"foe"` になっており、まもる・マジックコートに誤って阻害される状態だった。`target="self"` を明示することで修正する
- **`move_override_types` はミラータイプで新設済み**: `model/pokemon.py` の変更は不要（既存のフィールド・`types` プロパティ分岐・`reset_on_switch_out` でのリセットをそのまま利用する）
- **テクスチャー自身が一番上の技になりうる**: 第六世代以降はテクスチャー自身も「選ばれない技」から除外されている（選ばれる）。`moves[0]` がテクスチャー自身であっても、その `data.type`（ノーマル）が自分の現在のタイプと異なれば成功する
- **本プロジェクトの対象範囲は第一世代〜第五世代の分岐仕様（相手コピー・ランダム選択）を扱わない**: SVベースのため常に「一番上の技と同じタイプ」の第六世代以降ルールのみを実装する
- **スケッチ・のろいの除外は対象外**: いずれも第六世代以降では通常どおり選ばれるため、`data.type` を素直に参照するだけでよく、個別の除外ロジックは不要
- **タイプ無し技（めざめるダンス等）の扱いは第九世代のテクスチャーには言及がない**: `moves[0]` がタイプ無し技になるケースは通常起こりえない（テクスチャー自体が常にノーマルタイプの技として存在するため、少なくとも1つは通常のタイプを持つ技を選択画面に表示する前提だが、`moves[0]` の位置に来るかは編成次第）。本仕様書に明記がないため、実装では `data.type` をそのまま使用し、特別な例外処理は行わない
