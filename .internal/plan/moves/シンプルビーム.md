# 計画書: シンプルビーム

## 仕様要約

- タイプ: ノーマル / 変化技 / PP 16（champions/move_list.txt準拠） / 命中 100
- 効果: 相手の特性を「たんじゅん」に書き換える
- 「たんじゅん」特性の効果: 自分のランク変化の段階数が 2 倍になる（上昇・下降ともに）
  （実装済み。`data/ability.py` の「たんじゅん」エントリ、
  `handlers/ability.py` の `たんじゅん_double_stat` を参照）
- 失敗条件（`ON_BEFORE_APPLY_MOVE` priority 100）:
  - 対象の特性が「なまけ」である場合（`protected` フラグを持たないため個別チェックが必要）
  - 対象の特性が `protected` フラグを持つ場合
  - 対象の特性がすでに「たんじゅん」である場合
  - 対象の特性変更がとくせいガード等で防がれる場合
    （`battle.ability_manager.is_change_blocked`）

詳細は `.internal/spec/moves/シンプルビーム.md` を参照。

## 実装箇所（参考: 実装済みコード）

### `handlers/move_status.py`（五十音順: シ の位置）

```python
def シンプルビーム_can_apply(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """シンプルビームの失敗条件チェック。

    以下の場合は失敗する:
    - 対象の特性が「なまけ」である（`なまけ` には `protected` フラグが
      付与されていないため、`base_name` での個別チェックが必要）
    - 対象の特性が「上書きできない特性」（`protected` フラグ）を持つ
    - 対象の特性がすでに「たんじゅん」である
    - 対象の特性変更がとくせいガード等で防がれる
    """
    defender_ability = ctx.defender.ability.base_name
    if (
        defender_ability == "なまけ"
        or ctx.defender.ability.has_flag("protected")
        or defender_ability == "たんじゅん"
        or battle.ability_manager.is_change_blocked(ctx.defender)
    ):
        battle.add_event_log(
            ctx.attacker, LogCode.MOVE_FAILED,
            payload=FailureLogPayload(move=ctx.move.name, display_reason="シンプルビーム")
        )
        return HandlerReturn(value=False, stop_event=True)
    return HandlerReturn(value=value)


def シンプルビーム_change_ability(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """シンプルビームの効果: 相手の特性を「たんじゅん」に書き換える。"""
    battle.change_ability(ctx.defender, "たんじゅん")
    return HandlerReturn(value=value)
```

## 注意点・エッジケース（review-test 向け）

### 失敗条件

- 対象の特性が「なまけ」の場合は失敗する（なかまづくりと同じ個別チェックパターン）。
- 対象の特性が `protected` フラグを持つ場合は失敗する（いえき・なりきりと同じ判定）。
- 対象の特性がすでに「たんじゅん」の場合は失敗する
  （`battle.change_ability` 内部の `if mon.ability.base_name == ability: return`
  ガードに処理を委ねるのではなく、`シンプルビーム_can_apply` 側で明示的に失敗させ、
  `MOVE_FAILED` ログを出す）。
- 対象が「とくせいなし」状態（`volatile`）の場合でも特性オブジェクト自体は残るため、
  `ability.has_flag("protected")` や `base_name` 判定は通常通り動作する。

### 特性変更の副作用

- `battle.change_ability` が呼ばれると `ON_ABILITY_DISABLED` → ハンドラ解除 →
  `ON_ABILITY_ENABLED` の順でイベントが発火し、「たんじゅん」ハンドラが登録される。
- 交代すると `base_ability_name` に戻る（`change_ability` は `base_name` も更新しない）ため、
  交代後は元の特性に戻る。

### マジックコート・マジックミラー

- シンプルビームは変化技かつ対象が相手のため、マジックコート・マジックミラーで
  反射される対象となる。
- 反射処理は変化技に対する汎用処理（manager 内部）が扱うため、ハンドラ側で
  追加ガードは不要。反射後は使用者自身が対象になり、失敗条件が使用者の特性で
  再評価される。

### かがくへんかガス中の挙動

- かがくへんかガス中の場合、対象の特性は機能しないが特性オブジェクト自体は変わらない。
- `protected` フラグの判定はオブジェクトに基づくため、かがくへんかガス下でも
  `protected` 特性への使用は失敗する。
