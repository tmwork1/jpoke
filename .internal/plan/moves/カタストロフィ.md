# カタストロフィ 実装計画書

## 仕様要約（.internal/spec/moves/カタストロフィ.md 参照）

* タイプ: あく / 分類: 特殊 / 威力: -（固定ダメージ） / 命中: 90 / PP: 10
* 対象: 相手1体 / 優先度: 0 / 直接攻撃なし
* 効果: 相手の残りHP×1/2（切り捨て）の固定ダメージ。相手の残りHPが1の場合は1ダメージ。
* 判定: 急所×、命中判定○、追加効果×、まもる○、おうじゃのしるし(第五世代以降)○、マジックコート×

`いかりのまえば`（ノーマル・物理・直接攻撃あり）と全く同じダメージ計算方式（現在HPの半分固定ダメージ）を
共有する技であり、`.internal/spec/moves/カタストロフィ.md` の「技の仕様」節でも
`いかりのまえば#技の仕様` を参照するよう明記されている。

## 現状の実装状況

`src/jpoke/data/moves/move_ka.py` に既に登録済み。

```python
"カタストロフィ": MoveData(
    type="あく",
    category="special",
    pp=10,
    power=0,
    accuracy=90,
    flags={"fixed_damage"},
    handlers={
        Event.ON_MODIFY_MOVE_DAMAGE: h.MoveHandler(
            ha.half_damage,
            subject_spec="attacker:self",
        )
    }
),
```

`handlers/move_attack.py` の `half_damage`（既存の共通ハンドラ。いかりのまえばと共用）:

```python
def half_damage(battle: Battle, ctx: AttackContext, value: Any) -> HandlerReturn:
    """対象の現在HPの半分を与える固定ダメージを計算する。"""
    return HandlerReturn(value=max(1, ctx.defender.hp // 2))
```

### 今回のレビューで修正した点

**`priority=15` の追加（重要・挙動修正）**: `いかりのまえば`（`.internal/plan/moves/いかりのまえば.md`）で
既に修正済みの問題が、`half_damage` を共用する `カタストロフィ` 側には未反映のまま残っていた。
デフォルトの `priority=100` のままだと、`みがわり`（`ON_MODIFY_MOVE_DAMAGE` priority=20,
`data/volatile.py`）や `こらえる`（`ON_MODIFY_MOVE_DAMAGE` priority=95, `data/volatile.py`）よりも
**後に** `half_damage` が実行されてしまい、以下の実害がある。

- みがわり状態の相手に使用すると、`みがわり_block_damage`（priority=20）が
  先に「入力値0」を見て「ダメージ0」と判断し、みがわりのHPを消費しない。
  その後 `half_damage`（priority=100）が本体HPの半分を再計算し、
  その値がそのまま本体の実HPに適用されてしまう
  （みがわりが存在するのに本体が直接ダメージを受ける、という誤動作）。
- こらえる状態で残りHP1のときに使用すると、`こらえる_endure`（priority=95）が
  先に「入力値0」を見て何もせず、その後 `half_damage`（priority=100）が
  1ダメージを計算してそのまま適用し、こらえるが機能せず瀕死になる。

これは「固定ダメージ技は通常のダメージ計算式（`calc_damages`）を経由せず、
本来ダメージ計算前に確定しているはずの基礎ダメージ値を
`ON_MODIFY_MOVE_DAMAGE` ハンドラ内で初めて算出する」という実装方式に起因する。
そのため `half_damage` は「基礎ダメージの確定」という役割を持ち、
みがわり・こらえる等の「確定済みダメージへの修正」ハンドラより
**前に**実行される必要がある。

`.internal/spec/turn.md` の `ON_MODIFY_DAMAGE` 節にはこのケース（固定ダメージ技の
基礎ダメージ確定）は掲載されていないため、既存の同種ハンドラを参照して
priority を決定した（`いかりのまえば` と同一の根拠）。
- `みがわり`: priority=20（`data/volatile.py`）
- `こらえる`: priority=95（`data/volatile.py`）
- `ばけのかわ`: priority=10（`data/ability.py`、同じく「確定済み値を
  無視して上書きする」ハンドラ。ミミッキュはゴーストタイプのため
  あくタイプのカタストロフィとは実際には競合しない）

以上を踏まえ、`みがわり`(20) より前・`ばけのかわ`(10) と競合しない値として
`priority=15` を採用した（`いかりのまえば` と同値）。これにより、
- みがわり状態の相手に使用時: 本体HP基準の正しい半減ダメージが
  `half_damage`(15) で確定 → `みがわり_block_damage`(20) が
  みがわりのHPを上限として肩代わりし、超過分は本体に持ち越されない
  （本体HPは変化しない）。
- こらえる状態で使用時: 正しいダメージが先に確定 → `こらえる_endure`(95) が
  瀕死を防ぎ1HP残す。
という、本来の挙動に修正された。
（がんじょう・きあいのタスキは priority=100 だが、カタストロフィの
ダメージは常に現在HPの半分以下であり満タンHPから瀕死にはならないため、
この2つとの実害のある競合は生じない。）

## Priority 根拠

`.internal/spec/turn.md` に `ON_MODIFY_MOVE_DAMAGE`（固定ダメージ技のダメージ確定）は
個別掲載されていない未掲載イベントのため、既存の同種ハンドラ
（`みがわり`=20, `こらえる`=95, `ばけのかわ`=10）を参照し、
`みがわり`より前に基礎ダメージを確定させる必要から `priority=15` とした
（`いかりのまえば`と同一の根拠・同一の値）。

## 注意点・エッジケース（review-test 向け）

1. **ゴーストタイプへの無効化は非該当**
   - カタストロフィはあくタイプの技であり、いかりのまえば（ノーマル技）と異なり
     ゴーストタイプに対しても通常通り命中しダメージを与える。無効タイプなし。

2. **みがわり・こらえるとの相互作用**
   - 上記の priority 修正により正しく動作することをテストで確認済み。

3. **がんじょう・きあいのタスキ**
   - カタストロフィは現在HPの半分以下しかダメージを与えられないため、
     満タンHPから瀕死になることはなく、実害のある競合は発生しない。

4. **せいぎのこころ・びびり**
   - あくタイプの技であるため、`せいぎのこころ`（あく技を受けると攻撃ランク上昇）・
     `びびり`（あく技を受けるとすばやさランク上昇）を発動させる点が
     いかりのまえば（ノーマル技）との違いとして備考に記載されている。
     これらの特性は既存の共通ハンドラ（`ability.py`）で処理されるため、
     カタストロフィ側で個別対応は不要。
