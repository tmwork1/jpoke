# へんしん（揮発状態）実装計画書

> 本ファイルは、既存実装（技へんしん・特性かわりものの実装過程で先に作られた
> `battle.transform()` 基盤）を事後的に整理した計画書としてレビュー時に新規作成した。
> 実体の設計判断は `.internal/plan/moves/へんしん.md`「実装済み（追記）」節・
> `.internal/plan/abilities/かわりもの.md`「実装済み（追記）」節も参照。

## 概要

技「へんしん」・特性「かわりもの」が共有する変身機構のうち、「対象が変身中かどうか」を
判定するためのマーカー専用の揮発状態を実装する。変身の実体（コピー処理・退場時の復元）は
揮発状態のハンドラではなく `Battle.transform()`（`core/battle.py`）と
`Pokemon.reset_on_switch_out()`（`model/pokemon.py`）に実装する。

## 実装済み

### 1. 「へんしん」揮発状態のデータ定義（ハンドラなし）

**ファイル**: `data/volatile.py`

```python
"へんしん": VolatileData(),
```

ハンドラを一切持たない。`attacker.has_volatile("へんしん")` /
`defender.has_volatile("へんしん")` として、技へんしん・特性かわりものの失敗条件判定
（`handlers/move_status.py` の `へんしん_can_apply`、`handlers/ability.py` の
`かわりもの_transform_to_opponent`）からのみ参照される。

### 2. `Battle.transform(source, target)`（コピー処理の実体）

**ファイル**: `core/battle.py`

以下を `source` にコピーする。

- 特性: `source.ability.base_name != target.ability.name` のとき（＝実際に特性名が
  変わる場合）のみ、場に出ている場合は `ON_ABILITY_DISABLED` を発火してから
  `unregister_handlers`、特性を差し替えた後 `register_handlers` して `ON_ABILITY_ENABLED`
  を発火する。特性名が変わらない場合はこれらを一切行わない
  （無限再帰防止。「詳細な仕様」節を参照）。
- 技: `source.pre_transform_moves`（switchスコープの`memory`で管理するプロパティ）に
  変身前の技リストを退避し、`target.moves` の技名で `Move` を作り直す。PPは元の技の最大PPが
  1の場合のみ1、それ以外は一律5にする。
- 性別: `source.pre_transform_gender` に退避してから `target.gender` をコピーする。
- タイプ・体重: `source.data`（種族データ）自体は変更せず、`source.transform_types`/
  `source.transform_weight`（いずれもswitchスコープの`memory`で管理するプロパティ）に
  コピー先の値を保持し、`Pokemon.types`/`Pokemon.weight` のプロパティ算出時にこれらが
  `None` でなければ優先して使う。
- 実数値ステータス: `source.set_raw_stat(idx, target.get_raw_stat(idx))` をHPを除く5ステータス
  （idx=1..5）に対して行う。
- 能力ランク補正: `source.boosts = dict(target.boosts)` で丸ごと置き換える。

最後に `self.volatile_manager.apply(source, "へんしん")` で揮発状態を付与する。

成功条件・失敗条件（対象がみがわり状態・へんしん状態・オーガポン/テラパゴス関連の
テラスタル例外等）の判定は本メソッドの責務ではなく、呼び出し元
（`へんしん_can_apply`・`かわりもの_transform_to_opponent`）が行う。

### 3. `Pokemon.reset_on_switch_out()`（退場時の復元）

**ファイル**: `model/pokemon.py`

- `pre_transform_moves`が`None`でなければ、`moves`/`gender`を変身前の値に戻す。
- `memory["switch"]`スコープを丸ごとクリアすることで、`transform_types`/`transform_weight`/
  `pre_transform_moves`/`pre_transform_gender`もまとめてリセットされる
  （タイプ・体重のプロパティはこれらが`None`のとき`self.data`基準の値にフォールバックする）。
- 特性・実数値ステータス・能力ランクは、同メソッドが既に持つ交代時の既存リセット処理
  （`ability`を`base_ability`へ戻す・`update_stats()`で種族値ベースの実数値へ再計算・
  `boosts`を全て0に戻す）がそのまま兼ねるため、変身専用の追加処理は不要。

`reset_on_switch_out()` は `SwitchManager._switch_out()` から交代・瀕死交代の両方で
呼ばれるため、死に出しでも正しく変身前の状態へ復元される。

### 4. 呼び出し元の失敗条件判定・変身実行

**ファイル**: `handlers/move_status.py`（へんしん）・`handlers/ability.py`（かわりもの）

- `へんしん_can_apply`（`ON_BEFORE_APPLY_MOVE`, priority=100 = `MoveHandler` 既定値）:
  自身が既にへんしん状態／対象がみがわり状態／対象がへんしん状態／
  オーガポン・テラパゴス関連のテラスタル例外／自身のステラタイプテラスタルを判定する。
- `へんしん_apply`（`ON_STATUS_HIT`, priority=100 = `MoveHandler` 既定値）:
  `battle.transform(ctx.attacker, ctx.defender)` を呼ぶのみ。
- `かわりもの_transform_to_opponent`（`Event.ON_SWITCH_IN` のみ。優先度は
  `AbilityHandler` 既定値=100。他の場に出た時特性と同じ扱い）:
  正面の相手の不在・みがわり／へんしん状態・テラスタル関連例外を判定してから
  `battle.transform(mon, target)` を呼ぶ。

## 本レビューで修正した実装上の問題（2件）

いずれも `battle.transform()` と「かわりもの」特性の相互作用に起因する問題で、
揮発状態「へんしん」自体の実装には変更なし。

### 問題1: 無限再帰（スタックオーバーフロー）

修正前は `transform()` が特性名の変化有無に関わらず無条件で `ON_ABILITY_DISABLED`/
`ON_ABILITY_ENABLED` を発火していた。技へんしん・特性かわりもので、変身先（コピー元）の
特性が「かわりもの」だった場合、変身した側も特性「かわりもの」を獲得し、その
`ON_ABILITY_ENABLED`（修正前はここにも登録されていた）が発火して
`かわりもの_transform_to_opponent` が再度呼ばれ、`transform()` が再帰的に呼ばれる。
2回目の呼び出しでも特性名は変わらず（両者とも既に「かわりもの」）、修正前の実装では
そこでも無条件に `ON_ABILITY_ENABLED` を発火してしまうため、無限再帰でクラッシュしていた
（例: `へんしん`使用者が特性「かわりもの」を持つ相手に変身する、というだけで再現する）。

**修正**: `transform()` に「特性名が実際に変わる場合のみ」ガードを追加
（`AbilityManager.change_ability` の「特性名が同じなら何もしない」という既存の規約と統一）。

### 問題2: スキルスワップ/さまようたましいでかわりものを得た側が誤って変身する

修正前は「かわりもの」が `Event.ON_ABILITY_ENABLED` にも登録されていたため、
スキルスワップ（`AbilityManager.swap_ability`）やさまようたましい（同じく`swap_ability`を
使う）で「かわりもの」を新たに獲得した側が、獲得直後に相手へ変身してしまっていた。
`.internal/spec/abilities/かわりもの.md`「スキルスワップ/へんしん、さまようたましいに
よりこの特性を得ることはできるが、特性の効果は発動しない」という明記された仕様に反する。

**修正**: 「かわりもの」の登録を `Event.ON_SWITCH_IN` のみに限定し、
`Event.ON_ABILITY_ENABLED` から削除した（`data/ability.py`）。
