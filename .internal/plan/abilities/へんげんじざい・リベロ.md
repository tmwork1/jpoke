# へんげんじざい・リベロ 実装計画

## スコープ
- 優先度: H
- 仕様書: `.internal/spec/abilities/へんげんじざい.md`, `.internal/spec/abilities/リベロ.md`
- 実装方針: 同一効果のため、共通ハンドラ1本を `へんげんじざい` と `リベロ` の両方に登録する。

## 実装ゴール（完了定義）
- 技実行直前に、使用者のタイプがその技タイプへ変化する。
- 1回の場滞在につき1回だけ発動する（SV仕様）。
- 行動不能や技不発時は発動しない。
- 交代でタイプ変化状態と発動回数がリセットされる。
- `へんげんじざい` と `リベロ` で挙動差が出ない。

## 現状整理（実装済み）
- `src/jpoke/data/ability.py` の `へんげんじざい` / `リベロ` は、共通ハンドラ `h.へんげんじざい_change_type`（`src/jpoke/handlers/ability.py`）を
  `Event.ON_MOVE_CHARGE`（`subject_spec="attacker:self"`, `priority=100`）に登録している。
- 行動フローは `src/jpoke/core/move_executor.py` の `run_move()` → `_execute_move()` で進み、
  `Event.ON_TRY_ACTION`（行動可否判定） → `Event.ON_MOVE_CHARGE`（本ハンドラ） → `Event.ON_TRY_MOVE_1` の順に発火する。
  行動不能（ねむり・まひ・ひるみ等）は `ON_TRY_ACTION` で弾かれるため、`ON_MOVE_CHARGE` に到達しない。
- `src/jpoke/model/pokemon.py` の `Pokemon.ability_override_type`（`memory["switch"]` スコープ）に一時タイプを保持し、
  `types` プロパティで `テラスタル > move_override_types > ability_override_type > volatile_override_type > 元タイプ` の
  優先順で参照する。`memory["switch"]` は交代時にリセットされるため、交代でタイプ変化状態は自動的に戻る。
- 1回の場滞在につき1回という制限は、汎用フィールド `Ability.activated_since_switch_in`（`src/jpoke/model/ability.py`）を用いて管理する
  （同フィールドは他の「場に出てから1回だけ発動」特性〈おもかげやどし等〉とも共用するが、常にどちらか一方の特性しか装備できないため競合しない）。
  特性が書き換わる・スキルスワップで交換されるといった `AbilityManager.change_ability` / `swap_ability` は毎回新しい `Ability` インスタンスを
  生成するため、このフラグは自然にリセットされ、「特性を再取得すると再発動できる」という仕様を満たす。
  一方 `かがくへんかガス` 等による無効化はインスタンスを差し替えないため、無効化解除後も再発動しない（仕様どおり）。

## 仕様反映ポイント
- 発動タイミングは「行動成立後、威力/一致補正計算前」に固定する。
- 発火イベントは `.internal/spec/turn.md` に合わせて `Event.ON_MOVE_CHARGE` を採用する。
- テラスタル中は非発動（仕様書どおり）にする。
- 変化技も発動対象に含める。
- `かがくへんかガス` による特性無効化中は発動しない。
- すでにタイプ変化が成立した後に `かがくへんかガス` で特性が無効化されても、その場では元タイプに戻さない（交代時リセットのみ）。

## 変更対象（ファイル単位・実装済み）
- `src/jpoke/model/pokemon.py`
  - 特性由来の一時タイプを保持するプロパティ `ability_override_type`（`memory["switch"]` スコープ）。
  - `types` プロパティで `テラスタル > move_override_types > ability_override_type > volatile_override_type > 元タイプ` の順に参照する。
  - `added_types` / `removed_types` も同じく `memory["switch"]` スコープのため、交代で自動的に初期化される。
  - 特性の有効/無効切り替え（かがくへんかガス等）では一時タイプを変更しない（解除は交代時のみ）。
- `src/jpoke/handlers/ability.py`
  - 共通ハンドラ `へんげんじざい_change_type` を実装。
  - 発動条件: 技タイプが存在する（わるあがき等の無タイプ技を除外）／未発動／非テラスタル／現在タイプと技タイプが完全一致していない。
  - 発動時に `ability_override_type` を技タイプへ設定し、`added_types`（もりののろい・ハロウィン等）をリセットして単タイプにする。
  - `ability.activated_since_switch_in = True` で以後の再発動を防ぐ。
- `src/jpoke/data/ability.py`
  - `へんげんじざい` / `リベロ` の両方に同一ハンドラを `Event.ON_MOVE_CHARGE`（`subject_spec="attacker:self"`, `priority=100`）へ登録。
- `src/jpoke/data/moves/move_ma.py`（もえつきる） / `src/jpoke/data/moves/move_ta.py`（でんこうそうげき）
  - 「使用者が該当タイプを持たない場合は失敗する」判定を `Event.ON_TRY_MOVE_1`（へんげんじざい発動〈`ON_MOVE_CHARGE`〉より後）から
    `Event.ON_TRY_ACTION`（`priority=15`, へんげんじざい発動より前）へ移動。
    これにより「もえつきる/でんこうそうげきが元タイプ不足で失敗したときはへんげんじざいが発動しない」という
    仕様（`.internal/spec/abilities/へんげんじざい.md` 該当行）を満たす。
    移動前は `Event.ON_MOVE_CHARGE` が先に発火するため、失敗するはずの技でへんげんじざいがタイプを変え、
    結果的に判定をすり抜けて技が成功してしまう不具合があった。
- `tests/abilities/test_ability_ha.py` / `tests/abilities/test_ability_yarawa.py`
  - へんげんじざい・リベロの発動/非発動/リセット/相互作用テストを追加。

## テスト計画（実装済み）
- 発動ケース: 異なるタイプの攻撃技・変化技で先にタイプが変化する。
- 発動ケース: 複合タイプの片方が技タイプと一致していても単タイプに変化する。
- 発動ケース: もりののろいで追加されたタイプは発動時にリセットされる。
- 非発動ケース: 単タイプで既に技タイプと一致している場合は変化しない。
- 非発動ケース: 同一場滞在2回目は発動しない。
- 非発動ケース: テラスタル中は発動しない。
- 非発動ケース: `かがくへんかガス` 展開中は発動しない。
- 非発動ケース: わるあがき（タイプなし技）では発動しない。
- 非発動ケース: もえつきる/でんこうそうげきが元タイプ不足で失敗したときは発動しない。
- 回復ケース: 交代後に再度1回だけ発動できる。
- 同一性ケース: 同条件で `へんげんじざい` と `リベロ` の結果が一致する。

## リスクと対策
- リスク: 発火イベントが早すぎると、行動失敗時にもタイプが変わる。
  - 対策: `Event.ON_TRY_ACTION`（行動可否判定）で弾かれた場合は `Event.ON_MOVE_CHARGE` に到達しないため、
    行動不能時は自然に発動しない。
- リスク: もえつきる/でんこうそうげきのように「使用条件としてのタイプ判定」を持つ技で、
  判定がへんげんじざいのタイプ変化より後（`Event.ON_TRY_MOVE_1`）に発火すると、
  本来失敗するはずの技が成功してしまう。
  - 対策: 該当する技側の判定を `Event.ON_TRY_ACTION`（へんげんじざい発動より前）へ移動する。
- リスク: 一時タイプ状態のリセット漏れで交代後にタイプが残留する。
  - 対策: `ability_override_type` は `memory["switch"]` スコープに保持し、
    `reset_on_switch_in()` / `reset_on_switch_out()` の双方でクリアされることを交代テストで検出する。
- リスク: 特性有効状態の再計算時に、タイプ変化まで巻き戻してしまう。
  - 対策: タイプ変化状態は `ability.enabled`（かがくへんかガス等による無効化）と独立して保持し、
    解除トリガーを交代時に限定する。
- リスク: テラスタル中のタイプ計算と干渉する。
  - 対策: `types` の優先順を `テラスタル > move_override_types > 特性上書き > volatile上書き > 元タイプ` に固定し、
    回帰テストで担保する。
- リスク: もりののろい/ハロウィンによる追加タイプ（`added_types`）が、特性発動後も
  「技タイプ + 追加タイプ」として残ってしまう。
  - 対策: 発動時に `added_types` を明示的にリセットする。
