## スコープ

- 優先度: H
- 仕様書: `.internal/spec/abilities/なまけ.md`
- 実装状態: 実装済み（本レビューで ON_TRY_ACTION の priority を修正）
- 実装方針: ポケモンごとに `mon.ability.state`（`"can_act"` / `"skip_next"`）を管理し、
  `ON_TRY_ACTION` で `state == "skip_next"` のとき行動をスキップして `"can_act"` に戻し、
  `state == "can_act"` のときは行動を許可しつつ `"skip_next"` に反転させる。

## 実装ゴール（完了定義）

- 場に出た最初のターンは行動でき、次のターンはなまけて行動不能になる（以降交互）。
- なまけるターンに「なまけている!」メッセージが表示される。
- ねむり・こおり状態で行動できないターンは X を消費しない（第五世代以降仕様）。
- 既存テストを壊さない。

## 実装対象イベント

- `Event.ON_SWITCH_IN` / `Event.ON_ABILITY_ENABLED` — `なまけ_init` で状態を `"can_act"` に初期化する
  （スキルスワップ・かがくへんかガス解除等で特性が有効化されたタイミングでも同様にリセットされる）
- `Event.ON_TRY_ACTION` — 行動時に状態を確認し、`"skip_next"` のとき行動をスキップして `"can_act"` に反転させる

## Priority

| イベント | Priority | 根拠 |
|---|---|---|
| ON_TRY_ACTION | 15 | 一次情報「なまけているメッセージが現れるタイミングはねむり/こおり判定の後、まひの痺れ/こんらん/ひるみ判定の前になる」 |

turn.md の `Event.ON_CHECK_ACTION`（実装上は `Event.ON_TRY_ACTION` に対応）には
「10 | なまけのカウント消費」「10 | ねむりのカウント消費」「10 | こおりの回復判定」が
同じ priority=10 として並記されているが、これはタイブレークの実行順を保証しない
（`EventManager._sort_handlers` は `(priority, -speed)` でソートし、同点の場合は
登録順というハンドラ内部の実装詳細に依存する）。

一次情報は「なまけているメッセージはねむり/こおり判定の後に出る」と明記しており、
かつ「ねむり/こおり状態で行動できない間、第五世代以降はXを消費しない」ため、
`ねむり_check_action`（`src/jpoke/data/ailment.py`, priority=10）・`こおり_action`
（同, priority=10）が stop_event するときは、その後段の `なまけ_try_action` が
一切実行されない必要がある。両者と同点の priority=10 のままだと、ability 側の
ハンドラが ailment 側より先に登録される実装（`はやおき` の
`# ねむりカウント消費 (priority=10) の直前` というコメントと同様の前例がある）
のため、なまけが先に走ってしまい眠り・氷結中でも X が誤って消費されるバグが
実際に再現した（本レビューで発見・修正）。そのため `ねむり`/`こおり`(10) より
後、`PPが残っていない`(20) より前になるよう priority=15 とする。

なお `リチャージ`（技の反動で動けない、`src/jpoke/data/volatile.py` の
`リチャージ_block_action`、priority=10）との組み合わせ（なまけの行動可能ターンに
ギガインパクト等を使い反動で動けなくなる場合）は、一次情報上は「反動が優先され、
かつXは消費される」とされているが、`リチャージ_block_action` が stop_event する
実装のため、本レビューの範囲（なまけ単体）では未対応。今後 `リチャージ` /
`きょけんとつげき` 側の見直しが必要になった際に併せて対応する。

## 変更対象（ファイル/関数単位）

- `tests/abilities/test_ability_na.py` — テスト

## 実装ステップ（実装済み。以下は経緯の記録）

1. 仕様書 `.internal/spec/abilities/なまけ.md` で X の消費ルール（ねむり/こおり時は消費しない等）を確認する。
2. `mon.ability.state`（`AbilityState`。`"can_act"` / `"skip_next"`）で管理する。
3. `なまけ_try_action` を実装する（`state == "skip_next"` のとき「なまけている!」ログを出して行動をキャンセルし `"can_act"` に戻す、`state == "can_act"` のときは行動を許可しつつ `"skip_next"` に反転させる）。
4. `ON_SWITCH_IN` / `ON_ABILITY_ENABLED` で X を初期化する handler を登録する。
5. `src/jpoke/data/ability.py` の `なまけ` に handlers を登録する。
6. テストを追加し、全テストが通ることを確認する。

## テスト観点

1. 1 ターン目は行動でき、2 ターン目はなまけて行動不能になる。
2. なまけるターンに「なまけている!」メッセージが出る。
3. ねむり状態のターンは X を消費しない（起きたターンに行動でき、その後は行動→なまけの交互パターンに戻る）。

## リスクと対策

- リスク: 交代やかがくへんかガスによる X リセット漏れ。
  - 対策: `ON_SWITCH_IN` と `ON_ABILITY_ENABLED` で必ず X を初期化する。
- リスク: ねむり/こおりの行動不能判定との priority 同点によるタイブレーク依存。
  - 対策: 上記の通り priority=15 とし、ねむり/こおりの判定(priority=10)より後に実行されるようにする。
