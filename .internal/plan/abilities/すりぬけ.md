## 効果まとめ（第9世代・チャンピオンズシングル）

| 効果                                   | 対象フィールド                               | 実装 |
| -------------------------------------- | --------------------------------------------- | ---- |
| 壁のダメージ軽減を無視                 | リフレクター / ひかりのかべ / オーロラベール | ✓    |
| しんぴのまもりによる状態異常無効を無視 | しんぴのまもり                               | ✓    |
| しんぴのまもりによるこんらん・ねむけ無効を無視 | しんぴのまもり                       | ✓    |
| しろいきりによる能力低下ガードを無視   | しろいきり                                   | ✓    |
| みがわりを無視して攻撃できる           | みがわり（volatile）                         | ✓    |

## 設計方針

`CLAUDE.md` の責務設計原則に基づき、**効果を所持している側（フィールド／volatile 側）** が問い合わせ、
**すりぬけ側**が回答するイベント駆動の疎結合設計にする。

- フィールド・volatile 側は `ctx.can_bypass_screen(battle)` / `ctx.can_bypass_status_guard(battle)` /
  `battle.move_executor.check_hit_substitute(ctx)` で問い合わせるだけ
- すりぬけ側が対応するイベントに `True`（貫通する）を返すハンドラを登録する
- `field.py` / `volatile.py` が特性名 `"すりぬけ"` をハードコードしない
- 将来同挙動の特性が追加されても `data/ability.py` に追加するだけで済む

対象イベントは効果の種類ごとに 3 つに分かれる（context 型が異なるため）。

| イベント                       | 発火元                              | context 型      | 既定値  |
| ------------------------------ | ------------------------------------ | ---------------- | ------- |
| `ON_CHECK_BYPASS_SCREEN`       | `AttackContext.can_bypass_screen`    | `AttackContext`  | `False` |
| `ON_CHECK_BYPASS_STATUS_GUARD` | `EventContext.can_bypass_status_guard` | `EventContext` | `False` |
| `ON_CHECK_HIT_SUBSTITUTE`      | `MoveExecutor.check_hit_substitute`  | `AttackContext`  | `True`（みがわりに当たる＝貫通しない） |

## 実装内容（現状）

### ① `src/jpoke/enums/event.py`

```python
# emit: AttackContext（リフレクター等の壁を貫通するか問い合わせ）
# handle: ability.py（すりぬけ等）subject_spec="attacker:self"
ON_CHECK_BYPASS_SCREEN = auto()

# emit: EventContext（しんぴのまもり・しろいきり等の耐性を貫通するか問い合わせ）
# handle: ability.py（すりぬけ等）subject_spec="source:self"
ON_CHECK_BYPASS_STATUS_GUARD = auto()

# emit: core/move_executor.py（みがわりへのヒット可否。音技・bypass_substitute
#       フラグ技はこのイベントより前に move_executor 側で直接判定される）
# handle: ability.py（すりぬけ等）subject_spec="attacker:self"
ON_CHECK_HIT_SUBSTITUTE = auto()
```

### ② `src/jpoke/core/context.py`

```python
# EventContext
def can_bypass_status_guard(self, battle: Battle) -> bool:
    """発動元がしんぴのまもり・しろいきり等の耐性を貫通するかを返す。"""
    return battle.events.emit(Event.ON_CHECK_BYPASS_STATUS_GUARD, self, False)

# AttackContext
def can_bypass_screen(self, battle: Battle) -> bool:
    """攻撃側がリフレクター・ひかりのかべ等の壁を貫通するかを返す。"""
    return battle.events.emit(Event.ON_CHECK_BYPASS_SCREEN, self, False)
```

みがわり側は既存の `MoveExecutor.check_hit_substitute(ctx)` が
`Event.ON_CHECK_HIT_SUBSTITUTE`（既定 `True` ＝みがわりに当たる）を発火する。
すりぬけ側が `False` かつ `stop_event=True` を返すことで、みがわりへの命中を打ち消す
（＝みがわりを貫通して本体に当たる）。

### ③ `src/jpoke/data/ability.py`

```python
"すりぬけ": AbilityData(
    handlers={
        Event.ON_CHECK_HIT_SUBSTITUTE: h.AbilityHandler(
            h.すりぬけ_bypass_substitute,
            subject_spec="attacker:self",
        ),
        Event.ON_CHECK_BYPASS_SCREEN: h.AbilityHandler(
            h.すりぬけ_bypass_screen,
            subject_spec="attacker:self",
        ),
        Event.ON_CHECK_BYPASS_STATUS_GUARD: h.AbilityHandler(
            h.すりぬけ_bypass_status_guard,
            subject_spec="source:self",
        )
    }
),
```

`subject_spec` が指すロール（`attacker` / `source`）は、常に「技を撃ってすりぬけを
発揮する側」を指す。壁・みがわり判定は `AttackContext`（`attacker`）、
しんぴのまもり・しろいきり判定は状態異常・揮発性状態付与の `EventContext`（`source`）を使うため
イベントごとに `subject_spec` を切り替える。

### ④ `src/jpoke/handlers/ability.py`

```python
def すりぬけ_bypass_screen(battle: Battle, ctx: AttackContext, value: bool) -> HandlerReturn:
    """すりぬけ特性: リフレクター・ひかりのかべ等の壁を貫通する"""
    return HandlerReturn(value=True)


def すりぬけ_bypass_status_guard(battle: Battle, ctx: EventContext, value: bool) -> HandlerReturn:
    """すりぬけ特性: しんぴのまもり・しろいきり等の耐性を貫通する"""
    return HandlerReturn(value=True)


def すりぬけ_bypass_substitute(battle: Battle, ctx: AttackContext, value: bool) -> HandlerReturn:
    """すりぬけ特性: みがわりを無視して攻撃する。"""
    return HandlerReturn(value=False, stop_event=True)
```

`ability.enabled` の判定は `EventManager` が `subject_spec` 解決時に自動で行うため、
ハンドラ内で `ctx.attacker.ability.enabled` 等の重複ガードは不要（かがくへんかガス・
かたやぶり無効化の考慮もハンドラ側に書かない）。

### ⑤ `src/jpoke/handlers/field.py`

各ハンドラで `ctx.can_bypass_screen(battle)` / `ctx.can_bypass_status_guard(battle)` を
条件式に組み込み済み。

| 関数名                            | 使用メソッド                  |
| ---------------------------------- | ------------------------------ |
| `リフレクター_reduce_damage`      | `ctx.can_bypass_screen`       |
| `ひかりのかべ_reduce_damage`      | `ctx.can_bypass_screen`       |
| `オーロラベール_reduce_damage`    | `ctx.can_bypass_screen`       |
| `しんぴのまもり_prevent_ailment`  | `ctx.can_bypass_status_guard` |
| `しんぴのまもり_prevent_confusion`| `ctx.can_bypass_status_guard`（こんらん・ねむけ） |
| `しろいきり_prevent_stat_drop`    | `ctx.can_bypass_status_guard` |

味方の壁・しんぴのまもり・しろいきりは無視しない点は、`is_foe_target()` /
「対象の場が defender 側」という既存の条件（フィールドは元々敵味方いずれかの
片側にしか存在しない）で自然に担保されている。すりぬけの有無に関わらず、
自分の場のこれらの状態が自分の技に影響を与えることはない。

### ⑥ `src/jpoke/handlers/volatile.py`

`みがわり_immune`（`ON_BEFORE_APPLY_MOVE`）・`みがわり_block_damage`（`ON_MODIFY_MOVE_DAMAGE`）は
いずれも `battle.move_executor.check_hit_substitute(ctx)` を経由するため、すりぬけの貫通判定は
`MoveExecutor.check_hit_substitute` 内の `ON_CHECK_HIT_SUBSTITUTE` 発火で一元的に処理される
（`volatile.py` 側に `すりぬけ` の直接参照は不要）。

## スコープ外（対応しない）

- フィールドでの野生エンカウント確率低下（バトル外効果）
- マジックコート+すりぬけの複合挙動（マジックコートは相手の変化技を跳ね返す状態変化だが、
  跳ね返した技を「跳ね返した側の特性」で貫通判定する複合ケースは未実装）
- わたげ（Cotton Down）がすりぬけ持ちの攻撃で誘発された際、すりぬけ側自身の場のしろいきりを
  無視できるかどうかの相互作用。わたげの能力低下は `battle.modify_stats(attacker, ..., source=defender)`
  で処理され、`can_bypass_status_guard` は `source`（＝わたげ所有者）の特性を参照するため、
  すりぬけを持つ攻撃者自身の特性は現状参照されない。競技上の出現頻度が極めて低い複合ケースのため
  未対応（一次情報: `.internal/wiki/abilities/すりぬけ.html` 特性の仕様節）。
- たこがための連続ランク低下がすりぬけ使用者由来かどうかの判定（`たこがため` 技自体が未実装）
- へんしん・フリーフォールとみがわりの相互作用（`へんしん` は変身機構未実装のため対応保留、
  `フリーフォール` は Pokémon LEGENDS アルセウス固有でシングルバトル対象外）
