# 特性実装計画: ビーストブースト

更新日: 2026-07-12

## 仕様参照

- `.internal/spec/abilities/ビーストブースト.md`

## 実装ゴール

- 攻撃技で他のポケモンをひんしにさせたとき、最も実数値が高い能力が1段階上がる。
- 同値の場合は こうげき → ぼうぎょ → とくこう → とくぼう → すばやさ の優先順。
- すでに最大ランクの能力が最高実数値でも、2番目の能力に移ることはない。

## 実装方針

- `Event.ON_MOVE_KO`、`subject_spec="attacker:self"` で実装。
- `mon.stats["atk"], ["def"], ["spa"], ["spd"], ["spe"]` の実数値を比較して最大値の能力を決定。
- Python の `max` は同値のとき最初の要素を返すため、順序付きタプルで優先順を表現。
- `mon.stats` はランク補正・持ち物・味方の特性・すなあらしの影響を受けない生の実数値であるため、
  それらは比較に反映されない（仕様どおり）。
- パワーシェア・ガードシェア・パワートリック・スピードスワップは対象の実数値そのものを
  書き換える実装（`handlers/move_status.py`）のため、`mon.stats` を参照するだけで自動的に反映される。
- ワンダールームのみ例外: `handlers/field.py` の `ワンダールーム_def_modifier` はダメージ計算時に
  比率を掛けるだけで `mon.stats` 自体は書き換えない。そのためビーストブースト側で
  `battle.get_global_field("ワンダールーム").is_active` を見て、比較用の一時辞書上で
  `def` と `spd` を入れ替えてから最大値を決定する。
- **じしんかじょう・くろのいななき** と同じ `ON_MOVE_KO` パターン。うなぎのぼりの
  効果2はこのハンドラをそのまま再利用する（`data/ability.py` の `うなぎのぼり` エントリ）。

## 変更候補

- `src/jpoke/data/ability.py` — `ビーストブースト`（および `うなぎのぼり`）に handlers を登録する
- `src/jpoke/handlers/ability.py` — `ビーストブースト_boost_best_stat_on_ko` を実装する
- `tests/abilities/test_ability_ha.py` — テストを追加する

## テスト観点

1. 倒したとき、最高実数値の能力が1段階上がる。
2. 同値の場合は優先順（atk > def > spa > spd > spe）で決まる。
3. 最高実数値の能力がすでに最大ランクなら発動しない（2番目の能力には移らない）。
4. ワンダールーム下では def/spd の実数値が入れ替わった状態で比較される。

## 疑似コード

```python
_BEAST_BOOST_ORDER: tuple[str, ...] = ("atk", "def", "spa", "spd", "spe")

def ビーストブースト_boost_best_stat_on_ko(battle, ctx, value):
    mon = ctx.attacker
    stats = dict(mon.stats)
    if battle.get_global_field("ワンダールーム").is_active:
        stats["def"], stats["spd"] = stats["spd"], stats["def"]
    best_stat = max(_BEAST_BOOST_ORDER, key=lambda s: stats[s])
    if battle.modify_stats(mon, {best_stat: +1}, source=mon):
        _announce_ability_triggered(battle, mon)
    return HandlerReturn(value=value)
```
