# 特性実装計画: はやあし

更新日: 2026-07-12

## 仕様参照

- `.internal/spec/abilities/はやあし.md`

## 実装ゴール

- 状態異常（どく・もうどく・まひ・やけど・こおり・ねむり）のとき、すばやさが1.5倍になる。
- まひ状態になっても、すばやさの低下効果（1/2化）を無視する。
- しびれて動けなくなる効果（`ON_TRY_ACTION`側）は無視できない。
- こんらん・メロメロなどの状態変化（volatile）では発動しない。

## 実装方針

- `DomainEvent.ON_CALC_SPEED`（`subject_spec="source:self"`）で `mon.ailment.is_active` を確認し、
  状態異常時のみ倍率を適用する。
- 状態異常でまひ以外の場合: 固定小数点補正 `apply_fixed_modifier(value, 6144)`（=1.5倍・切り捨て）を
  適用する。こだわりスカーフ（`item.py` の `こだわりスカーフ_boost_speed`）と同じ手法で、
  素早さは常に整数であるべきなので `value *= 1.5` のような float 演算は使わない。
- まひの場合: まひによる `1/2` の切り捨て（`まひ_speed`, `handlers/ailment.py`）を後段で受けるため、
  本ハンドラでは `value *= 3` の整数倍を適用する。
  - `ON_CALC_SPEED` は `DomainEvent`（`enums/event.py`）であり、`EventManager.emit()` は
    `DomainEvent` に対して `_sort_handlers` によるpriorityソートを行わない
    （`isinstance(event, DomainEvent)` の分岐で登録順のまま発火する）。
    そのため本ハンドラに `priority` を指定しても実行順には影響しない。
  - 実際の実行順は「特性ハンドラ→アイテムハンドラ→状態異常ハンドラ」の登録順
    （`core/switch_manager.py` の `_register_handlers_on_switch_in` を参照）で決まり、
    はやあし（特性）は まひ_speed（状態異常）より必ず先に実行される。
  - この順序により、切り捨ては まひ_speed の `value // 2` の1回のみで完結し、
    `floor(素早さ * 3 / 2) == floor(素早さ * 1.5)` と一致する（奇数の素早さでも誤差が出ない）。
    仮に まひ_speed → はやあし の順で実行されると `(素早さ // 2) * 3` となり、
    奇数の素早さで1ずれるため、順序への依存はこの計画に明記しておく。
- 状態変化（こんらん・メロメロ等）は `mon.ailment` を経由しないため、
  何も判定を追加しなくても自然に対象外となる。

## テスト観点

1. どく・もうどく・やけど・ねむり・こおり状態のときすばやさが1.5倍（切り捨て）になる。
2. まひ状態のときまひのすばやさ低下を無視し、かつ1.5倍（切り捨て）になる。
3. 素早さが奇数になる個体（奇数の種族値）でも、まひ・まひ以外いずれの状態異常でも
   `floor(素早さ*1.5)` と一致し、小数値や1ずれが発生しない。
4. こんらん等の状態変化では発動しない。
5. 状態異常が回復するとすばやさが元に戻る。
