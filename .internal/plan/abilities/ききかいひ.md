# ききかいひ 実装計画

## スコープ
- 優先度: H
- 仕様書: `.internal/spec/abilities/ききかいひ.md`
- 実装の核: にげごしと同一の発動条件・処理（HP が「1/2超 -> 1/2以下」に落ちたとき、既存の
  割り込み交代基盤 `Interrupt.EMERGENCY` を立てて処理させる）。ハンドラ関数
  `ききかいひ_switch_on_half_hp` をにげごしと共用する。

## 事前確認済みの実装面
- `.internal/plan/abilities/にげごし.md` に同機構の設計判断がまとまっている。ききかいひは
  にげごしと完全に同じ発動条件・処理順のため、そちらの設計をそのまま踏襲する。
- `src/jpoke/core/turn_controller.py` に `Interrupt.EMERGENCY` の解決ポイントがあり、
  技実行後（`_run_action_phase`）とターン終了時（`_run_end_phase`）の各段で割り込み
  交代を処理する。順序は EJECTBUTTON → EMERGENCY → PIVOT → EJECTPACK(after_move) /
  EJECTPACK(turn_end) → FAINTED であり、仕様書の「だっしゅつボタンが先」「だっしゅつ
  パックはききかいひの後」と一致する。
- `src/jpoke/core/switch_manager.py` の `run_interrupt_switch()` が汎用的に割り込み
  交代を処理する。
- 連続攻撃技については、各ヒットごとに `battle.modify_hp(..., reason="move_damage")`
  が呼ばれ `Event.ON_HP_CHANGED` が都度発火するが、`_trigger_emergency_switch()` は
  フラグを立てるだけで実際の交代実行はターンコントローラー側の割り込み解決ポイント
  （技の全ヒット終了後）まで遅延される。そのため「連続攻撃技はすべてのヒットが
  終わってから発動判定」という仕様は現状のアーキテクチャで自然に満たされる。

## 実装方針（レビュー時点で確認・修正した内容）
1. `src/jpoke/handlers/ability.py` の `ききかいひ_switch_on_half_hp` を、にげごしと
   `data/ability.py` 側で共用登録する（`Event.ON_HP_CHANGED` / `subject_spec="target:self"`）。
2. 発動しない理由コードを `HPChangeReason` で判定する。
   - `self_attack`（こんらん自傷）
   - `pain_split`（いたみわけ。自身が使用した場合・相手から受けた場合の両方とも
     `いたみわけ_equalize_hp` が両者に同じ reason を付与するため区別不要）
   - `self_cost`（みがわり・はらだいこ・のろい(呪い)・ソウルビートなどの自己HP消費。
     マジックガードでも防げない性質の自己コストとして共通化されている）
3. **レビューで発見した不整合**: `はらだいこ_apply` / `みがわり_apply` /
   `ソウルビート_pay_hp_and_modify_attacker_stats` が `reason="self_cost"` を
   付与しておらず、デフォルトの `reason=""` のまま `battle.modify_hp()` を呼んで
   いたため、以下の二重のバグがあった。
   - ききかいひ/にげごし側の除外条件が `{"self_attack", "pain_split"}` のみで
     `"self_cost"` を含んでおらず、`のろい`（すでに `reason="self_cost"` を
     正しく付与していた）ですら誤って発動し得た。
   - `マジックガード_ignore_damage` は `reason` が
     `{"move_damage", "pain_split", "self_attack", "self_cost", "perish"}` の
     いずれでもない場合にダメージを無効化する実装のため、`reason=""` のままだと
     マジックガード持ちがみがわり/はらだいこ/ソウルビートを使ってもHPが消費され
     ない（無効化されてしまう）バグがあった。
   - 修正: `はらだいこ_apply` / `みがわり_apply` /
     `ソウルビート_pay_hp_and_modify_attacker_stats` の `battle.modify_hp()` 呼び出しに
     `reason="self_cost", source=mon` を追加し、`ききかいひ_switch_on_half_hp` の除外
     セットに `"self_cost"` を追加した。
4. 交代先の選択・実行そのものは既存の `Interrupt.EMERGENCY` 解決導線に委ねる。

## 実装タスク（完了状況）
1. `src/jpoke/data/ability.py` の `ききかいひ`・`にげごし` 双方に
   `ききかいひ_switch_on_half_hp` を登録済み。
2. `src/jpoke/handlers/ability.py` に `_crossed_half_hp`（閾値判定の共通関数）と
   `_trigger_emergency_switch`（割り込み予約の共通関数）を実装済み。
3. `self_cost` を除外条件に追加（本レビューで修正）。
4. `はらだいこ_apply` / `みがわり_apply` / `ソウルビート_pay_hp_and_modify_attacker_stats`
   に `reason="self_cost"` を付与（本レビューで修正）。

## 判定ルール
### 発動
- 攻撃技でダメージを受け、最終的に HP が半分以下になった（連続技は全ヒット終了後）。
- どく・やけど・天候・しおづけ・バインド・設置技等の定期/間接ダメージで HP が半分以下
  になった。
- 特性・アイテムの効果によるダメージで HP が半分以下になった。
- かげふみ/ありじごく/じりょくの影響や、にげられない/バインド/ねをはる/
  フェアリーロック状態を無視して発動する（これらは通常交代コマンドの可否判定であり、
  `Interrupt.EMERGENCY` の強制交代はそもそも `get_available_switch_commands()` の
  トラップ判定を経由しないため、現状の実装で自然に満たされる）。

### 非発動
- ダメージ前からすでに HP が半分以下だった。
- 瀕死になった。
- 交代先（控え）がいない。
- こんらん自傷（`self_attack`）。
- いたみわけ（自身の使用・被弾のどちらも `pain_split`）。
- みがわり・はらだいこ・のろい(呪い)・ソウルビートの自己HP消費（`self_cost`）。
- 特性無効化中（かがくへんかガス・かたやぶり関連はフレームワーク側で処理されるため
  ハンドラ側で明示チェックしない）。

## テスト計画
1. 単発攻撃で HP が「1/2超 -> 1/2以下」になった直後に `Interrupt.EMERGENCY` が立ち、
   技後の割り込み交代で引っ込む（実装済みテストあり）。
2. こんらん自傷では発動しない（実装済みテストあり）。
3. やけどダメージでも発動する（実装済みテストあり）。
4. 被弾前からHPが半分以下の場合は発動しない（実装済みテストあり）。
5. はらだいこ・みがわりの自己HP消費（`self_cost`）では発動しない（本レビューで追加）。
6. 相手のいたみわけ（`pain_split`）を受けても発動しない（本レビューで追加）。
7. 控えのポケモンがいない場合は発動しない（本レビューで追加）。

## 未解決点（既知の簡略化）
- 連続技の途中でHPが一度半分以下になった後、きのみ等で回復して技終了時には半分超に
  戻っている場合でも `Interrupt.EMERGENCY` フラグが取り消されず発動してしまう
  可能性がある（`run_interrupt_switch()` はフラグを立てた時点の状態のみを信頼し、
  解決時に現在HPを再検証しない）。この組み合わせはきわめて限定的（多段技＋
  きのみ発動タイミングの両立）であり、現状は未対応。
- オボンのみ等のきのみ回復とききかいひの発動順（「きのみが先に発動し、なお半分
  以下ならききかいひが発動する」）は、同一イベント上のハンドラ優先度
  （デフォルト優先度100・同速のため登録順依存）に委ねている。明示的な優先度
  指定は行っていない。
- 特性ちからずくの効果が発動した技のダメージを除外する仕様（第七世代由来の
  ごく限定的な仕様）は未実装。現行のチャンピオンズルールでの実用上の影響は
  極めて小さいと判断し、対応を見送る。
- とんぼがえり/ボルトチェンジ/クイックターン、ドラゴンテール/ともえなげ、
  後攻コアパニッシャー、レッドカード、だっしゅつボタン/だっしゅつパックとの
  細かな処理順の相互作用は、既存の `Interrupt` 優先順位（EJECTBUTTON →
  EMERGENCY → PIVOT → EJECTPACK）に委ねており、個別の技効果側の実装
  （各技の計画書・仕様書）で担保する。ききかいひ側で追加の特殊対応はしていない。
