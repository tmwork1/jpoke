# ノーガード の実装計画

## 概要
命中判定を必中化する特性。攻撃側がノーガード所持 **または** 防御側がノーガード所持のいずれかで、命中率をNoneに設定する。
一撃必殺技にも適用され、あなをほる等の半無敵状態にも命中する（.internal/spec/abilities/ノーガード.md参照）。

## 実装対象のイベント
1. **`ON_MODIFY_ACCURACY`** - 命中率補正イベント（priorityは .internal/spec/turn.md 未掲載のため、既存の同種ハンドラを参照して決定。下記「priorityの根拠」参照）
2. **`ON_TRY_MOVE_1`** - あなをほる/そらをとぶ/ダイビング/シャドーダイブ中の対象への回避判定（`handlers/volatile.py` の共通関数 `can_hit_hidden_target` に手を入れる。詳細は下記）

## 実装内容

### イベント: ON_MODIFY_ACCURACY（命中率補正）
- **担当**: `handlers/ability.py` に新規ハンドラを追加
- **処理**:
  - 攻撃側がノーガード所持 → None を返す（必中）
  - または防御側がノーガード所持 → None を返す（必中）
  - 条件（無視される）：
    - タイプ相性無効、まもる系、対象不在などは無視できない（別イベントで処理）
    - 半無敵状態（そらをとぶ/あなをほる中）でも命中する（`can_hit_hidden_target` 側で対応。下記参照）
    - 一撃必殺技の命中判定部分にも適用
    - `かたやぶり` の有無に関わらず適用（`mold_breaker_ignorable` フラグを付けない＝標準でかたやぶりの影響を受けない）

#### priorityの根拠・stop_eventの必要性
`ON_MODIFY_ACCURACY` には同一優先度（デフォルト100）のまま多数のハンドラ
（すながくれ・はりきり・ちどりあし・ゆきがくれ・ふくがん・しょうりのほし・こうかくレンズ・
ひかりのこな・フォーカスレンズ・ミクルのみ・じゅうりょく等）が登録されており、これらは
`apply_fixed_modifier(value, ...)` で value に対して直接乗算を行う。同優先度内の実行順は
「素早さの速いポケモン側のハンドラが先」という並び替えのため、**ノーガード側が先に実行されて
value=None を返すと、後続の乗算系ハンドラが `None * modifier` でクラッシュする**
（例: 攻撃側ノーガードが素早さで勝る場合、防御側すながくれのハンドラが後から
`apply_fixed_modifier(None, 3277)` を呼び TypeError）。

このため、ノーガードの必中確定時は `HandlerReturn(value=None, stop_event=True)` を返し、
以降の `ON_MODIFY_ACCURACY` ハンドラの実行そのものを止める。これは既存の同種「必中」実装
（`きょけんとつげき_guaranteed_hit` / `ちいさくなる_guaranteed_hit` / `ロックオン_guarantee_hit`、
いずれも `handlers/volatile.py`）と同じパターンであり、priorityの明示的な変更は不要
（`stop_event=True` により実行順に依存せず安全に必中を確定できる）。

### イベント: ON_TRY_MOVE_1（潜伏中の回避判定・半無敵状態のバイパス）
- **担当**: `handlers/volatile.py` の `can_hit_hidden_target`（あなをほる/そらをとぶ/ダイビング/
  シャドーダイブの4状態が共有する判定関数）
- **処理**: 攻撃側または防御側がノーガード所持なら、`HIDDEN_MOVE_ALLOWED_MOVES` による技限定の
  回避判定を経由せず、常に命中を許可する（`return HandlerReturn(value=value)` で早期リターン）
- **根拠**: ノーガードは半無敵状態の相手にも必中になる仕様（一次情報に明記）。この判定は
  `ON_MODIFY_ACCURACY` とは独立した別イベントのため、ノーガード側のハンドラ追加だけでは
  対応できず、共有関数側の修正が必要
- しれいとう（コマンダー）によるヘイラッシャ飲み込み中のシャリタツは対象外
  （ダブルバトル専用ギミックのため、シングル専用の本プロジェクトでは対象外）

### 実装順序
1. `handlers/ability.py` に `ノーガード_guarantee_hit` ハンドラを追加
2. 攻撃側と防御側の両方をチェック
3. `AbilityData` に handlers を **２つのハンドラとして登録**：
   - `subject_spec="attacker:self"` - 攻撃側がノーガード持ちの場合
   - `subject_spec="defender:self"` - 防御側がノーガード持ちの場合
4. `handlers/volatile.py` の `can_hit_hidden_target` にノーガードのバイパスを追加
5. テスト追加

## 実装の詳細ポイント
- `return HandlerReturn(value=None, stop_event=True)` で必中化かつ後続ハンドラの
  クラッシュを防止する（`value=None` を返すだけで `stop_event` を付けないと、
  他の命中率補正ハンドラが `None` に対して乗算しクラッシュしうる）
- 攻撃側: `ctx.attacker.ability.name == 'ノーガード'`
- 防御側: `ctx.defender.ability.name == 'ノーガード'`
- 複数条件は OR で結合（いずれかなら必中）
- 同じハンドラを２つ登録して、攻撃側・防御側の両方に対応
- 半無敵状態への必中は `can_hit_hidden_target` 側の対応で実現する（ノーガード側のハンドラのみでは不可）

## テスト項目
- 攻撃側ノーガード時に低命中技が必中になる
- 防御側ノーガード時に低命中技が必中になる
- すながくれ（すなあらし下）の命中率低下を無視して必中になる（stop_eventのクラッシュ回帰テストを兼ねる）
- はりきりの命中率低下を無視して必中になる
- ミラクルスキンの変化技命中率低下を無視して必中になる
- 一撃必殺技（こおりタイプでない使用者のぜったいれいど）でも必中になる
- 半無敵状態（そらをとぶ中）の相手にも命中する
