# だっしゅつパック 実装計画

## スコープ
- 優先度: -
- 仕様書: `.internal/spec/items/だっしゅつパック.md`
- 実装状態: **実装済み**（本計画書はレビューにより加筆）

## 実装方針

### 発動トリガー
- `Event.ON_MODIFY_STAT`（`.internal/spec/turn.md` 未掲載のイベント。ランク変化の実減少量を通知する
  イベントで、しろいハーブ・まけんき・かちき・ビビリだま・ものまねハーブなど既存の
  ON_MODIFY_STAT ハンドラと同じ用途のため、それらと同じ既定 priority(100) を採用する）
  を `subject_spec="target:self"` で購読し、`だっしゅつパック_reserve_switch`
  （`src/jpoke/handlers/item.py`）で判定する。
- `value` は `battle.modify_stats()` が実際に適用した `{stat: 変化量}` の辞書（`actual_changes`）。
  下降量そのものを見るため、ランクが下限（-6）に達していなくても、また元のランクが
  マイナスにならなくても（+2→+1 等）発動する。
- 交代先の有無判定には `battle.query.has_available_bench(player)` を使う
  （**通常の交代可否判定 `can_switch()` は使わない**）。`can_switch()` は
  にげられない・バインド・ねをはる・フェアリーロックや特性かげふみ・ありじごく・
  じりょくなどの「とらわれ状態」も見てしまうが、だっしゅつパックはこれらを無視して
  発動する仕様のため、控えの生存有無のみを見る `PokemonQuery.has_available_bench()`
  （`src/jpoke/core/query.py`）を新設して使用する。
- 発動条件を満たすと `battle.player_states[player].interrupt = Interrupt.EJECTPACK_REQUESTED`
  を立てる（即座には交代しない。交代は各フェーズの節目で `SwitchManager` がまとめて解決する）。

### とらわれ状態の無視（コマンド生成側の対応）
- `CommandManager.get_available_switch_commands()` は元々バトンタッチ（`Interrupt.PIVOT`）の
  ときだけとらわれ状態チェックをスキップしていたが、だっしゅつパックの割り込み交代
  （`interrupt.name.startswith("EJECTPACK")`）でも同様にスキップするよう拡張した。
  そうしないと、`だっしゅつパック_reserve_switch` でフラグを立てても、実際に交代コマンドを
  解決する段階でとらわれ状態により選択肢が0件になり例外・交代不発になる。

### すばやさ最速のみ発動（同時発動の裁定）
- `SwitchManager.override_ejectpack_interrupt()` は `EJECTPACK_REQUESTED` の
  プレイヤーをすばやさ順に走査し、最初に見つかった（＝一番速い）プレイヤーのみ
  対象フェーズのフラグ（`EJECTPACK_ON_START` 等）に昇格させる。
  **それ以外の `EJECTPACK_REQUESTED` は `Interrupt.NONE` に破棄する**
  （以前は昇格させずに放置していたため、`_process_events_after_switch` の
  再帰チェック `has_interrupt()` に引っかかり、後続フェーズで発動してしまうバグがあった。
  同時に複数のポケモンのだっしゅつパックが条件を満たした場合、すばやさが一番高い
  ポケモンのみが発動し、他は完全に不発になる仕様のため、この場で確定的に破棄する）。

### 発動後の再検証（マジシャン・わるいてぐせとの整合性）
- `SwitchManager.run_interrupt_switch()` は割り込み交代を実行する直前に、
  `Interrupt.required_item_name()`（`src/jpoke/enums/interrupt.py`）で
  「このフラグの発動に必要なアイテム名」を取得し、対象ポケモンが現在もそのアイテムを
  実際に持っているかを再検証する。持っていなければ（マジシャン・わるいてぐせ等で
  発動後に奪われていれば）交代を行わずフラグを破棄する。
  - `Event.ON_HIT`（自傷効果・攻撃対象への追加効果）は `Event.ON_DAMAGE_HIT`
    （マジシャン・わるいてぐせ・レッドカードが購読）より前に解決されるため、
    フラグを立てた時点ではまだアイテムを保持しているケースが多い。そのため
    「発動条件を満たした後、実際に交代する前にアイテムを失っていないか」を
    交代実行の直前で再チェックする必要がある。

### ミラーアーマーとの相互作用
- `ミラーアーマー_reflect_stat_drop`（`src/jpoke/handlers/ability.py`）はランク低下を
  跳ね返す際、`ctx.target`（ミラーアーマー所持者）自身のランクは変化させない
  （`battle.modify_stats()` 経由での実減少が発生しないため、通常は
  `Event.ON_MODIFY_STAT` が発火せずだっしゅつパックも発動しない）。
  しかし仕様上、ミラーアーマーで跳ね返した場合も所持者のだっしゅつパックは発動するため、
  `ミラーアーマー_reflect_stat_drop` 内で `ctx.target.item.name == "だっしゅつパック"` を
  直接判定し、`Interrupt.EJECTPACK_REQUESTED` を明示的に設定する特別対応を行っている
  （ビビリだま等、他の ON_MODIFY_STAT 系の反応効果はこの反射では発動しない点と対照的）。

### バトンタッチ・パワースワップ等での非発動
- バトンタッチ（`SwitchManager.run_switch()` 内の `baton_data["rank"]` 適用）、
  パワースワップ／ガードスワップ／じこあんじ／ひっくりかえす、くろいきりによる
  ランクリセットは、いずれも `mon.rank[stat]` へ直接代入する実装になっており
  `battle.modify_stats()` を経由しない。そのため `Event.ON_MODIFY_STAT` が発火せず、
  だっしゅつパックは発動しない（仕様どおり）。

## 今後の対応
- クリアスモッグ・シャドースチール（技）、きみょうなくすり（特性）による
  ランクリセットは、対象の技・特性自体が本シミュレータで未実装のため対応不要。
  実装される際は、くろいきりと同様に `mon.rank` への直接代入とし、
  `battle.modify_stats()` を経由させないこと。
- だっしゅつボタン・ききかいひ・にげごしと同時に発動条件を満たした場合の
  「だっしゅつパックは発動しない」規定は、ターン進行上のフェーズ順序
  （`TurnController._run_move_phase` で EJECTBUTTON→EMERGENCY→PIVOT→だっしゅつパックの順に解決）
  により、だっしゅつボタン側が先に交代を消費する場合が大半だが、
  それぞれ独立した割り込みが同一フェーズで両方成立するケース（例: 自傷技で自分の
  だっしゅつパックが、同時に相手のだっしゅつボタンも誘発する場合）で、
  だっしゅつパック側が本当に不発になるかまでは未検証。将来的な精査対象とする。
