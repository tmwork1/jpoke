# だっしゅつボタン 実装計画

## スコープ
- 優先度: -
- 仕様書: `.internal/spec/items/だっしゅつボタン.md`
- 実装状態: **実装済み**（本計画書はレビューにより加筆）

## 実装方針

### 発動トリガー
- `Event.ON_HIT`（`.internal/spec/turn.md` には「だっしゅつボタン」自体の priority 行はなく、
  ON_DAMAGE 節の末尾に Interrupt 注記としてのみ言及されている）を
  `subject_spec="defender:self"` で購読し、`だっしゅつボタン_reserve_switch`
  （`src/jpoke/handlers/item.py`）で判定する。
- 実装上の `Event.ON_DAMAGE_HIT` は `actual_damage<=0` のとき発火しないため採用しない
  （`move_executor.py` の `_execute_hit` で `if actual_damage <= 0: return` により
  `ON_DAMAGE_HIT` の発火自体がスキップされる）。ばけのかわ/アイスフェイスの肩代わりや
  こらえるでHP1のまま耐えたとき（実HPダメージ0）も発動する仕様
  （`.internal/spec/items/だっしゅつボタン.md` 詳細な仕様）を満たすため、いのちのたま
  （`いのちのたま_recoil`）と同様に常に発火する `Event.ON_HIT` を使用する。
  priority は turn.md に具体的な数値の記載がないため既定値（100）のまま使用する。

### 0ダメージでも発動するケース／発動しないケースの切り分け
`Event.ON_HIT` は「みがわりに被弾しても発動する」ため、以下を明示的にハンドラ内で
判定する必要がある。

- **発動する**: ばけのかわ/アイスフェイスの肩代わり、こらえるでHP1のまま耐えた場合
  （`Event.ON_MODIFY_MOVE_DAMAGE` の時点でダメージが0に補正されるため、
  `ON_HIT` に渡る `value`（実ダメージ）は0になるが、これらは「攻撃技を受けた」
  ことに変わりないため発動する）。
- **発動しない**: みがわりに阻まれた場合。`ctx.substitute_damage`
  （`src/jpoke/core/context.py`、`みがわり_block_damage` が設定する）が
  真の場合はみがわりが身代わりになったと判定してスキップする。
- **発動しない**: 特性ちからずくの効果が発動した技を受けた場合。
  `ctx.attacker.ability.name == "ちからずく" and ctx.move.has_flag("secondary_effect")`
  を判定する（`いのちのたま_recoil` 等と同一パターン）。
- **発動しない**: 受けた攻撃でひんしになった場合。`ctx.defender.fainted` を判定する。
  `Event.ON_HIT` は defender が瀕死になったかどうかに関わらず発火するため、
  ハンドラ内で明示的にガードしないと、瀕死交代（`Interrupt.FAINTED`）より前に
  誤って `Interrupt.EJECTBUTTON` を設定してしまう
  （`_run_move_phase` は各アクター行動直後に
  `run_interrupt_switch(Interrupt.EJECTBUTTON)` を呼ぶため、瀕死処理
  `run_faint_switch()`（ターン終了処理内）より先に実行される）。
- 相性・特性・まもる状態で技を無効化した場合は `_check_hit_by_type` や
  `ON_TRY_MOVE_2` 等で `_execute_hit` 自体に到達しないため、`Event.ON_HIT` も
  発火せず追加のガードは不要。変化技を受けた場合も `_execute_hit` の変化技分岐
  （`_execute_status_hit`）では `ON_HIT`/`ON_DAMAGE_HIT` を発火しないため対応不要。

### 交代先の有無判定（とらわれ状態の無視）
- 交代先の有無判定には `battle.query.has_available_bench(player)` を使う
  （**通常の交代可否判定 `can_switch()` は使わない**）。`can_switch()` は
  にげられない・バインド・ねをはる・フェアリーロックや特性かげふみ・ありじごく・
  じりょくなどの「とらわれ状態」も見てしまうが、だっしゅつボタンはこれらを無視して
  発動する仕様のため、控えの生存有無のみを見る `PokemonQuery.has_available_bench()`
  （`src/jpoke/core/query.py`、だっしゅつパックと共用）を使用する。

### とらわれ状態の無視（コマンド生成側の対応・レビューで修正）
- `CommandManager.get_available_switch_commands()` は元々 PIVOT・EJECTPACK系の
  割り込みでのみとらわれ状態チェックをスキップしており、`Interrupt.EJECTBUTTON`
  が対象外になっていた。そのため `だっしゅつボタン_reserve_switch` で
  `Interrupt.EJECTBUTTON` を設定しても、実際に交代コマンドを解決する段階で
  とらわれ状態により選択肢が0件になり、交代不発（あるいは交代先が選べない）と
  なるバグがあった。`state.interrupt != Interrupt.EJECTBUTTON` の条件を追加して
  修正した。

### アイテム消費とマジシャン・わるいてぐせ・はたきおとすとの整合性
- `Interrupt.EJECTBUTTON.requires_item_consumption()` は `True` を返し、
  `Interrupt.required_item_name()` は `"だっしゅつボタン"` を返す
  （`src/jpoke/enums/interrupt.py`）。
- `SwitchManager.run_interrupt_switch()` は割り込み交代を実行する直前に、
  対象ポケモンが現在も `だっしゅつボタン` を実際に持っているかを再検証する。
  はたきおとす・ほしがる・どろぼう・マジシャン・わるいてぐせは
  `Event.ON_DAMAGE_HIT`（`Event.ON_HIT` より後に発火）でアイテムを奪う／落とす
  ため、`だっしゅつボタン_reserve_switch`（`ON_HIT` で発動判定）が先にフラグを
  立てても、実行時点の再検証で「アイテムを失っていた」と判定され交代しない。
  仕様「はたきおとす・ほしがる・どろぼう・マジシャンの効果はだっしゅつボタンより
  早く発動する。このため攻撃を受けた側はもちものを失うか奪われ交代することが
  できない」を満たす。

### ドラゴンテール・ともえなげとの整合性
- `ドラゴンテール_force_switch`/`ともえなげ_force_switch`（`Event.ON_DAMAGE_HIT`）は
  `battle.run_switch()` を同期的に呼び出す。`SwitchManager.run_switch()` は
  冒頭で `state.interrupt = Interrupt.NONE` を実行するため、`ON_HIT` で先に
  立てていた `Interrupt.EJECTBUTTON` は強制交代の実行時に自動的に破棄される。
  結果として仕様「ドラゴンテール・ともえなげによる強制交代効果はだっしゅつボタン
  より早く発動する。このためだっしゅつボタンは消費されず、交代先を自由に選ぶ
  ことができない」を満たす（追加のコード対応は不要）。

### ききかいひ・にげごしとの整合性
- ききかいひ/にげごし（`ききかいひ_switch_on_half_hp`）は `Event.ON_HP_CHANGED`
  （`modify_hp()` 内部、`Event.ON_HIT` より前に発火）で `Interrupt.EMERGENCY` を
  設定する。同一ポケモンがだっしゅつボタンも所持していて同時に条件を満たした場合、
  `Event.ON_HIT` で `だっしゅつボタン_reserve_switch` が後から
  `state.interrupt = Interrupt.EJECTBUTTON` を設定するため、EMERGENCY フラグは
  上書きされる。`_run_move_phase` は `EJECTBUTTON` → `EMERGENCY` の順に解決するため
  結果的にだっしゅつボタンによる交代のみが行われる。仕様「だっしゅつボタンと
  ききかいひ/にげごしの発動条件を同時に満たした場合、だっしゅつボタンが先に発動して
  控えに引っ込む」を満たす（追加のコード対応は不要）。

## 対象外（プロジェクトスコープ外）
- フリーフォール（Pokémon LEGENDS アルセウス固有の状態）はシングルバトル対象外の
  ため未対応。
- ヘイラッシャ + 特性しれいとうの合体は、しれいとうがダブルバトル前提の仕様
  （味方が場に出ていることが発動条件）のため、シングルバトルでは原理的に発生しない。
- 「だっしゅつボタンを持つ複数のポケモンが全体攻撃技で同時に攻撃を受けた場合、
  すばやさが一番高いポケモンのみ発動する」はダブルバトルの全体攻撃技が前提のため
  シングルバトルでは発生しない。
- 「だっしゅつボタンとレッドカードの同時発動」「だっしゅつボタンとだっしゅつパックの
  同時発動」は、いずれも一方の条件が異なるポケモン（同時に場に出ている2体以上）に
  発生することを前提とした記述であり、シングルバトル（各プレイヤー同時に場に出る
  ポケモンは1体のみ）では、そもそも同一ポケモンが両方のアイテムを同時に持つことは
  できないため再現しない。ダブルバトル対応時に再検証すること。

## 今後の対応
- クリアスモッグ・シャドースチール等、ダブルバトル固有の相互作用は未対応
  （プロジェクトスコープ外）。
- 多段技で複数ヒットにまたがってみがわりが破壊されるケース（1ヒット目でみがわり
  破壊、2ヒット目が本体に直撃）では `ctx.substitute_damage` がヒットをまたいで
  リセットされないため、2ヒット目が実際には本体への直撃であっても
  `ctx.substitute_damage` の値が残っていれば発動しない可能性がある。これは
  `かいがらのすず_drain_on_hit` 等、既存の同パターンのハンドラにも共通する
  `AttackContext.substitute_damage` フィールドの設計上の制約であり、本チケットの
  スコープでは対応しない。
