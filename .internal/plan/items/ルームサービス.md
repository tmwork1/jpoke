# ルームサービス 実装計画

## 効果概要
トリックルームが有効な場にいる、または有効な場に繰り出したとき、すばやさ（S）を-1して消費する
（1度きり）。あまのじゃく・たんじゅんの補正やクリアボディ等の無効化を無視する挙動は、
`ON_BEFORE_MODIFY_STAT` / `ON_MODIFY_STAT` を購読する既存の特性ハンドラ（あまのじゃく・たんじゅん・
クリアボディ・ミラーアーマー等）が汎用的に処理するため、本アイテム側では意識する必要がない。

## 実装仕様

| 項目 | 内容 |
|------|------|
| イベント1 | `Event.ON_FIELD_ACTIVATE`（トリックルームが新規に発動したとき） |
| イベント2 | `Event.ON_SWITCH_IN`（トリックルームが有効な場に繰り出したとき） |
| subject_spec | いずれも `"source:self"` |
| priority（イベント1） | 既定値100。`.internal/spec/turn.md` に `Event.ON_FIELD_ACTIVATE` の記載がないため、
  同種の既存ハンドラ（かぜのり_boost_atk_on_tailwind_start、ふうりょくでんき_on_field_activate）に
  倣い既定値を採用する。 |
| priority（イベント2） | 120。`.internal/spec/turn.md` の `Event.ON_SWITCH_IN` に「120 ルームサービスの発動」と
  明記されている。場に出たときに発動する特性（多くは priority=100）より後、
  こだいかっせい・クォークチャージ等（priority=140）より先に発動する。 |
| once | 明示的な `once=True` は使わない。`_announce_and_consume_item` によるアイテム消費
  （`consumed` 無効化理由の付与）で以後の発動を防ぐ（既存の他アイテムハンドラと同じ方式）。 |
| 消費条件 | `battle.modify_stats(mon, {"spe": -1}, source=mon)` の返り値が空でない（実際にランクが
  変化した）場合のみ発動を通知し消費する。 |

### ハンドラロジック（疑似コード）
```python
def _ルームサービス_try_trigger(battle, mon):
    changes = battle.modify_stats(mon, {"spe": -1}, source=mon)
    if changes:
        _announce_and_consume_item(battle, mon)

def ルームサービス_drop_speed_on_trick_room(battle, ctx, value):
    mon = ctx.source
    if value.name == "トリックルーム":
        _ルームサービス_try_trigger(battle, mon)
    return HandlerReturn(value=value)

def ルームサービス_drop_speed_on_switch_in(battle, ctx, value):
    mon = ctx.source
    if battle.get_global_field("トリックルーム").is_active:
        _ルームサービス_try_trigger(battle, mon)
    return HandlerReturn(value=value)
```

### 設計上の要点
- `battle.modify_stats(..., source=mon)` で **source を所持者自身に明示する**。これにより
  `ON_MODIFY_STAT` を購読するまけんき・かちき（`ctx.source != ctx.target` を条件に発動）が、
  相手の技でトリックルームが発動した場合でも誤発動しない
  （仕様: 「他のポケモンによるトリックルームでルームサービスを発動させる場合でも、所持者自身が
  自分の能力を下げたことになる」「敵が使ったトリックルームでルームサービスを発動しても、特性
  まけんき・かちきは発動しない」）。同時に `source=mon` は
  クリアボディ／ミラーアーマー（`ctx.source is not None and ctx.source != ctx.target` の場合のみ
  ブロック・反射する）の判定にも影響しないため、ランク低下は無視されずに適用される。
- あまのじゃく・たんじゅんは `ON_BEFORE_MODIFY_STAT` を購読する既存の特性ハンドラが自動的に
  符号反転・倍加するため、本アイテム側は常に `{"spe": -1}` を渡すだけでよい。
- `modify_stats` の返り値（実際に変化したランクの辞書）が空の場合（すでにランク下限、または
  あまのじゃくで既にランク上限）は発動・消費しない（仕様:「すでに所持者のすばやさが最低まで
  下がっているときは発動しない」）。
- マジックルーム中・特性ぶきよう中は発動しない仕様だが、これは
  `マジックルーム_apply`（`src/jpoke/data/field/global_field.py` の `ON_FIELD_ACTIVATE` /
  `ON_SWITCH_IN`、priority=100既定）が `item_manager.add_disabled_reason(mon, "マジックルーム")` を
  付与し、`ぶきよう`特性も同様に道具を無効化する既存の汎用アイテム無効化機構
  （`_check_handler_validity` の `item.enabled_ignoring` チェック）で処理されるため、
  本アイテム側で個別対応は不要。マジックルームは同一 `ON_SWITCH_IN` イベント内で
  priority=100（マジックルーム側）→120（ルームサービス側）の順に評価されるため、
  同時に場に出た場合も正しく無効化される。

## 未対応・今後の課題
以下は一次情報に記載されているが、既存のイベント設計では正確な再現が難しい、または
シングルバトルでの発生頻度・影響が極めて限定的なため未対応とする:
- 設置技（まきびし等）の効果でダメージ・状態異常を受けた直後に、他の処理へ割り込んでルーム
  サービスが発動する仕様（本実装では通常の `ON_SWITCH_IN` 優先度順で処理されるのみ）。
- すばやさランク下限で不発だった場合に、後続の攻撃ダメージ・天候ダメージ等を受けた瞬間に
  即座に発動する仕様（本実装は `ON_FIELD_ACTIVATE` / `ON_SWITCH_IN` の2契機のみで判定する）。
- トリックルーム発動時に複数のポケモン（自分・相手双方）が同時に条件を満たした場合の
  発動順（一次情報: 手持ち画面のすばやさ実数値順、ランク・へんしん等を考慮しない）。
  本実装は `EventManager._sort_handlers` の既定ロジック（priorityの次に現在の実効すばやさ
  で比較）に委ねており、ランク補正等を考慮してしまう点が一次情報と異なる。
- トリック・リサイクル等でトリックルーム中に道具を新たに入手した際の即時発動。

## テスト観点
- トリックルームが発動した瞬間に S が-1 されること（`test_ルームサービス_トリックルームでS低下`）
- すでに S ランクが下限のときは発動・消費しないこと
  （`test_ルームサービス_すばやさが下限のときは発動しない`）
- トリックルーム状態の場に交代で繰り出したときも発動すること
  （`test_ルームサービス_トリックルーム中に交代で場に出るとS低下`）
- 相手が発動させたトリックルームによる低下でも、まけんきなど「相手による低下」を条件とする
  効果が誤発動しないこと（`test_ルームサービス_相手のトリックルームでもまけんきは発動しない`）
