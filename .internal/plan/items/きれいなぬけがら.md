# アイテム実装計画: きれいなぬけがら

更新日: 2026-07-04

## スコープ

- 仕様書: `.internal/spec/items/きれいなぬけがら.md`
- 実装状態: 実装済み
- 実装方針: `Event.ON_CHECK_TRAPPED` を priority=-100 でフックし、`value=False, stop_event=True` を返すことで、
  相手の特性（かげふみ/じりょく/ありじごく）・場の状態（フェアリーロック）・自身の揮発性状態
  （にげられない/バインド/ねをはる）による交代禁止判定を無条件に上書きする。

## 実装ゴール（完了定義）

- 相手がかげふみ/じりょく/ありじごくを持っていても、所持者は交代できる。
- 場がフェアリーロック状態でも、所持者は交代できる。
- 所持者自身がにげられない/バインド/ねをはる状態でも交代できる。
- 効果発動時にアイテムは消費されず、専用ログも出さない（仕様書「発動が明示されない」を反映）。
- 既存テストを壊さない。

## 実装対象イベント

- `Event.ON_CHECK_TRAPPED` — `core/query.py` の `is_trapped()`（`can_switch()` からのみ呼ばれる）が
  発火。交代可否のみに影響し、`.internal/spec/turn.md` には掲載されていない内部イベント。

### priority 決定根拠

`.internal/spec/turn.md` に `ON_CHECK_TRAPPED` の記載はないため、既存の同種ハンドラを参照して決定した。

- `かげふみ`/`じりょく`/`ありじごく`（`data/ability.py`）、`フェアリーロック`（`data/field/global_field.py`）、
  `にげられない`/`バインド`/`ねをはる`（`data/volatile.py`）は、いずれも `priority` を明示しておらず
  デフォルト値 `100`（`core/handler.py` の `Handler.priority` 既定値）で登録されている。
- `EventManager._sort_handlers` は `priority` 昇順（同値なら素早さ降順）でハンドラを実行し、
  `stop_event=True` を返した時点でそれ以降のハンドラは評価されずに即座に値を返す
  （`EventManager.emit` 参照）。
- きれいなぬけがらは「上記のどの交代禁止効果よりも優先して無条件に交代可能にする」効果のため、
  他の `ON_CHECK_TRAPPED` ハンドラ（すべて priority=100）より確実に先に実行される
  `priority=-100` を採用し、`stop_event=True` で以降の判定を打ち切る。

## テスト観点

1. 相手の特性（かげふみ等）による交代禁止を無視して交代できる（発動する）。
2. 何もない状態では通常通り交代できる（比較対象、既存の他アイテムテストで代替可）。
3. 消費アイテムではないため、発動後も所持し続ける（アイテムを直接検証する必要はなく、
   ハンドラが `once` を使用していないことで担保される）。
4. 自身の揮発性状態（にげられない等）による交代禁止も無視して交代できる（相互作用）。

## リスクと対策

- リスク: `is_trapped()` は `can_switch()` からのみ呼ばれており、野生ポケモン戦の「にげる」判定とは
  独立している。本プロジェクトはポケモンチャンピオンズシングルバトル（トレーナー戦のみ）を対象と
  しており「にげる」コマンド自体が存在しないため、仕様書にある「にげるには効果が及ばない」旨の
  但し書きは実装上考慮不要。
