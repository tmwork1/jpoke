# レッドカード 実装計画

## 効果概要
持たせたポケモンが攻撃技のダメージ（0を含む）を受けたとき、技を使ってきた相手を
ランダムな控えポケモンと強制交代させる（1回発動で消費）。
プレイヤーが交代先を選べないため Interrupt は不要。ふきとばし系の `blow()` と同パターン。

## 実装仕様

| 項目 | 内容 |
|------|------|
| イベント（実ダメージ>0） | `Event.ON_DAMAGE_HIT` |
| イベント（実ダメージ0の救済） | `Event.ON_HIT`（`value<=0` のときのみ処理） |
| subject_spec | `"defender:self"` |
| priority | 150（`.internal/spec/turn.md` ON_DAMAGE:「150 レッドカードの発動・交代」） |
| 消費条件 | 発動時（交代不可の場合も含む）に `battle.item_manager.consume_item(mon)` |

### なぜ2つのイベントに登録するのか

`Event.ON_DAMAGE_HIT` は `core/move_executor.py` の `_execute_hit` で
`actual_damage <= 0` のとき発火しない（`if actual_damage <= 0: return` で打ち切られる）。
ばけのかわ・アイスフェイスの肩代わりやこらえるでHP1のまま耐えたとき（実HPダメージ0）も
発動する仕様（詳細な仕様参照）を満たすため、いのちのたま・だっしゅつボタンと同様に
常に発火する `Event.ON_HIT` でもこのケース（`value<=0`）のみを処理する。

`Event.ON_HIT` は `_execute_hit` 内で `ON_MODIFY_MOVE_DAMAGE` 直後・`ON_DAMAGE_HIT` より前に
発火するため、実ダメージ(`value>0`)側の交代処理をここで行うと `battle.foe()` 解決など
ON_HIT 発火直後に続くボーナス処理（物理/特殊被弾数の記録・接触記録等、`ctx.attacker` が
まだ場に残っていることを前提とするコード）を壊してしまう。そのため実ダメージ>0の
通常ケースは、そのボーナス処理より後に発火する `Event.ON_DAMAGE_HIT` 側で処理し、
実ダメージ0の救済ケースのみ `Event.ON_HIT` 側で処理する（`_execute_hit` はこの分岐で
即 return するため、後続コードを壊さない）。

### ハンドラロジック（疑似コード）
```python
def _レッドカード_should_trigger(ctx) -> bool:
    mon = ctx.defender
    return not (
        mon.fainted
        or ctx.substitute_damage          # みがわりで防がれた
        or ctx.hit_index != ctx.hit_count  # 連続攻撃技は最後のヒットのみ判定
        or ctx.attacker.fainted           # 反動・自滅技で攻撃者が既にひんし
        or (
            ctx.attacker.ability.name == "ちからずく"
            and ctx.move.has_flag("secondary_effect")
        )
    )

def _レッドカード_try_force_switch(battle, ctx) -> None:
    foe = ctx.attacker
    opponent = battle.get_player(foe)
    state = battle.player_states[opponent]
    # とらわれ状態（にげられない・バインド・フェアリーロックや特性かげふみ・ありじごく・
    # じりょく等）を無視して発動するため、can_switch は使わず控えの生存有無のみを見る。
    commands = [
        Command.get_switch_command(i)
        for i, m in enumerate(state.team)
        if m is not state.active and m.alive
    ]
    if not commands:
        return  # 交代先がいなければ発動しない（アイテムも消費しない）

    _announce_and_consume_item(battle, mon)

    # ねをはる状態・特性きゅうばん/ばんけんの相手は交代させられないが、アイテムは消費される
    if foe.has_volatile("ねをはる") or foe.ability.name in ("きゅうばん", "ばんけん"):
        return

    command = battle.random.choice(commands)
    battle.run_switch(opponent, state.team[command.index])

def レッドカード_force_switch(battle, ctx, value):        # Event.ON_DAMAGE_HIT
    if _レッドカード_should_trigger(ctx):
        _レッドカード_try_force_switch(battle, ctx)
    return HandlerReturn(value=value)

def レッドカード_force_switch_on_zero_damage(battle, ctx, value):  # Event.ON_HIT
    if value <= 0 and _レッドカード_should_trigger(ctx):
        _レッドカード_try_force_switch(battle, ctx)
    return HandlerReturn(value=value)
```

## とらわれ状態の扱い（要注意ポイント）

- にげられない・バインド・フェアリーロックや特性かげふみ・ありじごく・じりょくは
  **無視して発動する**（`can_switch`/`is_trapped` は使わない）。
- ねをはる状態、特性きゅうばん・ばんけんの相手は**交代させられない**が、アイテムは消費される。
- しれいとうで合体したヘイラッシャの交代不可判定は本プロジェクトでは未実装（`しれいとう`
  特性自体に合体状態のロジックがない）ため対応しない。他の強制交代アイテム
  （だっしゅつパック・だっしゅつボタン）と同様のスコープ制限。

## 既知の制限

- 「いのちのたまを持ったポケモンからの攻撃に対してはレッドカードが先に発動するため、
  交代させられた相手はいのちのたまの反動ダメージを受けない」という細部は未対応。
  `いのちのたま_recoil` は `Event.ON_HIT`（priority=160）に登録されており、
  `_execute_hit` 内で `Event.ON_HIT` は常に `Event.ON_DAMAGE_HIT` より先に発火するため、
  実ダメージ>0の通常ケース（`レッドカード_force_switch`、`Event.ON_DAMAGE_HIT` 側）に
  対しては、いのちのたまの反動が必ず先に適用されてしまう。
  これを正しい順序にするには、実ダメージ>0のケースも `Event.ON_HIT` 側（優先度150、
  いのちのたまの160より先）で処理する必要があるが、その場合 `_execute_hit` 内で
  `Event.ON_HIT` 発火直後に続く「物理/特殊被弾数の記録・接触記録」等のボーナス処理が
  `ctx.attacker`（既に交代済みでベンチに下がっている）を前提としており、
  `battle.foe()` の解決に失敗してクラッシュする（要 `core/move_executor.py` 側の
  リファクタ）。影響範囲が本アイテム単体の実装を超えるため、今回は対応を見送り、
  既知の制限として記録する。

## テスト観点

- ダメージを受けたとき相手がランダムに強制交代されること
- 相手の控えがいない場合（最後の1体）は交代が起きないこと（アイテムも消費されない）
- 発動後にアイテムが消費されること
- 実ダメージ0（こらえる・ばけのかわ）でも発動すること
- みがわりに阻まれたとき（`ctx.substitute_damage`）は発動しないこと
- ちからずくの効果が発動した技を受けたときは発動しないこと
- とらわれ状態（バインド等）を無視して発動すること
- ねをはる状態・特性きゅうばん/ばんけんの相手は交代させられない（アイテムは消費される）こと
- 攻撃側がみがわり状態でも交代させられること
- 持たせたポケモンがひんしになった場合は発動しないこと
- 攻撃者が反動・自滅技でひんしになった場合は発動しないこと
- 連続攻撃技は最後のヒットの後にのみ発動すること
