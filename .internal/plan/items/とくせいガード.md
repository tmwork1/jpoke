# とくせいガード 実装計画

## スコープ
- 優先度: -
- 仕様書: `.internal/spec/items/とくせいガード.md`（作成済み）
- 実装状態: **実装済み**

## 効果概要
持たせたポケモンは、戦闘中に特性が変わることや、無効化されること、かたやぶり系の効果で無視されることがなくなる（非消費）。

## 実装方針

### 1. 特性の無効化ブロック
- `src/jpoke/core/ability_manager.py` の `add_disabled_reason()` は、無効化理由を追加する前に
  `Event.ON_CHECK_ABILITY_DISABLE` を発火し、`True` が返された場合は無効化をブロックする。
- `とくせいガード` はこのイベントに `subject_spec="source:self"` でハンドラを登録し、
  常に `True` を返して無効化（かたやぶり/いえき/かがくへんかガス等）をブロックする。
- priority は本イベントが `.internal/spec/turn.md` に掲載されない内部チェック専用イベントであり、
  唯一の関与ハンドラであるため、既存の他アイテムの内部チェック系ハンドラ（`ON_CHECK_ITEM_CHANGE` 等）に
  倣い `priority=200` とする（他ハンドラとの競合は想定されない）。

### 2. 特性の変更ブロック（今回のレビューで追加）
仕様書には「特性を変える技、特性を変える特性の効果で特性が書き変わることや、無効化されることがない」
と明記されているが、従来の実装は `add_disabled_reason` 経由の無効化ブロックのみで、
`change_ability()` / `swap_ability()` 経由の特性変更（シンプルビーム・なかまづくり・なやみのタネ・
スキルスワップ・なりきり・トレース等）は一切ブロックされていなかった。これを是正する。

- `AbilityManager.is_change_blocked(mon)` を新設し、`Event.ON_CHECK_ABILITY_DISABLE` を
  `EventContext(source=mon)` で発火してブロック可否を判定する（無効化チェックと同一イベントを再利用）。
- `change_ability()` は特性変更前に `is_change_blocked()` を確認し、ブロックされる場合は何もせず戻る。
  → なかまづくり・なやみのタネ・シンプルビーム・なりきり（使用者自身の特性変更）・トレースが
    この一箇所を経由するため、まとめて防がれる。
- `swap_ability()` はスキルスワップ・さまようたましいで使用される。どちらか一方でもブロックされる場合は
  入れ替え自体を行わない（仕様書: 「所持者に対するスキルスワップ、または所持者が使用するスキルスワップは
  失敗するため、もう片方に特性がコピーされることは無い」）。
- 上記に加え、以下の各技の `_can_apply` ハンドラ（`src/jpoke/handlers/move_status.py`）に
  `battle.ability_manager.is_change_blocked(...)` のチェックを追加し、既存の `protected` フラグと
  同様に技自体を「わざが失敗した」として扱う（PP消費のみで不発、MOVE_FAILEDログ）:
  - シンプルビーム: 対象がブロック対象なら失敗
  - なかまづくり: 対象がブロック対象なら失敗
  - なやみのタネ: 対象がブロック対象なら失敗
  - スキルスワップ: 使用者・対象いずれかがブロック対象なら失敗
  - なりきり: 使用者自身がブロック対象なら失敗（自分の特性を変えられないため）
- トレース（`トレース_copy_ability`）は `battle.change_ability()` を経由するため、上記の中央集権的な
  ブロックのみで対応する（個別の `_can_apply` は存在しないため追加不要）。トレースの「保留」状態など
  トレース自身の詳細仕様（`.internal/spec/abilities/トレース.md`）はトレース側の実装レビュー対象であり、
  本レビューでは「とくせいガード所持時に特性が変わらない」という最低限の挙動のみを保証する。

### 3. ぶきようによる無効化を受けない
仕様書:「所持者の特性がぶきようであるときでも道具の効果が発動し、特性の変更や無効化を防ぐ」。

- `ぶきよう` 特性は `battle.item_manager.add_disabled_reason(mon, "ぶきよう")` を通じて
  所持アイテムそのものを無効化する（`ItemDisabledReason = "ぶきよう" | "マジックルーム"`）。
  この無効化は `EventManager._check_handler_validity()` の `case "item":` で
  `subject.item.enabled` を見て動的にハンドラの発火可否を判定している。
- `とくせいガード` はこの一般的な無効化から除外される必要があるため、`Handler` に
  `ignored_disable_reasons: frozenset[str]` フィールドを追加し、`GameEffect.enabled_ignoring(reasons)` で
  指定理由を除いた無効化状態を判定できるようにした。
- `とくせいガード` のハンドラには `ignored_disable_reasons=frozenset({"ぶきよう"})` を指定する。
  これにより、`ぶきよう` によるアイテム無効化中でも `とくせいガード` のハンドラは発火し続けるが、
  `マジックルーム` やアイテム喪失（すりかえ/マジシャン等）による無効化は従来通り効果が消える
  （`.internal/spec/abilities/トレース.md` に「マジシャン/わるいてぐせ/マジックルームによりとくせいガードの
  効果が消えた場合でもトレースは有効にならない」との記述がある通り）。

### 4. メッセージ・アイテム公開
仕様書:「効果が発動した際は『<所持者>は とくせいガードの効果で 特性が 守られている!』というメッセージが出る」。

- `とくせいガード_check_ability_disable` ハンドラ内で `_announce_item_triggered(battle, ctx.source)` を
  呼び出し、`LogCode.ITEM_TRIGGERED` のログとアイテムの `revealed = True` 化を行う。
- 仕様書には「かがくへんかガスが発動している場に所持者が出てきたときは、メッセージは出ないものの、
  特性は守られている」という、より細かいコンテキスト依存のメッセージ抑制条件があるが、これは
  かがくへんかガス側の発動タイミングに依存する仕様であり、本アイテムの実装単体では区別できない。
  本実装では簡略化し、ブロックが発生した場合は常にメッセージを出す仕様とする（既知の簡略化点）。
