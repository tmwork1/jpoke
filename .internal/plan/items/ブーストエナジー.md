# ブーストエナジー 実装計画

更新日: 2026-07-04

## スコープ

- 仕様書: `.internal/spec/items/ブーストエナジー.md`
- 実装状態: 実装済み（本改訂で ON_CHECK_ITEM_CHANGE / なげつける / パラドックス以外への誤発動ガードを追加）

## 実装ゴール（完了定義）

- こだいかっせい/クォークチャージを持つポケモンが登場した直後にブーストエナジーを消費し、
  ランク補正込みで最も高い能力を1.3倍（素早さのみ1.5倍）にする。
- 場がにほんばれ（こだいかっせい）またはエレキフィールド（クォークチャージ）の場合は
  ブーストエナジーは発動しない（場由来のブーストが優先され、アイテムは温存される）。
- マジックルーム中は発動しない。解除後に即座に発動する。
- こだいかっせい/クォークチャージを持たないポケモンが持っていても何も起きない
  （バトル中に取得した場合も同様）。
- バトル中にブーストエナジーを新たに取得したとき（消費後の再取得等）、即座にブーストが発動する。
- こだいかっせい/クォークチャージ持ちが絡む場合、トリック・すりかえ・はたきおとす・どろぼう・
  マジシャン・わるいてぐせ等によるブーストエナジーの授受を防ぐ（渡す/奪われる双方）。
  ただし、ブースト発動に伴う自己消費は防がない。
- なげつけるはこだいかっせい/クォークチャージ持ちが使用すると失敗する。それ以外は威力30で成功する。

## 実装対象イベント

- `Event.ON_SWITCH_IN` / `Event.ON_ABILITY_ENABLED` / `Event.ON_FIELD_CHANGE` —
  こだいかっせい/クォークチャージ側（`data/ability.py`）に登録済み。
  priority=200（`.internal/spec/turn.md` に掲載がないイベントのため、エレキシード等の場設定アイテム
  （priority=120）やエレキメイカー等の場設定特性（priority既定100）より後に発動するよう、
  既存の同種ハンドラとの相対順序に基づき決定）。
  （旧 `Event.ON_REFRESH_PARADOX_BOOST` も同一ハンドラに登録されていたが、
  emit 呼び出しがコードベースのどこにも存在せず常にデッドコードだったため、
  コード品質レビュー（2026-07-12, #66）で当該イベント自体を削除した。上記3イベントで
  カバレッジは変わらない。）
- `Event.ON_ITEM_ENABLED` / `Event.ON_ITEM_GAINED` — アイテム側（`data/item.py`）に登録。
  マジックルーム解除時・バトル中のアイテム取得時にパラドックスブーストを再判定する。
  こだいかっせい/クォークチャージ側のハンドラと同じ関数
  （`ability_paradox.refresh_paradox_charge_state`）に委譲するため、
  複数ハンドラ間の実行順は問題にならない（優先度未指定＝デフォルト100）。
- `Event.ON_CHECK_ITEM_CHANGE` — アイテム側に新規登録。トリック・すりかえ・はたきおとす・
  どろぼう・マジシャン・わるいてぐせ等の共通判定イベント。優先度未指定＝デフォルト100
  （他の専用道具の同種ハンドラと同様、単独登録のため相対順序は問題にならない）。

## 実装詳細: `ブーストエナジー_prevent_item_change`

`ON_CHECK_ITEM_CHANGE` は `item_manager.can_change_item(target, source=None)` で発火し、
呼び出し経路によって `ctx.source` の意味が異なる:

- `remove_item(target, source=attacker)` のように単純消失処理（はたきおとす等）では
  `ctx.source` に実行者（≠ target）が入る。この経路では「渡す」ことはなく破壊されるだけなので、
  対戦相手側の特性は判定しない。
- `swap_items()` 経由（トリック・すりかえ・どろぼう・マジシャン・わるいてぐせ）では
  両者に対して `ctx.source=None` で判定が行われる。この経路は実際に相手へアイテムが渡るため、
  `battle.foe(mon)` で交換相手を取得し、相手の特性も判定する。
- ブーストエナジー消費に伴う自己消費（`consume_item(mon)` → `remove_item(mon, source=mon)`）は
  `ctx.source is mon`（同一オブジェクト）になるため、保持者自身の特性チェックの対象から除外する
  （このガードが無いとブーストの自己消費自体が阻害され、既存の登場時発動テストが壊れる）。

```python
def ブーストエナジー_prevent_item_change(battle: Battle, ctx: EventContext, value: bool) -> HandlerReturn:
    mon = ctx.target
    if mon is None:
        return HandlerReturn(value=value)

    paradox_abilities = ("こだいかっせい", "クォークチャージ")
    if mon.ability.name in paradox_abilities and ctx.source is not mon:
        return HandlerReturn(value=False, stop_event=True)

    if ctx.source is None:
        foe = battle.foe(mon)
        if foe is not None and foe.ability.name in paradox_abilities:
            return HandlerReturn(value=False, stop_event=True)

    return HandlerReturn(value=value)
```

登録:
```python
"ブーストエナジー": ItemData(
    removable=False,
    fling_power=30,
    handlers={
        Event.ON_ITEM_ENABLED: h.ItemHandler(
            h.ブーストエナジー_refresh_paradox_charge,
            subject_spec="source:self",
        ),
        Event.ON_ITEM_GAINED: h.ItemHandler(
            h.ブーストエナジー_refresh_paradox_charge,
            subject_spec="source:self",
        ),
        Event.ON_CHECK_ITEM_CHANGE: h.ItemHandler(
            h.ブーストエナジー_prevent_item_change,
            subject_spec="target:self",
        ),
    }
),
```

## テスト観点

1. こだいかっせい/クォークチャージのパラドックスポケモンが登場時にブーストエナジーを消費し発動する。
2. マジックルーム解除後にブーストが即座に発動する。
3. バトル中にブーストエナジーを新たに取得すると即座に発動する。
4. こだいかっせい/クォークチャージを持たないポケモンが取得しても何も起きない。
5. こだいかっせい/クォークチャージ持ちが持っている間ははたきおとす・どろぼうで奪われない。
6. こだいかっせい/クォークチャージ持ちが持っている間はトリックで交換されない。
7. こだいかっせい/クォークチャージ持ちへはトリックで渡すことができない。
8. なげつけるはこだいかっせい/クォークチャージ持ちが使用すると失敗し、それ以外は威力30で成功する。

（場（にほんばれ/エレキフィールド）優先時にアイテムが温存される挙動は
`tests/abilities/test_ability__group.py` の既存テストで確認済み。）

## リスクと対策

- リスク: 「なげつけるを使用すると失敗する」の判定対象は仕様書上「パラドックスポケモン」と
  記載されているが、ウガツホムラ/タケルライコ/テツノイワオ/テツノカシラの4種は例外的に
  授受制限が働かない（一次情報の脚注参照、Twitter投稿のみが出典のごく狭いバグ挙動）。
  この4種の例外は実装コストに対して影響範囲が極めて限定的なため、本改訂では実装しない
  （通常のこだいかっせい/クォークチャージ持ち同様に制限がかかる）。
- リスク: `refresh_paradox_charge_state` はこだいかっせい/クォークチャージ側と
  アイテム側の両方から呼ばれる共通関数のため、特性ガードを外す変更は両経路に影響する。
  → 全テスト実行で回帰がないことを確認済み。
