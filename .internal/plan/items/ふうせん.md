# ふうせん 実装計画

## 効果概要
浮遊状態になり地面技が当たらない。攻撃技が有効に命中すると（実際のダメージ量が0であっても）割れて消滅する（非消費だがダメージで除去される）。

## 実装仕様

### イベント 1: 浮遊判定を返す

| 項目 | 内容 |
|------|------|
| イベント | `Event.ON_CHECK_FLOATING` |
| subject_spec | `"source:self"` |
| priority | 100（デフォルト。`.internal/spec/turn.md` に本イベントの掲載なし。`ふゆう`・`うちおとす`等の既存 `ON_CHECK_FLOATING` ハンドラと同様にデフォルト値を採用） |
| once | False |

### イベント 2: 攻撃技が有効に命中したとき消滅

| 項目 | 内容 |
|------|------|
| イベント | `Event.ON_HIT` |
| subject_spec | `"defender:self"` |
| priority | 100（`.internal/spec/turn.md` の `Event.ON_HIT` に「100 \| ふうせんが割れる」と明記） |
| once | False |

`.internal/spec/turn.md` の `Event.ON_DAMAGE`（実装上の `Event.ON_DAMAGE_HIT` に相当）にも
「30 \| ふうせんが割れる」という記載があるが、`ON_DAMAGE_HIT` は実際のダメージ量が
`actual_damage <= 0` のとき発火しない（`core/move_executor.py` の
`if actual_damage <= 0: return`）。
`.internal/spec/items/ふうせん.md`「詳細な仕様」に明記されている通り、以下のケースでは
実ダメージが0でもふうせんは割れる必要がある。

- みがわり状態で攻撃を防いだ場合
- 特性ばけのかわ・アイスフェイスで攻撃技のダメージを肩代わりした場合
- ダメージ補正やオーバーフローの結果、ダメージが0になった場合

これらはいずれも `Event.ON_MODIFY_MOVE_DAMAGE` の時点でダメージが0に補正されるだけで、
`ON_BEFORE_APPLY_MOVE` で無効化されるわけではないため、`_execute_hit` は最後まで実行され
`Event.ON_HIT`（ダメージ量に関わらず必ず発火）は発火する。したがって `ON_DAMAGE_HIT` ではなく
`ON_HIT` を採用する（`いのちのたま`の反動処理と同様の判断。`src/jpoke/data/item.py` の
`いのちのたま` エントリのコメントを参照）。

一方、まもるやタイプ・特性による無効化は `ON_BEFORE_APPLY_MOVE` の時点で
`move_applied=False` となり `_execute_hit` 自体が実行されないため、`ON_HIT` も発火せず、
ふうせんは割れない（仕様と一致）。

### ハンドラロジック（疑似コード）
```python
def ふうせん_check_floating(battle, ctx, value):
    return HandlerReturn(value=True)


def ふうせん_pop_on_hit(battle, ctx, value):
    mon = ctx.defender
    # 攻撃技が有効に命中したとき消滅（アイテム発動ログ→消費の順で処理する）
    _announce_and_consume_item(battle, mon)
    return HandlerReturn(value=value)
```

## 既知の制約
- `どろぼう`・`ほしがる`（`Event.ON_HIT` に登録、`subject_spec="attacker:self"`、デフォルト
  `priority=100`）にヒットした場合、仕様上はふうせんが割れる効果が奪取より優先されるが、
  本実装では同一 `Event.ON_HIT` 内で `priority` が同値のためすばやさ順に実行される
  （`EventManager._sort_handlers`）。したがって奪う側が速い場合は稀に奪取が先に処理される
  可能性がある。この相互作用の解消は `どろぼう`／`ほしがる` 側のレビューで扱う。
- `特性マジシャン`は `Event.ON_DAMAGE_HIT` に登録されているため、`ON_HIT`（本実装）が
  常に先に発火し、ふうせんが割れてからマジシャンの奪取判定が行われるため、この組み合わせは
  仕様通り動作する。
- `さしおさえ`（Embargo）は本コードベースに未実装のため、`ItemDisabledReason` に対応する
  無効化理由が存在しない。実装時には `add_disabled_reason(mon, "さしおさえ")` 相当の処理を
  追加すれば、既存の `source="item"` ハンドラの無効化チェック（`_check_handler_validity`）
  により本アイテムは自動的に対応できる見込み。

## テスト観点
- ふうせん持ちに地面技が当たらないこと
- ダメージ技でふうせんが消滅し、その後地面技が当たること
- 変化技ではふうせんが消滅しないこと
- みがわりが攻撃を肩代わりした場合でもふうせんが割れること（本体はダメージを受けない）
- ダメージ量が0に補正された場合でもふうせんが割れること
- じゅうりょく状態下でふうせんの浮遊効果が打ち消されること
- ぶきよう特性によりふうせんの効果（浮遊・破損）が発動しなくなること
