# てきおうりょく レビュー結果

## 仕様書
- 変更なし。一次情報（ポケモンWiki `てきおうりょく`）と照合し、効果・テラスタル時の仕様
  （テラタイプ一致・元タイプ一致・失った元タイプ・ステラの各ケース）とも一致していることを
  確認した。

## 実装計画書
- 修正した。既存の計画書は「テラスタルにより失った元タイプの技には効果が発動しない」という
  仕様上の重要な分岐が明記されておらず、これが実装バグの一因と考えられたため、以下を補強した。
  - 実装ゴールに「テラスタルにより失った元タイプの技には発動しない（1.5倍のまま）」を明記。
  - 実装方針に、STAB補正のベース値（6144）だけでは「テラタイプ一致による6144」と
    「失った元タイプ一致による6144」を区別できない旨と、`ctx.attacker.active_tera_type` /
    `ctx.move.type` を直接比較して判定する方針を明記。
  - priority（`.internal/spec/turn.md` 未掲載の内部イベントのため `priority=100` を採用、
    同イベントを使う他ハンドラが存在せず根拠として妥当）を明記。
  - `HandlerReturn` は `value` のみ使用し `stop_event` は不使用である旨を明記。
  - テスト観点に「失った元タイプの技は1.5倍のまま」「テラタイプ一致のみで元タイプ不一致は
    2.0倍」「ステラは無補正」の3項目を追加。

## 実装（handlers/ / data/）
- 修正した。`src/jpoke/handlers/ability.py` の `てきおうりょく_modify_stab` にバグがあった。
  - **バグの内容**: STAB補正値が `6144`（1.5倍相当）であれば無条件に `8192`（2.0倍）へ
    補正していたが、テラスタル中は「テラタイプに一致するがステラでない元タイプ一致技
    （テラスタイプ不一致・元タイプ一致）」による `6144` と「テラタイプに一致した技」による
    `6144` の両方が同じ数値 `6144` として渡ってくるため区別できず、一次情報にある
    「テラスタルにより失った元タイプの技にはてきおうりょくの効果が発動しない
    （1.5倍のまま据え置き）」という仕様に反して、失った元タイプの技まで2.0倍に補正して
    しまっていた。
  - **修正内容**: `ctx.attacker.active_tera_type` と `ctx.move.type` を直接比較し、
    テラスタル中かつ技タイプがテラスタイプと不一致の場合は無補正（元の値をそのまま返す）
    にする分岐を追加した。ステラテラスタル時の無補正処理は変更なし。
  - `src/jpoke/data/ability.py` のハンドラ登録（`Event.ON_CALC_ATK_TYPE_MODIFIER`,
    `subject_spec="attacker:self"`）は仕様・計画と整合しており変更なし。

## テスト
- 追加した。`tests/abilities/test_ability_ta.py` の `test_てきおうりょく_STAB補正` に
  以下の3ケースを追加し、修正前の実装では失敗する（バグを検出できる）ことを確認した。
  1. テラスタイプが元タイプと不一致・技タイプがテラスタイプに一致 → 2.0倍（8192）。
  2. テラスタルにより失った元タイプ（元タイプ一致・テラスタイプ不一致）の技 → 1.5倍
     （6144）のまま補正されない。
  3. ステラタイプへのテラスタル → 効果が発動せず、ステラ補正込みの初回STAB（8192）の
     まま変化しない。

  無関係）で全テストが通ることを確認した。ログ:
