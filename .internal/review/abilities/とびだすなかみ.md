## 実装（handlers/ / data/）
- 既存実装（`src/jpoke/handlers/ability.py` の `とびだすなかみ_save_hp` /
  `とびだすなかみ_retaliate_on_ko`、`src/jpoke/data/ability.py` の登録）はロジック自体は
  仕様どおりだったが、HP保存の仕組みにアーキテクチャ上のリスクがあったため修正した。
  - **修正前**: モジュールレベルのグローバル辞書 `_とびだすなかみ_hp_before: dict[int, int]`
    を `id(defender)` をキーに使って ON_BEGIN_MOVE → ON_MOVE_KO 間でHPを受け渡していた。
    このパターンは本コードベースの既存の慣習（`AttackContext.substitute_damage` のように、
    同一技の実行中だけ有効な一時状態は `ctx` の専用フィールドに積む）と異なり、
    以下のリスクを抱えていた:
    - `id()` はオブジェクトの生存期間内でのみ一意なため、defender がGCされた後に別のPokemon
      インスタンスが同じ`id()`を再利用すると、無関係な戦闘・ポケモンへ値が混入しうる。
    - ON_MOVE_KO が発火しない（ひんしにならない）限りエントリが `pop` されず、
      プロセスの生存期間中辞書に残り続ける（メモリリーク）。
  - **修正後**: `AttackContext` に `defender_hp_before_move: int = 0` フィールドを追加し、
    ON_BEGIN_MOVE から ON_MOVE_KO まで同一の `ctx` インスタンスが使い回されることを利用して
    ctx 自身にHPを保存するよう変更（グローバル辞書を廃止）。
    - `src/jpoke/core/context.py`: `AttackContext.defender_hp_before_move` を追加
    - `src/jpoke/handlers/ability.py`: `とびだすなかみ_save_hp` / `とびだすなかみ_retaliate_on_ko`
      をこのフィールドを使うよう書き換え
  - ロジック面（KOのタイミング、みがわり無視、じばく等での不発、勝敗判定順序）は
    `battle.modify_hp` の既存の実装（`ON_MOVE_KO` が defender ひんし確定後にのみ発火し、
    `modify_hp` が volatile を経由せず実HPを直接書き換える）により仕様どおり動作することを確認した。
