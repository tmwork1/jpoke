## 重大バグ（修正済み）

### BUG-1: `_heal_at_pinch` — 無効状態の LethalState が分布から消失する

**ファイル**: `src/jpoke/handlers/lethal.py`

**修正前**:
```python
if heal_with == "ability" and not state.ability_enabled:
    continue  # ← new_dist に追加されないまま次へ
if heal_with == "item" and not state.item_enabled:
    continue
```

`continue` で次のループに進む際、`new_dist[state] += freq` が実行されないため、
「道具/特性が既に無効化された」状態が分布から**削除**される。
確率の総和が 1 より小さくなり、致死率が不正な値になる。

**修正後**:
```python
if heal_with == "ability" and not state.ability_enabled:
    new_dist[state] += freq  # 回復せずそのまま維持
    continue
if heal_with == "item" and not state.item_enabled:
    new_dist[state] += freq
    continue
```

この影響を受けるハンドラ:
- `オボンのみ_heal`（`heal_with="item"`, `consume=True`）  
  2ヒット目以降、`item_enabled=False` の状態が消失していた。

## 中程度の問題（修正済み）

### ISSUE-1: `hp_counter` プロパティでの情報欠落

**ファイル**: `src/jpoke/core/lethal.py:92`

**修正前**:
```python
@property
def hp_counter(self) -> dict[int, int]:
    return {state.hp: freq for state, freq in self.hp_dist.items()}
```

同じ HP 値を持つが `ability_enabled` / `item_enabled` が異なる複数の `LethalState` が存在する場合、
辞書の後勝ちで頻度が上書きされる。確率の総和が変わるため、`lethal_probability` が不正になる可能性がある。

**再現条件**: オボンのみを持つポケモンが 1/2 以下になりうる攻撃のとき。  
- `LethalState(hp=50, item_enabled=True)` と `LethalState(hp=50, item_enabled=False)` が共存しうる。

**修正後**:
```python
@property
def hp_counter(self) -> dict[int, int]:
    result: dict[int, int] = defaultdict(int)
    for state, freq in self.hp_dist.items():
        result[state.hp] += freq
    return dict(result)
```

### ISSUE-2: `_update_hp` が分布の最小値を使用する

[ユーザーコメント] これで問題ない。

**ファイル**: `src/jpoke/core/lethal.py:298`

```python
def _update_hp(mon: Pokemon, hp_dist: LethalDist):
    mon.hp = min(state.hp for state in hp_dist)
```

分布内の最小 HP を `mon.hp` にセットするため、ハンドラが `mon.hp` を参照して
ピンチ発動の判定などを行う場合、常に最悪ケースとして扱われる。

現状の実装はすべての分岐を最悪ケースに統一する保守的な戦略で、
一部の消耗アイテム（オボンのみなど）が過剰に発動しない限り問題にならない。  
ただし、「HPが閾値以上なら発動しない」系ハンドラを追加した際に
意図せず発動判定が変わるリスクがある。

## 軽微な指摘（修正済み）

### ISSUE-3: `LethalContext` がループ内で毎回生成される

**ファイル**: `src/jpoke/core/lethal.py:203`

**修正前**:
```python
for atk in range(1, max_attack + 1):
    for move, count in move_list:
        ctx = LethalContext(attacker, defender, move, critical=critical)
        for hit in range(count):
            ...
```

`attacker`, `defender`, `critical` はループ中に変化せず、`move` も `move_list` の各要素に対して固定のため、
`atk` のループごとに同じ `LethalContext` を毎回生成していたのは冗長。

**修正後**:
```python
ctx_list = [(move, count, LethalContext(attacker, defender, move, critical=critical))
            for move, count in move_list]

for atk in range(1, max_attack + 1):
    for move, count, ctx in ctx_list:
        for hit in range(count):
            ...
```

`move_list` に対応する `LethalContext` を `atk` ループの外で1回だけ生成し、以降は使い回す。

### ISSUE-4: `オボンのみ` の `event="any"` と `たべのこし` の `event="on_turn_end"` の非対称性

[ユーザーコメント] 現状の実装が正しい。

**ファイル**: `src/jpoke/data/item.py`

`オボンのみ` は `event="any"` のため `on_damage` / `on_move_secondary` / `on_turn_end` すべてで呼ばれる。  
`consume=True` + `item_enabled=False` で2回目以降の発動は防がれるが、
毎イベントでルーティングコストが発生する。

`event="on_damage"` に絞ることで意図を明示し、不要な呼び出しを減らせる。  
（ポケモンチャンピオンズでの実際の発動タイミングを確認した上で変更する）
